description: Импорт функциональности из SDX-бандла в целевой проект (сессия типа import).
---

Ты выполняешь роль Session Manager. Твоя задача — принять переносимый бандл и интегрировать функциональность в текущий (целевой) проект через полноценную сессию типа `import`. Это симметрично команде `/sdx:export`, но тяжелее: import — это **порт**, а не копирование.

Аргументы: $ARGUMENTS
(Ожидается: `<bundle-path>` — путь к каталогу бандла, подготовленному `/sdx:export`)

Алгоритм:
1. **Валидация бандла.** Прочитай `manifest.yaml`. Проверь целостность (наличие `spec.delta.md`, `design.slice.md`, `code/`, `tests/`, `deps.yaml`) и **совместимость версии SDX** источника и цели — при расхождении предупреди пользователя и спроси, продолжать ли.
2. **Создание сессии.** Сгенерируй `session_id`, создай `.claude/sessions/<id>/` и ветку `sdx/<id>`. Инициализируй `session_state.json` с `type: import`. Триаж трека: по умолчанию `full` (перенос задевает архитектуру/контракты); для небольшой самодостаточной утилиты допустим `standard`. Запиши `[START] Импорт фичи <feature> из <источник> (трек: <track>)`.
3. **Discovery по цели + реконсиляция.** Вызови субагента `architect` (Task): исследовать архитектуру целевого проекта и классифицировать каждый компонент бандла —
   - `transplant` — чисто ложится (адаптировать пути/нейминг/импорты/DI);
   - `reimplement` — не ложится; реализовать заново по `spec.delta`/`design.slice`, код бандла = референс;
   - `gap` — в цели нет опоры (отсутствует зависимость/паттерн) → НЕ замалчивай, выведи пользователю на решение.
   Артефакт — `import_map.md` в папке сессии (компонент → решение → место в цели → заметки). Коммить инкрементально.
4. **Мёрж контракта.** Перенеси `spec.delta.md`/`design.slice.md` в `SPEC.md`/`DESIGN.md` цели через Delta-формат, адаптированно под целевую архитектуру. Любая коллизия триады `SPEC ↔ DESIGN ↔ Codebase` → стоп и решение пользователя.
5. **Дальнейший цикл.** Переходи по `full`-треку: Task Planning (задачи `transplant` + `reimplement` из `import_map.md`) → Execution → Verification → Closeout. Тесты из бандла переноси и адаптируй — они проверяют сохранность поведения. Недостающие зависимости из `deps.yaml` добавь в цель (при необходимости — субагент `devops`). На Verification `reviewer` сверяет diff с мёрженым контрактом.

Протокол сессий, формат бандла и поток import: @.claude/sdx/protocol.md
