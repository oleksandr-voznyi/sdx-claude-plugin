---
description: Структурированная верификация поставки (3 оси) с fresh-eyes ревью.
---

Ты выполняешь роль Session Manager. Твоя задача — провести этап Verification: подтвердить корректность исполнением и запустить независимое fresh-eyes ревью поставки.

Алгоритм:
1. Прочитай `session_state.json`: `session_id`, `track`, `git_branch`. Определи объём верификации по треку:
   - **full** — обязательно; полная матрица против `SPEC.md`/`DESIGN.md`.
   - **standard** — суженно; матрица против `change_note.md`.
   - **patch** — по запросу; по умолчанию достаточно регрессионного теста (шаг 2), fresh-eyes ревью — опционально.
2. **Корректность исполнением.** Вызови субагента `qa` (Task): убедись, что тесты на изменения написаны и полный прогон зелёный (при необходимости — coverage). Если прогон красный — останови этап, сообщи пользователю, запиши `[ERROR]`.
3. **Diff поставки.** Вычисли diff ветки сессии против `main` целиком:
   ```bash
   git diff main...sdx/<session_id>
   ```
4. **Fresh-eyes ревью (всегда через Task, никогда инлайн).** Вызови субагента `reviewer`, передав ему СТРОГО ограниченный вход:
   - пути к артефактам-контракту (`SPEC.md`/`DESIGN.md`/`change_note.md`, критерии приёмки);
   - diff поставки из шага 3.

   НЕ передавай ему `session.log`, историю рассуждений или пояснения вида «мы так сделали, потому что…». Любое «почему» ломает чистоту взгляда. Сам тоже не выполняй это ревью инлайн — только делегированием, чтобы гарантировать стену контекста.
5. Ревьюер создаёт `verification_report.md` в папке сессии. Прочитай его.
6. **Гейт по результату:**
   - Если есть находки **FAIL** — этап не пройден. Сообщи пользователю сводку, направь на исправление (субагент `developer`) или `/sdx:backtrack`. Запиши `[ERROR] Verification: N FAIL`.
   - Если есть только **WARN** — выведи их пользователю и запроси явное квитирование (например, «сценарий X без теста — принять или покрыть?»). Без подтверждения этап не закрывается.
   - Если находок нет — этап пройден.
7. Логируй только итог-ошибку: при FAIL — `[ERROR]` (шаг 6). PASS и создание `verification_report.md` в лог НЕ пиши — отчёт коммитится в ветку, а переход дальше зафиксирует `[STAGE_CHANGE]` в `/sdx:next`.

Протокол сессий, оси верификации и контракт изоляции ревьюера: @.claude/sdx/protocol.md
