# Technical Design: Режим экстремального прототипирования (трек `vibe`, тип `proto`)

Сессия: `feat-vibe-track-20260720` (feature, full). Вход: `SPEC.md` (REQ-VIBE-1..13),
`context_report.md` (карта §3, развилки §4). Решение и инварианты фиксируются новым **ADR-018**
(`docs/DECISIONS.md`, раздел «Новый ADR-018» ниже).

## Архитектурный обзор

Вводится пятый профиль флоу `vibe` и пятый «специальный» тип сессии `proto`, жёстко привязанный
к нему на `/sdx:start` (прецедент ADR-017). Архитектурно поставка держится на **четырёх**
решениях; всё остальное — их следствия.

### Решение 1. Трек описывается ОДНОЙ строкой матрицы, и в ней нет `Closeout`

```
vibe|Prototype|-|no
```

Это единственная новая строка `SDX_STAGE_MATRIX` (`sdx/hooks/sdx-stage.sh:38-60`). Отсюда сразу:

- `cmd_init`: `matrix_stages vibe | head -1` = `Prototype` → старт сессии на `Prototype` легален,
  старт на любом другом этапе отклоняется кодом `2` без создания файла (существующая логика);
- `cmd_next`: `last` = `Prototype` = текущий этап → ветка `stage = last` срабатывает **всегда**,
  возвращая `OK no-op Prototype` (код `0`, файл не тронут). Трек физически не может продвинуться
  к `Closeout`, потому что `Closeout` не входит в его цепочку. Это и есть enforcement REQ-VIBE-8:
  **инвариант выражен формой данных, а не проверкой в коде** — нет ветки логики, которую можно
  обойти, забыть или сломать рефактором;
- `cmd_backtrack`: `Prototype` попадает в union имён этапов (`matrix_stage_exists`), поэтому
  `backtrack --to Prototype` из ДРУГОГО трека отбивается вторым шагом («этап не активен в треке
  `<track>` — нужна смена трека, не откат») — корректное сообщение, а не «имя не распознано».
  Внутри `vibe` `backtrack --to Prototype` — штатный no-op;
- `cmd_retrack`: у `Prototype` пустая цепочка предшественников (первый активный этап трека),
  поэтому вход в `vibe` скриптом не блокируется — запрет входа остаётся прозаическим
  (REQ-VIBE-11, симметрично запрету деэскалации в `doc`, ADR-017). Выход ИЗ `vibe` проходит
  штатный evidence-based forward-skip guard уже по цепочке ЦЕЛЕВОГО трека — см. Решение 3.

**Отличие от прецедента ADR-017, которое нельзя замалчивать.** Трек `doc` был «изменением нулевой
мощности» для процедурного кода. `vibe` — нет: он первым в истории фреймворка нарушает молчаливое
допущение «последний активный этап любого трека = `Closeout`». Допущение материализовано ровно в
одном месте — while-цикл входа в Closeout (`commands/archive.md:16-31`), и без правки этого места
Решение 1 превращается из фичи в баг (бесконечный цикл no-op-итераций). Правка цикла — не
опциональное сопровождение, а **неотъемлемая часть** Решения 1 (см. Решение 2).

### Решение 2. Защита цикла входа в Closeout — по контракту stdout, а не по коду возврата

Терминальность этапа детектируется по строке stdout `OK no-op <stage>` (контракт REQ-STAGE-5,
`sdx/protocol.md:124`), а не по «коммит ничего не закоммитил» или по коду возврата. Если скрипт
вернул no-op, а результирующий `cur_stage` ≠ `Closeout` — значит трек не имеет собственного
`Closeout`, дальнейшие итерации бессмысленны по построению: стоп с явной диагностикой
(«легализуй прототип через `/sdx:retrack`»). Проверка трек-агностична: она защищает и от любого
будущего трека без `Closeout`, а не только от `vibe`.

### Решение 3. Легализация = обычный `retrack`, где доказательством служит spec-after артефакт

`sdx-stage.sh retrack` проверяет наличие гейт-артефактов **на диске** (`[ -s "$path" ]`), а не в
git. Из этого следует, что реверс-инжиниринг требований из НЕКОММИЧЕННОГО кода прототипа
полностью укладывается в существующий guard без единой строки нового кода: сначала создаются
`change_note.md` (цель `standard`) либо `context_report.md`+`SPEC.md`+`DESIGN.md`+`PLAN.md` (цель
`full`), затем — первый коммит кода (REQ-VIBE-7), затем правка `track` + `retrack <id> Execution`.
Guard проходит по цепочке предшественников `Execution` в НОВОМ треке и пропускает переход именно
потому, что реверс-инжиниринг реально выполнен. Обратное тоже верно и ценно: **пропустить
легализацию нельзя** — без артефактов `retrack` в `Execution` отклоняется кодом `1` с указанием
конкретного непройденного этапа.

**Уточнение по факту поставки: guard не доказывает сам коммит кода.** `sdx-stage.sh retrack`
смотрит только на гейт-артефакты **на диске** — коммит первого кода прототипа (REQ-VIBE-7) в его
зону ответственности не входит и входить не может (guard ничего не знает о git-истории). Держит
REQ-VIBE-7 отдельная прозаическая пост-проверка в `commands/retrack.md` (ветка легализации,
шаг «Первый коммит кода» — см. ниже): она стоит ДО правки `track` и останавливает легализацию,
если коммит фактически не состоялся. Это два разных, дополняющих друг друга механизма
удержания триады, а не один и тот же guard, проверяющий обе вещи.

### Решение 4. Гейт принятия/отклонения — отдельная команда `/sdx:proto` с точечным откатом по baseline-снимку

Гейт не встраивается в `/sdx:next` (см. обоснование в «Компоненты», `commands/proto.md`).
Точечность отката (REQ-VIBE-5) обеспечивается **снимком baseline на входе в каждую попытку**:
git-ревизия + список untracked-файлов на этот момент. Список файлов прототипа = (tracked-дельта
против baseline-ревизии) ∪ (untracked сейчас ∖ untracked в baseline), с исключением
`.claude/sessions/**`. `git clean -fd` и `git checkout .` не применяются НИКОГДА — только
поимённые операции по вычисленному списку.

**Уточнение по факту поставки: блоки действия самодостаточны.** `AskUserQuestion` (шаг 5) и
`Task`-вызовы субагентов (легализация) — границы, за которыми shell-сессия текущего хода не
переживает: переменные и временные файлы `mktemp`, вычисленные ДО границы, к моменту действия
ПОСЛЕ неё уже не существуют. Поэтому каждый блок действия (откат в `/sdx:proto` шаг 6; коммит
кода в `retrack.md` ветка легализации) не переиспользует `$base_rev`/`$tmp_tracked`/`$tmp_new` из
более раннего шага, а самостоятельно читает `prototype_baseline.txt` и заново выполняет весь
пересчёт (см. «Вычисление списка файлов прототипа» ниже) внутри одного непрерывного bash-хода.
Результат идемпотентен — дерево между шагами не меняется, поэтому пересчёт даёт тот же список,
что был показан пользователю, — но фрагмент вычисления как код существует не в одном месте, а в
**четырёх дословно идентичных копиях** (`proto.md` шаг 3 и шаг 6; `retrack.md` ветка легализации
п.1 и п.3). Дублирование — осознанная цена самодостаточности блока, а не недосмотр; при правке
алгоритма вычисления все четыре копии правятся синхронно.

  (`fail_marker=no` у единственной строки) и остаётся как есть.

- **[MODIFIED] `sdx/hooks/stage-gate.sh`** — две строки:
  - `:41` комментарий → `# Code writes are legitimate during Execution, Deployment and Prototype stages.`
  - `:43` `Execution|Deployment) exit 0 ;;` → `Execution|Deployment|Prototype) exit 0 ;;`

  Сообщение deny (`:76`) не меняется: на `Prototype` оно недостижимо. Дополнительных ограничений
  на пути, доступные прототипу, НЕ вводится (прод-территорию независимо от стадии закрывает
  `prod-guard.sh`, дублировать её в stage-gate — избыточно).

- **[UNCHANGED, НАМЕРЕННО] `sdx/hooks/stop-gate.sh`** — правка кода **не требуется и не
  выполняется**. `case "$stage" in Execution|Verification) ;; *) exit 0 ;; esac`
  (`stop-gate.sh:21-27`) уже трактует `Prototype` как transparent — ровно то поведение, которое
  требует режим («получить результат целиком, не воюя с тест-полом на коде, тестов у которого
  ещё нет по построению»). Это **осознанное решение, а не пропуск**; оно фиксируется тремя
  способами, чтобы не быть перепутанным с забывчивостью: (а) абзацем в `sdx/protocol.md`,
  раздел «Enforcement-слой», пункт stop-gate; (б) инвариантом ADR-018; (в) регрессионным
  сценарием в `test-stop-gate.sh` (см. «Тест-стратегия»), который упадёт, если кто-то
  «на всякий случай» добавит `Prototype` в enforcement-ветку.
  Побочные свойства, которые надо знать: `SDX_STOP_GATE=1` по-прежнему форсирует прогон и на
  `Prototype` (аварийный/headless-путь сохранён); детерминированный тест-пол возвращается
  автоматически в момент легализации, когда сессия приземляется на `Execution` целевого трека.

- **[UNCHANGED, НАМЕРЕННО] `sdx/hooks/archive-verify.sh`** — вариант А означает, что сессия
  `track=vibe` физически не может дойти до вызова этого скрипта. Новый инвариант не вводится,
  нумерация 1/5/6 (закреплённая ADR-009/012/015) не трогается.

- **[UNCHANGED] `sdx/hooks/stage-write-guard.sh`, `prod-guard.sh`, `preflight.sh`,
  `hooks/hooks.json`** — трек- и стадия-агностичны.

### Команды

- **[ADDED] `commands/proto.md`** — новая команда `/sdx:proto`: гейт принятия/отклонения
  прототипа. Полный алгоритм — раздел «Схема данных / API».

  **Обоснование выбора «новая команда», а не шаг в `/sdx:next` и не подрежим `/sdx:retrack`:**
  1. Контракт `/sdx:next` — «продвинуть `stage` через `sdx-stage.sh`». На `vibe` продвижения нет
     по построению; навесив на `next` деструктивный интерактивный гейт, мы бы сделали смысл самой
     частой команды фреймворка трек-зависимым и поместили безвозвратную операцию в путь, который
     пользователь жмёт машинально.
  2. Ни один исход гейта не является сменой `stage`: «принять» передаёт эстафету `/sdx:retrack`,
     «отклонить» оставляет `stage` нетронутым. Оба исхода вне контракта `next`.
  3. Прецедент формы уже есть: `/sdx:manual` (ADR-006) — интерактивный шаг, специфичный
     стадии, оформлен отдельной командой и вызывается из `/sdx:verify`, а не растворён в ней.
  4. Идемпотентность: `/sdx:proto` можно вызывать сколько угодно раз — до ответа на
     `AskUserQuestion` он ничего не мутирует, только показывает дифф.
  5. Обнаружимость: у команды есть собственное `description`, она видна в списке `/sdx:*`;
     «скрытая ветка внутри `next`» не видна.

## Схема данных / API

### `SDX_STAGE_MATRIX` — новая строка (дословно)

Добавляется в конец матрицы, после блока `doc`:

```
vibe|Prototype|-|no
```

Формат не расширяется: ни нового значения `fail_marker`, ни новой ветки в `stage_artifact_ok()`.
Единственная новизна — **число строк трека равно единице и последняя из них не `Closeout`**.

### `session_state.json` (REQ-COMPAT-1: схема полей НЕ меняется, расширяются только enum'ы)

```
"type":  "feature | bug | refactor | init | import | grooming | retro | postmortem | intake | proto"
"track": "patch | standard | full | doc | vibe"
"stage": "Discovery | Business Spec | Technical Design | Task Planning | Prototype | Execution | Documentation | Verification | Deployment | Closeout"
```

Ни один enforcement-скрипт не читает `type`; `track` читают только `sdx-stage.sh` (через матрицу).
Новых полей не вводится — номер попытки и baseline живут в отдельном файле (ниже).

### `prototype_baseline.txt` — новый артефакт сессии (только трек `vibe`)

Путь: `.claude/sessions/<id>/prototype_baseline.txt`. Обычный tracked-артефакт сессии, коммитится
как любой другой файл сессии (запрет коммитов REQ-VIBE-3 касается **кода**, не служебных файлов
сессии). Машиночитаемый, поэтому намеренно `.txt`, а не `.md`.

```
# SDX vibe prototype baseline
attempt 1
started 2026-07-20 14:05:11
rev 9f3c1a2b4d5e6f708192a3b4c5d6e7f8091a2b3c
untracked notes/scratch.txt
untracked tmp/local.env
```

- `attempt N` — номер текущей попытки (растёт при каждом отклонении; лимита нет, REQ-VIBE-5);
- `rev` — коммит, относительно которого считается tracked-дельта прототипа;
- `untracked <path>` — 0..N строк: untracked-файлы, существовавшие ДО начала попытки. Именно
  эти строки гарантируют, что посторонние untracked-файлы пользователя **не попадут** в список
  отката. Пути project-relative, ровно как их печатает `git ls-files --others`.

**Важное свойство `rev`.** Захват `rev` происходит ДО коммита самого baseline-файла, поэтому `rev`
может оказаться на один коммит позади `HEAD`. Это безвредно и учтено: тот единственный коммит
затрагивает исключительно `.claude/sessions/**`, а этот префикс исключён из всех вычислений
списка прототипа.

### Захват baseline (одинаково: `/sdx:start` шаг 8 для `vibe` и `/sdx:proto` после отклонения)

```bash
sdir=".claude/sessions/<id>"
# Гард безопасности: путь с переводом строки сделал бы построчный формат неоднозначным.
n_nul=$(git ls-files --others --exclude-standard -z | tr -dc '\0' | wc -c)
n_ln=$(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' | wc -l)
if [ "$n_nul" != "$n_ln" ]; then
  echo "SDX proto: среди untracked-файлов есть путь с переводом строки — baseline не может быть записан безопасно." >&2
  exit 1
fi
{
  printf '# SDX vibe prototype baseline\n'
  printf 'attempt %s\n' "<N>"
  printf 'started %s\n' "$(date '+%Y-%m-%d %H:%M:%S')"
  printf 'rev %s\n' "$(git rev-parse HEAD)"
  git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' | sed 's|^|untracked |'
} > "$sdir/prototype_baseline.txt"
git add "$sdir/prototype_baseline.txt" && git commit -m "sdx(<id>): vibe baseline (attempt <N>)"
```

При первом захвате `<N>` = 1; при повторном — `attempt` из старого файла + 1 (читается ДО
перезаписи). Если гард сработал — попытка не стартует, пользователю предлагается убрать/
переименовать проблемный путь; это честный отказ, а не тихая деградация к `git clean`.

**Уточнение по факту поставки: два разных представления `<N>`.** В `start.md` (первый захват —
всегда попытка 1) номер попытки — литеральная строка `"1"`, без параметризации: там нет
предыдущего `attempt`, который нужно было бы прочитать. В `proto.md` шаг 6 (повторный захват
после отклонения) используется вычисленное значение `<N+1>`, где `<N>` — `attempt`, запомненный
на шаге 2 ДО пересчёта (номер только что завершившейся попытки); символ `<N>` без пометки в
других местах того же шага (лог, коммит отклонения) обозначает именно эту, завершившуюся попытку,
а не новую. Двойной смысл `<N>` (описание тезиса «завершилась» и «стартует» одним символом)
устранён поставкой — оба употребления сопровождаются явной пометкой, какая попытка имеется в
виду.

### Вычисление списка файлов прототипа (`/sdx:proto`, и он же — вход легализации)

Канонический вид фрагмента — ниже. По факту поставки он вставлен дословно (не вызовом общей
функции — в командах `.md` её негде разместить) в четыре точки использования: `proto.md` шаг 3
(показ на гейте) и шаг 6 (перед откатом — самодостаточный пересчёт, см. Решение 4); `retrack.md`
ветка легализации п.1 (вход в реверс-инжиниринг) и п.3 (перед первым коммитом кода — тоже
самодостаточный пересчёт). Все четыре копии обязаны оставаться побайтово идентичными.

```bash
sdir=".claude/sessions/<id>"
base_rev="$(awk '$1=="rev"{print $2; exit}' "$sdir/prototype_baseline.txt")"
git rev-parse --verify --quiet "${base_rev}^{commit}" >/dev/null || {
  echo "SDX proto: baseline-ревизия ${base_rev} не найдена — точечный откат недоказуем." >&2; exit 1; }

tmp_tracked="$(mktemp)"; tmp_new="$(mktemp)"

# (1) tracked-дельта: изменённые, удалённые и добавленные-в-индекс файлы против baseline.
#     `git diff --name-only <rev>` сравнивает РАБОЧЕЕ ДЕРЕВО с ревизией, поэтому покрывает
#     и staged, и unstaged. Каталог артефактов сессии исключён — он живёт своей жизнью.
#     `-c core.quotePath=false` — иначе не-ASCII пути печатаются в C-кавычках
#     (`"\320\272..."`) и не совпадают с реальными именами файлов ниже по пайплайну.
git -c core.quotePath=false diff --name-only "$base_rev" -- . ':!.claude/sessions/**' > "$tmp_tracked"

# (2) новые untracked = untracked сейчас МИНУС untracked, записанные в baseline.
comm -23 \
  <(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' \
      | grep -v '^\.claude/sessions/' | LC_ALL=C sort) \
  <(sed -n 's/^untracked //p' "$sdir/prototype_baseline.txt" | LC_ALL=C sort) \
  > "$tmp_new"
```

**Почему это точечно и безопасно:** на входе в попытку дерево либо чисто (первая попытка,
гарантируется `start.md` шаг 4), либо приведено к чистоте откатом (повторная попытка), а всё, что
пользователь имел до попытки в виде untracked, поимённо перечислено в baseline и вычитается
шагом (2). Ни одна операция отката не обращается к путям вне `$tmp_tracked` ∪ `$tmp_new`.

### Отклонение прототипа — точечный откат (`/sdx:proto`)

Выполняется ТОЛЬКО после явного ответа пользователя в `AskUserQuestion`, которому предшествует
показ полного списка и диффа. **Самодостаточность блока (уточнение по факту поставки, см.
Решение 4):** `AskUserQuestion` — граница, за которой `$base_rev`/`$tmp_tracked`/`$tmp_new`,
вычисленные на шаге показа, больше не существуют, поэтому фактический блок действия открывается
их пересчётом (дословно фрагмент «Вычисление списка файлов прототипа» выше) и лишь затем
переходит к операциям (а)-(г) ниже — весь пересчёт+откат исполняется одним непрерывным bash-ходом.

```bash
# (а) tracked: восстановить содержимое baseline; файлы, созданных прототипом и попавших
#     в индекс, в baseline отсутствуют — их надо удалить, а не «восстанавливать».
while IFS= read -r f; do
  [ -z "$f" ] && continue
  if git cat-file -e "$base_rev:$f" 2>/dev/null; then
    git checkout "$base_rev" -- "$f"
  else
    git rm -f --quiet -- "$f"
  fi
done < "$tmp_tracked"

# (б) новые untracked: удалить поимённо. НИКАКОГО `git clean -fd`.
while IFS= read -r f; do
  [ -z "$f" ] && continue
  rm -f -- "$f"
done < "$tmp_new"

# (в) прибрать каталоги, опустевшие после (б); непустые каталоги не трогаются.
while IFS= read -r f; do
  d="$(dirname -- "$f")"
  [ "$d" = "." ] || rmdir -p --ignore-fail-on-non-empty -- "$d" 2>/dev/null || true
done < "$tmp_new"

# (г) верификация отката: вне каталога сессии дерево обязано стать чистым.
leftover="$(git status --porcelain -- . ':!.claude/sessions/**')"
rm -f "$tmp_tracked" "$tmp_new"
```

Если `leftover` непуст — откат **не** объявляется успешным: пользователю выводится остаток
дословно. `.gitignore` НЕ называется вероятной причиной: `git status --porcelain` без `--ignored`
игнорируемые файлы структурно не показывает, эта причина недостижима. Вероятные причины (называть
прозаически, не додумывая другие): (а) путь изменён/создан уже после вычисления списка — окно
между вычислением `$tmp_tracked`/`$tmp_new` и проверкой (в) не защищено блокировкой; (б) сбой
отдельной операции отката (`git checkout`/`git rm`/`rm` для конкретного файла в циклах (а)/(б) не
выполнился, а код возврата цикла не проверяется построчно); (в) untracked-файлы, поимённо
записанные в `prototype_baseline.txt` и потому намеренно исключённые из `$tmp_new` (`comm -23`) —
они не считаются частью прототипа и остаются как есть, поэтому появляются в `leftover` как
`?? path` даже при полностью успешном откате. Автодоуборка не выполняется ни в одном случае.
Сессия при этом всё равно остаётся на `Prototype`; ручная доуборка (или подтверждение, что остаток
— ожидаемый файл из причины (в)) — решение пользователя.

Далее: новый baseline (`attempt N+1`), строка в лог и коммит:

```bash
echo "[$(date '+%Y-%m-%d %H:%M:%S')] [CHECKPOINT] Прототип отклонён (попытка <N>), откат по списку из <K> файлов; новая точка — попытка <N+1>" >> "$sdir/session.log"
git add "$sdir/prototype_baseline.txt" "$sdir/session.log" && git commit -m "sdx(<id>): vibe prototype rejected (attempt <N>), baseline reset"
```

**Категория лога.** Используется существующая `CHECKPOINT` («фиксация точки восстановления») —
отклонение буквально устанавливает новую точку восстановления. Пятая категория лога **не
вводится**: enum ADR-005 (`START`/`STAGE_CHANGE`/`CHECKPOINT`/`ERROR`) остаётся закрытым.
`STAGE_CHANGE` здесь был бы неверен — этап не меняется, `stage` не пишется вообще, `sdx-stage.sh`
не вызывается.

### `/sdx:proto` — команда целиком

Frontmatter:
```yaml
---
description: Гейт принятия/отклонения прототипа (трек vibe): показать результат, принять с легализацией или откатить точечно.
---
```

Алгоритм:

1. Прочитай `session_state.json`. Если `track != "vibe"` → стоп: «команда применима только к
   сессиям трека `vibe`». Если `stage != "Prototype"` → стоп с указанием текущего этапа
   (сессия уже легализована → `/sdx:next`).
2. Прочитай `prototype_baseline.txt`. Отсутствует/не содержит `rev`/ревизия не резолвится →
   **fail-closed**: стоп, откат не предлагается. Baseline никогда не «додумывается» из `HEAD` —
   это ровно тот класс ошибки, который уничтожил бы чужие файлы.
3. Вычисли `$tmp_tracked` и `$tmp_new` (выше). Оба пусты → сообщи «прототип не произвёл
   изменений» и заверши без гейта (нечего принимать и нечего откатывать).
4. Предъяви результат пользователю:
   - сводка: `attempt N`, число изменённых tracked-файлов, число новых файлов, `git diff --stat "$base_rev" -- . ':!.claude/sessions/**'`;
   - полный список новых файлов (пути + число строк каждого);
   - дифф tracked-файлов: `git diff "$base_rev" -- . ':!.claude/sessions/**'`. Если он длиннее
     ~400 строк — покажи `--stat` + первые 200 строк и предложи развернуть полностью.
5. **`AskUserQuestion` — безусловно, независимо от `gate_mode` (REQ-VIBE-10).** Если
   `gate_mode == "auto"` — предварительно выведи одну строку: «стоп-рубрика протокола:
   деструктивная операция + продуктовое суждение — авторежим здесь не действует». В
   `auto_decisions.md` НИЧЕГО не пишется: дефолта, который можно было бы принять, нет.
   Три варианта ответа:
   - **Принять** — прототип достиг цели, идти в легализацию;
   - **Отклонить** — откатить показанный список;
   - **Продолжить работу** — выйти без действий (неразрушающий третий выход: пользователь не
     обязан принимать бинарное решение прямо сейчас).
6. **Отклонить** → выполнить откат (см. выше), новый baseline, лог, коммит; сообщить итог
   (сколько файлов восстановлено, сколько удалено, остаток если есть); напомнить, что сессия
   осталась на `Prototype` и попытка №N+1 может начинаться немедленно. Повторного подтверждения
   НЕ запрашивается: список и дифф были предъявлены как часть самого решения (REQ-VIBE-5), а
   второе «вы уверены?» обесценивает первое.
7. **Принять** → вторым `AskUserQuestion` определи целевой трек легализации: варианты
   `standard` и `full`, с **рекомендованным** значением, которое ты формулируешь прозаически —
   тем же суждением, каким выбираешь трек на `/sdx:start`: `full`, если прототип задел публичные
   контракты/схемы данных/архитектуру либо получился крупным; `standard` иначе. Затем выполни
   алгоритм `/sdx:retrack <выбранный трек>` (ветка легализации `commands/retrack.md` шаг 3).
   Автоматически, без подтверждения, трек НЕ меняется — целевой трек это второе решение
   пользователя, а не следствие первого.

### Легализация — `commands/retrack.md`, ветка «из `vibe`» (порядок операций строгий)

Предусловия: `track == "vibe"`, `stage == "Prototype"`, прототип принят на `/sdx:proto`, целевой
трек — `standard` или `full`.

1. **Вычисли дифф прототипа** тем же кодом, что и `/sdx:proto` (`prototype_baseline.txt` →
   `$tmp_tracked`/`$tmp_new`). Это единственный допустимый источник «что было сделано» —
   не память диалога и не `git status`.
2. **Spec-after реверс-инжиниринг — ДО коммитов.** `retrack` проверяет артефакты на диске, а не
   в git, поэтому порядок «сначала артефакты, потом коммиты» полностью легален:
   - цель **`standard`** — `change_note.md` создаётся **инлайн основной сессией** (симметрия с
     треком `standard`, где `Discovery`/`Change` идут инлайн, без `architect`). Содержимое:
     что и зачем сделано (переформулировано как требование, а не как пересказ диффа),
     перечень затронутых файлов, сознательно отложенное. Если объём прототипа таков, что инлайн
     реверс-инжиниринг честно не помещается, — это сигнал, что целевой трек `full`, а не повод
     писать тонкий `change_note.md`;
   - цель **`full`** — последовательно, через Task: `architect` → `context_report.md`
     (ретроспективный: что прототип задел в кодовой базе, какие интеграционные точки и допущения
     принял де-факто); `ba` → `SPEC.md` (Delta-формат, `[ADDED]`); `architect` → `DESIGN.md`;
     `lead-dev` → `PLAN.md` (задачи вида «покрыть тестами / довести до контракта / убрать
     срезанные углы», не «реализовать с нуля»). Всем четверым на вход даются ПУТИ + команда
     получения диффа, а не тело диффа (экономия токенов, ADR-005), кроме случая, когда дифф
     мал и передать его дешевле.
3. **Первый коммит кода (REQ-VIBE-7) + пост-проверка, что он состоялся, — только по вычисленному
   списку, никогда `git add -A`.** Между п.2 (`Task`-вызовы субагентов) и этой точкой shell-сессия
   не переживает границу — блок открывается самодостаточным пересчётом `$tmp_tracked`/`$tmp_new`
   (дословно тот же фрагмент, что и в п.1 и в `proto.md` шаг 3/6 — см. «Вычисление списка файлов
   прототипа» и уточнение к Решению 4), затем объединяет оба списка в один union-список `$tmp_all`
   (ровно те файлы, за которые отвечает пост-проверка ниже) и коммитит по нему:
   ```bash
   tmp_all="$(mktemp)"
   cat "$tmp_tracked" "$tmp_new" | sed '/^$/d' | LC_ALL=C sort -u > "$tmp_all"

   if [ -s "$tmp_all" ]; then
     xargs -d '\n' git add -- < "$tmp_all"
     git commit -m "sdx(<id>): vibe prototype code (legalized)"
     commit_rc=$?
   else
     commit_rc=0
   fi
   ```
   `xargs -d '\n'` вместо `$(cat "$tmp_all")` — иначе пути с пробелами ломаются word-splitting'ом
   (та же защита нужна и в пост-проверке ниже — не только в `git add`). Пустой `$tmp_all`
   (прототип с последнего baseline ничего не изменил) — легитимный случай, не ошибка: `git
   add`/`git commit` вовсе не вызываются, `commit_rc` считается `0` явно (иначе `git add --` без
   аргументов молча вернул бы `0`, ничего не застейджив, а `git commit` упал бы «nothing to
   commit»). Запрет `git add -A`/`git add .` симметричен запрету слепого авто-коммита в
   `switch.md` и служит той же цели: посторонние untracked-файлы пользователя не должны попасть в
   ветку сессии.

   **Пост-проверка REQ-VIBE-7 (уточнение по факту поставки).** Guard `sdx-stage.sh retrack`
   (Решение 3) фактом коммита не занимается — он проверяет только гейт-артефакты на диске.
   Единственное, что удерживает REQ-VIBE-7 «коммит кода состоялся ДО смены трека», — эта
   прозаическая пост-проверка, выполняемая ДО п.5 (смены `track`/`stage`). Критерий —
   **единственный**: фактическое состояние файлов ИЗ ВЫЧИСЛЕННОГО СПИСКА (`leftover_code =
   git status --porcelain` по `$tmp_all`, через тот же `xargs -d '\n'`), а не код возврата `git
   commit`:
   ```bash
   if [ -s "$tmp_all" ]; then
     leftover_code="$(xargs -d '\n' git status --porcelain -- < "$tmp_all")"
   else
     leftover_code=""
   fi
   ```
   Ненулевой `commit_rc` при пустом `leftover_code` — **легитимный сценарий, не отказ** (например,
   код прототипа уже был закоммичен пользователем вручную во время `Prototype`: `git add` не
   застейджил ничего нового, `git commit` упал с «nothing to commit», но файлы списка уже чисты
   относительно `HEAD` — REQ-VIBE-7 фактически выполнен). Проверяется РОВНО вычисленный список
   (`$tmp_all`), а не всё рабочее дерево — иначе проверка ложно спотыкается о посторонние
   untracked-файлы пользователя, зафиксированные в baseline (тот же класс дефекта, что и W-2
   второго прогона ревью, уже устранённый). При непустом `leftover_code` легализация
   останавливается с сообщением об ошибке (остаток `git status` выводится дословно), `track` в
   файле состояния в этой точке ещё не тронут — рассинхронизации между `track`/`stage` не
   возникает.
4. **Второй коммит — артефакты реверс-инжиниринга:**
   ```bash
   git add .claude/sessions/<id> && git commit -m "sdx(<id>): legalize prototype — spec-after artifacts"
   ```
   Порядок «код → артефакты» выбран для читаемости `git log`: первый коммит — точное замыкание
   прототипа (`git show` даёт ровно его), второй — выведенный из него контракт. Для `reviewer`
   порядок безразличен: `verify.md` шаг 4 берёт дифф всей ветки против основной.
5. **Смена трека и этапа** — существующий механизм `retrack.md` шаг 4 без изменений:
   `Edit` поля `track` (`vibe` → `standard|full`), сброс `gate_mode` в `interactive` при `auto`
   (легализация = эскалация), затем
   ```bash
   echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] Смена трека vibe → <new_track> (легализация прототипа)" >> .claude/sessions/<id>/session.log \
     && bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<id>" "Execution"
   ```
   Как это проходит evidence-based guard:
   - `standard`: предшественники `Execution` — `Discovery` (артефакт `-`, засчитывается всегда)
     и `Change` (`change_note.md` ИЛИ пара `SPEC.md`+`DESIGN.md`) → шаг 2 их создал → код `0`;
   - `full`: предшественники — `Discovery`/`Business Spec`/`Technical Design`/`Task Planning`
     → нужны все четыре файла → шаг 2 их создал → код `0`;
   - если шаг 2 выполнен не полностью — код `1`, stderr называет конкретный непройденный этап;
     это и есть механическая невозможность «легализоваться на словах». Fallback — `Discovery`
     нового трека (пустая цепочка предшественников) с последующим штатным `/sdx:next`.
6. Дальше — обычный жизненный цикл целевого трека: `Execution` (доводка, `developer`;
   с этого момента stop-gate снова активен, а норма инкрементальных коммитов действует без
   исключений — REQ-VIBE-9) → `Verification` (`qa` + fresh-eyes `reviewer`) → `Closeout`
   (`/sdx:archive`), уже с собственным `Closeout` целевого трека.

### `commands/archive.md` — цикл входа в Closeout после правки (дословно)

```bash
sid="<session_id>"; sdir=".claude/sessions/$sid"
noop_terminal=""
while :; do
  out="$(bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" next "$sid")"; rc=$?
  [ "$rc" -eq 0 ] || break
  cur_stage="$(jq -r '.stage' "$sdir/session_state.json")"
  case "$out" in
    "OK no-op "*)
      # Терминальный этап трека: next НИКОГДА не продвинется дальше, повтор итерации
      # был бы бесконечным no-op. Единственный корректный терминал протокола — Closeout;
      # любой другой означает трек без собственного Closeout (vibe, ADR-018).
      [ "$cur_stage" = "Closeout" ] || noop_terminal="$cur_stage"
      break
      ;;
  esac
  git add "$sdir/session_state.json" "$sdir/session.log"
  git commit -m "sdx($sid): $out"
  [ "$cur_stage" = "Closeout" ] && break
done
```

Разбор изменений относительно текущего кода (`commands/archive.md:16-31`):

- **Детект терминальности — по форме stdout**, а не по коду возврата и не по «коммит ничего не
  закоммитил». `OK no-op <stage>` — часть контракта REQ-STAGE-5, а значит устойчивое, объективное
  условие. Это и закрывает найденный риск: сегодня итерация на `vibe|Prototype` давала
  `rc=0`, `stage` не менялся, `git commit` тихо падал «nothing to commit», и цикл шёл по кругу
  вечно.
- **Условие остановки трек-агностично**: `noop_terminal` сработает для ЛЮБОГО трека, чей
  последний активный этап не `Closeout`, — не только для `vibe`.
- **Побочно устранён паразитный коммит**: раньше вызов `/sdx:archive` на сессии, уже стоящей
  на `Closeout`, порождал бессмысленную пару `git add`/`git commit` (падавшую с «nothing to
  commit»); теперь no-op выходит из цикла до неё.

Прозаическая обработка после цикла:
- `noop_terminal` непуст → **стоп, чек-лист Closeout не начинается**. Сообщение: «Трек
  `<track>` не имеет собственного `Closeout`: сессия остановлена на терминальном этапе
  `<noop_terminal>`. Для трека `vibe` это ожидаемо — прототип обязан пройти легализацию:
  `/sdx:proto` (если решение принять/отклонить ещё не принято), затем `/sdx:retrack
  standard|full`. Закрыть сессию можно только после легализации (REQ-VIBE-8/ADR-018)».
  В лог — `[ERROR] Ошибка перехода: трек без собственного Closeout, этап <noop_terminal>`.
- прочие ветки (`rc=1`, `rc=2`) — без изменений.

**Проверка того же класса риска в других командах.** `grep -n "while" commands/*.md` даёт ровно
одно вхождение — этот цикл. `commands/next.md` вызывает `sdx-stage.sh next` **однократно**, без
цикла; там правится только формулировка no-op и добавляется маршрутизация на `/sdx:proto`
(риска зацикливания нет). `backtrack.md`/`retrack.md` циклов не содержат.

### Правки `sdx/protocol.md` (семь точек)

1. Блок «Состояние сессии» — enum'ы `type`/`track`/`stage` (см. выше).
2. Таблица «Профили флоу» — новая строка. **Формулировка ячейки «Активные этапы» важна для
   sanity-теста 34** (он вырезает завершающую скобку и сравнивает цепочку дословно):
   ```
   | **vibe** | экспериментальное прототипирование «в один промпт» (тип `proto`, жёстко привязан к этому треку на `/sdx:start` — ADR-018) | Prototype (единственный активный этап; собственного Closeout нет — выход только легализацией через `/sdx:retrack`) | — |
   ```
   Ограничение по конструкции: аннотация в ячейке цепочки — ровно одна пара скобок в конце и
   без вложенных скобок, иначе sanity-тест не срежет её.
3. Примечание под таблицей — формулировка sanity-теста расширяется на пять треков.
4. Блок «Пояснения» — новый буллет про `vibe`: параллельная ветка вне шкалы
   `patch → standard → full`; отсутствие собственного `Closeout` как структурный инвариант;
   именованное исключение из нормы инкрементальных коммитов (REQ-VIBE-9); гейт `/sdx:proto`
   безусловно интерактивен; вход только через `/sdx:start proto`.
5. Раздел «Enforcement-слой»: в пункте **stage-gate** — «запись кода открыта на `Execution`,
   `Deployment` и `Prototype`»; в пункте **stop-gate** — явный абзац, что на `Prototype` хук
   прозрачен по общему правилу `*) exit 0`, и это **намеренно** (тестов на прототипе нет по
   построению; пол возвращается на `Execution` целевого трека после легализации; `SDX_STOP_GATE=1`
   форсирует прогон и здесь).
6. Раздел «Контракт закрытия сессии (Closeout)»: фраза «Closeout — обязательный завершающий этап
   для ВСЕХ треков» уточняется: «…для всех треков линейной шкалы и `doc`. Трек `vibe`
   собственного `Closeout` не имеет вовсе (ADR-018): сессия обязана сперва легализоваться
   через `/sdx:retrack`, после чего закрывается по правилам целевого трека». Плюс строка в
   «Гейт входа в Closeout»: при `track == "vibe"` — стоп с направлением на легализацию.
7. Раздел «Верификация» — фраза, что `vibe` не имеет собственной верификации и наследует объём
   целевого трека; `reviewer` видит код прототипа в диффе ветки наравне с обычным кодом.

## Обработка ошибок и Граничные случаи

- **`/sdx:archive` на непринятом прототипе** — детерминированно останавливается (no-op-детект,
  выше). Пользователь получает ровно одно действие для продолжения.
- **`/sdx:next` на `vibe|Prototype`** — легитимный no-op; сообщение направляет на `/sdx:proto`
  или `/sdx:retrack`, коммитить нечего (файл состояния скриптом не тронут).
- **`/sdx:backtrack --to <любой этап>` на `vibe`** — все цели, кроме `Prototype`, отбиваются
  кодом `1` с существующим сообщением «не активен в треке — нужна смена трека, не откат».
  `--to Prototype` — no-op.
- **`backtrack --to Prototype` из чужого трека** — код `1` («не активен в треке `<track>`»),
  потому что `Prototype` теперь есть в union имён, но не в цепочке этого трека. Проверяется
  негативным сценарием [55].
- **Baseline утрачен/повреждён** — `/sdx:proto` fail-closed: откат не предлагается вовсе,
  пользователю сообщается, что точечность недоказуема, и предлагается разобрать дерево вручную
  (`git status`, `git diff`). Ни при каких условиях не подставляется `HEAD` вместо `rev` и не
  выполняется `git clean`.
- **Путь с переводом строки среди untracked** — гард на захвате baseline; попытка не стартует.
- **Файлы прототипа под `.gitignore`** — вне замыкания `--exclude-standard`, откатом не
  затрагиваются. Названная граница: игнорирование файла — это утверждение проекта, что файл
  не является исходником; удалять такие файлы автоматически опаснее, чем оставить. Такие файлы
  структурно не могут быть причиной непустого `leftover` шага (г) — `git status --porcelain` без
  `--ignored` игнорируемые файлы не печатает вовсе (см. диагностику непустого `leftover` в
  «Отклонение прототипа» выше); они просто продолжают существовать нетронутыми.
- **Коммит кода при легализации не состоялся** (`retrack.md` ветка легализации, п.3) — детектируется
  пост-проверкой по единственному критерию: `leftover_code` (`git status --porcelain` РОВНО по
  вычисленному union-списку `$tmp_all`, не по всему дереву) непуст. Ненулевой `commit_rc` при
  пустом `leftover_code` — легитимный сценарий (код уже был закоммичен вручную), а не отказ, и
  сам по себе легализацию не останавливает. Легализация останавливается ДО правки `track`/`stage`;
  пользователю выводится остаток `git status` дословно, повторный вызов возможен после ручного
  разбора. Guard `sdx-stage.sh retrack` этот случай не ловит — он проверяет только артефакты
  реверс-инжиниринга на диске (Решение 3), а не git-историю.
- **Прототип успел закоммитить код** (нарушение REQ-VIBE-3, например ручным `git commit`
  пользователя) — `git diff --name-only $base_rev` всё равно перечислит такие файлы (сравнение
  с ревизией, а не с индексом), и `git checkout $base_rev -- <path>` вернёт их содержимое; но
  коммиты в истории ветки останутся. Это честно сообщается пользователю: «откат восстановил
  содержимое файлов, но коммиты <sha...> остались в истории ветки — при необходимости
  используй `git reset`». Автоматический `reset` не выполняется (переписывание истории —
  деструктив, не покрытый данным решением).
- **Прототип удалил tracked-файл** — попадает в `$tmp_tracked`, восстанавливается
  `git checkout`. Покрыто конструкцией, отдельной ветки не требуется.
- **Прототип создал файл и сделал `git add`** — не виден `ls-files --others`, но виден
  `git diff --name-only $base_rev`; в baseline-ревизии его нет → ветка `git rm -f` (шаг (а)).
- **Повторные попытки** — `attempt` инкрементируется, новый baseline перезаписывает старый
  (история старых baseline'ов доступна через `git log -p -- prototype_baseline.txt`; отдельный
  архив попыток не ведётся — сознательно, см. решение по `init.md`).
- **`/sdx:switch` с грязного `vibe`-дерева** — гард срабатывает, штатный путь — `git stash`;
  формулировка сообщения правится (см. «Компоненты»). Технически стэш безопасен для baseline:
  `rev` указывает на коммит, а не на состояние дерева.
- **`gate_mode: auto` на `proto`** — принимается, но практической поверхности до легализации не
  имеет; при легализации сбрасывается в `interactive` как при любой эскалации. Пользователю это
  сообщается один раз на старте.
- **Коллизия триады на легализации** — обычная стоп-рубрика: если реверс-инжиниринг вскрывает
  противоречие прототипа с существующими `docs/specs/`/`docs/designs/`, работа останавливается
  и решение запрашивается у пользователя (CLAUDE.md §4), в любом `gate_mode`.

## Безопасность

- **Инвариант «нет мёржа без легализации» выражен структурой данных.** Отсутствие
  `vibe|Closeout` в матрице — не проверка, которую можно обойти, а отсутствие пути. Обход
  потребовал бы прямой правки `track` в `session_state.json` (легитимная операция для
  `retrack`, но она и ЕСТЬ легализация — с обязательным прохождением evidence-based guard) либо
  ручного мёржа ветки мимо `/sdx:archive` (вне модели угроз фреймворка: пользователь всегда
  может сделать `git merge` руками, никакая проза и никакой хук этого не запрещают).
- **Деструктив выполняется только по явно предъявленному списку.** Ни `git clean -fd`, ни
  `git checkout .`, ни `git reset --hard` в поставке не появляются. Все операции — поимённые,
  над множеством, вычисленным из baseline. Файлы вне множества недостижимы для кода отката
  по построению.
- **Гейт деструктива безусловно интерактивен** (REQ-VIBE-10) — совпадает со стоп-рубрикой
  протокола («деструктивные/необратимые операции»); `gate_mode: auto` не смягчает.
- **Fail-closed при недоказуемом baseline** — отсутствие/повреждение снимка блокирует откат
  целиком, а не переводит его в «широкий» режим. Асимметрия выбрана сознательно: цена
  ручной уборки ниже цены уничтожения чужих файлов.
- **Первый коммит кода — по списку, не `-A`** — исключает захват посторонних untracked-файлов
  в ветку сессии.
- **stage-gate расширяется ровно на одну стадию** нового трека; прод-территория остаётся за
  `prod-guard.sh`, который стадия-агностичен и на `Prototype` работает как обычно.
- **Ослабление тест-пола ограничено стадией `Prototype`** — единственная стадия единственного
  трека, где stop-gate прозрачен по существующей общей логике. Компенсация — обязательные
  тесты и полная верификация после легализации (REQ-VIBE-6), без которых сессия не закрывается.
- **Схема `session_state.json` не расширяется** (REQ-COMPAT-1): новые данные (baseline, номер
  попытки) вынесены в отдельный файл, что исключает конфликт со `stage-write-guard.sh`.

## Тест-стратегия

### `sdx/hooks/test-sdx-stage.sh`

**Сценарий 34 (sanity: матрица ↔ таблица протокола) — обобщается, но НЕ достаточен.**
Механически он обобщается правкой одного слова: `for track in full standard patch doc` →
`for track in full standard patch doc vibe`. Цикл параметричен, вырезание завершающей скобочной
аннотации работает и для однокомпонентной цепочки, разделитель ` → ` в ней просто отсутствует
(`sed 's/ → /\n/g'` оставляет одну строку). Единственное требование — формулировка ячейки
протокола (см. выше): ровно одна пара скобок, в конце, без вложенности.

структурный инвариант REQ-VIBE-8 будет уничтожен молча. Поэтому вводится отдельный sanity:

- **[49] sanity: трек `vibe` состоит ровно из одного этапа `Prototype` и не имеет `Closeout`.**

## Новый ADR-018 (тезисы для `docs/DECISIONS.md`)

**Заголовок:** `## ADR-018. Трек vibe и тип сессии proto: прототипирование без церемонии + обязательная легализация`

- **Контекст.** Любая сессия, производящая код, несёт церемонию своего трека и норму
  инкрементальных коммитов (ADR-005/REQ-SESS-1). Для исследовательской работы, где ценность идеи
  ещё не доказана, это заставляет платить спеками и коммитами за результат, который может быть
  отброшен целиком (`docs/backlog/PROC-004`).
- **Решение.** Пятый профиль флоу `vibe` с **единственным** активным этапом `Prototype` и пятым
  типом сессии `proto`, жёстко привязанным к нему на `/sdx:start` (прецедент ADR-017). Выход из
  трека — только `/sdx:retrack` на `standard`/`full` (легализация: spec-after реверс-инжиниринг →
  первый коммит кода → тесты → верификация → Closeout целевого трека). Интерактивный гейт
  принятия/отклонения — отдельная команда `/sdx:proto`; отклонение выполняет точечный откат по
  списку, вычисленному от baseline-снимка попытки.
- **Ключевое решение — enforcement формой данных.** У трека `vibe` НЕТ строки `Closeout` в
  `SDX_STAGE_MATRIX`. «Нет мёржа без легализации» становится не проверкой, а отсутствием пути:
  `cmd_next` навсегда возвращает `OK no-op Prototype`. Процедурный код `sdx-stage.sh` не
  меняется ни на строку.
- **Именованное исключение из ADR-005/REQ-SESS-1.** Приостановление обязательности
  инкрементальных коммитов действует ТОЛЬКО на стадии `Prototype` трека `vibe` и снимается
  первым коммитом кода при легализации. Это исключение, а не отмена нормы: на всех прочих
  треках/стадиях и на самой сессии после легализации норма действует без изменений.
- **Альтернативы (отклонены):**
  1. *Проверка `track != "vibe"` в `archive-verify.sh`.* Скрипт вызывается ПОСЛЕ мёржа (шаги
     6→7 Closeout) — сработала бы слишком поздно; плюс потребовала бы нового инварианта в
     закреплённой нумерации 1/5/6 (ADR-009/012/015).
  2. *Только прозаический гейт входа в Closeout.* PROC-004 требует именно инварианта
     enforcement; проза этого не даёт. Оставлена как defense-in-depth ПОВЕРХ структурного
     решения, не вместо.
  3. *Гейт принятия/отклонения внутри `/sdx:next`.* Смысл самой частой команды стал бы
     трек-зависимым, а деструктивная операция оказалась бы в машинально нажимаемом пути.
  4. *`git clean -fd` при отклонении.* Уничтожил бы посторонние untracked-файлы пользователя;
     заменён точечным откатом по baseline (REQ-VIBE-5).
  5. *Fresh-eyes `reviewer` на гейте принятия.* Противоречит духу режима («один промпт»);
     компенсируется обязательным ревью на Verification после легализации.
  6. *Лимит числа попыток отклонения.* Отклонён по решению пользователя на Discovery-гейте;
     при необходимости вводится отдельной доработкой.
- **Последствия.**
  - Первый трек, нарушающий допущение «последний активный этап = `Closeout`»; допущение жило в
    while-цикле `commands/archive.md` и потребовало обязательной защиты по контракту stdout.
    Защита сделана трек-агностичной.
  - Первая процедурная (не только матричная) правка enforcement-хука ради нового трека:
    `stage-gate.sh` открывает запись кода на `Prototype` (одна строка `case`).
  - На `Prototype` нет детерминированного тест-пола (stop-gate прозрачен по общей логике) —
    осознанно; пол возвращается на `Execution` целевого трека.
  - Запрет входа `{patch,standard,full,doc} → vibe` и запрет `vibe → patch` — прозаические, не
    enforced скриптом (тот же известный класс, что запрет деэскалации в `doc`, ADR-017).
  - Отклонённая попытка не оставляет постоянного следа в `docs/` — сознательно.
- **Инварианты.**
  - У трека `vibe` ровно один активный этап; `Closeout` в его цепочке отсутствует —
    проверяется sanity-сценарием [49] `test-sdx-stage.sh`.
  - Единственный выход из трека — `/sdx:retrack` на `standard`/`full`; он же обязан произвести
    первый коммит кода прототипа.
  - Гейт `/sdx:proto` интерактивен в любом `gate_mode`.
  - Откат ограничен списком, вычисленным от `prototype_baseline.txt`; `git clean`/`checkout .`/
    `reset --hard` не применяются; при недоказуемом baseline откат не выполняется вовсе.
  - Схема `session_state.json` не расширяется — только enum'ы значений.

## Риски и границы поставки

1. **Прозаические, не enforced границы: запрет входа в `vibe` и запрет `vibe → patch`.**
   Скрипт `retrack` пропустит вход в `vibe` (у `Prototype` пустая цепочка предшественников).
   Держится текстом `commands/retrack.md`. Принятый trade-off, тот же класс, что ADR-017/DEBT-007;
   кандидат в `DEBT-` при обнаружении реального нарушения.
2. **Отсутствие тест-пола на `Prototype`.** Прототип может завершиться с красными тестами
   проекта — по замыслу режима. Риск: пользователь принимает прототип, не заметив регрессии в
   существующей функциональности. Компенсация — обязательная верификация после легализации;
   регрессия обнаружится там, но позже и дороже.
3. **Отсутствие fresh-eyes ревью до легализации** (SPEC «Риски»): решение принять прототип
   целиком доверено пользователю.
4. **Дрейф дублирующейся копии матрицы** в `docs/designs/stage-transition-guard.md` —
   унаследованный риск (не покрыт sanity-тестом, синхронизируется вручную). Новой поставкой не
   решается; кандидат в `DEBT-` (sanity-тест на третью копию матрицы).
5. **REQ-VIBE-12 — параллельное прототипирование вне объёма** (зависимость `PROC-010`: гонки
   параллельных субагентов за общие файлы сессии). Одна сессия `proto` = один прототип.
6. **REQ-VIBE-13 — структурированный вход материала вне объёма** (зависимость `PROC-009`):
   цель прототипа передаётся текстом промпта `/sdx:start`.
7. **Файлы под `.gitignore`, созданные прототипом, переживают откат** — названная граница
   (см. «Обработка ошибок»), не дефект реализации.
8. **Ручные коммиты пользователя во время `Prototype`** делают откат неполным на уровне
   истории (содержимое восстанавливается, коммиты остаются). Автоматический `git reset` не
   выполняется сознательно.
9. **Объём прозы.** Основная масса поставки — инструкции команд и агентов, не покрываемые
   юнит-тестами; риск смягчается сценариями `MANUAL_TEST.md` и fresh-eyes ревью.
10. **DEBT-017 как активное предупреждение.** Перечисления треков правятся во ВСЕХ восьми
    `agents/*.md`; для `reviewer.md` принято мотивированное решение НЕ добавлять `vibe` в enum —
    это должно быть явно квитировано на Verification, иначе будет прочитано как повторение
    того же дефекта.

## Версионирование плагина

Рекомендуемый bump — **MINOR**: `1.5.1 → 1.6.0`. Новый трек + новый тип сессии + новая команда +
новый ADR — по значимости сопоставимо с ADR-017 (`1.3.0 → 1.4.0`). Ломающих изменений контракта
нет: существующие треки, схема состояния и коды возврата `sdx-stage.sh` не меняются.
