# Manual Test Guide: Режим экстремального прототипирования (трек `vibe`, тип `proto`)

Сессия: `feat-vibe-track-20260720`. Исполняется интерактивно через `/sdx:manual`.

## Зачем этот файл

Автотесты (`sdx/hooks/test-*.sh`) покрывают ТОЛЬКО машиночитаемую матрицу состояний
(`sdx-stage.sh`) и два enforcement-хука (`stage-gate.sh`, `stop-gate.sh`). Основная масса
поставки — `/sdx:proto` целиком, маршрутизация `/sdx:next`, защита цикла в `/sdx:archive`,
запреты направления в `/sdx:retrack` и правки `agents/*.md` — это проза инструкций для Session
Manager, не исполняемый код; юнит-тестов для неё в фреймворке нет (DESIGN, «Что тестами НЕ
покрывается»). Сценарии ниже — прямая компенсация этого пробела, минимум по заданию Verification
плюс два дополнительных сценария (F, G), закрывающих критерии приёмки SPEC, для которых при
составлении этого плана не нашлось ни автоматического, ни изначально запланированного ручного
покрытия (см. итоговую таблицу покрытия в отчёте `qa`).

## Подготовка окружения

Все сценарии выполняются **вне текущего рабочего дерева** этого репозитория — сессия
`feat-vibe-track-20260720` активна на ветке `sdx/feat-vibe-track-20260720` (этап Verification),
и команды `/sdx:start`/`/sdx:proto`/`/sdx:archive`/`/sdx:retrack` меняют состояние сессии и ветку.
Запуск их поверх ЖИВОЙ сессии, которую сейчас верифицирует `qa`, создал бы конфликт состояния и
рисковал бы данными этой самой сессии.

1. Подготовь отдельную песочницу — git-репозиторий вне этого чекаута (например,
   `/tmp/sdx-vibe-manual-test/`), с установленным SDX-плагином той же версии, что и эта ветка
   (`CLAUDE_PLUGIN_ROOT` указывает на актуальный чекаут `sdx-claude-plugin` этой ветки, либо
   отдельная установка плагина из неё).
2. Инициализируй в песочнице SDX (`/sdx:init`).
3. Перед КАЖДЫМ `/sdx:start` убедись, что `git status --porcelain` в песочнице пуст.
4. Сценарии сгруппированы в блоки A–G; блоки B и C переиспользуют одну и ту же сессию
   (см. предусловия шагов 12–13). Остальные блоки независимы и каждый начинается с новой
   `proto`-сессии, если явно не указано иное.

## Блок A: `/sdx:archive` на непринятом vibe-прототипе — единственная остановка без зацикливания

Покрывает критерии приёмки SPEC 5 и 7 (нет мёржа без легализации; `/sdx:archive` без
легализации останавливается с указанием на `/sdx:retrack`). Прямая проверка риска, найденного
в DESIGN («сегодня итерация на `vibe|Prototype` давала `rc=0`, `stage` не менялся, `git commit`
тихо падал… и цикл шёл по кругу вечно»).

### Шаг 1: Старт vibe-сессии

**Действие:** В песочнице выполнить `/sdx:start proto "тестовый прототип: страница статуса"`.
**Предусловия:** рабочее дерево песочницы чистое.
**Ожидаемый результат:** создан `session_state.json` с `type: "proto"`, `track: "vibe"`,
`stage: "Prototype"`; в `session.log` есть строка `[START]`; диалог о выборе альтернативного
трека НЕ показан.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 2: Минимальное изменение без прохождения гейта

**Действие:** Создать/изменить один произвольный файл в песочнице (например, `app.js` с любым
содержимым), не вызывая `/sdx:proto`.
**Предусловия:** сессия из шага 1 активна, `stage = Prototype`.
**Ожидаемый результат:** изменение вносится без ошибок enforcement-хуков (запись кода открыта
на `Prototype`, REQ-VIBE-3).
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 3: Прямой вызов `/sdx:archive` без легализации

**Действие:** Выполнить `/sdx:archive`.
**Предусловия:** прототип не принят и не отклонён (гейт `/sdx:proto` ещё не вызывался).
**Ожидаемый результат:** команда останавливается ОДНИМ сообщением, указывающим на необходимость
пройти `/sdx:proto` (если решение ещё не принято) либо `/sdx:retrack standard|full` (если уже
принято); чек-лист Closeout НЕ начинается; `session_state.json.stage` остаётся `Prototype`;
НЕ создаётся ни одного нового git-коммита (`git log` до и после вызова идентичен); команда
завершает выполнение за разумное время, не зависает и не выводит повторяющихся строк
`OK no-op Prototype` по кругу.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 4: Повторный вызов `/sdx:archive` — регрессия детерминированности

**Действие:** Немедленно повторно выполнить `/sdx:archive`.
**Предусловия:** шаг 3 выполнен, состояние сессии между вызовами не менялось.
**Ожидаемый результат:** поведение идентично шагу 3 (то же сообщение, тот же стоп, `stage`
по-прежнему `Prototype`, новых коммитов нет) — подтверждает, что остановка детерминирована.
**Фактический результат:** _(заполняется при исполнении)_

## Блок B: `/sdx:proto` — предъявление списка ДО деструктива и точечность отката

Покрывает критерий приёмки SPEC 4 (список+дифф предъявлены до отката; откат ограничен списком;
файлы вне списка не затронуты).

### Шаг 5: Старт новой vibe-сессии и работа прототипа

**Действие:** `/sdx:start proto "тестовый прототип: экспорт CSV"`; затем создать/изменить 1–2
файла как имитацию работы прототипа (например, `export.js`).
**Предусловия:** чистое рабочее дерево песочницы.
**Ожидаемый результат:** сессия на `Prototype`, изменения внесены, промежуточных коммитов кода
нет (REQ-VIBE-3).
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 6: Создание постороннего файла вне контекста прототипа

**Действие:** В отдельном терминале (не через ассистента) создать файл, не относящийся к
прототипу — например, `scratch/user-notes.txt` с произвольным текстом.
**Предусловия:** шаг 5 выполнен, сессия ещё на `Prototype`, гейт `/sdx:proto` ещё не вызывался.
**Ожидаемый результат:** файл присутствует в рабочем дереве как untracked
(`git status --porcelain` показывает `??`).
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 7: Вызов гейта — проверка предъявления ДО деструктива

**Действие:** Выполнить `/sdx:proto`.
**Предусловия:** шаги 5–6 выполнены.
**Ожидаемый результат:** команда показывает сводку (`attempt 1`, число tracked/новых файлов),
ПОЛНЫЙ список новых файлов (включая `scratch/user-notes.txt`) и дифф — ДО того, как задать
вопрос о решении; `AskUserQuestion` с тремя вариантами (Принять/Отклонить/Продолжить работу)
появляется только после показа списка.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 8: Неразрушающий выход

**Действие:** Выбрать «Продолжить работу».
**Предусловия:** шаг 7 выполнен, вопрос показан.
**Ожидаемый результат:** ни один файл не изменён и не удалён; `git status` идентичен состоянию
до вызова; сессия осталась на `Prototype`; `attempt` в `prototype_baseline.txt` не изменился.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 9: Изъятие постороннего файла из вычисления

**Действие:** Переместить посторонний файл за пределы рабочего дерева, например
`mv scratch/user-notes.txt /tmp/user-notes.txt.bak`.
**Предусловия:** шаг 8 выполнен.
**Ожидаемый результат:** файл отсутствует в рабочем дереве песочницы; `git status --porcelain`
больше не показывает его.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 10: Точечный откат — «Отклонить»

**Действие:** Повторно выполнить `/sdx:proto`, выбрать «Отклонить».
**Предусловия:** шаг 9 выполнен (посторонний файл вне дерева на момент вычисления списка).
**Ожидаемый результат:** показанный на этот раз список содержит ТОЛЬКО файлы прототипа
(`export.js` и т.п.), без постороннего файла; после отклонения файлы прототипа откатываются
точечно (созданные — удалены, изменённые — восстановлены до состояния baseline); `attempt`
увеличивается на 1; в `session.log` — запись `[CHECKPOINT]`; создаётся коммит
`vibe prototype rejected`. Никаких `git clean -fd`/`git checkout .`/`git reset --hard` не
выполняется (наблюдаемо по факту: посторонние пути вне вычисленного списка не тронуты).
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 11: Возврат постороннего файла — подтверждение неучастия в откате

**Действие:** Вернуть файл обратно: `mv /tmp/user-notes.txt.bak scratch/user-notes.txt`.
**Предусловия:** шаг 10 выполнен.
**Ожидаемый результат:** файл на месте с неизменным содержимым — откат шага 10 физически не мог
его затронуть (не входил ни в `$tmp_tracked`, ни в `$tmp_new` на момент вычисления).
**Фактический результат:** _(заполняется при исполнении)_

**Важное честное примечание (для протокола, не блокирует прохождение блока B).** Этот сценарий
демонстрирует РЕАЛЬНУЮ гарантию дизайна: список предъявляется ДО деструктива (шаг 7), и у
пользователя есть неразрушающий выход (шаг 8) для ручного изъятия постороннего файла ДО отклонения.
Дизайн НЕ гарантирует автоматическую защиту файла, который появился в дереве уже ПОСЛЕ старта
попытки и остаётся в дереве на момент вычисления списка (шаг 3 алгоритма `/sdx:proto`) — такой
файл будет классифицирован как «новый» и попадёт в список отката наравне с файлами прототипа. Это
согласуется с явно названным риском SPEC («Широта отката при отклонении»: «ошибка в его
вычислении… задевает файлы, не относящиеся к нему») и не является дефектом реализации — но стоит
явно квитировать это понимание на Verification, чтобы не читалось как необнаруженный пробел.

## Блок E: `/sdx:retrack standard` без `change_note.md` — отказ, затем успех

Покрывает критерий приёмки SPEC 6 (легализация обязана произвести реверс-инжиниринг требований
до того, как перейти в `Execution`). Дублирует на живой сессии то, что автоматические сценарии
56–58 `test-sdx-stage.sh` уже проверяют на синтетической фикстуре — здесь цель именно
интеграционная: подтвердить, что тот же guard одинаково срабатывает через реальные файлы реальной
сессии, а не только в изолированном тестовом окружении.

### Шаг 17: Прямой вызов легализации без артефакта реверс-инжиниринга

**Действие:** В сессии на `Prototype` (можно переиспользовать сессию из блока C, `attempt 3`)
вручную поменять `track` на `standard` в `session_state.json` (имитация шага 5 `retrack.md` без
предшествующего шага 2), затем выполнить:
```bash
bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<session_id>" "Execution"
```
**Предусловия:** `track` вручную выставлен в `standard`; ни `change_note.md`, ни пара
`SPEC.md`+`DESIGN.md` в каталоге сессии не существуют.
**Ожидаемый результат:** код возврата `1`; stderr явно называет непройденный этап `Change` (не
общий/неконкретный код ошибки); `stage` в `session_state.json` не изменился.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 18: Создание артефакта реверс-инжиниринга

**Действие:** Создать непустой `change_note.md` в каталоге сессии (краткое описание того, что
фактически сделал прототип — переформулированное как требование, не как пересказ диффа).
**Предусловия:** шаг 17 выполнен (отказ зафиксирован).
**Ожидаемый результат:** файл создан и непустой.
**Фактический результат:** _(заполняется при исполнении)_

### Шаг 19: Повторный вызов легализации — успех

**Действие:** Повторно выполнить:
```bash
bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<session_id>" "Execution"
```
**Предусловия:** шаг 18 выполнен, `track = standard`.
**Ожидаемый результат:** код возврата `0`; `stage` в `session_state.json` становится
`Execution`; в `session.log` появляется строка `[STAGE_CHANGE]`.
**Фактический результат:** _(заполняется при исполнении)_

## Блок G [дополнительный сценарий]: `/sdx:retrack` в `vibe` извне — отклоняется

Покрывает критерий приёмки SPEC 8. Ни автотестом, ни исходным минимальным списком DESIGN не
покрыт; сам DESIGN («Риски и границы поставки», п.1) прямо называет этот запрет прозаическим
(«скрипт `retrack` пропустит вход в `vibe`… держится текстом `commands/retrack.md`») — то есть
единственная существующая линия защиты — это как раз проверяемая здесь прозаическая инструкция.

### Шаг 21: Попытка входа в `vibe` из другого трека

**Действие:** В песочнице открыть обычную (не `vibe`) сессию, например
`/sdx:start standard "тестовая фича"`, затем выполнить `/sdx:retrack vibe`.
**Предусловия:** активная сессия трека `standard` (допустимо также `patch`/`full`/`doc`),
чистое рабочее дерево.
