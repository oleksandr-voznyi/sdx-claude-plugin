# Context Report: Режим экстремального прототипирования (vibe)

Сессия: `feat-vibe-track-20260720` (type: `feature`, track: `full`, stage: `Discovery`).
Источник требования: `docs/backlog/PROC-004-vibe-prototyping-mode.md`.

## 1. Суть требования и границы

### Суть (дословно из PROC-004)
Новый режим "в один промпт": реализация фичи/эпика без церемонии спеков и без
инкрементальных коммитов (быстрый откат грязного дерева `git checkout/clean`), но с
обязательной **легализацией** после утверждения прототипа — реверс-инжиниринг дельт в
SPEC/DESIGN, покрытие тестами, verify, и только затем мёрж. Рекомендация PROC-004 уже
называет конкретные элементы:
- тип сессии `proto`, трек `vibe`;
- стадия `Prototype` (stage-gate открыт, stop-gate off, дерево не коммитится —
  осознанное исключение из REQ-SESS-1);
- гейт утверждения: отклонено → `git checkout/clean`; принято → обязательный ретрек в
  легализацию (spec-after по образцу `/sdx:init --existing` → тесты `qa` → verify →
  Closeout);
- инвариант enforcement: сессия трека `vibe` не может пройти `/sdx:archive` без
  легализации;
- новый ADR (исключение из ADR-005/REQ-SESS-1 фиксируется явно).

### Что входит (по прочтению CLAUDE.md §3, sdx/protocol.md — трек как профиль флоу)
- Новый профиль флоу `vibe` в `SDX_STAGE_MATRIX` (машиночитаемый источник истины,
  `sdx/hooks/sdx-stage.sh:38-60`) и его человекочитаемая проекция в таблице «Профили
  флоу» (`sdx/protocol.md:51-58`).
- Новый тип сессии `proto` (по аналогии с `grooming`/`retro`/`postmortem`/`intake` из
  ADR-017), жёстко привязанный к `track: vibe` на `/sdx:start` — по прецеденту жёсткой
  привязки типа к треку.
- Механика «принять/отклонить» прототип и деструктивный откат.
- Механика легализации: ретрек `vibe → standard/full`, spec-after реверс-инжиниринг.
- Открытие stage-gate на новой стадии `Prototype` (аналог `Execution`/`Deployment`).
- Отключение/оставление в покое stop-gate на `Prototype` (по умолчанию transparent —
  см. §4.3 ниже, дополнительного кода, скорее всего, не требуется).
- Точка enforcement «нельзя закрыть `vibe`-сессию без легализации».
- Новый ADR (аналог ADR-017), фиксирующий исключение из ADR-005/REQ-SESS-1.
- Обновления команд (`start.md`, `next.md`, `retrack.md`, `archive.md`, возможно новая
  команда/подрежим для гейта принятия/отклонения), протокола, агентов (аналогично
  DEBT-017 для доп. трека — избежать повторения того же класса дефекта).

### Что НЕ входит (границы, требующие явного решения пользователя, не домысливаются)
- Возможность **параллельного** vibe-прототипирования нескольких вариантов одной фичи
  (PROC-004 говорит про одну реализацию "в один промпт", не про сравнение вариантов).
  Если понадобится — пересекается с открытым `PROC-010` (гонки параллельных субагентов
  за общие файлы сессии), но это отдельный вопрос, не обязательный для этой поставки.
- Обратный вход `{patch, standard, full, doc} → vibe` через `/sdx:retrack` — по
  прецеденту ADR-017 (доп. трек `doc` запрещает деэскалацию в себя прозаически) резонно
  ожидать симметричного запрета, но это архитектурная развилка, а не решённый факт (см.
  §4.6).
- Структурированное место для входящих материалов (BRD/макеты) — это зависимость
  `PROC-009`, вне скоупа `PROC-004` и этой сессии.
- Собственный fresh-eyes ревью НА этапе Prototype (до легализации) — вероятно
  отсутствует по духу режима («один промпт»), но нужно явное подтверждение
  архитектурного решения на Technical Design.

**Ключевой архитектурный вывод прецедента** (см. `docs/designs/session-types-doc-track.md:1-58`
и ADR-017 в `docs/DECISIONS.md:395-537`): новый трек был спроектирован как **изменение
нулевой мощности для процедурного кода** — описан данными (4 строки матрицы), а не новой
веткой логики, за счёт переиспользования уже существующих паттернов (`Discovery|-` как у
`standard`; `Verification|...|yes` как у всех треков; совпадение имени файла-артефакта
`change_note.md` между `Update`/`Change`/`Execution`, что даёт бесплатную эскалацию).
**Для `vibe` этот приём воспроизводим лишь частично** — см. §4: у `vibe`, в отличие от
`doc`, есть минимум одна процедурная точка (открытие stage-gate на новой стадии) и минимум
одна структурная особенность (трек без собственного `Closeout` в матрице), которых у `doc`
не было. Это следует явно проговорить как отличие precedents на Technical Design, а не
имитировать "нулевую мощность" там, где её нет.

## 4. Ключевые архитектурные развилки для Technical Design

### 4.1. Совместимость «дерево не коммитится» с REQ-SESS-1/ADR-005 и гардами
- `sdx/protocol.md:25-46` (модель "сессия = ветка", ADR-012) и ADR-005
  (`docs/DECISIONS.md:42-51`) описывают инкрементальные коммиты как **норму**, а не
  формальный инвариант, проверяемый скриптом напрямую по коммитам — единственная
  скриптовая проверка "дерево должно быть чистым" находится в `archive-verify.sh`
  (инвариант 1, `archive-verify.sh:16-20`) и работает ТОЛЬКО в момент Closeout, уже ПОСЛЕ
  мёржа шагов 1–5. `git status --porcelain` в `switch.md` — тоже не про историю коммитов, а
  про грязное дерево при переключении.
- Отсюда следует: технически исключение «не коммитить во время Prototype» **не требует
  правки ни одного enforcement-хука** — не коммитить и так разрешено в любой момент любой
  сессии (просто до сих пор было прозаической нормой коммитить часто). Единственное, что
  меняется, — это ПРОЗАИЧЕСКАЯ инструкция для `Prototype`: явно НЕ звать коммит после
  каждого шага (в отличие от `Execution`, где `developer`/оркестратор коммитит по ходу).
- **Развилка А**: считать это не нарушением REQ-SESS-1, а его локальным, точечным
  приостановлением НА ОДНОЙ стадии одного трека — тогда новый ADR фиксирует это явно как
  "исключение, а не отмену" (символично тому, как ADR-008 документирует прецедент разворота
  решения). К моменту Closeout (после легализации) REQ-SESS-1 действует снова без исключений
  — легализация обязана произвести первый коммит кода.
- **Развилка Б**: ввести отдельное явное поле (например, `session_state.json` не трогаем —
  REQ-COMPAT-1 — но добавить прозаическое правило "на `vibe|Prototype` коммит НЕ ДЕЛАЕТСЯ
  до явного `/sdx:retrack`") без нового ADR-исключения — слабее, не соответствует
  требованию PROC-004 "ADR фиксируется явно".
- **Рекомендация к рассмотрению**: развилка А ближе к букве PROC-004 ("ADR (исключение...
  фиксируется явно)").

### 4.2. Где и как enforced «vibe не проходит /sdx:archive без легализации»
Три варианта, с трейд-оффами:

- **Вариант А — структурный (данными, не кодом)**: НЕ включать `Closeout` в набор активных
  этапов `vibe` в `SDX_STAGE_MATRIX` вообще (единственная строка `vibe|Prototype|-|no`).
  Тогда `cmd_next` (`sdx-stage.sh:240-243`) на `stage=Prototype` для `track=vibe` всегда
  возвращает `OK no-op Prototype` (последний = единственный активный этап), НИКОГДА не
  продвигаясь к `Closeout`. Единственный способ покинуть `vibe` — `/sdx:retrack` на другой
  трек (что структурно ЕСТЬ легализация). Плюс: соответствует духу ADR-016/017 ("данные, не
  процедурная логика", `docs/designs/session-types-doc-track.md:11-13`) — ZERO новых веток
  в `stage_artifact_ok`/`matrix_*`. Минус (найденный, конкретный, не гипотетический): ломает
  скрытое допущение `commands/archive.md:16-31` — while-цикл предполагает, что у ЛЮБОГО
  трека `next`, повторяемый до упора, ВСЕГДА достигает `Closeout`; для `vibe` он зациклится
  на бесконечных no-op итерациях без ошибки. Требует обязательной правки `archive.md`
  (детектировать "no-op, но не Closeout" → явная остановка с инструкцией легализовать).

- **Вариант Б — явная проверка в `archive-verify.sh`**: добавить проверку `track != "vibe"`
  как новый FAIL до инвариантов 1/5/6 (или расширение существующего). Минус: `archive-verify.sh`
  работает уже ПОСЛЕ мёржа — то есть проверка сработала бы СЛИШКОМ ПОЗДНО (мёрж уже
  произошёл к моменту вызова скрипта по текущему контракту Closeout, `commands/archive.md:50-58`
  предшествует `archive-verify.sh` по шагам 6→7) — экономически хуже, чем блокировка ДО входа
  в Closeout. Также нарушает зафиксированную нумерацию инвариантов 1/5/6 (`ADR-015`,
  `docs/DECISIONS.md:296-297`: "Нумерация пунктов Closeout-чек-листа и инвариантов
  archive-verify не меняется расширениями бэклога" — хотя это про бэклог-расширения
  конкретно, прецедент показывает высокую цену изменения нумерации).

- **Вариант В — гейт входа в Closeout (прозаический, аналог `gate_mode`-дисклоуза)**: по
  образцу `sdx/protocol.md:271-273` ("Гейт входа в Closeout") добавить прозаическое условие
  "трек `vibe` — направь на `/sdx:retrack`" в `commands/archive.md` шаг 2, BEZ изменения
  скрипта. Минус: НЕ enforced — held only by prose (аналог принятого, но явно названного
  риска "деэскалация в doc запрещена прозаически", `ADR-017`,
  `docs/DECISIONS.md:499-501`), а PROC-004 явно требует "инвариант enforcement" — сильнее
  намекает на вариант А, а не В.

**Рекомендация к рассмотрению**: вариант А (структурный) как основной механизм +
обязательная защитная правка `archive.md` от зацикливания (это не альтернатива, а
неотъемлемая часть варианта А, иначе он превращается в баг, а не фичу). Вариант В — как
дополнительный defense-in-depth (сообщение до входа в изнурительный цикл), не вместо А.

### 4.3. Стадия `Prototype`: что открывать
- **stage-gate**: нужна правка `stage-gate.sh:41-44`, добавить `Prototype` в список
  разрешённых для кода стадий (`case "$stage" in Execution|Deployment|Prototype) exit 0 ;;`).
  Открытый вопрос: разрешить ЛЮБОЙ код (как на `Execution`) или дополнительно ограничить
  каким-то дополнительным признаком (например, запретить прод-относящиеся пути) — вероятно
  избыточно, `prod-guard.sh` уже отдельно покрывает Bash-деплой-команды независимо от stage.
- **stop-gate**: `stop-gate.sh:21-27` уже трактует любую стадию, не входящую в
  `Execution|Verification`, как transparent — `Prototype` автоматически подпадает под это,
  БЕЗ правки кода. Нужно явно задокументировать это в DESIGN как намеренное, не забытое
  (симметрично тому, как для `doc` в ADR-017 явно оговорено, что stop-gate НЕ становится
  "прозрачным по построению" — здесь наоборот, вопрос именно в "прозрачно и это осознанно").
- **stage-write-guard**: без изменений — уже трек-агностичен (сравнивает распарсенное
  значение `stage` до/после правки, не завязан на конкретные имена стадий), новая стадия
  `Prototype` не требует новой ветки логики.

### 4.4. Механика отклонения прототипа (`git checkout/clean`)
- Это первый прецедент **явного деструктивного шага внутри самого жизненного цикла
  сессии** (а не в Closeout, где деструктив уже есть — `git branch -d`,
  `archive-verify.sh:62`, но выполняется скриптом ПОСЛЕ трёх доказанных инвариантов). Здесь
  деструктив — до какого-либо доказательства, по прямому решению пользователя.
- Стоп-рубрика (`sdx/protocol.md:151-159`) требует остановки на "деструктивных/необратимых
  операциях" **безусловно**, независимо от `gate_mode`. Значит: гейт "принять/отклонить"
  ОБЯЗАН быть интерактивным (`AskUserQuestion`) даже если сессия стартовала с `--auto`.
  Это нужно явно прописать как ИСКЛЮЧЕНИЕ из общего правила "Prototype — не гейт" — сама
  реализация в Prototype идёт без подтверждений (единый промпт), но ВЫХОД из Prototype —
  всегда с подтверждением.
- Открытый вопрос по объёму `git clean`: `-fd` затронет ЛЮБЫЕ untracked-файлы в рабочем
  дереве, не только те, что относятся к прототипу — если пользователь параллельно
  экспериментировал с чем-то ещё вне SDX-потока, это будет случайно уничтожено. Нужно либо
  ограничить `clean` конкретными путями (список изменённых/добавленных файлов от начала
  Prototype, вычисляемый как разница отслеживаемого дерева на момент входа в стадию), либо
  явно предупредить пользователя о широте `git clean -fd` перед выполнением.
- Что происходит с САМОЙ веткой/сессией после отклонения — неопределено: удалить ветку
  сразу (сессия так и не произвела ценности) или оставить как "черновик", доступный для
  повторной попытки? Если удалить — это ЕЩЁ один Closeout-подобный путь, отдельный от
  штатного 8-пунктного чек-листа (`sdx/protocol.md:263-291`), потому что там нет кода для
  переноса и нет мёржа. См. вопрос пользователю в §5.

### 4.5. Механика легализации и evidence-based forward-skip guard в `retrack`
- Текущий guard (`sdx/protocol.md:127-137`, реализация `sdx-stage.sh:428-469`) требует, чтобы
  у КАЖДОГО этапа НОВОГО трека, предшествующего цели, гейт-артефакт **существовал на диске
  как файл** — существование определяется `[ -s "$path" ]`, а не наличием в git-индексе или
  коммите. Это значит: файл-доказательство МОЖЕТ быть некоммиченным на момент вызова
  `sdx-stage.sh retrack` — механизм retrack УЖЕ безразличен к git-статусу файла, только к
  его наличию на файловой системе. Отсюда следует нетривиальный, но корректный вывод: **spec-after
  реверс-инжиниринг (создание `change_note.md` или пары `SPEC.md`+`DESIGN.md` по итогам
  осмотра прототипа) МОЖЕТ произойти, пока код прототипа ещё не закоммичен** — сам гейт
  `retrack` это не блокирует.
- Развилка — **в какой трек легализовать по умолчанию**:
  - **standard**: легче, единственный гейт-артефакт цепочки предшественников —
    `change_note.md` ИЛИ пара `SPEC.md`+`DESIGN.md` (эквивалент для `Change`, `sdx-stage.sh:104-118`)
    — ретрек `vibe → standard|Execution` требует только их. Соответствует духу PROC-004
    ("тесты qa → verify → Closeout" без явного упоминания `PLAN.md`/Task Planning).
  - **full**: требует физического наличия ВСЕХ артефактов цепочки — `context_report.md`,
    `SPEC.md`, `DESIGN.md`, `PLAN.md` (если цель — `full|Execution` или дальше) — тяжелее,
    но соответствует масштабу, если прототип оказался крупной фичей.
  - **Рекомендация к рассмотрению**: сделать выбор прозаическим суждением оркестратора на
    гейте легализации (аналогично обычному триажу `patch/standard/full` на `/sdx:start`,
    `commands/start.md:19-24`), НЕ жёстко фиксировать `vibe → standard` — это симметрично
    тому, что PROC-004 не называет конкретный целевой трек, только "тесты qa → verify →
    Closeout" как обязательную последовательность.
- **Первый коммит кода**: ни `retrack.md`, ни `sdx-stage.sh` сегодня не производят коммитов
  кода — `retrack.md` шаг 4.3 коммитит только `session_state.json`/`session.log`
  (`commands/retrack.md:31`). Легализация вводит НОВУЮ обязанность — закоммитить сам код
  прототипа (это будет первый коммит содержательных изменений на ветке `sdx/<id>`,
  предшествующий обычному потоку `Execution`). Нужно решить: отдельный явно поименованный
  коммит ("sdx(<id>): commit vibe prototype code (legalized)") ДО коммита
  SPEC/DESIGN/change_note, вместе с ним, или после — влияет на читаемость истории и на то,
  что увидит `reviewer` в diff на последующей Verification (`commands/verify.md:21-25` берёт
  diff всей ветки против основной, так что порядок коммитов внутри ветки не критичен для
  этого конкретного шага, но важен для читаемости при `git log`).

### 4.6. Взаимодействие с `gate_mode: auto`
- Пока сессия находится в `vibe|Prototype`, прозаических гейтов "по ходу" почти нет (весь
  прототип — один промпт) — `gate_mode` практически не имеет поверхности применения ДО
  гейта принятия/отклонения.
- Гейт принятия/отклонения — стоп-рубрика, безусловно интерактивен (§4.4) — `gate_mode: auto`
  здесь НЕ смягчает (симметрично `/sdx:manual`, `sdx/protocol.md:157`, "шаги ручного
  тестирования... интерактивен по определению").
- Легализация — это переход `vibe → {standard, full}` — по аналогии с любым переходом ИЗ
  специального трека в кодовый (симметрично `doc → {patch,standard,full}`,
  `commands/retrack.md:22`) её стоит трактовать как "эскалацию" для целей сброса
  `gate_mode → interactive` — даже если пользователь запускал `/sdx:start proto ... --auto`.
  Нужно явное решение, распространить ли этот сброс и на `vibe` (вероятно да, по прецеденту).
- Отдельный вопрос: имеет ли смысл вообще разрешать `/sdx:start --auto` для `type: proto`? Раз
  единственный содержательный гейт (`Prototype`→accept/reject) и так безусловно интерактивен,
  `--auto` не даёт vibe-сессии никакого практического эффекта до легализации — стоит явно
  сообщить об этом пользователю при попытке, а не молча игнорировать флаг.

## 5. Открытые вопросы к пользователю

1. **Целевой трек легализации по умолчанию** — `standard` (лёгкий, `change_note.md`) или
   решение принимает оркестратор на месте по масштабу получившегося прототипа (§4.5)?
   PROC-004 не называет трек явно.
2. **Судьба ветки/сессии при отклонении прототипа** — удалить ветку `sdx/<id>` сразу
   (мини-Closeout без чек-листа, без кода для переноса) или оставить как черновик для
   повторной попытки внутри той же сессии (перезапуск Prototype)? Если повторная попытка
   разрешена — сколько попыток, есть ли предел (аналог loop-guard `stop-gate`)?
3. **Объём `git clean`** при отклонении — ограничивать ли откат только файлами, реально
   тронутыми на стадии `Prototype` (требует отслеживания "снимка" untracked-файлов на входе
   в стадию), или принять риск `git clean -fd` по всему дереву с явным предупреждением
   пользователю перед выполнением?
4. **Запрет входа `{patch,standard,full,doc} → vibe` через `/sdx:retrack`** — вводить ли
   симметричный ADR-017 прозаический запрет (по мысли "vibe создаёт uncommitted-дерево, что
   не имеет смысла для сессии, уже накопившей историю") — подтверди/опровергни это
   допущение.
5. **Есть ли fresh-eyes ревью ДО легализации** (осмотр прототипа независимым `reviewer` на
   момент принятия решения accept/reject) или только пользователь лично судит прототип, а
   `reviewer` появляется штатно позже, уже на Verification легализованного трека? PROC-004
   не упоминает `reviewer` на этапе принятия решения вовсе.
6. **Название типа сессии** — PROC-004 предлагает `proto`; уточни, не конфликтует ли это
   с уже занятыми типами (`grooming`/`retro`/`postmortem`/`intake`/`feature`/`bug`/`refactor`/
   `init`/`import`) — по факту не конфликтует, но стоит явно подтвердить нейминг перед
   Business Spec, чтобы не менять его задним числом (ID типа входит в `session_state.json` и
   во множество мест прозы).
7. **Разрешить ли параллельный vibe** (несколько версий прототипа одной задачи) — явно вне
   скоупа по умолчанию (см. §1 "Что НЕ входит"), но стоит явно подтвердить, что это
   сознательно отложено, а не забыто (пересекается с открытым `PROC-010`).
