### [WARN] [Корректность] `$(cat "$tmp_all")` в коммите и в пост-проверке ломается на путях с пробелами; документированная компенсация покрывает только `git add`

- Где: `commands/retrack.md:80,95,107`
- Суть: и `git add -- $(cat "$tmp_all")`, и `git status --porcelain -- $(cat "$tmp_all")` подвержены
  word-splitting. Прогон в временном репозитории: файл прототипа `my file.js` → `fatal: pathspec 'my'
  did not match any files`, ничего не застейджено, `git commit` падает, пост-проверка корректно
  отбивает легализацию (**fail-closed, потери данных нет**). Однако `git status --porcelain -- my
  file.js` возвращает пустой вывод с кодом `0` — второй симптом на таких путях слеп; поставку спасает
  только `commit_rc`. Приписка «(при путях с пробелами — построчный `xargs -d'\n' git add --`)»
  предлагает компенсацию **только для `git add`**: применив её буквально, исполнитель получит
  успешный коммит и одновременно ослеплённую пост-проверку.
- Ожидалось: оба места, зависящие от списка, обрабатывают путь одинаково безопасно (например,
  `xargs -d'\n'` и там, и там).
- Фактически: асимметричная подсказка в прозе; риск узкий (fail-closed в поставленном виде), поэтому
  WARN, а не FAIL. Циклы отката в `proto.md` (`while IFS= read -r f`) от этой проблемы свободны —
  проверено.

### [WARN] [Корректность] Tracked-файлы с не-ASCII именами не откатываются: `git diff --name-only` печатает их в C-кавычках

- Где: `commands/proto.md:79,90-97` и все четыре копии фрагмента (`DESIGN.md:402` — канон)
- Суть: ветка untracked безопасна (`git ls-files … -z | tr '\0' '\n'` кавычек не ставит), а вот
  tracked-дельта считается `git diff --name-only "$base_rev"` **без `-z`**, поэтому при
  `core.quotePath=true` (значение по умолчанию) путь `файл.txt` печатается как
  `"\321\204\320\260\320\271\320\273.txt"`. Проверено исполнением: `git cat-file -e "$base_rev:$f"`
  → FAIL, следовательно цикл (а) уходит в ветку `git rm -f -- "$f"`, которая тоже отбивается
  (`pathspec … did not match`); коды возврата в цикле не проверяются, файл остаётся изменённым.
- Ожидалось: точечный откат восстанавливает все файлы вычисленного списка либо честно отказывает.
- Фактически: откат молча неполон, но **fail-loud на следующем шаге** — проверка (г) `leftover` ловит
  остаток и запрещает объявлять откат успешным (причина (б) в новой диагностике `proto.md:115`
  описывает ровно этот класс). Файлы вне списка не затрагиваются, инвариант безопасности не нарушен.
  `DESIGN.md` предусматривает гард на путь с переводом строки, но не на квотирование; лечится
  добавлением `-c core.quotePath=false` (или `-z`) в `git diff --name-only`. Находка унаследованная,
  не привнесена фиксами третьего прогона.
