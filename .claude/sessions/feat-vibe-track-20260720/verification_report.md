## Находки

### [WARN] [Корректность] `xargs -d '\n'` — GNU-расширение; на не-GNU `xargs` (macOS/BSD) блок легализации отказывает «наружу», а пост-проверка REQ-VIBE-7 слепнет

- Где: `commands/retrack.md:86`, `commands/retrack.md:106` (и парный канон `DESIGN.md:563,588`)
- Суть: опция `-d` у `xargs` отсутствует в BSD/macOS-реализации. На такой платформе цепочка
  разворачивается так: `xargs -d '\n' git add --` завершается ошибкой «illegal option», ничего не
  стейджится; `git commit` падает «nothing to commit» (`commit_rc≠0`, но он по решению T40 больше НЕ
  является критерием); `leftover_code="$(xargs -d '\n' git status --porcelain -- < "$tmp_all")"` тоже
  падает и даёт **пустую строку** — единственный критерий пост-проверки выполняется, и легализация
  продолжается со сменой `track`/`stage` при фактически незакоммиченном коде прототипа.
- Ожидалось: механизм, удерживающий REQ-VIBE-7 (по `DESIGN.md` «Решение 3» — единственный), при отказе
  своего инструмента ведёт себя fail-closed, либо требование к платформе/инструменту зафиксировано в
  артефактах. Портируемая альтернатива, дающая заодно и корректное квотирование, — `git add
  --pathspec-from-file=<file>` / `git status --porcelain --pathspec-from-file=<file>`.
- Фактически: ни `DESIGN.md`, ни `commands/retrack.md`, ни ADR-018 не называют GNU-зависимость; в
  остальном enforcement-слое (`sdx/hooks/*.sh`) GNU-специфичных опций нет — это первая такая
  зависимость в критическом пути. На GNU/Linux (среда прогона этой сессии) поведение корректно,
  подтверждено исполнением, поэтому WARN, а не FAIL; решение о поддержке платформ — за пользователем.

### [WARN] [Корректность] Остаточный класс квотирования: `-c core.quotePath=false` не снимает C-кавычки с путей, содержащих `"` или `\`; в связке с единственным критерием пост-проверки это даёт fail-open

- Где: `commands/proto.md:83`, `commands/retrack.md:39,68`, канон `DESIGN.md:404`; пост-проверка
  `commands/retrack.md:105-116`
- Суть: `core.quotePath` управляет только экранированием байтов > 0x7F. Пути с двойной кавычкой или
  бэкслэшем git печатает в C-кавычках всегда. Проверено исполнением:
  `git -c core.quotePath=false diff --name-only <rev>` для файлов `we"ird.txt` и `back\slash.txt` даёт
  `"we\"ird.txt"` и `"back\\slash.txt"`, а `git cat-file -e "<rev>:<такой путь>"` → MISSING.
  Последствия асимметричны:
  - в `/sdx:proto` (откат) — **fail-loud**: `git checkout`/`git rm` по такому пути отбиваются, но
    проверка (г) `leftover` ловит остаток и запрещает объявлять откат успешным;
  - в `/sdx:retrack` (легализация) — **fail-open**: прогон во временном репозитории, где единственным
    файлом списка был изменённый `we"ird.txt`: `git add` → `fatal: pathspec ... did not match`,
    `commit_rc=1` (игнорируется), `leftover_code=[]` при реальном `git status` = ` M "we\"ird.txt"` →
    пост-проверка пропускает, `track` меняется, код не закоммичен.
- Ожидалось: единственный механизм удержания REQ-VIBE-7 не должен терять детектирующую способность на
  тех же путях, на которых теряет способность коммит. Полное решение того же класса — `-z` (или
  `--pathspec-from-file` с NUL), а не `core.quotePath`.
- Фактически: сужение критерия до одного (`leftover_code`, T40) само по себе корректно устранило ложный
  обрыв на ручном коммите, но убрало второй симптом (`commit_rc`), который в этом узком сценарии ранее
  давал fail-closed. Сценарий экзотичен (имена файлов с `"`/`\`), потери данных нет, поэтому WARN;
  граница нигде в артефактах не названа — гард на перевод строки описан, гард на квотирование — нет.
