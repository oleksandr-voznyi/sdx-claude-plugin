# Change Note: закрытие DEBT-021 и DEBT-022

Сессия `fw-reconcile-debt-20260720`, трек `standard`, тип `refactor`. Бизнес- и технические
решения зафиксированы слитно (ADR-003).

## Проблема

Обе записи бэклога порождены первым реальным прогоном `/sdx:reconcile --scan-only`
(отчёт — `docs/sdx-reconcile-report.md`) и касаются одного механизма, поэтому закрываются
одной сессией.

- **DEBT-021.** В репозитории нет ни одного маркера `<!-- SDX-MIGRATED: → <ID> (<дата>) -->`:
  миграция находок аудита в `docs/backlog/` выполнялась вручную (2026-07-19, сессия
  `fw-backlog-20260719`) до появления команды. Механический пропуск уже перенесённого
  (шаг 4 команды) в этом репозитории не работает — все 23 кандидата отсеиваются
  исключительно прозаической дедупликацией по смыслу (шаг 5). Дедупликация сработала верно,
  но она суждение модели, а не проверка факта.
- **DEBT-022.** Список источников шага 2 не называет `.sdx/bundles/` ни как источник, ни как
  исключение — поведение не определено. При этом класс «gitignored транспортный артефакт с
  уникальным содержимым» реален: в этом репозитории бандл долгое время был единственной
  полной спецификацией отложенных требований Фаз 2–4 (ровно риск находки `C1`/`DEBT-008`).

## Решение

### 1. DEBT-022 — бандлы становятся сканируемым источником с именованным исключением

`.sdx/bundles/**` вносится в список источников шага 2 `commands/reconcile.md` **с двумя
явными оговорками**:

- **Что является кандидатом.** Бандл описывает функциональность для переноса, а не задачи
  проекта. Кандидатами являются ТОЛЬКО незакрытые/отложенные требования и открытые вопросы
  из бандла (типично — `roadmap`-разделы, `assumptions.md`, отложенные `REQ-*`), а не его
  содержимое как таковое: срезы `SPEC`/`DESIGN`, код и тесты кандидатами не являются
  никогда — для них существует штатный путь `/sdx:import`.
- **Маркеры в бандлы НЕ пишутся — третье поведение, отличное и от «сканируется», и от
  «исключено».** Бандл переносим по своей природе: он может быть скопирован в другой
  проект, где маркер со ссылкой на `DEBT-0NN` ЭТОГО проекта не значит ничего и вводит в
  заблуждение. Идемпотентность повторного скана для бандлов обеспечивается дедупликацией
  (шаг 5), а не маркером; отчёт каждого прогона называет просканированные бандлы явно, чтобы
  отсутствие маркеров не выглядело пропуском.

Выбор «сканировать» против «исключить» обоснован тем, что `/sdx:import` покрывает случай
«бандл принят в работу», но не случай «бандл лежит и забыт» — а риск потери порождает
именно второй.

### 2. DEBT-021 — разовая расстановка маркеров в этом репозитории

В `docs/audit-2026-07-01-recommendations.md` рядом с каждой из 23 находок (`A1`–`A6`,
`B1`–`B4`, `C1`–`C7`, `D1`–`D2`, `E1`–`E4`) ставится маркер вида
`<!-- SDX-MIGRATED: → BUG-001 (2026-07-19) -->`.

- **Дата в маркере — дата фактической миграции** (2026-07-19, сессия `fw-backlog-20260719`),
  а не дата этой сессии: маркер документирует, когда фрагмент был перенесён, а не когда
  кто-то это заметил.
- **Целевые ID берутся из уже существующего маппинга** — поля `source` записей
  `docs/backlog/`, сверенного прогоном `--scan-only` (полный, без пропусков).
- **Исходный текст находок не меняется** — ни статусные строки, ни формулировки: файл
  остаётся историческим снапшотом ревизии, каким его объявляет собственный баннер.
- **Бандл `.sdx/bundles/upgrade_2026-06-27/` маркерами НЕ помечается** — по правилу из п.1.

### 3. Сознательные не-цели

- **Маркер уровня файла** (единственная пометка «файл мигрирован целиком» вместо N
  пофрагментных) не вводится: это новая синтаксическая сущность в контракте команды ради
  экономии 22 строк в одном историческом файле. Если случай повторится на практике —
  повод для отдельной записи бэклога, а не для расширения контракта сейчас.
- **Статусные строки аудита не синхронизируются** с бэклогом (часть из них разошлась:
  `A3`/`A6`/`E1` помечены открытыми при закрытых `DEBT-001`/`DEBT-004`/`PROC-002`). Файл
  сам объявляет, что актуальные статусы ведутся только в `docs/backlog/` — расхождение
  предусмотрено его конвенцией, а не является дрейфом.

## Границы решения

- Маркеры остаются конвенцией в тексте, не enforced хуком (унаследованная граница
  `DEBT-021` не устраняется этой сессией — устраняется лишь её отсутствие в данных).
- Идемпотентность для бандлов держится на дедупликации по смыслу, то есть на суждении
  модели, — сознательный трейд-офф в пользу переносимости бандлов.
- Скан бандлов остаётся прозаическим: полнота извлечения отложенных требований из
  45-килобайтного бандла не гарантируется, как и для любого другого источника.

## Дополнение (правки по итогам верификации)

Ревью дало PASS без блокирующих находок; три WARN закрыты в том же круге (четвёртый —
`status: open` у самих `DEBT-021`/`DEBT-022` — не дефект: закрытие записей выполняется
пунктом 4 Closeout-чек-листа):

1. **[Когерентность] Инвариант «перенос ⇒ пометка» остался абсолютным в двух местах.**
   Шаг 7 `commands/reconcile.md` объявлял шаги 8–9 неделимыми с обоснованием «иначе
   повторный скан завёл бы дубль» — для кандидата из бандла ложны и утверждение, и
   обоснование, а исключение жило только в шаге 9 без ссылки отсюда. То же в
   `sdx/protocol.md` («перенесённые источники помечаются на месте»). Оба места дополнены
   явной отсылкой к именованному исключению. Находка ценна тем, что показывает цену
   введения третьего поведения: правило пришлось обмягчить везде, где оно формулировалось
   абсолютно, а не только там, где вводилось исключение.
2. **[Когерентность] Односторонняя ссылка про бандлы в отчёте.** Обязанность отчёта
   называть просканированные бандлы была записана в `REQ-RECON-7`, но не в `REQ-RECON-10`
   (собственно требовании к отчёту). Ссылка сделана двусторонней.
3. **[Когерентность] Шапка постоянной спеки** утверждала «версия плагина 1.5.0» при
   содержании, отражающем 1.5.1. Переписана так, чтобы фиксировать историю версий
   механизма, а не версию на момент написания — спека постоянная и переживёт следующие
   правки.
