# Business Specification: Типы сессий grooming / retro / postmortem и трек doc

## Контекст и Цели

Работа над самим бэклогом (пересмотр приоритетов, актуализация статусов), периодическая
ретроспектива завершённых сессий и разбор инцидентов (постмортем) сейчас не имеют своего
типа сессии и трека в SDX — делаются вне процесса, «по наитию», без структурированного входа,
выхода и следа в бэклоге (`docs/backlog/PROC-002`).

Цель — ввести три новых типа сессии (`grooming`, `retro`, `postmortem`), которые проходят по
общему лёгкому треку `doc` (Discovery → Update → Verification (лёгкая) → Closeout, без стадий
Execution/кода) и производят прослеживаемый результат: изменения `docs/backlog/`, а для
`retro`/`postmortem` — дополнительно постоянный документ разбора. Это распространяет
дисциплину SDX (гейты, трассируемость, Closeout-чек-лист) на процессную/рефлексивную работу,
которая раньше её не проходила.

Решения о треке `doc`, его этапах и гейт-артефакте `Update` = `change_note.md` уже приняты
пользователем на Discovery-гейте этой сессии и здесь не переоткрываются — эта спецификация
формализует их в бизнес-терминах и раскрывает детали входа/выхода каждого типа.

## Требования

### Назначение и вход трёх типов сессий

- [ADDED] **REQ-DOC-1 (grooming — назначение и вход).** Тип `grooming` — сессия пересмотра
  ТЕКУЩЕГО состояния бэклога: актуализация статусов/приоритетов/волн (`wave`) существующих
  записей `docs/backlog/`, закрытие неактуальных, устранение дублей, перегруппировка. Вход —
  срез текущего открытого/deferred бэклога (`docs/backlog/README.md` + сами записи); сессия НЕ
  привязана к конкретному периоду времени или конкретной завершённой сессии.

- [ADDED] **REQ-DOC-2 (retro — назначение и вход).** Тип `retro` — сессия разбора набора уже
  ЗАВЕРШЁННЫХ SDX-сессий за выбранный период (или по явному списку `session_id`, заданному
  пользователем на старте): цель — выявить системные наблюдения (повторяющиеся паттерны,
  что сработало хорошо/плохо в процессе) и превратить их в новые записи бэклога. Вход —
  `docs/history/sessions-log.md` (срез за период/по списку) плюс записи `docs/backlog/`,
  которые эти сессии породили или закрыли.

- [ADDED] **REQ-DOC-3 (postmortem — назначение и вход).** Тип `postmortem` — сессия разбора
  ОДНОГО конкретного инцидента (не обязательно привязанного к SDX-сессии — может быть
  прод-инцидент, процессный сбой, найденный дефект с серьёзными последствиями): цель —
  зафиксировать хронологию, корневую причину и план действий. Вход — описание инцидента,
  собираемое интерактивно от пользователя на этапе Discovery, плюс релевантные записи
  `docs/backlog/`/`docs/history/sessions-log.md`, если применимо.

### Выход

- [ADDED] **REQ-DOC-4 (обязательный минимум — записи бэклога).** Каждая doc-сессия
  (независимо от типа) обязана произвести минимум одно наблюдаемое изменение
  `docs/backlog/` — новую запись либо правку статуса/приоритета/волны существующей.
  Отсутствие такого изменения на момент Closeout означает, что сессия не достигла цели.

- [ADDED] **REQ-DOC-5 (дополнительный постоянный документ для retro/postmortem).** Для типов
  `retro` и `postmortem`, помимо записей бэклога, обязателен постоянный документ разбора,
  переживающий закрытие сессии: он ведётся в каталоге сессии как `retro.md`/`postmortem.md` по
  единому шаблону и на Closeout переносится в постоянное хранилище — `docs/history/retro/` и
  `docs/history/postmortem/` соответственно (по аналогии с уже существующим
  `docs/history/plans/` для `PLAN.md`), с именем файла по `session_id`. Для типа `grooming`
  отдельного постоянного документа не создаётся — записи бэклога являются единственным и
  достаточным выходом.

### Поведение трека `doc`

- [ADDED] **REQ-DOC-6 (активные этапы, единые для всех трёх типов).** Все три типа сессии
  проходят по единому треку `doc`: Discovery → Update → Verification (лёгкая, обязательная) →
  Closeout. Состав и порядок активных этапов определяется треком, а не типом сессии — сам тип
  влияет только на содержание (что собирается на Discovery и что производится на Update), не
  на маршрут.

- [ADDED] **REQ-DOC-7 (жёсткая привязка типа к треку).** При `/sdx:start` со значением типа
  `grooming`, `retro` или `postmortem` трек `doc` выбирается безусловно, без диалога об
  альтернативном треке — в отличие от триажа `patch`/`standard`/`full`, здесь связь тип↔трек
  жёсткая 1:1 по построению задачи.

- [ADDED] **REQ-DOC-8 (Discovery на doc-треке).** Discovery выполняется инлайн, без вызова
  субагента `architect`, и состоит в сборе входного среза, специфичного для типа сессии
  (REQ-DOC-1/2/3): для `grooming` — обзор текущего бэклога; для `retro` — выборка из
  `sessions-log.md` за период/по списку; для `postmortem` — интервью пользователя об
  инциденте. Собственного гейтуемого файла-артефакта у Discovery на этом треке нет.

- [ADDED] **REQ-DOC-9 (Update на doc-треке — единый гейт-артефакт).** `change_note.md` —
  единственный объективно проверяемый гейт-артефакт этапа Update для всех трёх типов. Правки
  `docs/backlog/` выполняются НА ЭТОМ этапе (не откладываются до Closeout); для `retro`/
  `postmortem` на этом же этапе параллельно заполняется содержательный, но не гейтуемый
  скриптом документ (`retro.md`/`postmortem.md`). `change_note.md` фиксирует краткое резюме
  сессии и список ID затронутых записей бэклога (и ссылку на `retro.md`/`postmortem.md`, если
  применимо).

- [ADDED] **REQ-DOC-10 (обязательная лёгкая Verification).** doc-трек включает обязательный
  лёгкий этап Verification: fresh-eyes прогон `reviewer` (через делегирование Task) против
  `change_note.md` — оси корректности и когерентности (не противоречит ли новая/изменённая
  запись бэклога уже существующим записям и решениям, зафиксированным в постоянных
  документах). Ось полноты — в границах `change_note.md`. Субагент `qa` не вызывается,
  автоматизированного тестового прогона нет — на этом треке нет кода.

### Закрытие сессии (Closeout)

- [ADDED] **REQ-DOC-11 (гейт входа в Closeout).** Как и на треке `patch`, вход в Closeout для
  `doc`-сессии требует существования `verification_report.md` без находок `FAIL`; при
  отсутствии — стоп и направление на `/sdx:verify`.

- [ADDED] **REQ-DOC-12 (адаптация чек-листа Closeout).** Стандартный чек-лист Closeout
  применяется к `doc`-сессии целиком, со следующими типовыми отметками:
  - п.2 (перенос дельт в `docs/specs/`/`docs/designs/`) — как правило `(n/a)`: doc-сессия не
    производит `SPEC.md`/`DESIGN.md`. Исключение: если сессия по итогам разбора приходит к
    выводу об изменении самого процесса/протокола SDX (правки `sdx/protocol.md`, `CLAUDE.md`,
    новый ADR) — такие правки переносятся по п.2 штатно, и пункт не `(n/a)`.
  - п.3 (архив `PLAN.md`) — всегда `(n/a)`: doc-трек не включает Task Planning.
  - п.4 (глобальный лог + бэклог) — основная точка выхода doc-сессии: актуализация статусов
    закрытых/изменённых записей бэклога и обновление индекса (REQ-DOC-4), а для `retro`/
    `postmortem` — перенос постоянного документа разбора в `docs/history/retro/` или
    `docs/history/postmortem/` (REQ-DOC-5) как часть обновления `docs/history/`.
  - п.1, п.5–п.8 (коммиты, удаление каталога сессии до мёржа, мёрж, enforcement-проверка) —
    применяются без изменений, трек-агностичны.

- [ADDED] **REQ-DOC-13 (расширение допустимых значений типа сессии).** Поле `type` в
  `session_state.json` расширяется значениями `grooming`, `retro`, `postmortem`. Поле остаётся
  описательным (не валидируется enforcement-скриптами), но становится значимым для
  прозаической логики `/sdx:start` (жёсткая привязка к треку `doc`, REQ-DOC-7) и для выбора
  шаблона на этапе Update (REQ-DOC-9).

## Критерии приемки

- [ ] `/sdx:start grooming <goal>` (аналогично `retro`, `postmortem`) создаёт сессию с
      `track: "doc"`, стартовый `stage: "Discovery"`, без запроса подтверждения выбора трека.
- [ ] На старте сессии типа `retro`/`postmortem` в каталог сессии копируется соответствующий
      шаблон (`retro.md`/`postmortem.md`); для `grooming` дополнительный шаблон-артефакт не
      создаётся.
- [ ] `/sdx:next` на `doc`-треке проводит переходы строго в порядке Discovery → Update →
      Verification → Closeout и не предлагает переход на Business Spec, Technical Design, Task
      Planning, Documentation или Deployment.
- [ ] Переход из Update в Verification блокируется механизмом гейта при отсутствии непустого
      `change_note.md` в каталоге сессии.
- [ ] `/sdx:verify` на `doc`-сессии создаёт `verification_report.md` по результатам fresh-eyes
      прогона `reviewer` против `change_note.md`; субагент `qa` не вызывается; отчёт содержит
      находки по осям корректности и когерентности.
- [ ] `/sdx:archive` для `doc`-сессии не начинает чек-лист Closeout без существующего
      `verification_report.md` без находок `FAIL`.
- [ ] На момент прохождения Closeout `docs/backlog/` содержит минимум одно изменение (новая
      запись либо изменённый `status`/`priority`/`wave`), помеченное `session: <session_id>`
      этой сессии.
- [ ] Для сессий типа `retro`/`postmortem` на момент завершения Closeout существует файл
      `docs/history/retro/<session_id>.md` или `docs/history/postmortem/<session_id>.md`
      соответственно, а исходный `retro.md`/`postmortem.md` в каталоге сессии удалён вместе с
      остальными файлами сессии по п.5 чек-листа.
- [ ] Печать чек-листа Closeout для типовой `doc`-сессии показывает п.2 и п.3 отмеченными
      `(n/a)`, кроме случая, когда сессия правит сам протокол/`CLAUDE.md` — тогда п.2 содержит
      реальные изменения и не `(n/a)`.
- [ ] `session_state.json.type` новых сессий принимает значения `grooming`/`retro`/`postmortem`
      наравне с уже существующими `feature`/`bug`/`refactor`/`init`/`import`, и для всех трёх
      новых значений `track` в момент старта равен `doc` без исключений.

## Риски и Ограничения

- **Единый `change_note.md` для трёх разных по смыслу типов.** Один и тот же файл служит
  гейт-артефактом и для «резюме груминга», и для «резюме ретро/постмортема со ссылкой на
  отдельный содержательный документ» — при недисциплинированном заполнении он может стать
  формальной отпиской вместо содержательного резюме. Компенсация — обязательный fresh-eyes
  прогон Verification (REQ-DOC-10), который проверяет когерентность содержания, а не только
  факт существования файла.
- **Отсутствие Execution/кода на этом треке** означает, что единственная защита от «пустой»
  doc-сессии — прозаический критерий REQ-DOC-4 (минимум одно изменение бэклога) и Verification;
  ни один enforcement-хук это не проверяет механически.
- **Перенос постоянного документа разбора** (`docs/history/retro/`, `docs/history/postmortem/`)
  — новый паттерн хранения (ранее в `docs/history/` был только `sessions-log.md` и
  `plans/`); решение зафиксировано этой спецификацией по аналогии с `docs/history/plans/`, но
  подлежит подтверждению на этапе Technical Design как техническое решение по неймингу и месту.
- **Retro как источник данных зависит от актуальности `docs/history/sessions-log.md`.** Если
  журнал ведётся с пропусками, `retro`-сессия не сможет опереться на полный список завершённых
  сессий за период — это ограничение входа, не устраняемое данной задачей.

## Открытые вопросы

Явных неоднозначностей, требующих решения пользователя, на уровне бизнес-требований не
осталось — ключевые развилки (имя трека `doc`, состав активных этапов, единый гейт-артефакт
`change_note.md`, обязательность лёгкой Verification, расширение `/sdx:start` вместо новых
команд) закрыты решениями пользователя на Discovery-гейте. Единственное авторское решение,
принятое в этой спецификации без отдельного подтверждения пользователя, — точное постоянное
расположение документов `retro`/`postmortem` (`docs/history/retro/`, `docs/history/postmortem/`,
REQ-DOC-5) по аналогии с `docs/history/plans/`; при желании альтернативной раскладки (например,
плоские файлы `docs/history/retro-<id>.md` без подкаталогов) это стоит скорректировать на этапе
Technical Design.
