# Context Report: fw-session-types-20260720

## Задача

Реализовать `docs/backlog/PROC-002-grooming-retro-postmortem-session-types.md`: три новых
типа сессий (`grooming` / `retro` / `postmortem`) на **общем лёгком doc-only треке**
(«Discovery-lite → Update → Closeout», без Execution-кода), с шаблонами артефактов
(`retro.md`, `postmortem.md`); выход всех трёх — записи `docs/backlog/`. Требуется ADR.
Зависимость PROC-002 → E2 закрыта (`PROC-003`, ADR-015, закрыта сессией
`fw-backlog-20260719`) — блокеров нет.

## Текущее состояние (что уже есть)

### Треки и матрица
`sdx/hooks/sdx-stage.sh:38-56` — единственный машиночитаемый источник истины
`SDX_STAGE_MATRIX` (heredoc, формат `track|stage|artifact|fail_marker`), сейчас три трека:
`full` (9 этапов), `standard` (5 этапов: Discovery→Change→Execution→Verification→Closeout),
`patch` (3 этапа: Execution→Verification→Closeout). Артефакт `-` = гейт не проверяется
скриптом (прозаическое суждение оркестратора); `change_note.md`/`SPEC.md`+`DESIGN.md` —
взаимозаменяемая пара для объединённого этапа `Change` (`stage_artifact_ok`,
`sdx-stage.sh:100-114`).

Дублирующая копия той же матрицы (для читаемости) — `docs/designs/stage-transition-guard.md:77-103`
(это DESIGN исходной задачи DEBT-001/011, а не сам источник истины) и человекочитаемая
проекция — таблица «Профили флоу» в `sdx/protocol.md:51-55`.

`type` в `session_state.json` (`sdx/protocol.md:9`, `commands/start.md:2,8`) — **чисто
описательное поле**: НЕ валидируется ни одним скриптом (грепнул `sdx-stage.sh`,
`stage-gate.sh`, `archive-verify.sh`, `resolve-session.sh` — нигде `.type` не читается
кроме `init`, который просто кладёт значение в JSON без проверки допустимых значений).
Это значит: добавление `grooming`/`retro`/`postmortem` как значений `type` — не требует
правок кода/схемы, только документации и логики `/sdx:start` (какой трек выбрать для
такого `type`).

### Гейты и enforcement
- `stage-gate.sh` (PreToolUse Write/Edit/MultiEdit) всегда открывает `docs/**`, `.claude/**`,
  `*.md` независимо от стадии (`stage-gate.sh:59-62`) — doc-only сессия ничего не пишет вне
  этих путей, поэтому **никаких правок stage-gate не требуется**, весь новый трек будет
  прозрачен для этого хука на любом этапе.
- `archive-verify.sh` проверяет только инварианты 1/5/6 (чистое дерево, ветка слита, каталог
  сессии не tracked) — они трек-агностичны, **правок не требует**.
- `stage-write-guard` защищает поле `stage` от прямой правки — трек-агностичен, не требует
  правок.
- Sanity-тест `sdx/hooks/test-sdx-stage.sh:792-814` (сценарий 34) сверяет упорядоченные
  цепочки этапов из таблицы `protocol.md` и матрицы **только для `for track in full standard
  patch`** (жёстко зашитый список, `test-sdx-stage.sh:796`) — добавление нового трека в
  матрицу этот тест НЕ сломает (новый трек просто не проверяется), но и не даст
  автоматической защиты от рассинхронизации для него. Рекомендация — расширить цикл до
  `full standard patch <новый-трек>` заодно с добавлением строки в таблицу протокола, чтобы
  новый трек получил ту же гарантию симметрии.

## Открытые развилки и рекомендации

### 1. Название и состав нового трека
Рекомендация: имя трека — **`doc`** (однословно, по конвенции patch/standard/full;
альтернативы `note`/`review` хуже — `review` пересекается по смыслу с агентом `reviewer`).
Активные этапы — ровно как задано текстом бэклога, без добавлений: `Discovery → Update →
Closeout`. Новые строки матрицы:
```
doc|Discovery|-|no
doc|Update|change_note.md|no
doc|Closeout|-|no
```
Discovery — инлайн, без вызова субагента `architect` (по аналогии со `standard`,
`protocol.md:61`) — эти сессии по духу обзорные/рефлексивные, а не архитектурные; вызывать
architect на «груминг бэклога» избыточно.

Ключевая развилка внутри — **имя и природа гейт-артефакта этапа `Update`**. Матрица
привязана к треку, а не к `type`, поэтому единого файла на все три типа сессий требует сам
механизм (`sdx-stage.sh` не знает о `type`). Варианты:
- (a) Переиспользовать `change_note.md` как универсальный гейт-артефакт для ВСЕХ трёх типов
  (grooming тоже, у которого нет собственного шаблона) — минимальные изменения кода/логики,
  консистентно с уже существующей ролью `change_note.md` как «аккумулятора дельт лёгких
  треков» (ADR-003). Для retro/postmortem `change_note.md` может быть кратким резюме +
  ссылкой на параллельно создаваемые `retro.md`/`postmortem.md` (эти два — содержательные, но
  НЕ гейтуемые скриптом файлы, только «прозаические» ожидаемые артефакты, аналогично тому как
  `PLAN.md` не гейтует Execution).
- (b) Заводить три отдельных трека (`grooming`, `retro`, `postmortem`) с разными гейт-
  артефактами (`change_note.md`, `retro.md`, `postmortem.md` соответственно) — точнее
  семантически, но противоречит явной формулировке бэклога «общий... трек» и утраивает
  матрицу/тесты/документацию без выигрыша в безопасности (гейт всё равно только
  существование+непустота).

**Рекомендация: (a)** — она буквально соответствует тексту PROC-002 («общий... трек»), не
плодит сущности в матрице и полностью переиспользует уже провалидированный контракт
`change_note.md`. `retro.md`/`postmortem.md` — шаблонные, содержательные, но НЕ
объективно-гейтуемые файлы (как `PLAN.md`, `MANUAL_TEST.md` и т.п. уже в системе).

### 2. Closeout-чек-лист на doc-track сессии
- П.1 (коммиты), п.5 (`git rm` сессии до мёржа), п.6-8 (мёрж + `archive-verify.sh`) —
  применяются без изменений, трек-агностичны.
- П.2 («дельты в `docs/specs/`/`docs/designs/`») — **`(n/a)`** в общем случае: doc-only
  сессия не производит `SPEC.md`/`DESIGN.md`. Исключение — если postmortem-сессия приходит к
  выводу, что нужно поменять сам процесс/протокол (как эта самая сессия PROC-002!), тогда её
  выход — правки `sdx/protocol.md`/`CLAUDE.md`/ADR, которые естественно ложатся именно в п.2
  по аналогии с любой другой процессной правкой; тогда п.2 — не n/a. Это нормально: чек-лист
  уже поддерживает вариативность через `(n/a)`-пометку, ничего чинить не нужно.
- П.3 (`PLAN.md`) — всегда **`(n/a)`**: doc-track никогда не доходит до Task Planning
  (стадии нет в матрице).
- П.4 (глобальный лог + бэклог) — **это и есть основной выход** doc-track сессий: для
  grooming — массовые правки статусов/приоритетов/wave существующих записей; для
  retro/postmortem — новые `DEBT-`/`IDEA-`/`PROC-` записи. По духу эта работа уже сделана на
  этапе `Update` (через `/sdx:backlog`), но пункт 4 остаётся явной точкой финальной сверки
  (индекс README, `session` guid проставлен и т.д.) — не переносить её целиком в Update,
  сохранить как контрольную точку.
- `archive-verify.sh` не читает содержимое артефактов (только инварианты 1/5/6) — doc-track
  сессия проходит его без изменений скрипта.

### 3. Verification на doc-track
Литеральная формулировка бэклога — `Discovery-lite → Update → Closeout`, **без**
Verification. Это сознательно отличается от ADR-014 (обязательная лёгкая Verification для
`patch`), где обоснованием было «риск смержить код без единого взгляда». Для doc-track:
- Контент (retro/postmortem/groom-решения) по конструкции пишется **вместе с пользователем в
  интерактивной сессии** — не автономно агентом, как код на patch. Роль независимого взгляда
  частично закрыта самим присутствием пользователя в реальном времени.
- Нет кода/тестов — прогонять `qa` нечего; единственный смысл fresh-eyes review здесь был бы
  «когерентность» (не противоречит ли новая запись бэклога существующим), что для 1-3 файлов
  дешевле сделать самим оркестратором инлайн на этапе Update, чем городить отдельный этап +
  вызов `reviewer` через Task.

**Рекомендация**: НЕ включать Verification в новый трек — следовать тексту бэклога буквально.
Зафиксировать это явно одной фразой в `protocol.md` («doc-track не проходит Verification —
контент формируется интерактивно с пользователем, замены независимой проверке не требуется»)
чтобы не выглядело как непреднамеренный пропуск на фоне ADR-014. Это решение стоит явно
включить пунктом ADR-017 (не unilateral solo-решение архитектора — фиксируемое как осознанный
trade-off).

### 4. Как стартуют сессии
Рекомендация: **расширить `/sdx:start <type> <goal>`**, не заводить `/sdx:groom`/`/sdx:retro`/
`/sdx:postmortem` как отдельные команды — сохраняет единую точку входа и уже существующий
паттерн «type → триаж трека» (`start.md` шаг 2). Конкретные правки:
- `start.md` шаг 2 («Триаж трека») — добавить: если `type` ∈
  {`grooming`,`retro`,`postmortem`} → трек `doc` **безусловно** (не гипотеза, не требует
  уточняющего вопроса — в отличие от patch/standard/full триажа, здесь тип и трек жёстко
  связаны 1:1, т.к. вся суть PROC-002 — что это специализированные типы с фиксированным
  форматом).
- `start.md` шаг 5 — стартовый `stage` для `doc` = `Discovery` (как standard/full).
- `start.md` шаг 7 — новая ветка: Discovery-lite инлайн (без architect) → переход к `Update`;
  при `type=retro`/`type=postmortem` — скопировать соответствующий шаблон из
  `sdx/templates/retro.md`/`postmortem.md` в каталог сессии как отправную точку; при
  `type=grooming` — сразу к работе с `/sdx:backlog`.
- `sdx/protocol.md:9` — enum `type` расширить: `feature | bug | refactor | init | import |
  grooming | retro | postmortem`.
- `next.md:15` — фраза «первый содержательный гейт сессии (Discovery/Change)» для
  предложения авторежима — расширить на «(Discovery/Change/Update)», иначе doc-track не
  получит предложение включить `gate_mode: auto`.

### 5. Выход = записи бэклога, связь с п.4 Closeout
Уже разобрано в п.2 выше. Дополнительно: поскольку `/sdx:backlog add`/edit уже поддерживает
запись на текущую ветку сессии по ходу работы (не только на Closeout), Update-этап doc-track
сессии должен явно инструктировать: правки бэклога делать СРАЗУ на этапе Update (не
откладывать до Closeout) — тогда `change_note.md` этого этапа фиксирует, какие записи
созданы/изменены (список ID), а Closeout п.4 сверяет это по факту (индекс, `status`, `session`
поля) вместо повторной содержательной работы.

## Риски

1. **Дрейф дублирующихся копий матрицы.** Три места несут одни и те же данные
   (`sdx-stage.sh` — источник истины, `protocol.md` — человекочитаемая проекция,
   `stage-transition-guard.md` — DESIGN-документация исходной задачи DEBT-001/011). Только
   первые два защищены sanity-тестом (и то не полностью — см. риск 2). Третий — не
   автоматизирован, чисто дисциплина ревью.
2. **Sanity-тест сценарий 34 не покрывает новые треки автоматически** — жёстко перечисляет
   `full standard patch`. Если трек `doc` добавить в матрицу и таблицу протокола, но забыть
   добавить в цикл теста, тест продолжит зелёный, создавая ложное чувство защищённости
   (рассинхронизация `doc`-строк между матрицей и `protocol.md` не будет поймана). Рекомендую
   расширить цикл сразу в этой же сессии.
3. **Смешение семантики `change_note.md`.** Решение переиспользовать `change_note.md` как
   гейт-артефакт `Update` для ВСЕХ трёх типов (включая grooming, у которого он и есть
   единственный артефакт) размывает роль файла — раньше он был «дельты лёгких треков кода»,
   теперь ещё и «резюме груминга/ретро/постмортема». Не структурная проблема (ADR-003 уже
   определяет его широко — «единый аккумулятор дельт лёгких треков»), но стоит явно
   зафиксировать это расширение роли в ADR-017, а не оставить implicit.
4. **Отсутствие Verification на doc-track — сознательное отступление от прецедента ADR-014**
   (там обязательность верификации на дешёвом треке была явным уроком «no one looked at the
   patch»). Нужно явное обоснование в ADR-017, почему doc-track не наследует этот урок
   (аргумент — интерактивность/присутствие пользователя, см. развилку 3), иначе выглядит как
   регресс защиты, а не осознанный trade-off.
5. **Версия плагина / релизная практика.** Последний коммит истории — bump 1.2.2 → 1.3.0 и
