## 3. Независимая ручная проверка (исполнение в изолированной фикстуре)

Проверка выполнена напрямую вызовами `sdx/hooks/sdx-stage.sh` во временном каталоге
(`CLAUDE_PROJECT_DIR` указывал на фикстуру, не на этот репозиторий), в обход
слэш-команд/LLM-прослойки — чтобы независимо от `test-sdx-stage.sh` подтвердить поведение
скрипта. Фактические коды возврата:

1. `init "doc-manual-check" grooming doc Discovery interactive sdx/doc-manual-check` →
   `OK - -> Discovery`, **exit 0**. `session_state.json` содержит `type:"grooming"`,
   `track:"doc"`, `stage:"Discovery"`.
2. `next` (Discovery → Update, без гейт-артефакта) → `OK Discovery -> Update`, **exit 0**.
3. `next` (Update → Verification, `change_note.md` отсутствует) →
   `гейт не пройден — не найден/пуст 'change_note.md' ...`, **exit 1**, стадия осталась
   `Update`.
3b. То же с 0-байтовым `change_note.md` (создан `touch`) → идентичная ошибка, **exit 1**.
4. `next` после заполнения `change_note.md` непустым текстом →
   `OK Update -> Verification`, **exit 0**.
5. `next` (Verification → Closeout, отчёт содержит `### [FAIL] Something wrong`) →
   `гейт не пройден — 'verification_report.md' содержит находки FAIL. ...`, **exit 1**, стадия
   осталась `Verification`.
6. `next` после замены отчёта на чистый (`All good.`) → `OK Verification -> Closeout`,
   **exit 0**.
7. `next` при `stage=Closeout` (терминальный) → `OK no-op Closeout`, **exit 0**, стадия не
   изменилась.

Итог: полный цикл `doc` (Discovery → Update → Verification → Closeout), включая оба отказа на
непройденных гейтах (Update без `change_note.md`, Verification с FAIL-маркером), воспроизведён
вручную и совпадает с ожиданиями SPEC/DESIGN и с автотестами [36]-[43].

Дополнительно проверена retrack-эскалация в отдельной фикстуре:
- `doc → standard`, цель `Change`, `change_note.md` накоплен на `doc|Update` →
  `OK Update -> Change`, **exit 0** — подтверждён «бесплатный перенос» файла через совпадение

- `doc → full`, цель `Business Spec`, без `context_report.md` на диске →
  `'Business Spec' недостижим — этап 'Discovery' трека 'full' не подтверждён (не найден/пуст

- `doc → full`, цель `Discovery` (собственный первый активный этап `full`) →

## 7. Находки

### [WARN] Некорректная remediation-подсказка при FAIL-гейте на треке `doc`
При отказе перехода `Verification -> Closeout` из-за находки `### [FAIL]` в
`verification_report.md`, `sdx-stage.sh` выводит **захардкоженный** (не параметризованный по
треку) текст: `«Исправь их и вызови /sdx:backtrack --to Execution»` (строка 261
`sdx/hooks/sdx-stage.sh`, код не тронут этой сессией — унаследован из `patch`/`standard`/
`full`, где стадия `Execution` в треке реально существует). На треке `doc` стадии `Execution`
нет вовсе (`Discovery → Update → Verification → Closeout`) — буквальное следование этой
подсказке приводит ко второй ошибке. Воспроизведено вручную:
```
$ sdx-stage.sh backtrack <sid> Execution   # (тот же трек doc)
SDX sdx-stage: этап 'Execution' не активен в треке 'doc' — нужна смена трека, не откат:
/sdx:retrack <track>.
exit=1
```
`commands/next.md:26` и `commands/archive.md:30` явно инструктируют оркестратора «сообщи
пользователю текст stderr дословно... не формулируй заново» — то есть некорректная подсказка
дословно дойдёт до конечного пользователя doc-сессии. Правильная ремедиация для `doc` —
`/sdx:backtrack --to Update` (стадия, где реально редактируется `change_note.md`), не
`Execution`. Не покрыто ни одним сценарием `test-sdx-stage.sh` (сценарий [41] проверяет только
факт `exit 1` и наличие подстроки `FAIL` в stderr, не корректность текста ремедиации).
**Не блокирует** ни один критерий приёмки SPEC.md буквально (ни один критерий не описывает
текст этого сообщения) — поэтому не `FAIL`, но это реальный, воспроизводимый UX-дефект,
экспонированный (не внесённый) добавлением трека `doc`. Рекомендация: параметризовать
сообщение по треку (для `doc` — `Update`) отдельной точечной правкой либо завести
`DEBT-`-запись в бэклог на Closeout, если немедленный фикс сочтён избыточным для этой сессии.
