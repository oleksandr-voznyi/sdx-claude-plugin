# Business Specification: Рефакторинг работы с сессиями (версионирование артефактов, git worktree, автоопределение основной ветки)

## Контекст и Цели

Аудит фреймворка от 2026-07-01 выявил два связанных дефекта в жизненном цикле сессий SDX:

- **C7** — протокол (ADR-005) декларирует, что артефакты сессии коммитятся инкрементально в
  ветку `sdx/<id>`, но каталог `.claude/sessions/` целиком находится в `.gitignore` — коммит
  физически невозможен. Сессии живут только локально, не переносятся между машинами/участниками
  команды, а заявленное основание ADR-005 («git — авторитетный реестр артефактов») сейчас ложно.
- **C2** — `/sdx:switch` выполняет слепой `git add -A && git commit` в общем рабочем дереве перед
  переключением веток, что закоммичивает посторонний мусор/секреты под шумным сообщением и не
  обрабатывает конфликтные случаи.

Обе находки решаются одной связкой: переход на модель **«одна сессия = свой git worktree на
ветке `sdx/<id>`»**, внутри которого версионируются содержательные артефакты сессии. Отдельно, по
дополнительному запросу пользователя (2026-07-03), фреймворк не должен хардкодить имя основной
ветки (`main`) — часть репозиториев использует `master` или иное имя.

Ожидаемый результат: содержательные артефакты сессии видны команде через git и переживают
Closeout ровно так, как заявляет протокол; переключение между параллельными сессиями не портит
общее рабочее дерево; фреймворк одинаково работает в репозиториях с любым именем основной ветки.

## Требования

### Версионирование артефактов сессии (C7)

- [ADDED] **REQ-SESS-1**: Содержательные артефакты сессии (`session_state.json`, `session.log`,
  `SPEC.md`, `DESIGN.md`, `PLAN.md`, `change_note.md`, `verification_report.md`, `checkpoint.md`,
  `MANUAL_TEST.md`, `import_map.md` и любые другие `.md`/`.json`-артефакты, порождаемые командами
  `/sdx:*`) коммитятся в ветку `sdx/<id>` инкрементально по ходу работы над сессией — а не
  оставлены неверсионируемыми.
- [ADDED] **REQ-SESS-2**: Эфемерные scratch-файлы enforcement-слоя (`.stopgate.count`,
  `.stopgate.out` и любые другие временные буферы хуков) НЕ версионируются ни при каких условиях —
  они не несут исторической ценности и меняются на каждом ходу/прогоне.
- [ADDED] **REQ-SESS-3**: После успешного Closeout и слияния ветки сессии `sdx/<id>` в основную
  ветку в git-состоянии основной ветки (индекс/HEAD, а не только рабочее дерево на диске)
  отсутствуют файлы каталога сессии `<id>`. Свежий `clone`/`checkout` основной ветки не
  восстанавливает файлы сессии.
- [ADDED] **REQ-SESS-4**: История артефактов сессии (кто/когда/что менял) остаётся доступной через
  git-историю ветки `sdx/<id>` после Closeout (ветка не удаляется до момента, определённого
  протоколом закрытия — см. существующий инвариант 5 archive-verify).

### git worktree как механизм изоляции сессий (C2)

- [ADDED] **REQ-WT-1**: Каждая сессия создаётся в собственном git worktree, привязанном к ветке
  `sdx/<id>`. Артефакты сессии физически размещаются внутри этого worktree.
- [ADDED] **REQ-WT-2**: `/sdx:switch` перестаёт выполнять `git add -A`/`commit`/`checkout` в общем
  рабочем дереве. Переключение на другую сессию не мутирует состояние текущего рабочего дерева
  никаким образом, не относящимся явно и осознанно к этой сессии.
- [ADDED] **REQ-WT-3**: Работа над конкретной сессией физически изолирована от остальных
  параллельно активных сессий — правки в одной сессии (включая незакоммиченные, «грязные»,
  изменения рабочего дерева) не видны и не мешают другой одновременно активной сессии.
- [REMOVED] Слепой авто-коммит всего рабочего дерева при переключении сессии (текущее поведение
  `switch.md`) — устраняется полностью, без частичной замены на `git stash`.
- [ADDED] **REQ-WT-4**: Общая инфраструктура фреймворка (код хуков `.claude/sdx/hooks/*.sh`,
  `.claude/settings.json`, `.claude/agents/`, `.claude/commands/`), закоммиченная в основную ветку,
  остаётся видимой и работоспособной одинаково из любого worktree сессии (стандартное поведение
  git worktree — общий `.git`, разделяемое отслеживаемое дерево по каждой ветке).
- [ADDED] **REQ-WT-5**: После Closeout worktree сессии корректно освобождается штатным механизмом
  git worktree (а не только удалением каталога с диска) — `git worktree list` не показывает
  «осиротевшие»/prunable записи для закрытых сессий.

### Автоопределение основной ветки (новое требование, 2026-07-03)

- [ADDED] **REQ-BRANCH-1**: Ни один скрипт, хук или инструкция агента фреймворка не содержит
  жёстко зашитого имени основной ветки (`main`). Имя основной ветки определяется динамически для
  каждого проекта.
- [ADDED] **REQ-BRANCH-2**: На репозитории, где основная ветка называется `master`, инвариант
  «ветка сессии слита в основную» (проверяемый `archive-verify.sh` при `/sdx:archive`) корректно
  проходит/падает в соответствии с фактическим состоянием слияния — без ручной правки скриптов
  или инструкций под конкретный проект.
- [ADDED] **REQ-BRANCH-3**: На репозитории с основной веткой `master` материализация diff для
  fresh-eyes ревью (`/sdx:verify`, `git diff <основная>...sdx/<id>`) корректно вычисляет разницу
  относительно фактической основной ветки, а не относительно `main`.
- [ADDED] **REQ-BRANCH-4**: Определение имени основной ветки не требует индивидуальной настройки
  или правки инструкций/конфигурации под каждый отдельный проект при развёртывании фреймворка
  (`/sdx:init`) — определение происходит по общему для всех проектов правилу/механизму.

## Критерии приемки

- [ ] REQ-SESS-1: На типовом прогоне сессии (`start` → несколько `next`/`checkpoint` →
      `archive`) в git-логе ветки `sdx/<id>` присутствуют коммиты, содержащие `session_state.json`,
      `session.log` и хотя бы один из содержательных `.md`-артефактов, созданных по ходу сессии.
- [ ] REQ-SESS-2: После нескольких ходов агента (в т.ч. срабатываний `stop-gate.sh`) `git status`
      внутри worktree сессии не показывает untracked/modified `.stopgate.count`/`.stopgate.out` как
      кандидатов на коммит (файлы либо отсутствуют в индексе, либо явно игнорируются).
- [ ] REQ-SESS-3: После `/sdx:archive` на основной ветке `git log -- .claude/sessions/<id>` не
      показывает файлов, присутствующих в HEAD (историю коммитов ветки `sdx/<id>` это не отменяет);
      `git ls-tree -r main` (или иная основная ветка) не содержит путей `.claude/sessions/<id>/*`.
- [ ] REQ-WT-1/REQ-WT-4: `git worktree list` после `/sdx:start` показывает отдельную запись для
      новой сессии на ветке `sdx/<id>`; файлы `.claude/sdx/hooks/*.sh` идентичны и работоспособны
      при вызове из этого worktree.
- [ ] REQ-WT-2: После рефакторинга в тексте команды `switch.md` отсутствуют вызовы `git add -A`,
      `git commit` (авто-коммит без явного пользовательского намерения) и `git checkout` в общем
      дереве в рамках логики переключения.
- [ ] REQ-WT-3: Одновременное наличие двух активных сессий с незакоммиченными правками в каждой не
      приводит к конфликтам/потере данных при работе с любой из них по отдельности.
- [ ] REQ-WT-5: После успешного `/sdx:archive` `git worktree list` не содержит записи для закрытой
      сессии (ни активной, ни prunable).
- [ ] REQ-BRANCH-1: Полнотекстовый поиск литерала `main` как имени ветки (не как части другого
      слова/пути) по `.claude/sdx/hooks/*.sh` и `.claude/commands/sdx/*.md` не находит мест, где имя
      используется как жёстко зашитое значение основной ветки (за исключением документации/примеров,
      явно помеченных как иллюстративные).
- [ ] REQ-BRANCH-2: На тестовом репозитории, инициализированном с `git init -b master`, полный
      прогон `patch`-сессии (`start` → `execution` → `archive`) успешно проходит инвариант 5
      `archive-verify.sh` без установки оверрайд-переменной `SDX_ARCHIVE_NO_BRANCH_OK`.
- [ ] REQ-BRANCH-3: На том же тестовом репозитории с `master` команда `/sdx:verify` успешно
      материализует diff и передаёт его на fresh-eyes ревью без ошибок разрешения ссылки на ветку.
- [ ] REQ-BRANCH-4: `/sdx:init` на новом проекте с произвольным именем основной ветки не требует от
      пользователя вручную указывать это имя нигде в конфигурации фреймворка.

## Риски и Ограничения

- Требование REQ-WT-2/REQ-WT-3 зависит от эмпирически не до конца подтверждённого поведения
  Claude Code harness: следует ли `CLAUDE_PROJECT_DIR` за `cd` в рамках одного работающего
  процесса, или он фиксируется на каталоге запуска CLI (см. context_report.md §7 вопрос 1, §8).
  Проектирование должно исходить из консервативного допущения («env фиксируется на каталоге
  запуска»), что меняет ожидаемую пользовательскую семантику `/sdx:switch` (см. открытые вопросы
  ниже) — это не блокирует бизнес-требования выше, но сильно сужает пространство технических
  решений.
- Изменение модели версионирования артефактов сессии затрагивает существующий Closeout-контракт
  (порядок шагов, инвариант 1/5 archive-verify) — риск регресса в существующем тест-сьюте
  `test-archive-verify.sh`, который сейчас полагается на допущение «каталог сессии невидим для
  git».
- Требование «не хардкодить `main`» расширяется на документацию и ADR (`docs/DECISIONS.md`,
  `CLAUDE.md`), где упоминания `main` носят описательный характер — важно не переусердствовать и
  не заменить обоснованные примеры бессмысленными абстракциями там, где это не несёт
  функционального смысла (см. критерий приёмки REQ-BRANCH-1 с оговоркой про иллюстративные
  упоминания).
- Переход на worktree-модель — потенциально breaking change для уже существующих (созданных до
  внедрения этого рефакторинга) сессий, если такие остаются активными на момент раскатки (см.
  «Вне объёма» ниже и открытый вопрос №6 в context_report.md).

## Вне объёма

- Миграция уже существующих (созданных до внедрения worktree-модели) активных не-worktree сессий
  на новую модель. Предполагается чистый cutover с явным breaking change, если на момент раскатки
  такие сессии обнаружатся — решение о необходимости миграционной логики выносится на Technical
  Design (см. открытый вопрос ниже), но полноценный миграционный инструмент/скрипт в объём этой
  сессии не входит, если Design не покажет, что это тривиально.
- Изменения в `prod-guard.sh` и `preflight.sh` — по находкам Discovery (context_report.md §4, §7
  вопрос 8) эти хуки worktree-нейтральны (не определяют сессию по ветке/пути `.claude/sessions/`)
  и не требуют правок в рамках этого рефакторинга.
- Прочие находки аудита 2026-07-01, не относящиеся к C7/C2/автоопределению ветки (C1, C3, C4, C5,
  C6, D1 и др.) — вне объёма данной сессии.
- Расширение `/sdx:status` листингом сессий через `git worktree list` — упомянуто в
  context_report.md (§6) как приятный побочный эффект, закрывающий часть находки C5, но не
  является обязательным требованием этой сессии; может быть включено в Design как факультативное
  улучшение, не блокирующее приёмку.

## Открытые вопросы для Design

Ниже — технические развилки, зафиксированные в Discovery (context_report.md §7), которые
Business Spec сознательно не решает и передаёт на Technical Design:

1. **Расположение worktree** — внутри репозитория (gitignored `.sdx/worktrees/<id>/`) или снаружи
   (`../<repo>-sessions/<id>/`). Компромиссы описаны в context_report.md §5, §7 вопрос 2.
2. **Порядок удаления файлов сессии относительно мёржа** (§3, §7 вопрос 3) — вариант A (коммит
   удаления на ветке `sdx/<id>` до мёржа в основную ветку) vs вариант B (пост-мёрж коммит удаления
   прямо в основную ветку). Выбор определяет переписывание `archive-verify.sh` и порядок шагов
   Closeout-чек-листа.
3. **Точный состав версионируемых файлов внутри `.claude/sessions/<id>/`** — нужен ли явный
   allowlist/паттерн игнора для `.stopgate.*` (per-directory `.gitignore` внутри каталога сессии
   или паттерн в корневом `.gitignore`), см. §7 вопрос 4.
4. **Судьба каталога `.claude/sessions/` как корневого имени** — сохранить текущее относительное
   расположение внутри worktree, или мигрировать на унификацию с `.sdx/worktrees/<id>/` целиком
   (см. §7 вопрос 5) — более инвазивная правка, затрагивающая 9+ команд и весь протокол.
5. **Семантика `/sdx:switch` под консервативным допущением о `CLAUDE_PROJECT_DIR`** (§8) — если
   env действительно фиксируется на каталоге запуска CLI, `/sdx:switch` перестаёт быть
   автоматизируемым in-process шагом оркестратора и становится инструкцией пользователю
   открыть/запустить CLI в каталоге другого worktree. Это меняет пользовательский опыт команды —
   требует явного решения и, вероятно, отдельного подтверждения у пользователя на Design-гейте.
6. **Миграционная логика для уже существующих не-worktree сессий** (§7 вопрос 6) — нужен ли
   graceful fallback, или чистый cutover с breaking change, зафиксированным отдельным ADR.
7. **Модель тестирования `archive-verify.sh` под worktree** (§7 вопрос 7) — отдельный test suite
   с фикстурой `git worktree add`, либо доказательство worktree-агностичности хука через
   параметризацию `$CLAUDE_PROJECT_DIR` на обычном репозитории.
8. **Источник истины для автоопределения основной ветки** (§6a, §7 вопрос 9) — `origin/HEAD` vs
   `git config init.defaultBranch` vs эвристика существования `main`/`master`, порядок фолбэков,
   и точка общего шелл-хелпера, переиспользуемого хуками и prose-командами.

## Связь с существующими решениями (ADR)

- **ADR-005** («Артефакты коммитятся в ветку сессии инкрементально») — данный рефакторинг не
  отменяет решение, а делает его физически исполнимым (устраняет противоречие с `.gitignore`,
  описанное как C7). После реализации формулировка ADR-005 становится буквально истинной.
- **Кандидат ADR-009** («git worktree + tracked session artifacts») — Technical Design должен
  зафиксировать новым ADR решение о переходе на модель «сессия = worktree = ветка», со ссылкой на
  ADR-005 (уточняет/операционализирует его), по аналогии с прецедентом ADR-008 (новый ADR при
  развороте/уточнении предыдущего решения, а не правка задним числом).
