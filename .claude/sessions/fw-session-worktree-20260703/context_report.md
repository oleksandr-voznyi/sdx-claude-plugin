# Context Report: Рефакторинг работы с сессиями (C7 + C2, git worktree)

## Источники находок
- `docs/audit-2026-07-01-recommendations.md`, находки **C7** (стр. 202–216) и **C2** (стр. 156–164).
- Приоритизация: пункт 9 таблицы (`C7`, трек standard/full, «делать в связке с C2»), пункт 8 (`C2`).

## Целевое поведение (жёсткие требования пользователя)
1. Файлы папки сессии версионируются и коммитятся **только в ветку `sdx/<id>`**.
2. После Closeout+мёржа `.claude/sessions/<id>/` **отсутствует в `main`** (как и сейчас).
3. Механизм — **git worktree**: одна сессия = свой worktree на ветке `sdx/<id>`, закрывает заодно C2.

## 1. Текущий жизненный цикл файлов сессии (по командам)

| Команда | Что делает с `.claude/sessions/<id>/` |
|---|---|
| `start.md` (шаг 3–6) | Создаёт директорию, создаёт ветку `sdx/<id>`, пишет `session_state.json`, пишет `session.log`. Шаг 8 явно предписывает **коммитить артефакты инкрементально в ветку сессии** — но директория в `.gitignore`, коммит невозможен (см. §2). |
| `switch.md` | Шаг 2: **слепой** `git add -A && git commit -m "sdx: auto-save session before switch"` в текущей ветке, затем `git checkout sdx/$1` — это C2. Поскольку `.claude/sessions/` игнорируется, `add -A` его не подхватывает, но подхватывает весь остальной грязный код в дереве без разбора (мусор/секреты) под шумным сообщением. Не обрабатывает конфликты (checkout невозможен, та же ветка). |
| `checkpoint.md` | Пишет `checkpoint.md` в папку сессии, лог `[CHECKPOINT]`. Только Write, коммит не делает. |
| `status.md` | Read-only, не мутирует. |
| `next.md` / `retrack.md` / `backtrack.md` | Мутируют `session_state.json` (`stage`/`track`), лог `[STAGE_CHANGE]`/`[ERROR]`. Коммит не делают явно — полагаются на общий принцип «коммить инкрементально» из протокола, который тоже упирается в `.gitignore`.
| `verify.md` | Создаёт `verification_report.md` (через `reviewer`), `MANAKL_TEST.md`/manual-прогон. Diff вычисляется командой `git diff main...sdx/<id>` (шаг 4) — конструкция уже подразумевает, что ветка сессии существует как отдельная ref, совместима с worktree без изменений.
| `archive.md` | Единственная команда, которая **удаляет** папку сессии — делегирует в `archive-verify.sh` (см. §3) **после** мёржа в `main`. |

**Вывод:** сам протокол (`start.md` шаг 8, ADR-005) уже заявляет требование «коммитить инкрементально», но оно физически невыполнимо из-за `.gitignore`. Это ровно противоречие C7 — подтверждено чтением кода, не только текста аудита. `switch.md` (C2) — отдельный, но связанный дефект: слепой `add -A` вне зависимости от worktree/gitignore.

## 2. `.gitignore`

```
# SDX: данные активных сессий не версионируются
.claude/sessions/

# SDX: переносимые бандлы import/export (транспортные артефакты)
.sdx/bundles/

# Локальные настройки Claude Code
.claude/settings.local.json
```

Снятие `.claude/sessions/` целиком сделает трекаемыми не только «содержательные» артефакты (`session_state.json`, `session.log`, `SPEC.md`/`DESIGN.md`/`PLAN.md`/`change_note.md`, `verification_report.md`, `checkpoint.md`, `MANUAL_TEST.md`, `import_map.md`), но и **эфемерные scratch-файлы enforcement-слоя**:
- `.stopgate.count` (loop-guard `stop-gate.sh`),
- `.stopgate.out` (буфер вывода тест-прогона, `stop-gate.sh`, шаг «Per-session temp file»).

Эти файлы меняются на каждом ходу/прогоне и не несут ценности как история — коммитить их означало бы шум в истории ветки сессии и грязное дерево на каждом Stop-хуке (что сломало бы инвариант 1 archive-verify — «дерево чистое»). **Нужна дифференциация**: трекать содержательные `.md`/`.json`-артефакты, игнорировать `.stopgate.*` (например, через `.claude/sessions/*/。gitignore`-паттерн `.stopgate.*` или явный allowlist).

## 3. `archive-verify.sh` и его тест — конфликт последовательности

Текущая последовательность (`archive-verify.sh`):
1. Invariant 1: `git status --porcelain` пусто → иначе FAIL, abort.
2. Invariant 5: ветка `sdx/<id>` **точным именем** (`grep -qx`) входит в `git branch --merged main` → иначе FAIL, abort (включая случай отсутствия ветки — недоказуемость слияния, `SDX_ARCHIVE_NO_BRANCH_OK=1` для явного оверрайда).
3. Только если 1 и 5 прошли — **деструктив**: `rm -rf "$sdir"`, проверка отсутствия каталога, `git branch -d "sdx/$sid"`.

`test-archive-verify.sh`, `setup_clean_repo()` явно **гитигнорит** `.claude/sessions/` в тестовом временном репо (`printf '.claude/sessions/\n' > "$TMPPROJ/.gitignore"`) — то есть весь тест-сьют (все 6 сценариев) сконструирован вокруг допущения «каталог сессии невидим для git status». Это ровно то допущение, которое требование пользователя (п.1/2) отменяет.

**Ключевой конфликт при переходе на tracked-модель:** если файлы сессии закоммичены в ветку и ветка смёржена в `main` **как есть** (со всем содержимым `.claude/sessions/<id>/`), то:
- Invariant 1 (чистое дерево) после мёржа — выполнится (в дереве всё закоммичено, включая файлы сессии).
- Но `rm -rf "$sdir"` после этого **не коммитится** — рабочее дерево `main` становится грязным (удалённые tracked-файлы), а скрипт не делает финальный commit. Итог: `main` физически на диске чист, но git-состояние (индекс/HEAD) всё ещё содержит файлы сессии → требование пользователя п.2 («в main их быть не должно») не выполняется на уровне репозитория (git log/clone/checkout `main` вернёт файлы обратно).

**Варианты для Technical Design** (не решаю здесь — Discovery):
- **(A) Удаление коммитом на ветке сессии ДО мёржа.** В Closeout, после переноса дельт в `docs/` (чек-лист п.2), сделать явный коммит на `sdx/<id>`: `git rm -r .claude/sessions/<id>/ && git commit`. Затем мёржить в `main` — merged-HEAD уже не содержит файлы сессии, `rm -rf` в хуке становится избыточным/страховочным (можно оставить как no-op-проверку «действительно отсутствует» вместо активного удаления). Плюс: `main` никогда не видит файлы сессии даже мимолётно; git-история их всё равно хранит (в ветке `sdx/<id>`, которая по инварианту 5 всё равно живёт как минимум до удаления `git branch -d`). Минус: меняет обязательный порядок шагов чек-листа Closeout (удаление до мёржа, а не после), обязывает архитектора явно развести «коммит дельт» и «коммит удаления» на ветке.
- **(B) Пост-мёрж коммит удаления прямо в `main`.** Оставить `rm -rf` в хуке, но добавить `git add -A && git commit` сразу после (прямой коммит в `main`, не через merge). Плюс: минимальные правки последовательности. Минус: нарушает обычный паттерн «`main` получает только merge-коммиты»; смешивает роль скрипта (verify → mutate git history) с эффектом «пишет в `main` напрямую» — модель ответственности хуков в протоколе описывает их как gate/verify, не как источник новых коммитов в `main`.
- И (A), и (B) требуют переписать `test-archive-verify.sh` (гитигнор-допущение в фикстурах больше не отражает реальность) — либо завести отдельный набор сценариев «tracked session files».

## 4. Worktree-совместимость хуков — проверено по коду

Все 5 хуков определяют активную сессию **по имени git-ветки**, а не по cwd:

- `stage-gate.sh`: `branch="$(git -C "$proj" branch --show-current ...)"`, где `proj="${CLAUDE_PROJECT_DIR:-.}"`. Путь к состоянию: `"$proj/.claude/sessions/${sid}/session_state.json"`.
- `stop-gate.sh`: идентичная схема + temp-файлы `"$proj/.claude/sessions/${sid}/.stopgate.count"` / `.stopgate.out` — комментарий в коде уже отмечает «worktree-safe» (per-session, не общий temp).
- `prod-guard.sh`: не завязан на ветку/сессию вообще (глобальный per-project `conf`), worktree-нейтрален.
- `preflight.sh`: SessionStart, только проверка `jq`, worktree-нейтрален.
- `archive-verify.sh`: `git -C "$proj" ...`, `sdir="$proj/.claude/sessions/$sid"` — вызывается ПОСЛЕ мёржа, из `/sdx:archive`, предполагаемо из основного (main) окружения.

**Критический для Design вопрос, который аудит закрывает утверждением «хуки уже worktree-совместимы» — я его не могу подтвердить эмпирически из кода одного проекта:**

Все пути и `git -C` в хуках построены вокруг `$CLAUDE_PROJECT_DIR`, а не вокруг cwd процесса. `git -C "$proj" branch --show-current` возвращает ветку, **которая checked out конкретно в директории `$proj`** — это корректно только если `$CLAUDE_PROJECT_DIR` во время работы Claude Code над сессией-в-worktree **указывает на директорию этого worktree**, а не остаётся зафиксированным на корне основного репозитория. Это допущение про поведение самого Claude Code harness (не про код фреймворка) и **не верифицируется чтением этого репозитория** — см. открытый вопрос №1 ниже. Если допущение неверно (harness фиксирует `CLAUDE_PROJECT_DIR` на исходном каталоге запуска независимо от `cd`), то `stage-gate`/`stop-gate`/`archive-verify` при работе из worktree будут детектировать ветку `main` основного worktree (или ветку того каталога, где реально был найден `.git`), а не `sdx/<id>` — то есть **весь enforcement-слой молча деградирует в no-op** для worktree-сессий (safe-by-default маскирует поломку — то же семейство риска, что C5 «тихая деградация»).

Пути `.claude/sessions/${sid}/...` внутри хуков строятся от `$proj`, а не от cwd — тоже завязано на то же допущение.

## 5. git worktree — как ложится на модель «ветка = сессия»

- **Создание** (`start.md`, шаг 4): вместо (или в дополнение к) `git checkout -b sdx/<id>` — `git worktree add <path> -b sdx/<id>`. Открытый вопрос: `<path>` внутри репозитория (`.sdx/worktrees/<id>/`, потребует добавления в `.gitignore` — сам каталог worktree не должен трекаться основным деревом) или снаружи (`../<repo>-sessions/<id>/` — классическая практика, избегает самовложенности worktree внутрь родительского рабочего дерева и конфликтов с рекурсивными сканами/IDE watchers).
- **`session_state.json`/`session.log`** при tracked-модели физически будут жить **внутри worktree**, на ветке `sdx/<id>` — то есть их видно только из этого worktree (что и требуется: сессия = worktree = ветка = свой набор файлов). `.claude/settings.json` (проводка хуков) и `.claude/sdx/hooks/*.sh` (сам код хуков) — общая инфраструктура, трекается на `main` и по обычному git-worktree-поведению видна во всех worktree одинаково (это не session-specific).
- **`switch.md`**: под worktree-моделью не нужен `git checkout` вовсе — «переключение» становится сменой рабочего каталога (`cd` на нужный worktree) вместо checkout в общем дереве. Это устраняет C2 полностью (не нужен ни `add -A`, ни авто-коммит, ни stash — каждая сессия физически изолирована в своей директории). Открытый критический вопрос — может ли **один и тот же процесс Claude Code** (текущий агент) сменить эффективный `CLAUDE_PROJECT_DIR` через `cd` в рамках одного диалога, или переключение сессии по факту означает «пользователь должен перезапустить/переоткрыть Claude Code CLI в каталоге другого worktree» (т.е. `switch.md` превращается в инструкцию человеку, а не в автоматизированный шаг оркестратора). Наблюдение из окружения этого агента: Bash-инструмент документирует «working directory persists between commands, but shell state does not» — то есть `cd` в Bash-вызовах данного агента переживает между вызовами, но неясно, следует ли за этим `$CLAUDE_PROJECT_DIR` (переменная окружения, а не cwd) внутри хуков — это то же допущение, что в §4.
- **Closeout** (`archive.md`): после успешного `archive-verify.sh` понадобится `git worktree remove <path>` (а не просто `rm -rf`, иначе `git worktree list` будет висеть на «prunable» состоянии до `git worktree prune`) — это дополнение к текущей логике хука, которая сейчас просто `rm -rf` директорию сессии внутри общего дерева.
- **Риски**:
  - Вложенные worktree (если класть `.sdx/worktrees/<id>/` внутри самого репозитория) — нужно решить, что происходит, если внутри worktree кто-то попытается создать ещё один worktree (маловероятно при однa-сессия-один-worktree, но стоит зафиксировать запрет).
  - Пути хуков (`$CLAUDE_PROJECT_DIR/.claude/sdx/hooks/...` в `settings.json`) — если `CLAUDE_PROJECT_DIR` корректно следует за worktree, пути разрешатся нормально (хуки физически идентичны на любой ветке, т.к. `.claude/sdx/hooks/` трекается и на `main`, и на `sdx/<id>` — если ветка сессии форкнута от актуального `main`, скрипты синхронизированы; если ветка сессии устарела относительно `main` — хуки на её ветке могут быть версией на момент форка, это уже существующий риск, не новый от worktree).
  - `checkpoint`/`/clear`-восстановление — должно продолжать работать, поскольку `checkpoint.md` и остальные `.md` физически лежат в worktree-каталоге сессии; риск только если механизм `/clear` в Claude Code как-то завязан на исходный launch-каталог, а не на актуальный cwd/`CLAUDE_PROJECT_DIR` worktree — тот же класс допущения.
  - `stop-gate.sh` per-session temp-файлы (`.stopgate.count/.out`) уже помечены в коде как «worktree-safe» — но если эти файлы станут tracked (см. §2), они будут «мусорить» git-статус на каждом ходу; нужно явно НЕ трекать их (per-directory `.gitignore` внутри `.claude/sessions/<id>/`, или паттерн в корневом `.gitignore` вида `.claude/sessions/*/.stopgate.*`).

## 6a. Дополнительное требование: автоопределение основной ветки (не хардкодить `main`)

Требование пользователя (2026-07-03): фреймворк во многих местах жёстко предполагает, что
основная ветка называется `main`, но в части репозиториев (напр. старых) она — `master` или
иная. Нужно, чтобы агент/скрипты **ориентировались по ситуации**, без индивидуальной правки
инструкций под каждый проект.

Точки хардкода `main` (найдены по коду):
- `.claude/sdx/hooks/archive-verify.sh`: `git branch --merged main` (инвариант 5) — критично,
  при `master` инвариант «ветка слита» ложно провалится или пропустится.
- `.claude/commands/sdx/verify.md`: `git diff main...sdx/<id>` (материализация diff для reviewer).
- `.claude/sdx/protocol.md`: упоминания `main` в §Fresh-eyes (`git diff main...sdx/<id>`),
  §Контракт закрытия (п.5 «слита в main»), §Git-флоу.
- `docs/DECISIONS.md`, `CLAUDE.md`: упоминания «слияние в main».
- `archive.md`, `start.md`: `git checkout main`, мёрж в `main`.

**Механизм автоопределения (для Design):** определить дефолтную ветку динамически, напр.
`git symbolic-ref --quiet refs/remotes/origin/HEAD | sed 's@^refs/remotes/origin/@@'`
(если есть remote), фолбэк на `git config init.defaultBranch`, затем эвристика «существует ли
`main`, иначе `master`». Единая точка резолва (общий шелл-хелпер для хуков + документированное
правило для агента в prose-командах). Внести в объём этой сессии как связанное требование
(вопрос №9 ниже).

## 7. Открытые вопросы для Technical Design

1. **[КРИТИЧНО] Поведение `$CLAUDE_PROJECT_DIR` в worktree.** Следует ли `CLAUDE_PROJECT_DIR` за `cd` внутри одного диалога/сессии Claude Code, или он фиксируется при старте процесса на исходном каталоге? От ответа зависит, работает ли вообще связка «хуки определяют сессию по `git -C "$proj" branch --show-current`» в worktree-модели, и является ли `/sdx:switch` автоматизируемым шагом оркестратора или инструкцией пользователю перезапустить Claude Code в другом каталоге. Нужна эмпирическая проверка (например, тестовый worktree + `echo $CLAUDE_PROJECT_DIR` до/после `cd`) до фиксации архитектуры хуков.
2. **Расположение worktree.** Внутри репозитория под gitignored `.sdx/worktrees/<id>/`, либо снаружи репозитория (`../<repo>-sessions/<id>/`). Компромиссы: «внутри» — предсказуемые относительные пути для хуков/документации, но риск самовложенности и IDE/watcher-шума; «снаружи» — чище git-модель, но ломает допущение «всё внутри `.claude/sessions/`» и усложняет discovery для `/sdx:status`/новых участников.
3. **Последовательность удаления файлов сессии** (см. §3, варианты A/B): коммит-удаление на ветке сессии до мёржа (main никогда не видит файлы) vs пост-мёрж коммит прямо в `main` (rm-rf + commit после merge). Выбор определяет переписывание `archive-verify.sh` и чек-листа Closeout.
4. **Что именно версионировать внутри `.claude/sessions/<id>/`.** Полный набор `.md`/`.json` артефактов — да; эфемерные `.stopgate.count`/`.stopgate.out` — нет (нужен точечный игнор). Нужна ли директория `checkpoint.md` тоже в git (вероятно да, для восстановления после `/clear` на другой машине/участнике — это фактически расширяет пользу C7 «сессии видны команде», упомянутую в самой находке).
5. **Судьба `.claude/sessions/` как корневого каталога.** Оставить текущее имя/расположение (файлы живут в worktree по тому же относительному пути) — минимальные правки документации; либо мигрировать корень сессий на `.sdx/worktrees/<id>/` целиком, унифицировав с уже существующим `.sdx/bundles/` (транспортные артефакты import/export) — тогда `.claude/sessions/` можно упразднить, но это более инвазивная правка (много упоминаний по всему фреймворку: 9+ команд, protocol.md, все хуки, CLAUDE.md).
6. **`/sdx:switch` без worktree для уже существующих (не-worktree) старых сессий.** Нужна ли миграционная логика/грейсфул fallback для сессий, созданных до внедрения worktree-модели (если такие есть в проекте на момент раскатки), или это чистый cutover с явным breaking change, зафиксированным в новом ADR.
7. **Модель тестирования `archive-verify.sh` под worktree.** Текущие тесты создают временный **не-worktree** git-репозиторий (`git init` + branch). Нужно решить, тестировать ли worktree-специфичное поведение отдельным сьютом (`git worktree add` в фикстуре) или абстрагировать хук так, чтобы он был worktree-агностичен (полагается только на переданный `$CLAUDE_PROJECT_DIR`, что делает тест с обычным repo репрезентативным) — согласуется с вопросом №1.
8. **`prod-guard`/`preflight`** — worktree-нейтральны по коду (§4), правок не требуют; явно зафиксировать это в DESIGN, чтобы не трогать зря.
9. **Автоопределение основной ветки** (см. §6a): единый механизм резолва дефолтной ветки для хуков и команд вместо хардкода `main`; выбрать источник истины (`origin/HEAD` vs `init.defaultBranch` vs эвристика) и точку общего хелпера.

## 8. Эмпирическая проверка вопроса №1 (проведена оркестратором, 2026-07-03)

Зонд в текущем окружении (см. лог сессии):

1. **`CLAUDE_PROJECT_DIR` в этом окружении НЕ установлен** (`<не установлен>`). Хуки используют
   `proj="${CLAUDE_PROJECT_DIR:-.}"` → при отсутствии переменной `proj` резолвится в `.` (cwd
   процесса хука). Проводка в `settings.json` (`$CLAUDE_PROJECT_DIR/.claude/sdx/hooks/...`)
   вычисляется harness'ом — в проде переменная выставляется Claude Code на каталог запуска CLI.
2. **`git -C <worktree> branch --show-current` работает пер-каталогово корректно:** из каталога
   worktree вернулась `sdx/probe-throwaway`, из корня основного репо — ветка основного дерева.
   То есть детект сессии по ветке верен ТОЛЬКО если `$proj` указывает на каталог нужного worktree.
3. **`.claude/sdx/hooks/` виден внутри worktree** (трекается на `main`, git-worktree расшаривает
   рабочее дерево по ветке — скрипты хуков идентичны во всех worktree).

**Вывод для архитектуры (ключевой):** worktree-совместимость держится на том, что `$proj`
(=`CLAUDE_PROJECT_DIR`, а в проде это **каталог запуска CLI**) указывает на worktree сессии.
Практически это означает модель **«одна сессия = свой worktree = свой запущенный экземпляр
Claude Code, стартованный в каталоге этого worktree»**. In-process `/sdx:switch` через `cd`
между worktree в рамках одного работающего экземпляра **НЕ** переустанавливает
`CLAUDE_PROJECT_DIR` (env фиксируется при старте процесса) — значит хуки продолжат резолвиться
на исходный каталог запуска. Следствие: `/sdx:switch` в worktree-модели перестаёт быть
in-process git-checkout и становится **инструкцией открыть/запустить CLI в каталоге другого
worktree** (новая вкладка/окно терминала). Это чисто закрывает C2 (в общем дереве больше нет
ни `git checkout`, ни `add -A`), но меняет семантику switch — вынести на подтверждение
пользователю на гейте Business Spec/Design. Полная эмпирическая верификация поведения harness
(следует ли `CLAUDE_PROJECT_DIR` за процессом) в этом песочном окружении невозможна — но
проектировать безопасно под допущение «env фиксируется на каталоге запуска», т.к. это
консервативный (worst-case) сценарий, при котором дизайн обязан быть корректным.
