>    границы) шире: помечаются ВСЕ этапы строго после `target`, до конца активных
>    этапов трека, без верхней границы по текущему этапу. Формулировка ниже приведена
>    к фактической.



## Архитектурный обзор

Сегодня `stage` в `session_state.json` пишут **шесть независимых прозаических путей**
(`start`, `import`, `next`, `backtrack`, `retrack`, `archive`) — каждый со своей (или
никакой) логикой гейта. Решение вводит единственный детерминированный писатель —
CLI-скрипт `sdx/hooks/sdx-stage.sh` — и техническую границу, блокирующую прямую правку
поля `stage` в обход него.

Два новых артефакта enforcement-слоя:

1. **`sdx/hooks/sdx-stage.sh`** — не hook в терминах `hooks.json` (не подключается к
   событию), а **CLI-скрипт с четырьмя подкомандами**, вызываемый командами `/sdx:*`
   через инструмент `Bash` (тот же паттерн, что `archive.md` уже использует для
   `archive-verify.sh <id>`). Единственный код, кто пишет `.stage` в
   `session_state.json`. Несёт внутри себя (REQ-STAGE-3) машиночитаемую матрицу
   «трек → упорядоченные активные этапы → гейт-артефакт».
2. **`sdx/hooks/stage-write-guard.sh`** — новый PreToolUse-хук (`Write|Edit|MultiEdit`,
   третья запись в существующем matcher-е `hooks/hooks.json`), детерминированно
   блокирующий прямую правку поля `stage` в `session_state.json` вне зависимости от
   инструмента.

Ключевое структурное свойство, снимающее необходимость в специальном «пропуске для
скрипта» внутри хука: **`sdx-stage.sh` мутирует файл через `Bash` (printf/jq/mv), а не
через `Write`/`Edit`/`MultiEdit`.** Хук `stage-write-guard.sh` матчится только на эти три
инструмента — легитимный путь физически вне его зоны видимости, а не «специальный
случай, который он обязан распознать и пропустить». Это прямое следствие структуры
`PreToolUse`-матчеров Claude Code, не допущение.

Обе шесть прозаических точек записи сводятся к четырём вызовам:

| Команда | Подкоманда `sdx-stage.sh` |
|---|---|
| `/sdx:start`, `/sdx:import` | `init` |
| `/sdx:next`, `/sdx:archive` (вход в Closeout) | `next` |
| `/sdx:backtrack --to <stage>` | `backtrack` |
| `/sdx:retrack <track>` | `retrack` |

`/sdx:archive` не получает отдельного код-пути: `Closeout` в матрице — обычная
последняя строка активных этапов трека, поэтому вход в Closeout — такой же
`sdx-stage.sh next <sid>`, как любой другой forward-переход (REQ-CLOSEOUT-ENTRY-1).

Поля `track`/`gate_mode`/`artifacts`/`history` **не входят** в скоуп `sdx-stage.sh` и
deny-хука (REQ-DENY-2) — `/sdx:retrack` по-прежнему правит `track`/`gate_mode` напрямую
через `Edit` (легитимно, не блокируется), и лишь ЗАТЕМ вызывает
`sdx-stage.sh retrack <sid> <target>` для поля `stage`.

## Интерфейс `sdx/hooks/sdx-stage.sh` (CLI-контракт)

Вызывается командами так же, как уже вызывается `archive-verify.sh`:
```bash
bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" <subcommand> <args...>
```
Уважает `${CLAUDE_PROJECT_DIR:-.}` (тот же паттерн, что `archive-verify.sh`).

### `init` — первичная установка (REQ-STAGE-1, `/sdx:start`/`/sdx:import`)
```
sdx-stage.sh init <sid> <type> <track> <stage> <gate_mode> <git_branch>
```
Требует, чтобы `session_state.json` сессии **ещё не существовал** (иначе `exit 2` —
`init` не для повторной инициализации). Валидирует, что `<stage>` — первый активный
этап `<track>` в матрице. Создаёт `.claude/sessions/<sid>/session_state.json` целиком
(`session_id`, `type`, `track`, `stage`, `gate_mode`, `git_branch`, `artifacts: []`,
`history: []`) и **атомарно же** дописывает в `session.log` строку
`[START] Инициализация сессии <sid> (трек: <track>)` (создаёт `session.log`, если его
ещё нет). `start.md`/`import.md` после вызова просто коммитят оба файла — отдельный шаг
«запиши в лог» из их прежней прозы исключается (пишет скрипт).

### `next` — forward-переход (REQ-STAGE-2, `/sdx:next`, `/sdx:archive` вход в Closeout)
```
sdx-stage.sh next <sid>
```
**Без явного целевого этапа** — сознательное решение (см. «Развилка: `next` с целью или
без»). Скрипт сам вычисляет «следующий активный этап трека» из встроенной матрицы
(REQ-STAGE-3) и проверяет гейт-условие **уходящего** (текущего) этапа. `/sdx:archive`
вызывает ровно ту же подкоманду — `Closeout` в матрице является обычной последней
строкой активных этапов трека, отдельного кода не требуется (REQ-CLOSEOUT-ENTRY-1).

### `backtrack` — backward-переход (REQ-BACKTRACK-1/2, `/sdx:backtrack --to <stage>`)
```
sdx-stage.sh backtrack <sid> <target-stage>
```
Явная цель обязательна. Без проверки гейт-артефактов уходящего этапа. Помечает outdated
артефакты промежуточных этапов (см. «Backward-переход» ниже).

### `retrack` — пересчёт под новый трек (REQ-RETRACK-1, `/sdx:retrack <track>`)
```
sdx-stage.sh retrack <sid> <target-stage>
```
Явная цель обязательна, **вычисляется вызывающей стороной** (`retrack.md`: ближайший
активный этап нового трека, не дальше, чем позволяют фактически существующие
гейт-артефакты — см. «Развилка: кто вычисляет цель `retrack`»). _Пересмотр прохода 4:
прежняя формулировка «как правило текущий этап, если активен в новом треке» удалена как
неверная — при эскалации `patch → full` этап `Execution` формально активен в `full`, но
отклоняется до появления артефактов предшествующих этапов._ Скрипт вызывается
**после** того, как `retrack.md` уже поправил поле `track` в файле напрямую через `Edit`
(легитимно, REQ-DENY-2) — `retrack`-подкоманда читает уже обновлённый `track` и
неизменённый (пока) `stage`, проверяет «цель активна в новом треке» И (REQ-RETRACK-2,
добавлено на Verification, переписано на 4-м проходе — см. «Forward-skip guard в
`retrack`» ниже) **evidence-based потолок**: цель допустима, только если гейт-артефакты
всех предшествующих ей этапов нового трека фактически существуют на диске.

> **Пересмотр (проход 4).** Изначально потолок выводился из позиции `stage` на общей
> межтрековой шкале рангов (`SDX_CANON_ORDER`). Ревью нашло храповик: вход в трек по
> клэмпу засчитывался как накопленный прогресс, и парой `retrack` можно было подняться
> мимо всех гейтов. Ранговая таблица удалена; позиция `stage` — утверждение о
> прогрессе, артефакт — доказательство.

### Коды возврата (единые для всех подкоманд)
| Код | Значение |
|---|---|
| `0` | Успех, включая безопасный no-op (идемпотентный вызов, REQ-STAGE-4) |
| `1` | Переход отклонён (не пройден гейт / невалидная цель) — `stage` не изменён |
| `2` | Ошибка окружения/вызова (нет `jq`, нет `session_state.json`, повторный `init`, плохие аргументы) — `stage` не изменён |

### stdout/stderr (REQ-STAGE-5)
- Успех: `stdout` — одна строка `OK <old-stage> -> <new-stage>` (или `OK no-op <stage>`
  для идемпотентного вызова). Для `backtrack` дополнительно по одной строке
  `OUTDATED: <path>` на каждый помеченный артефакт — `backtrack.md` ретранслирует их
  пользователю без повторного чтения файлов (REQ-BACKTRACK-2 наблюдаемость).
- Отказ (код `1`/`2`): `stderr` — одна developer/user-facing строка на русском, **всегда**
  называющая (а) чего не хватает и (б) какой командой чинить — по образцу существующих
  `deny()`-сообщений `stage-gate.sh`/`prod-guard.sh`. Примеры:
  - `SDX sdx-stage: гейт не пройден — не найден/пуст 'DESIGN.md' в .claude/sessions/<sid>/. Заверши Technical Design, затем повтори /sdx:next.`
  - `SDX sdx-stage: гейт не пройден — 'verification_report.md' содержит находки FAIL. Исправь их и вызови /sdx:backtrack --to Execution.`
  - `SDX sdx-stage: 'X' не распознан как имя этапа протокола SDX. Проверь опечатку.`
  - `SDX sdx-stage: этап 'X' не активен в треке '<track>' — нужна смена трека, не откат: /sdx:retrack <track>.`
  - `SDX sdx-stage: 'X' позже текущего этапа '<stage>' в порядке трека — это не откат. Для движения вперёд используй /sdx:next.`
  - `SDX sdx-stage: jq не найден — переход отклонён (fail-closed), файл не изменён. Установите jq.`

Логирование `[STAGE_CHANGE]`/`[START]` в `session.log` выполняет **скрипт**, атомарно с
мутацией `session_state.json` (рекомендация из постановки задачи принята: «переход + лог»
— одна операция, не два раздельных хода агента, и не может рассинхронизироваться, если
второй Edit-ход прервётся).

## Матрица «трек → этапы → гейт-артефакты» (REQ-STAGE-3)

Единственный источник истины — внутри `sdx-stage.sh`, heredoc с `|`-разделителем
(имена этапов содержат пробелы, поэтому не `:`):

```bash
# Формат строки: track|stage|artifact|fail_marker
#   artifact     — относительный путь внутри каталога сессии; "-" = гейт не проверяем
#                  объективно (условие остаётся прозаическим суждением оркестратора,
#                  REQ-STAGE-2 «Ограничение»).
#   fail_marker  — "yes": дополнительно требуется отсутствие "^### \[FAIL\]" в artifact
#                  (формат reviewer, agents/reviewer.md); "no" — только существование+непустота.
# Порядок строк ВНУТРИ трека = порядок активных этапов (используется для next/backtrack).
SDX_STAGE_MATRIX='
full|Discovery|context_report.md|no
full|Business Spec|SPEC.md|no
full|Technical Design|DESIGN.md|no
full|Task Planning|PLAN.md|no
full|Execution|-|no
full|Documentation|-|no
full|Verification|verification_report.md|yes
full|Deployment|-|no
full|Closeout|-|no
standard|Discovery|-|no
standard|Change|change_note.md|no
standard|Execution|-|no
standard|Verification|verification_report.md|yes
standard|Closeout|-|no
patch|Execution|change_note.md|no
patch|Verification|verification_report.md|yes
patch|Closeout|-|no
'
```

Обоснование пустых (`-`) гейтов — прямая проекция вывода Discovery §3 «Общий вывод по
гейтам»: `Execution→Documentation` (декомпозиция PLAN.md), `Documentation→Verification`,
`Deployment→Closeout` сегодня не имеют объективно проверяемого критерия и не получают
его этим решением (SPEC явно не требует вводить формат чекбоксов PLAN.md — «Риски и
Ограничения» п.3 SPEC). `Business Spec→Technical Design` проверяет только существование
`SPEC.md`, не `status: approved` (DEBT-007, явно вне скоупа).

**Важное следствие унификации:** гейт `Verification→Closeout` (проверка
`verification_report.md` без `FAIL`) — **одна и та же строка матрицы для всех трёх
треков** (для `patch`/`standard` она же и есть гейт входа в Closeout; для `full` перед
Closeout ещё стоит `Deployment` без объективного гейта). Это делает архивную прозу
`archive.md` про «на треке patch — verification_report.md без FAIL» избыточной: она была
частным случаем общего правила «выйти из Verification нельзя без чистого отчёта» —
теперь это универсально enforced скриптом для всех треков одинаково, без ветвления по
`track` в `archive.md` (см. «Изменения в командах» ниже).

**Компромисс, не гарантия (Discovery риск 2).** `protocol.md` хранит человекочитаемую
проекцию этой же таблицы (уже существующую, строки 51–55) — риск рассинхронизации
снижается (одна каноническая копия вместо двух текстовых), но не устраняется технически.
Решение: `test-sdx-stage.sh` получает **sanity-сценарий** — парсит из `protocol.md`
цепочки этапов из строк таблицы и сверяет их с матрицей (grep + `sed` разбор
`→`-цепочки, не полноценный парсер). _Пересмотр: изначально проектировался только для
трека `full` и как сверка множеств; по факту поставлен по всем трём трекам и **с учётом
порядка** — усилен на проходе 2 после находки W-5, поэтому ловит и перестановку этапов,
а не только добавление/удаление._

## Механика записи `stage` (атомарность)

Общая для `init`/`next`/`backtrack`/`retrack` функция:

```bash
write_stage() {   # $1 = новое значение stage
  local new="$1" tmp
  tmp="$(mktemp "${state}.XXXXXX")" || {
    echo "SDX sdx-stage: не удалось создать временный файл рядом с $state." >&2; exit 2; }
  if ! jq --arg s "$new" '.stage = $s' "$state" > "$tmp"; then
    rm -f "$tmp"
    echo "SDX sdx-stage: jq не смог обновить $state — файл НЕ изменён." >&2
    exit 2
  fi
  mv "$tmp" "$state"    # mktemp в той же директории, что $state -> тот же fs -> mv атомарен
}
```

`mktemp "${state}.XXXXXX"` намеренно создаёт временный файл **в том же каталоге**, что и
целевой (`.claude/sessions/<sid>/`), а не в `$TMPDIR` — иначе `mv` между разными
файловыми системами не гарантирует атомарность (может выродиться в copy+unlink с окном
частичной записи). Частичная запись невозможна: либо `jq` целиком отработал во временный
файл и `mv` целиком заменил оригинал, либо оригинал остался нетронутым.

**Поведение без `jq` — для самого скрипта, отказ, а не деградация.** В отличие от
`stage-write-guard.sh` (fail-open, REQ-DENY-4 — компромисс пользователя), сам
`sdx-stage.sh` **обязан отказать** (`exit 2`, файл не тронут) при отсутствии `jq` —
проверка `command -v jq` выполняется в самом начале скрипта, до разбора подкоманды:
```bash
command -v jq >/dev/null 2>&1 || {
  echo "SDX sdx-stage: jq не найден — переход отклонён (fail-closed), файл не изменён. Установите jq." >&2
  exit 2
}
```
Обоснование асимметрии: у `sdx-stage.sh` ЕСТЬ шанс защитить целостность state (это его
единственная работа), в отличие от read-only `stage-gate.sh`, которому терять код-фриз
безопаснее, чем ломать сессию отказом писать вообще ничего. У deny-хука шанс тот же, что
у `sdx-stage.sh` — но SPEC явно (REQ-DENY-4, решение пользователя 2026-07-20) выбрал для
НЕГО противоположный компромисс: см. «Deny-хук» ниже.

**Гонки/параллельные вызовы.** Не актуально при однопоточной модели «одна сессия — один
CLI в основном рабочем дереве» (ADR-012, нет параллельных write-путей к одному
`session_state.json` в рамках одной сессии). `mktemp` с суффиксом `XXXXXX` на случай
повторного вызова в одном и том же ходе не оставляет коллизий имён.

## Deny-хук `sdx/hooks/stage-write-guard.sh`

### Проводка (`hooks/hooks.json`)
Третья запись в **существующем** matcher-е `PreToolUse` / `Write|Edit|MultiEdit`
(та же матрица, что уже даёт `stage-gate.sh`) — не отдельный matcher:
```json
{
  "matcher": "Write|Edit|MultiEdit",
  "hooks": [
    { "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/sdx/hooks/stage-gate.sh" },
    { "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/sdx/hooks/stage-write-guard.sh" }
  ]
}
```
Порядок внутри массива не влияет на исход (если любой хук вернёт `deny`, вызов
блокируется) — важен только для диагностики (какой хук первым дал `deny` в логах).

### Резолюция сессии и цели
Использует общую `lib/resolve-session.sh` (см. ниже): branch → sid → путь к
`session_state.json`. Не-`sdx/*`-ветка → `exit 0` (REQ-DENY-3). Нормализация `\`→`/`
для `target`/`proj` — как в `stage-gate.sh` (BUG-006), поскольку сравнение путей должно
быть устойчиво к Windows.

```bash
target="$(printf '%s' "$input" | jq -r '.tool_input.file_path // empty')"
[ -z "$target" ] && exit 0
target="${target//\\//}"; proj="${proj//\\//}"
sid="$(resolve_sid "$proj")"           # lib/resolve-session.sh; empty если не sdx/*
[ -z "$sid" ] && exit 0
state="$proj/.claude/sessions/${sid}/session_state.json"
[ "$target" != "$state" ] && exit 0    # хук специфичен ИМЕННО к этому файлу
```

### Первичное создание — не блокируется (REQ-DENY-2)
```bash
[ -f "$state" ] || exit 0   # файла ещё нет -> это создание, не правка -> пропуск всегда
```
Это разрешает Write-創ание файла **в принципе**, но по факту первичное создание уже не
идёт через `Write` — `start.md`/`import.md` создают файл через `sdx-stage.sh init`
(вызов `Bash`, вне matcher-а хука в принципе). Проверка выше — **defense in depth**, а не
единственная линия: она нужна (а) чтобы буквально удовлетворить критерий приёмки SPEC
(«первичное создание... не блокируется» проверяется тестом через `Write`-сценарий), и
(б) на случай, если в будущем какой-то путь создания снова начнёт использовать `Write`
напрямую — хук не должен стать регрессией для этого класса операций.

### Три инструмента — единый принцип обнаружения (apply-and-compare)

> Актуализировано на Verification (проход 2, FAIL-1) по факту поставленного кода —
> см. сноску в начале документа.

Все три инструмента используют один и тот же принцип: получить полное содержимое
файла ДО и ПОСЛЕ операции, распарсить `.stage` из обоих и сравнить **значения**, а не
искать текстовые паттерны. Для `Write` содержимое «после» уже дано целиком инструментом;
для `Edit`/`MultiEdit` содержимое «после» получается СИМУЛЯЦИЕЙ — правка применяется к
реально прочитанному с диска содержимому `$state` тем же способом, каким её применил бы
настоящий инструмент.

| Инструмент | Источник содержимого «после» | Как получено |
|---|---|---|
| `Write` | `.tool_input.content` — **весь** новый текст файла | Дано целиком инструментом |
| `Edit` | `.tool_input.old_string`/`.tool_input.new_string` | Симулируется: `content_before` (диск) с применённой заменой `old_string → new_string` |
| `MultiEdit` | `.tool_input.edits[]` (массив правок) | Симулируется: правки применяются **последовательно** к одной и той же копии содержимого, как это делает настоящий `MultiEdit` |

Во всех трёх случаях итог один: `check_stage_change(before, after)` парсит `.stage` из
`before` и `after` через `jq` и денит **только если значение изменилось**. Ни один путь
не ищет подстроку `"stage"` в тексте правки — реализация полностью отказалась от
паттерн-поиска по ключу в пользу сравнения распарсенных значений.

**`Write` (точный путь):**
```bash
content="$(printf '%s' "$input" | jq -r '.tool_input.content // empty')"
new_stage="$(printf '%s' "$content" | jq -r '.stage // empty' 2>/dev/null)"
if [ $? -ne 0 ]; then
  deny "новое содержимое не распознано как валидный JSON — не могу доказать, что 'stage' не меняется"
fi
[ "$new_stage" != "$old_stage" ] && deny "..."
```
Не-парсящийся контент трактуется консервативно («не могу доказать, что `stage` не
меняется») → `deny`.

**`Edit`/`MultiEdit` (симулированный путь):**
```bash
literal_replace() {   # $1=content $2=search $3=replace $4=all(0|1)
  # awk index()/substr() — байт-в-байт ЛИТЕРАЛЬНАЯ (не regex/glob) замена первого
  # вхождения (или всех, если $4=1), в точности как это делает Edit-инструмент.
  # Пусто на stdout + exit 1, если $2 не найден в $1 (как и настоящий Edit).
}
content_before="$(cat "$state")"
if content_after="$(literal_replace "$content_before" "$old_string" "$new_string" "$replace_all")"; then
  check_stage_change "$content_before" "$content_after"
fi
# иначе: old_string не найден -> настоящий Edit сам провалился бы до записи -> нечего гейтить
```
Для `MultiEdit` та же `literal_replace` применяется **последовательно** к каждому
элементу `edits[]`, начиная от одного и того же `content_before` (каждая следующая
правка видит результат предыдущей — как в реальном инструменте); сравнение `.stage`
делается **один раз**, по итоговому содержимому после всех правок батча (`MultiEdit`
атомарен на уровне инструмента — либо применяются все правки, либо ни одной).

**Итог смены механизма.** Ложноположительный случай из прежней (регэксп-по-ключу)
версии — реформатирование пробелов вокруг `"stage"` без изменения значения — этим
устранён: `literal_replace` + сравнение распарсенных значений не блокирует правку, если
`.stage` фактически не изменился (`test-stage-write-guard.sh` сценарий 9 явно
утверждает `pass`, не `deny`, для этого случая — зафиксировано как контракт).

**Исправлено на Verification (проход 3, F-2) — три находки прохода 2, зафиксированы
как факт кода, а не как оставшиеся ограничения:**
- **Бывший WARN-1 (`literal_replace`/`tool_failure_deny`, `stage-write-guard.sh`).**
  `literal_replace` передаёт содержимое файла через переменную окружения awk
  (`ENVIRON["SWG_CONTENT"]`); при превышении лимита длины строки окружения ОС (Linux
  `MAX_ARG_STRLEN` ≈ 131072 байт) `awk`-процесс не запускается и завершается кодом,
  ОТЛИЧНЫМ от штатных `0`(найдено)/`1`(не найдено) `literal_replace`. Это больше не
  трактуется как «нечего гейтить»: любой третий код возврата ведёт в
  `tool_failure_deny()` → **`deny`** с явным сообщением о сбое инструмента («не могу
  доказать, что `stage` не меняется»), не пропуск. Смягчающее по-прежнему в силе:
  документированная схема файла — восемь коротких полей, такой размер недостижим при
  штатном использовании; но исход при недостижимом крайнем случае теперь консервативен
  (deny), а не тихий обход.
- **Бывший WARN-2 (`check_stage_change`/`is_valid_json`, `stage-write-guard.sh`).**
  Карве-аут «не блокировать восстановление уже повреждённого файла» проверяется теперь
  ТОЛЬКО по содержимому файла **ДО** операции (`is_valid_json(before)`), а не по
  требованию «и before, и after не парсятся». Типичный случай «файл сломан,
  `Edit`/`Write` его чинит» (before — не JSON, after — валидный ремонт) теперь
  пропускается (не денится), а не блокирует саму починку. `is_valid_json` определена
  узко и явно: не-JSON или пусто/пробельно на входе — считается повреждением; валидные
  скалярные JSON-документы `null`/`false` повреждением НЕ считаются (иначе файл,
  содержащий буквально `null`, ошибочно попадал бы в карве-аут и пропускал реальную
  правку `stage`) и защищены обычным путём сравнения. Единственный путь довести файл до
  состояния «уже повреждён» вне этого карве-аута — правка через `Bash`, уже принятая
  граница (DEBT-003); штатными `Write`/`Edit`/`MultiEdit` испортить файл невозможно
  (каждая из этих попыток сама денится).
- **Бывший WARN-3 (`normalize_path`, `stage-write-guard.sh`).** Сравнение цели с
  `$state` теперь применяет `normalize_path()` к обеим сторонам перед точным сравнением
  строк: сегменты `/./` и повторные `/` схлопываются. Путь с сегментом `/./` к тому же
  файлу **совпадает** с `$state` и хук срабатывает — это больше не обход.
  `normalize_path()` осознанно НЕ полный канонизатор: `..` не резолвится, относительный
  путь в абсолютный не превращается (для этого нужно было бы угадывать cwd, у хука
  нет надёжного способа его узнать) — эта граница остаётся реальной (см. ниже).

**Реально оставшиеся, не устранённые границы:**
- **Сегмент `..` не нормализуется.** `normalize_path()` схлопывает только `/./` и
  повторные `/`, не резолвит `..` — путь вида `.../s1/../s1/session_state.json`
  по-прежнему обходит хук (не совпадает с `$state` буквально). Практически
  малодостижимо: контракт инструментов `Write`/`Edit`/`MultiEdit` в Claude Code требует
  абсолютных путей без относительной навигации, но технически возможно.
- **Bash-обход (DEBT-003), уже принятая, не расширяемая этим решением граница.**
  `session_state.json` остаётся редактируемым напрямую через `Bash`
  (`sed -i`/`jq` in-place) — хук матчится только на `Write`/`Edit`/`MultiEdit`. Это
  единственный практически достижимый путь довести файл до состояния, которым
  пользуется карве-аут восстановления выше (расширение поверхности карве-аута
  относительно прохода 2 отмечено верификацией как W-4 — граница не новая по классу,
  но не была явно названа в артефактах до этого пересмотра).

Три исправления выше подтверждены на Verification прямым воспроизведением (аномально
большой файл → `deny`; two-step атака «сломать JSON → починить другим `stage`»
воспроизведена и не проходит; `/./`-путь → `deny`) — детали в `verification_report.md`.

### Отсутствие `jq` (REQ-DENY-4 — fail-open, решение пользователя 2026-07-20)
Проверка `jq` выполняется **после** того, как хук убедился, что операция вообще
затрагивает `session_state.json` конкретной SDX-сессии (та же оптимизация порядка, что
`prod-guard.sh` применяет к своему conf — не тратить фейл-клоуз-ветку там, где она не
нужна):
```bash
if ! command -v jq >/dev/null 2>&1; then
  echo "SDX stage-write-guard: jq недоступен — защита поля 'stage' временно ОТКЛЮЧЕНА (fail-open), правка session_state.json НЕ проверяется. Установите jq." >&2
  exit 0
fi
```
Асимметрия с `sdx-stage.sh` (там — fail-closed) осознанная и зафиксирована REQ-DENY-4:
цена самоблока (нельзя починить состояние без `jq`, единственный путь — тот самый обход,
которого решение пытается избежать) выше цены пропуска случайной правки в уже
деградированной среде. `preflight.sh` уже предупреждает об отсутствии `jq` на
`SessionStart`.

## Общая библиотека `sdx/hooks/lib/resolve-session.sh`

Discovery отметил дублирование резолюции сессии (branch → sid → путь state) в
`stage-gate.sh` (28–38) и `stop-gate.sh` (9–18). С добавлением ДВУХ новых скриптов,
которым нужна та же логика, копий стало бы четыре — порог для выноса пройден. Решение:
**выносим сейчас**, новые скрипты используют библиотеку с первого дня; существующие
`stage-gate.sh`/`stop-gate.sh` — **рекомендованный, не блокирующий** рефакторинг той же
сессии (поведение-сохраняющий, низкий риск, см. «Риски и границы»).

Паттерн — как `lib/default-branch.sh`, но **sourceable** (функция, не отдельный
подпроцесс — резолюция вызывается на каждый `PreToolUse`/`Stop`, это горячий путь;
`default-branch.sh` вызывается редко, на `archive`, там оверхед subshell'а не важен):

```bash
#!/usr/bin/env bash
# SDX session resolver — shared by hooks that gate on `.stage`.
# Source, don't execute: `. "$here/lib/resolve-session.sh"`.
# resolve_sid <proj>  -> echoes sid on stdout if on branch sdx/<id>, else empty.
resolve_sid() {
  local proj="$1" branch
  branch="$(git -C "$proj" branch --show-current 2>/dev/null || true)"
  case "$branch" in
    sdx/*) printf '%s' "${branch#sdx/}" ;;
    *)     printf '' ;;
  esac
}
```
`stage-gate.sh`/`stop-gate.sh`/`sdx-stage.sh`/`stage-write-guard.sh` при рефакторинге
делают `. "$here/lib/resolve-session.sh"; sid="$(resolve_sid "$proj")"` вместо inline
`case`. Библиотека **не** резолвит путь к state-файлу (это однострочник, не стоит
абстракции) — только branch→sid.

## Backward-переход (закрытие DEBT-011)

### Правила (REQ-BACKTRACK-1) — исполнимая форма
```
sdx-stage.sh backtrack <sid> <target>:
  1. matrix ⊇ union(track_stages) — target ∈ union?  нет -> "имя не распознано" (REQ-STAGE-4)
  2. target ∈ active_stages(current_track)?           нет -> "нужна смена трека: /sdx:retrack" (REQ-BACKTRACK-1)
  3. target == current_stage?                          да -> OK no-op, exit 0 (REQ-STAGE-4)
  4. index(target) > index(current_stage) в порядке трека?  да -> "не откат — target позже; используй /sdx:next"
  5. иначе (index(target) < index(current_stage)):    без проверки гейт-артефактов -> перейти к маркировке
```

### Маркировка outdated (REQ-BACKTRACK-2) — механизм
**Решение:** HTML-комментарий-баннер, вставляемый **первой строкой** артефакта, файл не
переименовывается и не перемещается — это прямое прочтение SPEC («файл не удалён, не
переименован в бэкап-копию, но явно помечен как неактуальный»):
```bash
mark_outdated() {   # $1 = path, $2 = target stage (для текста баннера)
  local f="$1" tgt="$2"
  [ -f "$f" ] || return 0
  head -c 200 "$f" | grep -q '<!-- SDX-OUTDATED' && return 0   # уже помечен -> идемпотентно
  local tmp; tmp="$(mktemp "${f}.XXXXXX")" || return 1
  { printf '<!-- SDX-OUTDATED: устарело откатом /sdx:backtrack --to "%s" (%s). Актуализируйте перед продолжением; история версии — `git log -p -- %s`. -->\n\n' \
      "$tgt" "$(date '+%Y-%m-%d %H:%M:%S')" "$f"
    cat "$f"; } > "$tmp" && mv "$tmp" "$f"
  echo "OUTDATED: $f"
}
```
**Скоуп маркировки (пересмотрено на Verification, проход 3, F-3):** артефакты **строго
после** `target` в порядке трека (эксклюзивно), **БЕЗ верхней границы** — до конца
активных этапов трека, а не только до текущего этапа. Прежняя формулировка этого
раздела («до текущего этапа включительно») устарела и не совпадает ни с кодом
(`cmd_backtrack` идёт по ВСЕМ оставшимся строкам матрицы после `target`, комментарий
в коде — «No upper bound here on purpose»), ни с `SPEC.md` REQ-BACKTRACK-2 («артефакты,
относящиеся к этапам после новой точки» — тоже без верхней границы). Мотив расширения:
артефакт этапа, который в порядке трека формально ПОЗЖЕ текущего `stage`, может
реально существовать на диске (например, остался от предыдущего цикла Verification,
пока `stage` был откинут раньше другим `backtrack`) — если его не пометить, читающий
его человек или агент примет устаревшее содержимое за актуальное. **Важно:** баннер
НЕ блокирует гейты — помеченный артефакт продолжает засчитываться как доказательство
и для `next`, и для `retrack` (тест «outdated-marked artifact still counts as
evidence»). Это сигнал, а не механическое ограничение; см. «Границы доказательности»
в `sdx/protocol.md`. _Прежняя формулировка этого абзаца утверждала обратное («молча
продолжит проходить forward-гейты как актуальный») — исправлено на проходе 5._

Та же матрица даёт список `(stage, artifact)` пар для прохода. Пример: откат с `Execution` на `Technical
Design` в треке `full` метит `PLAN.md` (этап `Task Planning`, строго после `Technical
Design`) — именно тот случай, который SPEC явно называет («включая PLAN.md, если откат
происходит на этап проектирования или раньше»): `Task Planning` лежит строго после
`Technical Design` в матрице, значит его артефакт естественно попадает в диапазон без
специального случая для `PLAN.md` — частный пример SPEC оказывается следствием общего
правила «строго после target, без верхней границы», а не отдельным правилом. Артефакт
САМОГО `target`-этапа (например, `DESIGN.md` при откате именно на `Technical Design`)
баннером **не** метится — пользователь возвращается ревизировать именно его, он не
«устарел», он — текущий предмет работы.
`change_note.md` (patch/standard) маркируется тем же механизмом, если откат происходит
до `Change`/`Execution`.

Идемпотентность (повторный `backtrack` на ту же цель) не дублирует баннер — проверка
`grep -q '<!-- SDX-OUTDATED'` в первых 200 байтах.

## Forward-skip guard в `retrack` (REQ-RETRACK-2, закрывает WARN-4)

> **Добавлено на Verification (проход 2).** Первоначальная поставка Execution не имела
> симметричной `backtrack`-у защиты в `retrack`: подкоманда проверяла только «имя
> распознано» + «активно в новом треке», без проверки «не позже текущего». Ревьюер
> воспроизвёл эксплойт: трек `full`, `stage=Discovery`, ни одного артефакта —
> `sdx-stage.sh retrack <sid> Closeout` возвращал `rc=0` и записывал `stage=Closeout`,
> минуя все гейты (WARN-4 в `verification_report.md`, второй проход). Требование
> `REQ-RETRACK-2` (Delta в `SPEC.md`) и первая версия кода ниже — фикс этой находки,
> сделанный на Verification, а не на Execution.

> **Пересмотр (Verification, проход 3, F-1).** Позиционное решение прохода 2
> (`SDX_CANON_ORDER`, ниже) само оказалось эксплуатируемым: ранг, единожды поднятый
> одним `retrack`, «запоминался» через самодекларируемую позицию `stage` и мог расти
> дальше вторым легитимным `retrack` без единого пройденного гейта — храповик, а не
> монотонный барьер (репро — `verification_report.md`, F-1). Причина корневая: ранг
> выводился из ТОГО, ГДЕ `stage` самодекларированно НАХОДИТСЯ, а не из того, что
> ДОКАЗАНО на диске — то есть из позиции, а не из доказательства. Раздел ниже переписан
> под фактический код (`sdx-stage.sh` `cmd_retrack`/`stage_artifact_ok`) —
> **evidence-based** правило без какой-либо ранговой шкалы; исходное позиционное
> решение сохранено ТОЛЬКО в следующем подразделе как маркированная история отменённого
> решения, а не как действующая механика.

### Проблема: `stage`/`target` могут принадлежать разным трекам

`REQ-BACKTRACK-1` решает «не позже текущего» простым сравнением позиций **внутри одного
и того же трека** (`matrix_index`). Для `retrack` это не работает буквально: `target`
проверяется в **новом** треке, а текущий `stage` может быть именем, которого в новом
треке вообще нет (например, `Change` не существует в `full`; `Business Spec` не
существует в `standard`/`patch`). Нужна проверка, не зависящая от того, откуда
пришли — а не единая межтрековая шкала позиций (см. пересмотр выше — именно шкала и
была источником храповика).

### Отменённое решение (проход 2, история): `SDX_CANON_ORDER` — общая каноническая шкала рангов

> Ниже — то, что было реализовано на проходе 2 и удалено на проходе 3 (F-1). Сохранено
> как история решения — почему позиционный подход был отвергнут. Таблица `SDX_CANON_ORDER`
> и функция `canon_rank`, на которых он основывался, **полностью удалены** из
> `sdx-stage.sh`; отдельный sanity-тест (`test-sdx-stage.sh`, «sanity: SDX_CANON_ORDER /
> canon_rank dead code fully removed») страхует от их случайного возвращения.

Отдельная (от `SDX_STAGE_MATRIX`) таблица `<имя-этапа>|<ранг>` внутри `sdx-stage.sh`,
дающая каждому имени этапа, встречающемуся хоть в одном треке, числовой ранг на общей
шкале:

```
Discovery|1
Business Spec|2
Technical Design|3
Change|3
Task Planning|4
Execution|5
Documentation|6
Verification|7
Deployment|8
Closeout|9
```

Правило было: `target` допустим, если `canon_rank(target) <= canon_rank(current_stage)`
(либо, если у нового трека нет активного этапа с рангом не выше текущего, потолком
становится собственный первый активный этап нового трека — «войти в трек с его
стартовой точки всегда можно»). Дефект: `canon_rank(current_stage)` читается из
самодекларированного `stage`, который сам мог быть поднят предыдущим `retrack` без
единого пройденного гейта — отсюда храповик F-1.

### Текущее решение: evidence-based guard (без ранговой шкалы)

Цель `target` в новом треке достижима, только если у **каждого** этапа этого трека,
предшествующего `target` в порядке `SDX_STAGE_MATRIX` (`matrix_index < idx_target`),
собственный гейт объективно подтверждён на диске — той же функцией
(`stage_artifact_ok`), которую `cmd_next` применяет к уходящему этапу: файл-артефакт
существует, непуст, и (где применимо) не содержит маркера `[FAIL]`. `stage`/старый трек
в этой проверке **не участвуют вовсе** — ни как источник ранга, ни как способ
клэмповать потолок. У собственного первого активного этапа трека цепочка
предшественников пуста по построению, поэтому войти в трек с его стартовой точки
всегда можно (тот же практический эффект, что давал клэмп прохода 2, но не через
самодекларируемую позицию, а как тривиальное следствие пустой цепочки); приземлиться
дальше требует тех же артефактов, которые потребовал бы реальный проход через
`/sdx:next` — кроме этапа `Change` (см. «Эквивалентность `Change`» ниже): он
засчитывается и по паре `SPEC.md`+`DESIGN.md`, тогда как `next` на patch/standard
требует именно `change_note.md`.

**Эквивалентность `Change`.** У объединённого этапа `Change` (patch/standard) нет
собственного файла-артефакта — его гейт считается подтверждённым, если существует
непустой `change_note.md` **ИЛИ** одновременно непустые `SPEC.md` и `DESIGN.md`. Это не
ослабление, а прямое следствие того, что уже верно по построению: `retrack.md` шаг 3
безусловно промоутит `change_note.md` в оба этих файла ПЕРЕД вызовом скрипта при
эскалации ИЗ `Change`; симметрично, сессия `full`, дошедшая (или прошедшая) своим ходом
до `Technical Design`, сделала строго БОЛЬШЕ, чем когда-либо требовал бы этап `Change` —
отказывать ей в зачёте этой работы при деэскалации в `standard`/`patch` было бы чистой
церемонией, а не защитой (закрывает и W-6 прохода 2: `Change` теперь достижим из
`Business Spec` через ЛЮБОЕ из двух доказательств, если фактически есть на диске, не
только через боковой переход с совпадающим рангом).

### Правило (исполнимая форма, по факту `cmd_retrack`)

```
sdx-stage.sh retrack <sid> <target>:
  1. matrix ⊇ union(track_stages) — target ∈ union?    нет -> "имя не распознано"
  2. target ∈ active_stages(новый_track)?               нет -> "не активен в треке"
  3. idx_target = позиция target в новом треке (matrix_index)
     для каждого этапа s нового трека с позицией i < idx_target (в порядке матрицы):
       stage_artifact_ok(s)?  нет -> "'<target>' недостижим — этап 's' не подтверждён"
       (для s == "Change": ok если непуст change_note.md ИЛИ (непуст SPEC.md И непуст DESIGN.md))
  4. target == current_stage?                            да -> OK no-op
     (проверяется ПОСЛЕ шага 3 намеренно — иначе цель, чьё ИМЯ совпадает с текущим stage,
      но трек которой реально сменился, проскакивала бы guard как "уже здесь", F-1 sub-finding)
  5. иначе: write_stage(target) — без проверки гейт-артефактов уходящего этапа и самого
     target (REQ-RETRACK-1 не меняется — это по-прежнему не forward-гейт)
```

Ключевое отличие от `backtrack` и от отменённого решения прохода 2: сравнение здесь
**артефактное**, не позиционное — `retrack` теперь проверяет существование
`context_report.md`/`SPEC.md`/`DESIGN.md`/... у всех этапов ПЕРЕД целью (это не
противоречит REQ-RETRACK-1: он по-прежнему не проверяет гейт самого `target` и не
проверяет гейт уходящего этапа — только цепочку предшественников новой цели). Guard
закрывает именно класс «попасть на этап, для которого физически не пройден ни один из
предшествующих гейтов», не завися от того, каким путём и сколькими шагами `retrack`
сессия туда добралась — двух легитимных вызовов подряд (F-1) больше недостаточно, т.к.
каждый вызов заново читает диск, а не предыдущий вывод самого себя.

### Пример: почему «клэмп к первому этапу» остаётся корректным без ранговой шкалы

`patch` не содержит `Discovery`/`Business Spec`/... вообще — его жизненный цикл
начинается сразу с `Execution` **по замыслу трека**, а не потому что что-то
пропущено. Деэскалация `full`(`Discovery`, без единого артефакта) → `patch` должна
суметь приземлиться на `Execution` — иначе `retrack` в `patch` вообще никогда не
работал бы для самого частого случая деэскалации. В evidence-based правиле это работает
без клэмпа как отдельного механизма: у `patch`/`Execution` нет ни одного предшествующего
активного этапа в матрице `patch` (`Execution` — первая строка), поэтому цикл шага 3
просто не итерирует ни разу — цель разрешена тривиально, тем же способом, каким любой
трек разрешает собственную стартовую точку. Приземлиться дальше — на
`patch`/`Verification` — цикл потребовал бы гейт `patch`/`Execution` (артефакт `-`,
то есть безусловно пройден по матрице) и т.д.; конкретно для `patch` это не блокирует,
но для `full`(`Discovery`, ноль артефактов) → `full`/`Task Planning` дальше того же
`retrack` цепочка потребует реально существующие `context_report.md`, `SPEC.md`,
`DESIGN.md` — которых на диске нет, поэтому цель отклоняется, а не проходит по ранее
поднятому рангу (это и есть исправление F-1: см. `test-sdx-stage.sh`, сценарии на
композицию `retrack`+`retrack`, которые прямо воспроизводят прежний эксплойт и
проверяют, что он теперь отклонён).

> `next` продвигает ровно на один этап, и код `0` не означает «дошли до `Closeout`».
> Одиночный вызов позволял начать Closeout-чек-лист на `Verification` — регресс гейта
> ADR-014, найденный ревью (F-1 прохода 1). Каждая итерация коммитится отдельно. Дисклоуз `auto_decisions.md` (для
`gate_mode: auto`) остаётся прозаическим шагом **до** вызова скрипта, без изменений —
это не артефактный гейт, скрипт про него ничего не знает и не должен.

## Развилки, зафиксированные архитектурным решением

### Развилка: создание `session_state.json` — через `sdx-stage.sh init` или карве-аут в хуке
**Выбрано: оба, но с разной ролью.** `start.md`/`import.md` создают файл через
`sdx-stage.sh init` (Bash) — это делает REQ-STAGE-1 («ни одна команда не пишет `stage`
напрямую прозой») истинным буквально, не только по факту недостижимости деном. Карве-аут
«файла ещё нет → хук пропускает» в `stage-write-guard.sh` остаётся **дополнительно**,
как defense-in-depth и потому что явно проверяется тестом/критерием приёмки через
`Write`-сценарий (REQ-DENY-2). Отвергнутая альтернатива — «только карве-аут в хуке, без
`init`-подкоманды» — оставляла бы `start.md`/`import.md` прозаическими писателями по
факту (просто необнаружимыми деном), что противоречит духу REQ-STAGE-1, а не только
букве REQ-DENY-2.

### Развилка: `next` — с явной целью или без
**Выбрано: без.** Матрица — единственный источник истины (REQ-STAGE-3) внутри скрипта;
если бы `next.md` вычислял и передавал целевой этап явно, знание о порядке этапов
дублировалось бы в прозе `next.md` — именно то дублирование, которое REQ-STAGE-3 просит
устранить. Побочный эффект: повторный вызов `next` подряд не «идемпотентен» в буквальном
смысле REQ-STAGE-4 (нет цели для сравнения) — но безопасен: второй вызов проверяет гейт
УЖЕ нового текущего этапа и естественно отклоняется, если его гейт-артефакт ещё не
готов (не проскакивает вперёд молча). REQ-STAGE-4 в буквальной форме («цель равна
текущему») применяется к `backtrack`/`retrack`, где явная цель есть по смыслу команды.
Собственный idempotent-кейс `next`: текущий этап уже последний в треке (`Closeout`) →
`exit 0` no-op (нечего продвигать дальше).

### Развилка: кто вычисляет цель `retrack`
**Выбрано: оркестратор (прозаически), скрипт только валидирует.** Соответствие
«`Change` ≈ `Business Spec`+`Technical Design` слитно» — семантическое, не позиционное:
нельзя вывести его из одной только матрицы позиций без знания, ЧТО было фактически
сделано (промоушен `change_note.md` в `SPEC.md`/`DESIGN.md` — контентная операция,
уже описанная в `retrack.md` шаг 3, не меняется этим решением). Это ровно то, что SPEC
относит к «смысловым условиям, остающимся суждением оркестратора» (REQ-STAGE-2
«Ограничение»). Скрипт лишь не даёт суждению приземлиться на невалидный/неактивный в
новом треке этап — объективная часть.

## Схема данных / API — сводка контрактов

| Скрипт | Тип | Вход | Выход |
|---|---|---|---|
| `sdx-stage.sh init` | Bash CLI | 6 позиционных аргументов | `exit 0/2`, stdout `OK - -> <stage>` |
| `sdx-stage.sh next` | Bash CLI | `<sid>` | `exit 0/1/2`, stdout/stderr см. выше |
| `sdx-stage.sh backtrack` | Bash CLI | `<sid> <target>` | `exit 0/1/2` + `OUTDATED:` строки |
| `sdx-stage.sh retrack` | Bash CLI | `<sid> <target>` | `exit 0/1/2` |
| `stage-write-guard.sh` | PreToolUse hook (`Write\|Edit\|MultiEdit`) | JSON на stdin | JSON `permissionDecision` на stdout, всегда `exit 0` (тот же контракт, что `stage-gate.sh`/`prod-guard.sh` — НЕ `exit 2`) |

Схема `session_state.json` **не меняется** (REQ-COMPAT-1) — ни `init`, ни какая-либо
другая подкоманда не добавляет новых полей (в частности, маркировка outdated НЕ живёт в
JSON, см. «Backward-переход»).

## Обработка ошибок и Граничные случаи

- Вне ветки `sdx/<id>` — `stage-write-guard.sh` no-op (REQ-DENY-3); `sdx-stage.sh` не
  резолвит ветку вообще (принимает `sid` явным аргументом от вызывающей команды,
  симметрично `archive-verify.sh`) — вызывать его вне контекста активной сессии
  бессмысленно, но технически безопасно (просто не найдёт state-файл → `exit 2`).
- `session_state.json` отсутствует при `next`/`backtrack`/`retrack` → `exit 2`,
  сообщение направляет на `/sdx:start`/`/sdx:import`.
- Повторный `init` на уже существующий файл → `exit 2` (защита от случайной
  реинициализации активной сессии).
- Нет `jq`: `sdx-stage.sh` — fail-closed, `exit 2`, файл не тронут (REQ явно к скрипту
  из постановки задачи). `stage-write-guard.sh` — fail-open с громким предупреждением
  (REQ-DENY-4).
- Невалидный JSON в `Write`-контенте (deny-хук) — консервативно трактуется как deny (не
  можем доказать, что `stage` не меняется).
- Windows backslash-пути — нормализуются в `stage-write-guard.sh` тем же паттерном, что
  `stage-gate.sh` (BUG-006 прецедент), до сравнения с `$state`.
- Идемпотентные вызовы (`target == current` для `backtrack`/`retrack`; `next` на
  терминальном `Closeout`) — успешный no-op, файл не трогается вообще (даже `mtime` не
  меняется — `write_stage` не вызывается на этой ветке).
- Существующие открытые сессии (включая саму `fw-stage-guard-20260720`) — работают без
  миграции (REQ-COMPAT-1): следующий вызов `/sdx:next`/`backtrack`/`retrack`/`archive`
  после мёржа этой сессии в `main` обязан пойти через `sdx-stage.sh` — деном заблокирует
  прежний прозаический путь, если кто-то по инерции попробует править `stage` через
  `Edit`.

## Безопасность

- **Граница намеренного обхода явно не расширяется.** `session_state.json` остаётся
  редактируемым через `Bash` (DEBT-003) — deny покрывает только `Write`/`Edit`/
  `MultiEdit`. `sdx-stage.sh`, вызванный напрямую с произвольным `<target>`, не проверяет
  «кто и почему» его вызвал — только готовность артефактов. Это унаследованная, явно
  принятая граница (SPEC «Явная граница ответственности»), не новый риск.
- **Экранирование.** `deny()` в `stage-write-guard.sh` — тот же `jq -Rs .`-паттерн, что
  `stage-gate.sh`/`prod-guard.sh` (инъекция в `permissionDecisionReason` исключена).
- **Атомарность мутаций.** `write_stage`/`mark_outdated` используют `mktemp` в целевой
  директории + `mv` — нет окна частичной записи, читаемой другим процессом/ходом.
- **Деструктив.** Backward-переход намеренно НЕ деструктивен (REQ-BACKTRACK-2) — ни один
  путь `sdx-stage.sh` не удаляет и не зануляет содержимое файлов; git-история сессии —
  сеть безопасности (ADR-012, REQ-SESS-1).

## Решения и обоснования (ADR)

### ADR-016 (проект для `docs/DECISIONS.md`) — Единственный писатель `stage` + deny-хук (закрытие DEBT-001, DEBT-011)

- **Контекст.** `stage` в `session_state.json` писали шесть прозаических путей без
  единообразной проверки гейта; `backtrack` не проверял вообще ничего и содержал
  нерешённую развилку («бэкап или обнуление» PLAN.md) — DEBT-001 + DEBT-011.
- **Решение.** `sdx/hooks/sdx-stage.sh` (4 подкоманды: `init|next|backtrack|retrack`) —
  единственный писатель, со встроенной машиночитаемой матрицей «трек → активные этапы →
  гейт-артефакт» (REQ-STAGE-3). `sdx/hooks/stage-write-guard.sh` — новый PreToolUse-хук
  (`Write|Edit|MultiEdit`), блокирующий прямую правку поля `stage`, узко (REQ-DENY-2:
  только `stage`, не весь файл; создание нового файла не блокируется). Backward-переход
  помечает артефакты строго после целевой точки HTML-баннером в самом файле (без
  бэкапа/зануления, REQ-BACKTRACK-2). `sdx-stage.sh` fail-closed без `jq`;
  `stage-write-guard.sh` fail-open без `jq` (REQ-DENY-4, явное решение пользователя,
  пересмотр исходной формулировки).
- **Обоснование.** Единая точка входа — предпосылка для доказуемого гейта;
  деном-граница технически закрывает случайный (не намеренный) обход тем же инструментом,
  каким писали раньше. Матрица внутри скрипта устраняет дублирование знания о порядке
  этапов, остававшееся ранее только в прозе `protocol.md`+`next.md`.
- **Инварианты.**
  - Ни одна команда `/sdx:*` не пишет `.stage` в `session_state.json` напрямую
    Write/Edit/MultiEdit-инструментами — только через `sdx-stage.sh`.
  - Deny-хук специфичен полю `stage`; правки `track`/`gate_mode`/`artifacts`/`history`
    остаются легитимными без прохождения через скрипт.
  - Матрица «трек → этапы → гейт-артефакт» существует в ЕДИНСТВЕННОМ машиночитаемом
    месте — внутри `sdx-stage.sh`; `protocol.md` — человекочитаемая проекция.
  - Backward-переход не деструктивен: маркировка — банер в файле, не удаление/бэкап.
  - `retrack` не приземляет `stage` дальше, чем позволяют фактические гейт-артефакты на
    диске (REQ-RETRACK-2, добавлено на Verification, закрывает WARN-4; переписано на
    evidence-based правило на Verification, проход 3, F-1 — см. пересмотр в начале
    документа): целевой этап нового трека допустим, только если у КАЖДОГО
    предшествующего ему в матрице этапа гейт-артефакт объективно подтверждён на диске;
    самодекларируемая позиция `stage`/старый трек в проверке не участвуют. Форвард-
    движение мимо гейта остаётся исключительно прерогативой `next`.
  - Схема `session_state.json` не меняется (REQ-COMPAT-1).
- **Связь.** Закрывает DEBT-001, DEBT-011. Пересекается, но не решает DEBT-003
  (Bash-обход) и DEBT-007 (`status`/`artifacts`/`history` мёртвые поля) — оба явно
  зафиксированы как унаследованные границы. Расширяет паттерн `deny()` из ADR-1
  (`docs/designs/phase1-enforcement-routing.md`) на новый класс операции (диф значения
  поля, а не только путь файла).

_(Финальный номер/включение в `docs/DECISIONS.md` — на Closeout этой сессии, ADR
проектируется здесь по образцу существующих записей.)_

7. `MultiEdit`, где хотя бы одна из последовательно применённых правок (apply-and-compare,
   симулированных на реальном содержимом файла) в итоге меняет распарсенное значение
   `.stage` → `deny` для ВСЕЙ операции целиком (даже если остальные правки в пачке сами
   по себе легитимны — `MultiEdit` атомарен на уровне инструмента).
8. Без `jq` в `$PATH`, при этом операция реально правит `stage` файла существующей
   сессии → пропуск (`exit 0`) + предупреждение в stderr, а НЕ deny (REQ-DENY-4).
9. **(Пересмотрено, FAIL-1 — было «документируем ожидаемый false positive», стало
   «false positive устранён».)** `Edit`, реформатирующий пробелы вокруг `"stage"` БЕЗ
   изменения самого значения (например, `"stage":"Execution"` → `"stage" :  "Execution"`)
   → **`pass`** (apply-and-compare парсит одинаковое значение до/после и не денит) — это
   прямое, ожидаемое следствие перехода от поиска подстроки ключа к сравнению
   распарсенных значений, а не регрессия; фиксируется тестом как контракт.
10. Backslash-путь (Windows-style) к `session_state.json` — деном срабатывает так же,
    как с forward-slash путём (нормализация, BUG-006-паттерн).

Оба сьюта подхватываются `.claude/sdx/verify-cmd.sh` автоматически (глоб `test-*.sh`,
без правки самого `verify-cmd.sh`).

## Риски и границы

1. **Что ломается у существующих открытых сессий.** Ничего структурно — схема файла не
   меняется (REQ-COMPAT-1). Единственное наблюдаемое изменение: со следующего
   `/sdx:next`/`backtrack`/`retrack`/`archive` **после** мёржа этой сессии в `main`,
   прямая прозаическая правка `stage` (по старой памяти модели, если она «помнит» старый
   паттерн из предыдущего контекста) будет заблокирована деном — ожидаемо, штатный путь
   (`sdx-stage.sh`) даёт тот же результат за один Bash-вызов вместо Edit.
2. **Ложное срабатывание deny — эскейп-хетч.** Единственный путь при реальном ложном
   срабатывании (например, кейс 9 теста выше, если он проявится не в тесте, а в
   реальной сессии на нетипичном контенте) — прямая правка через `Bash`
   (`sed -i`/`jq` in-place). Это **тот же самый** путь, что уже задокументирован как
   осознанно принятая граница DEBT-003 — решение не расширяет и не сужает эту границу,
   лишь использует её как явный, названный резервный клапан. Столкновение с ложным
   срабатыванием — сигнал по протоколу «Непрерывное улучшение»: озвучить пользователю
   одной строкой, предложить `/sdx:backlog add`.
3. **Самоприменение к текущей сессии.** `fw-stage-guard-20260720` сама попадёт под
   деном сразу после Execution этой сессии (как только `stage-write-guard.sh` окажется
   подключён через `hooks/hooks.json`, что происходит уже на Execution, т.к.
   `hooks.json` — код, пишется в Execution) — дальнейшие переходы ЭТОЙ ЖЕ сессии
   обязаны идти через новый `sdx-stage.sh`, а не прежним прозаическим Edit-путём. Учтено
   при формулировке PLAN.md (задача на переход к Verification должна сама пройти через
   новый механизм — «ест собственную еду» сразу).
4. **Рекомендованный рефакторинг `stage-gate.sh`/`stop-gate.sh` на `lib/resolve-session.sh`
   — не блокирующий.** Если Execution не успевает/не считает нужным его сделать в
   рамках этой сессии, это НЕ нарушает ни один REQ (оба скрипта продолжают работать как
   есть, дублирование остаётся, но не растёт дальше двух старых копий бесконтрольно —
   новые скрипты уже используют общую библиотеку). Явно не блокирует критерии приёмки.
5. **DEBT-007 остаётся открытой дырой.** `status: approved` для гейта
   `Business Spec → Technical Design` по-прежнему непроверяем — `sdx-stage.sh` не
   пытается его проверить (пустой `-` в матрице для этого условия), риск принят
   осознанно, идентично формулировке SPEC.
6. **DEBT-003 не решается, явно не расширяется.** `session_state.json` остаётся
   Bash-редактируемым; deny покрывает только `Write`/`Edit`/`MultiEdit`, что зафиксировано
   и в SPEC, и здесь дважды (см. п.2 выше и «Безопасность»).
