# Context Report: DEBT-001 — детерминированный владелец поля `stage`

## 0. Резюме задачи
DEBT-001: enforcement (`stage-gate.sh`, `stop-gate.sh`) читает `stage` из
`.claude/sessions/<id>/session_state.json`, но это поле сегодня пишет прозой
(инструкцией в `commands/*.md`) сама модель — никакого детерминированного барьера
между «оркестратор решил перейти на следующий этап» и «файл состояния изменился»
нет. Подтверждённый пользователем объём решения — две части:
1. `sdx/hooks/sdx-stage.sh <session> <new-stage>` — единственный писатель `stage`,
   сам проверяющий гейт-артефакты перехода.
2. PreToolUse-deny на `Write|Edit|MultiEdit` по `session_state.json` — правки
   состояния только через скрипт (Bash), не напрямую.

Ниже — фактура для DESIGN, без проектных решений.

## 1. Кто сейчас пишет `session_state.json` (и что именно)

Все точки записи — **прозаические инструкции** в `commands/*.md`, исполняемые
основной сессией (роль Session Manager). Ни одна не идёт через скрипт.

| Файл:строка | Команда | Что пишет |
|---|---|---|
| `commands/start.md:26` | `/sdx:start` | Создаёт файл целиком: `type`, `track` (по триажу), стартовый `stage` (`Execution` для `patch`, `Discovery` для `standard`/`full`), `gate_mode` (`auto` при `--auto`, иначе `interactive`), `git_branch`. Коммитится шагом 6 (`sdx(<id>): init session state`). |
| `commands/import.md:13` | `/sdx:import` | Аналогично создаёт файл с `type: import`, трек по умолчанию `full` (или `standard`). |
| `commands/next.md:27` (+ commit на строке 29) | `/sdx:next` | Прямая правка `stage` на «следующий активный этап трека» **после** прозаической проверки гейта (шаг 3). Коммит `session_state.json` + `session.log` одним ходом. Это основной, «штатный» писатель. |
| `commands/backtrack.md:12-13` | `/sdx:backtrack --to <stage>` | Прямая правка `stage` на **произвольный** этап, указанный пользователем. Проверка на шаге 1 — только «этап активен в треке», НЕ проверка гейт-артефактов вперёд/назад. Это самый слабый писатель — по сути открытый unset любого этапа. |
| `commands/retrack.md:19` | `/sdx:retrack <track>` | Пишет одновременно `track` (новый) и `stage` («как правило текущий этап, если активен в новом треке; иначе ближайший активный»). При эскалации дополнительно сбрасывает `gate_mode` в `interactive`. |
| `commands/archive.md` (шаг 2, «Гейт входа в Closeout») | `/sdx:archive` | Переводит `stage` в `Closeout` **до** начала чек-листа закрытия, после гейта входа (patch: `verification_report.md` без FAIL; auto-mode: дисклоуз `auto_decisions.md` с подтверждением). Ещё один прямой писатель, отдельный от `/sdx:next`. |

**Поля кроме `stage`:**
- `track` — пишется при создании (`start`/`import`) и при `/sdx:retrack`.
- `gate_mode` — пишется при создании (`start --auto`), может предлагаться к
  включению оркестратором на первом гейте (`next.md` шаг 2а, через
  `AskUserQuestion`), сбрасывается в `interactive` при эскалации трека
  (`retrack.md`).
- `type`, `git_branch`, `session_id` — write-once при создании, далее только
  читаются.
- `status` (`draft|review|approved|executing`) — объявлено в схеме
  (`protocol.md:12`) и упомянуто **один раз** как условие гейта Business Spec
  (`next.md:18`: «`SPEC.md` и подтверждение пользователя (`status: approved`)»),
  но **ни одна команда не пишет это поле** — grep по `commands/`, `agents/`,
  `sdx/` не находит записи `status` нигде, кроме этого упоминания и схемы.
  Условие гейта сегодня фактически непроверяемо даже прозой — это то же самое
  наблюдение, что и в DEBT-007 (см. §7).
- `artifacts`, `history` — объявлены в схеме (`protocol.md:15-16`), нигде не
  читаются и не пишутся (подтверждено grep) — состав артефактов и история по
  ADR-005 выводятся из git. Тоже DEBT-007.

**Вывод:** сегодня минимум **5 разных команд** (`start`, `import`, `next`,
`backtrack`, `retrack`, `archive` — фактически 6) прямо мутируют `stage` каждая
своей прозаической логикой гейта (или без неё, как `backtrack`). Единый скрипт
должен покрыть все шесть путей записи, а не только линейный `/sdx:next`.

## 2. Кто читает `stage`

Только два детерминированных потребителя (оба — hooks, plugin-shipped,
`${CLAUDE_PLUGIN_ROOT}/sdx/hooks/`):

- **`sdx/hooks/stage-gate.sh`** (PreToolUse `Write|Edit|MultiEdit`, `hooks/hooks.json:15-21`):
  - Резолвит сессию по имени текущей git-ветки (`sdx/<id>`, строки 28-32); не
    ветка `sdx/*` → transparent no-op (exit 0).
  - Читает `.claude/sessions/<sid>/session_state.json`, `jq -r '.stage // empty'`
    (строки 35-38); файл отсутствует или `stage` пуст → no-op.
  - `Execution|Deployment` → всегда разрешено (строка 42).
  - `Verification` → дополнительно разрешены тестовые каталоги (строки 53-57).
  - Иначе → `docs/*`, `.claude/*`, `*.md` разрешены всегда (строка 61); прочее —
    сверка с per-project `stage-gate.allow` (строки 65-73); иначе `deny`.
- **`sdx/hooks/stop-gate.sh`** (Stop, `hooks/hooks.json:33-41`):
  - Та же резолюция сессии по ветке (строки 9-13) и чтение `stage` (строки 16-18).
  - Enforcement (тест-пол) активен только на `Execution|Verification`, если не
    форсировано `SDX_STOP_GATE=1` (строки 20-26).
- **`archive-verify.sh`** — `stage` НЕ читает вообще; инварианты 1/5/6 проверяются
  только через git (чистое дерево, ветка слита, каталог не tracked). Значит,
  DEBT-001 не влияет напрямую на archive-verify — но `archive.md` пишет
  `stage: Closeout` прозой перед его вызовом, и это тоже надо провести через
  будущий скрипт для консистентности писателя.
- **`prod-guard.sh`** — `stage` не читает вообще (независимая ось, Bash-команды
  по regex-паттернам).

Остальные «читатели» — прозаические: `status.md`, `switch.md`, `verify.md`,
`next.md`, `backtrack.md`, `retrack.md`, `archive.md` читают `stage`/`track`
инструкцией (LLM Read), это не enforcement, а информирование пользователя/самого
оркестратора.

## 3. Матрица переходов по трекам и проверяемость гейтов

Треки и активные этапы — `protocol.md:51-55`. Ниже — какие гейты **объективно
проверяемы файлом/паттерном** (годятся для детерминированного скрипта) и какие
сегодня оценивает только прозой оркестратор (см. `next.md:16-24`).

### patch: `Execution → Verification → Closeout`
| Переход | Условие по next.md | Проверяемость |
|---|---|---|
| `Execution → Verification` | «правка завершена и зафиксирована в `change_note.md`» | Частично: наличие/непустота `change_note.md` — проверяемо; «завершена» — прозаическое суждение. |
| `Verification → Closeout` | `verification_report.md` без FAIL (лёгкая верификация обязательна, ADR-014) | Объективно: файл существует + grep на маркер `FAIL` (формат отчёта задаёт `reviewer`, см. `protocol.md:172-175` — нужно зафиксировать точный маркер в DESIGN). |
| Backward: `Verification → Execution` (через `/sdx:backtrack --to Execution`) | Явно упомянуто в комментарии `stage-gate.sh:50` как путь исправления FAIL-находок | Скрипту нужен отдельный (более мягкий) режим для «отката» — не тот же чек-лист, что вперёд. |

### standard: `Discovery (лёгкий, инлайн) → Change → Execution → Verification → Closeout`
| Переход | Условие по next.md | Проверяемость |
|---|---|---|
| `Discovery → Change` | «зафиксированных результатов инлайн-исследования» — **никакого обязательного файла не названо** | Не проверяемо файлом сегодня — целиком прозаическое суждение оркестратора. Открытый вопрос для DESIGN: либо оставить непроверяемым (скрипт пропускает гейт-чек для этого перехода), либо ввести минимальный артефакт. |
| `Change → Execution` | `change_note.md` с описанием решения | Существование/непустота — проверяемо; смысловая полнота — нет. |
| `Execution → Verification` | Как у patch | Как у patch. |
| `Verification → Closeout` | Как у patch | Как у patch. |

### full: `Discovery → Business Spec → Technical Design → Task Planning → Execution → Documentation → Verification → Deployment → Closeout`
| Переход | Условие по next.md | Проверяемость |
|---|---|---|
| `Discovery → Business Spec` | наличие `context_report.md` | Объективно: файл существует. |
| `Business Spec → Technical Design` | `SPEC.md` + `status: approved` | `SPEC.md` существует — проверяемо; `status: approved` — **сегодня никем не пишется** (см. §1/§7, DEBT-007) — значит объективно непроверяемо, пока поле не специфицировано. |
| `Technical Design → Task Planning` | `DESIGN.md`, «согласованный со SPEC.md» | Файл существует — проверяемо; согласованность с SPEC — это ровно ось когерентности, которую сегодня проверяет только `reviewer` на Verification, а не на этом гейте. Открытый риск: скрипт может проверить только наличие файла, не смысловую когерентность. |
| `Task Planning → Execution` | `PLAN.md` с декомпозицией | Файл существует — проверяемо; декомпозиция — нет (формат `PLAN.md` не специфицирован машиночитаемо — нет стандартных чекбоксов/схемы). |
| `Execution → Documentation` | «все задачи выполнены… отмечены в PLAN.md» | Потенциально проверяемо, ЕСЛИ `PLAN.md` использует единообразный чекбоксовый формат (`- [ ]`/`- [x]`) — сегодня это не гарантировано никакой схемой/шаблоном. Нужно решение в DESIGN: либо зафиксировать формат PLAN.md как контракт для скрипта, либо не проверять. |
| `Documentation → Verification` | «соответствующие артефакты этапа готовы» | Не специфицировано какие именно артефакты — целиком прозаично. |
| `Verification → Deployment` | `verification_report.md` без FAIL; WARN квитированы (интерактивно или авто через `auto_decisions.md`) | FAIL-отсутствие — проверяемо; квитирование WARN в auto-режиме — проверяемо (запись в `auto_decisions.md`); в interactive-режиме квитирование — только через диалог с пользователем, скрипт не может это верифицировать сам (артефакта-доказательства нет). |
| `Deployment → Closeout` | «артефакты этапа готовы» | Не специфицировано — прозаично. |

### Общий вывод по гейтам
Объективно проверяемые файлом гейты сегодня: наличие `context_report.md`,
`SPEC.md`, `DESIGN.md`, `PLAN.md`, `change_note.md` (существование/непустота) и
отсутствие `FAIL` в `verification_report.md`. Остальные условия («согласовано»,
«завершена», «готовы», `status: approved`) — прозаическое суждение оркестратора,
которое скрипт **не может** объективно проверить без дополнительной
спецификации. Это прямое ограничение объёма `sdx-stage.sh`: он способен
гарантировать «файл существует / не пуст / нет маркера FAIL», но не «решение
согласовано по смыслу» — тот же зазор, что и у enforcement-слоя в целом
(прото-инвариант vs проза, см. `protocol.md` Enforcement-слой).

Также важно: **`/sdx:backtrack`** сегодня переводит `stage` на произвольный этап
без проверки гейта вообще (только «этап активен в треке») — значит новый скрипт
должен явно поддерживать два разных режима вызова: forward-transition (с
проверкой гейта уходящего этапа) и backward-transition/reset (без гейта или с
более мягким набором проверок). DEBT-011 (backtrack недоспецифицирован) прямо
пересекается здесь.

## 4. Существующая инфраструктура хуков

- **Проводка**: `hooks/hooks.json` — единый файл плагина с `SessionStart`
  (`preflight.sh`), `PreToolUse` (два matcher-а: `Write|Edit|MultiEdit` →
  `stage-gate.sh`, `Bash` → `prod-guard.sh`), `Stop` (`stop-gate.sh`). Пути через
  `"${CLAUDE_PLUGIN_ROOT}"/sdx/hooks/<script>.sh`. Для нового deny-правила на
  `session_state.json` естественное место — третий hook в том же
  `PreToolUse`/`Write|Edit|MultiEdit` matcher-е (доп. entry в массиве `hooks`
  того же matcher-а, или отдельный matcher — оба варианта валидны по формату
  Claude Code hooks).
- **Механизм блокировки** (единый для всех PreToolUse-хуков, задокументирован
  `protocol.md:124`): JSON на stdout
  `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"<причина>"}}`,
  **exit 0** (НЕ exit 2 — явно закомментировано «подтверждено на бинарнике
  Claude Code 2.1.195» в `stage-gate.sh:4` и `prod-guard.sh:4`). Новый deny-хук
  на `session_state.json` должен следовать тому же паттерну `deny()`.
- **Общие библиотеки**: `sdx/hooks/lib/default-branch.sh` — единственная
  разделяемая библиотека сегодня (резолюция default-ветки в 3 шага: symbolic-ref
  `origin/HEAD` → `init.defaultBranch` → эвристика main/master). Используется
  `archive-verify.sh` и (документированно) прозой в `verify.md`. Прецедент для
  вынесения общей логики резолюции сессии (branch → sid → state-path), которая
  сейчас **дублируется** внутри `stage-gate.sh` (строки 28-38) и `stop-gate.sh`
  (строки 9-18) — кандидат на вынос в `lib/`, если `sdx-stage.sh` и новый
  deny-хук её тоже используют.
- **Деградация в no-op**: единообразный паттерн во всех хуках — не
  `sdx/*`-ветка → exit 0; нет `session_state.json` → exit 0; пустой `stage` →
  exit 0. Единственное исключение — `prod-guard.sh`, которое при
  сконфигурированной защите и отсутствующем `jq` уходит в **fail-closed**
  (блокирует). `preflight.sh` на `SessionStart` предупреждает при отсутствии
  `jq`, но не отказывает жёстко (закомментированный `exit 2`).
- **Зависимость от `jq`**: все PreToolUse-хуки и логика чтения `stage` завязаны
  на `jq -r`. Новый `sdx-stage.sh` и deny-хук по аналогии тоже будут зависеть от
  `jq` — стоит явно решить в DESIGN его поведение при отсутствии `jq` (по
  аналогии с существующими: deny-хук на запись состояния логично делать
  fail-closed, т.к. у него есть шанс защитить целостность state, в отличие от
  read-only stage-gate).
- **Per-project конфиги** (`.claude/sdx/`): `prod-guard.conf`,
  `stage-gate.allow`, `verify-cmd.sh`, разворачиваются `/sdx:init` из
  `sdx/templates/`. У `stage-gate.allow` есть шаблон
  (`sdx/templates/stage-gate.allow`) — если новому deny-хуку понадобится
  per-project allow-list (например, разрешить прямую правку `session_state.json`
  для отладки/миграции старых сессий), это тот же паттерн: файл в
  `.claude/sdx/`, шаблон в `sdx/templates/`.

## 5. Тестовая инфраструктура

- Каждый hook-скрипт имеет соседний `sdx/hooks/test-<script>.sh` (self-contained
  bash, без внешних фреймворков): `test-stage-gate.sh` (192 строки),
  `test-stop-gate.sh` (294), `test-prod-guard.sh` (170), `test-archive-verify.sh`
  (344), `test-default-branch.sh` (126).
- **Стиль** (по `test-stage-gate.sh`): хелперы `pass()/fail()` со счётчиками,
  `setup_sdx_repo <branch> <stage>` создаёт временный git-репозиторий
  (`mktemp -d`), инициализирует ветку `sdx/<sid>` и пишет
  `.claude/sessions/<sid>/session_state.json` вручную (`printf '{"stage":"%s"}'`
  — то есть тесты не зависят от реальных команд создания состояния), `run_hook`
  пайпит JSON stdin в скрипт с `CLAUDE_PROJECT_DIR=$TMPPROJ`, `cleanup()` чистит
  временный каталог. В конце — сводка `Results: N passed, M failed`, `exit 0`
  при всех PASS, иначе `exit 1`.
- **Запуск**: `.claude/sdx/verify-cmd.sh` (per-project, этот репозиторий
  дэфудит сам себя — DEBT-004, закрыт мёржем `fw-dogfood-verifycmd-20260720`) —
  глоб `sdx/hooks/test-*.sh`, гоняет все по очереди, фейлится при первом же
  `FAIL`; **новый `test-sdx-stage.sh` подхватится автоматически** (glob, не
  хардкод-список) — писать его нужно строго по этому шаблону, чтобы попасть в
  прогон без правки `verify-cmd.sh`.
- Для `sdx-stage.sh` тестовый сценарий будет сложнее, чем у read-only хуков: там
  придётся не просто читать `session_state.json`, а **изменять его и
  проверять результат** (новый `stage`, идемпотентность при повторном вызове,
  отказ при отсутствующих гейт-артефактах, коды возврата для success/gate-fail/
  invalid-transition) — стоит заложить в DESIGN отдельные fixtures для каждого
  трека (patch/standard/full) с разным набором присутствующих/отсутствующих
  артефактов.

## 6. Риски и открытые вопросы (для Business Spec / Technical Design)

1. **`/sdx:start` создаёт файл впервые.** PreToolUse-deny на
   `Write|Edit|MultiEdit` по `session_state.json` заблокирует и **первичное
   создание** файла (`start.md:26`, `import.md:13`), если matcher не различает
   «файл не существует» (создание) от «файл существует» (правка). Нужно решить:
   (а) создание тоже идёт через `sdx-stage.sh` (например, подкоманду `init`), или
   (б) deny-хук пропускает случай отсутствующего файла. Первое чище (единая точка
   входа), но требует, чтобы `start.md`/`import.md` тоже перестали писать JSON
   инструкцией и звали скрипт.
2. **Не-`stage` поля.** Deny по всему файлу заблокирует и легитимные правки
   `track`/`gate_mode` (`retrack.md`), и (если когда-то специфицируют) `status`.
   Нужно решить объём: скрипт `sdx-stage.sh` расширяется до общего «писателя
   состояния» (умеет менять `track`/`gate_mode` тоже — тогда имя `sdx-stage.sh`
   вводит в заблуждение, возможно `sdx-session.sh`?), или deny специфичен только
   к полю `stage` (требует диффа JSON до/после в самом хуке — сложнее, чем
   блокировка по пути файла) — открытый архитектурный вопрос.
3. **`artifacts`/`history`** — раз они мёртвые (DEBT-007), стоит решить их
   судьбу ДО или ВМЕСТЕ с DEBT-001: если убрать их из схемы, поверхность,
   которую нужно защищать/разрешать в state-файле, сужается.
4. **`status`** — сегодня фактически не используется, но упомянут как условие
   гейта Business Spec (`next.md:18`). Если DEBT-001 вводит единственного
   писателя `stage`, естественно заодно решить, специфицируется ли `status`
   (когда его пишет кто) — иначе гейт `Business Spec → Technical Design`
   остаётся дырой (approve никогда реально не записывается, значит скрипт не
   сможет объективно проверить это условие).
5. **`/sdx:retrack` меняет `stage` не по линейной цепочке**, а «на ближайший
   активный этап нового трека» — это не «следующий» и не «предыдущий» этап, а
   произвольный пересчёт. `sdx-stage.sh`, спроектированный вокруг
   forward/backward transition, должен явно поддержать этот третий режим (или
   `retrack.md` вызывает скрипт с явным целевым этапом без гейт-проверки, как
   `backtrack`).
6. **`/sdx:backtrack` — откат без гейта.** Сегодня backtrack не проверяет
   ничего, кроме «этап активен в треке» — сознательно отличается от forward
   `/sdx:next`. Нужно решить: должен ли `sdx-stage.sh` в backward-режиме
   выполнять вообще какие-то проверки (например, «нельзя откатиться на этап,
   которого не было пройдено» — но это неважно, если цель отката ниже текущего
   этапа) — вероятно да, но не гейт-артефакты уходящего этапа, а просто
   валидность целевого этапа в треке. Пересекается с DEBT-011.
7. **`/sdx:archive` пишет `stage: Closeout`** отдельно от `/sdx:next` (шаг 2
   `archive.md`) — если `sdx-stage.sh` жёстко завязан на «следующий активный
   этап трека» (next-semantics), `archive.md` нужно либо звать его с явной целью
   `Closeout`, либо `Closeout` — специальный случай, всегда достижимый как
   «этап после последнего активного» (уже так формулирует `next.md:26`: «если
   активных этапов больше нет — следующим всегда идёт Closeout»). Вероятно
   `archive.md` может просто звать тот же `sdx-stage.sh <id> Closeout` как
   forward-transition — не отдельный код-путь. Стоит унифицировать вызов из
   `archive.md` с вызовом из `next.md`, а не оставлять archive.md с собственной
   прямой записью.
8. **Уже существующие сессии** (в частности — эта самая сессия
   `fw-stage-guard-20260720`, и любые другие активные ветки `sdx/*` на момент
   выката) имеют `session_state.json`, созданный старым (прозаическим) путём.
   Deny-хук начнёт блокировать прямые правки для них тоже — это ожидаемо и
   безопасно (они просто обязаны пользоваться новым скриптом с этого момента),
   но нужно явно проговорить в DESIGN: миграции формата не требуется, т.к.
   схема JSON не меняется, меняется только «кто пишет».
9. **Путь конфигурации: плагин vs per-project.** Сам `sdx-stage.sh` —
   однозначно plagin-shipped (`sdx/hooks/sdx-stage.sh`), т.к. это часть
   enforcement-слоя, поставляемого фреймворком (симметрично `stage-gate.sh`,
   `stop-gate.sh`, `archive-verify.sh`). Deny-правило на
   `session_state.json` — тоже логично как ещё один hook-скрипт в
   `sdx/hooks/`, подключённый через `hooks/hooks.json` (не нужен per-project
   conf/allow, в отличие от `prod-guard.conf`/`stage-gate.allow`, — но
   опционально стоит решить, нужен ли escape-hatch по аналогии с
   `stage-gate.allow`, если пользователю понадобится ручная правка state в
   исключительной ситуации).
10. **Формат распознавания транзакции gate-check внутри `sdx-stage.sh`.**
    Скрипту нужно знать активные этапы трека и порядок (сегодня это знание
    живёт только в `protocol.md` таблице как проза + повторено в `next.md`
    шаге 3 как текст условий). Потребуется машиночитаемое представление
    (встроенное в скрипт case/массивы, или конфиг) — иначе скрипт не сможет
    определить «следующий активный этап» программно. Дублирование этого знания
    между `protocol.md` (человекочитаемо) и `sdx-stage.sh` (машиночитаемо) —
    риск рассинхронизации, стоит явно отметить как компромисс в DESIGN (или
    сделать `protocol.md` источником, а скрипт — производным с тестом на
    соответствие).
11. **Идемпотентность и параллельные вызовы** — не критично для одного
    CLI-инстанса (ADR-012, нет параллельных сессий в одном рабочем дереве), но
    стоит явно решить поведение `sdx-stage.sh` при вызове с текущим ==
    целевым `stage` (no-op success или ошибка?) и при невалидном имени этапа
    (опечатка) — конкретных требований от бизнес-заказчика ещё нет, вопрос для
    Business Spec.

## 7. Связанные записи бэклога

- **DEBT-003** (обход stage-gate через Bash не задокументирован как граница) —
  ортогонален DEBT-001: даже с детерминированным `sdx-stage.sh` и deny на
  Write/Edit, `session_state.json` остаётся редактируемым через Bash
  (`echo > .claude/sessions/<id>/session_state.json`, `jq` in-place и т.п.) — то
  же ограничение, что и у stage-gate по коду. **Рекомендация DEBT-001 явно это
  признаёт**: «закрывает случайный обход, не претендуя на защиту от
  намеренного». В DESIGN стоит явно сослаться на DEBT-003 при описании границ
  нового deny-хука (тот же стандарт честности документации).
- **DEBT-007** (мёртвые поля `artifacts`/`history`, недоспецифицированный
  `status`) — пересекается напрямую (см. §1, §6.3-6.4): решение DEBT-001
  определяет, требуется ли схема `session_state.json` для сужения/расширения
  до начала работы над DEBT-001, или можно отложить.
- **DEBT-010** (тихая деградация хуков не видна) — новый `sdx-stage.sh` и
  deny-хук должны деградировать так же безопасно и по возможности видимо
  (не jq/не sdx-ветка → что происходит?) — стоит включить в
  «здоровье enforcement» блок `/sdx:status`, если/когда DEBT-010 берётся в
  работу; в рамках этой сессии как минимум не усугублять непрозрачность.
- **DEBT-011** (`/sdx:backtrack` недоспецифицирован) — прямая зависимость:
  DEBT-001 не может спроектировать backward-режим `sdx-stage.sh`, не решив
  заодно (или явно отложив) вопросы DEBT-011 (пункт 4 «пометь outdated ИЛИ
  перемести в бэкап», пункт 5 «обнули PLAN.md» — деструктив без правила).
  Рекомендуется на Technical Design явно решить: либо DEBT-011 закрывается
  той же сессией (естественная точка, т.к. `sdx-stage.sh` меняет семантику
  вызова backtrack), либо явно выносится за скоуп с пометкой «наследуемое
  поведение сохраняется как есть».

## Открытые вопросы к Business Spec / Technical Design (сводка)
- Кто пишет `session_state.json` при первом создании (`/sdx:start`,
  `/sdx:import`) — тоже скрипт или отдельный код-путь? (риск 1)
- Объём deny: весь файл целиком или только поле `stage`? Как быть с
  `track`/`gate_mode`, которые тоже легитимно меняются прозой? (риск 2)
- Судьба `artifacts`/`history`/`status` — решать вместе с DEBT-007 или после?
  (риск 3, 4)
- Как `sdx-stage.sh` поддерживает три разных семантики вызова: forward
  (`/sdx:next`), backward (`/sdx:backtrack`), retrack-пересчёт (`/sdx:retrack`),
  и прямой переход в Closeout (`/sdx:archive`)? Один интерфейс с флагом режима
  или несколько подкоманд? (риски 5, 6, 7)
- Нужен ли per-project escape-hatch (аналог `stage-gate.allow`) для ручной
  правки state в исключительных случаях? (риск 9)
- Где хранится машиночитаемая матрица трек→активные этапы→гейт-артефакты —
  дублируется в скрипте или выводится из единого источника с protocol.md?
  (риск 10)
- Объём закрытия DEBT-011 в этой же сессии или отдельно? (см. §7)
