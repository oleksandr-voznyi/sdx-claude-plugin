## Находки

### [FAIL] [Корректность/Когерентность] F-1. `/sdx:archive` начинает Closeout-чек-лист на этапе, который не является `Closeout` — обязательный гейт верификации (ADR-014) обходится
- Где: `commands/archive.md:16-22` (в diff — блок, заменивший прежнюю проверку); механизм — `sdx/hooks/sdx-stage.sh:429-486`.
- Суть: подкоманда `next` продвигает ровно на ОДИН активный этап трека и возвращает `0`. `archive.md` в ветке «Код `0`» безусловно утверждает «`stage` уже переведён в `Closeout` … продолжай к шагу 3 (чек-лист)». Если `/sdx:archive` вызван раньше времени (штатная ошибка оркестратора/пользователя), скрипт вернёт `0`, переведя сессию на промежуточный этап, а команда начнёт закрытие сессии.
- Воспроизведено: трек `patch`, `stage=Execution`, есть `change_note.md` → `sdx-stage.sh next p` → `OK Execution -> Verification`, rc=0. По прозе `archive.md` дальше печатается чек-лист Closeout, хотя `verification_report.md` не существует вовсе.
- Отягчающее: прежняя явная проверка («на треке `patch`: `verification_report.md` существует и не содержит FAIL») из `archive.md` **удалена** этим diff, а замена её не эквивалентна. То есть гейт ADR-014 на patch-треке стал слабее, чем до поставки.
- Ожидалось (SPEC, критерий «Переход в `Closeout` (последний активный этап трека пройден) выполняется тем же механизмом»; REQ-CLOSEOUT-ENTRY-1): вход в Closeout возможен только когда предыдущий активный этап действительно последний.
- Фактически: код `0` не отличает «перешли в Closeout» от «перешли куда-то ещё»; ни скрипт (нет режима/цели), ни команда (нет проверки stdout `-> Closeout`) этого не различают.

### [FAIL] [Корректность] F-2. Deny-хук не блокирует `Edit`, меняющий значение `stage` без упоминания ключа `"stage"`
- Где: `sdx/hooks/stage-write-guard.sh:759-783` (`STAGE_KEY_RE='"stage"[[:space:]]*:'`, ветка `Edit`).
- Суть: детекция построена на присутствии ключа в `new_string`. Совершенно штатная для LLM форма правки — `Edit(old_string: "\"Execution\"", new_string: "\"Closeout\"")` (значение уникально в плоском state-файле) — ключа не содержит и проходит **без deny**.
- Воспроизведено: ветка `sdx/probe`, существующий `session_state.json` со `stage: "Execution"`; вход `{"tool_name":"Edit","tool_input":{"file_path":"…/session_state.json","old_string":"\"Execution\"","new_string":"\"Closeout\""}}` → пустой stdout, `exit 0` (нет deny). То же для варианта без кавычек.
- Ожидалось (REQ-DENY-1 + критерий приёмки «Прямая правка поля `stage` через `Edit` … блокируется»): любая правка значения `stage` через `Edit` детерминированно блокируется; SPEC целится ровно в **случайный** обход, а это и есть случайный обход.
- Фактически: блокируется только форма, где фрагмент содержит подстроку `"stage":`. `DESIGN.md:368-373` документирует ложноотрицательный случай, но описывает **другой** сценарий («ключ переименован более ранней правкой… не бывает в штатном флоу») и не покрывает описанный здесь реалистичный. Тестом этот случай тоже не закрыт (`test-stage-write-guard.sh` проверяет только формы с ключом).

### [FAIL] [Корректность] F-3. `sdx-stage.sh retrack` принимает нераспознанное имя этапа как regex и записывает в state невалидный `stage`
- Где: `sdx/hooks/sdx-stage.sh:584-587` — `printf '%s\n' "$stages" | grep -qx "$target"` (без `-F`, без предварительной union-проверки `matrix_stage_exists`, которая есть только в `cmd_backtrack:510`).
- Воспроизведено: `retrack probe 'Executio.'` → `OK Execution -> Executio.`, rc=0, в файле `"stage": "Executio."`; далее `retrack probe '.*'` → `"stage": ".*"`. После этого `next` отвечает `exit 2 — состояние повреждено`, а починить `stage` через `Edit` мешает новый deny-хук (остаётся только Bash-обход).
- Ожидалось (REQ-STAGE-4: «вызов с именем этапа, не входящим в список этапов протокола, отклоняется **без изменения файла**, с сообщением, что имя этапа не распознано»; DESIGN «Схема данных» — схема state не деградирует): невалидное имя → `exit 1`, файл не тронут.
- Фактически: (а) метасимволы regex дают ложное совпадение и порчу состояния; (б) даже для простой опечатки сообщение — «этап 'X' не активен в треке» вместо предусмотренного REQ-STAGE-4 «не распознан как имя этапа протокола».
- Тестового покрытия нет: `test-sdx-stage.sh` сц. 16 проверяет только валидное имя, неактивное в треке.

## Сводка по осям

- **Когерентность.** ADR-016 в `docs/DECISIONS.md` согласован с реализацией и вставлен корректно (после ADR-015, нумерация `ADR-0NN`).
