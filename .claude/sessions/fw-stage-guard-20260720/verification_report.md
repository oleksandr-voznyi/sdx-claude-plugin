## Находки

### [FAIL] F-1 (Корректность) `retrack` + `retrack`: клэмп «войти в трек с его стартовой точки» работает как храповик и восстанавливает обход WARN-4

- Где: `sdx/hooks/sdx-stage.sh:436-471` (`cmd_retrack`, вычисление `idx_equiv` и ветка
  `[ -z "$idx_equiv" ] && idx_equiv=1`); требование `SPEC.md` REQ-RETRACK-2;
  `DESIGN.md:614-625` («Пример: почему клэмп к первому этапу нового трека необходим»).
- Суть: клэмп повышает `stage` до `Execution` при деэскалации в `patch` (это разрешено
  требованием и само по себе корректно), но повышенная позиция немедленно засчитывается как
  «накопленный прогресс» при следующем `retrack`. Инвариант REQ-RETRACK-2 не монотонен:
  ранг растёт без единого пройденного гейта.
- Ожидалось: «`retrack` переводит `stage` только на этап … **не дальше**, чем позволяет уже
  накопленный прогресс» (SPEC REQ-RETRACK-2); «форвард-движение мимо гейта остаётся
  исключительно прерогативой `next`» (`DESIGN.md:855-857`).
- Фактически: сессия трека `full` на этапе `Discovery` **без единого артефакта** двумя
  вызовами `/sdx:retrack` попадает на `full`/`Execution` (или `full`/`Task Planning`),
  после чего `/sdx:next` свободно доводит её до `full`/`Verification` — гейты
  `context_report.md`, `SPEC.md`, `DESIGN.md`, `PLAN.md` не проверялись никогда.
  Дополнительный эффект: `stage-gate.sh:42-43` разрешает запись в код на `Execution`/
  `Deployment` — код-фриз снимается на сессии, у которой нет ни спеки, ни дизайна.
  Это ровно класс риска DEBT-001 (случайный обход при легитимной последовательности
  «деэскалировали → передумали → эскалировали обратно»), а не намеренная атака.
- Воспроизведение (фактический вывод скрипта приведён в комментариях):
  ```bash
  P=$(mktemp -d); mkdir -p "$P/.claude/sessions/s1"
  cat > "$P/.claude/sessions/s1/session_state.json" <<'J'
  {"session_id":"s1","type":"feature","track":"full","stage":"Discovery","gate_mode":"interactive","git_branch":"sdx/s1","artifacts":[],"history":[]}
  J
  ST="$P/.claude/sessions/s1/session_state.json"; S=sdx/hooks/sdx-stage.sh
  setk(){ t=$(mktemp); jq --arg v "$1" '.track=$v' "$ST" >"$t"; mv "$t" "$ST"; }

  setk patch   # шаг 4.2 /sdx:retrack — правка track через Edit (легитимна, REQ-DENY-2)
  CLAUDE_PROJECT_DIR="$P" bash $S retrack s1 Execution         # OK Discovery -> Execution   (клэмп)
  setk full    # пользователь передумал: эскалация обратно в full
  CLAUDE_PROJECT_DIR="$P" bash $S retrack s1 "Task Planning"   # OK Execution -> Task Planning
  CLAUDE_PROJECT_DIR="$P" bash $S retrack s1 Execution         # (или сразу Execution — тоже разрешено)
  CLAUDE_PROJECT_DIR="$P" bash $S next s1                      # OK Execution -> Documentation
  CLAUDE_PROJECT_DIR="$P" bash $S next s1                      # OK Documentation -> Verification
  ls "$P/.claude/sessions/s1"                                  # session.log  session_state.json — и всё
  ```
- Сопутствующее: проверка no-op (`sdx-stage.sh:431-434`) стоит **до** guard-а, поэтому
  вариант «сменить трек, оставив то же имя этапа» вообще не доходит до проверки потолка —
  второй, ещё более короткий путь того же обхода (`patch`/`Execution` → `full`/`Execution`
  проходит как `OK no-op Execution`).
- Перебор всех 34 пар (текущий трек × этап → новый трек) выполнен. **Тупиков нет**: в каждой
  паре разрешён хотя бы один целевой этап, легитимные эскалации/деэскалации, описанные в
  `sdx/protocol.md` («Профили флоу», адаптивность трека), не блокируются. Ранг `Change`
  согласуется с заявленным (`Change` ↔ `Technical Design`) в обе стороны: из `standard`/`Change`
  в `full` разрешены `Discovery`/`Business Spec`/`Technical Design` и отклоняется `Task Planning`;
  из `full`/`Technical Design` в `standard` разрешён `Change`, из `full`/`Business Spec` — нет.
  Единственная аномалия перебора — рост ранга в строке `full/Discovery → patch: Execution`
  и симметричных, что и есть механизм храповика.
