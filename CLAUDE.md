# SDX (Spec-Driven X) Framework for Claude Code

Данный файл определяет правила и стандарты разработки в этом проекте на основе методологии Spec-Driven Development (SDD). Все агенты и участники проекта обязаны следовать этим инструкциям.

Этот репозиторий — **исходники SDX-плагина Claude Code** (ADR-013): корень репо является плагином (`.claude-plugin/plugin.json`) и одновременно marketplace (`.claude-plugin/marketplace.json`). Дистрибуция — установка плагина (`/plugin marketplace add <репо>` → `/plugin install sdx@sdx --scope user`), а не копирование `.claude/` по проектам. В рантайме команды/агенты/хуки исполняются из установленной копии плагина; после правок фреймворка нужен `/plugin marketplace update sdx`.

## 1. Языковая политика (Language Policy)
- **Язык общения в чате**: Русский
- **Инструкции агентов (agents/*.md)**: Русский
- **CLAUDE.md и правила**: Русский
- **Спецификации (SPEC.md) и Дизайн (DESIGN.md)**: Русский
- **Пользовательская документация**: Русский
- **Техническая документация (Swagger/OpenAPI)**: Английский
- **Код (имена переменных, комментарии)**: Английский

## 2. Ролевая модель (Agents)
В проекте используются специализированные субагенты (`agents/` плагина, вызываются через инструмент Task):
- **Business Analyst (`ba`)**: Отвечает за `SPEC.md` (бизнес-требования).
- **Architect (`architect`)**: Discovery-исследование (`context_report.md`) и `DESIGN.md` (техническое проектирование).
- **Lead Developer (`lead-dev`)**: Отвечает за `PLAN.md` (декомпозиция задач).
- **Developer (`developer`)**: Пишет код и Unit-тесты.
- **QA Agent (`qa`)**: Корректность исполнением — пишет интеграционные тесты, инструкции для ручного тестирования и обеспечивает зелёный прогон.
- **Reviewer (`reviewer`)**: Независимое fresh-eyes ревью поставки по трём осям (полнота/корректность/когерентность) на основе только артефактов и diff, без истории решений. Контракт изоляции контекста — вызывается ТОЛЬКО через Task.
- **Tech Writer (`tech-writer`)**: Генерирует документацию (Swagger, README, User Guides).
- **DevOps Agent (`devops`)**: Управляет инфраструктурой, конфигами и деплоем.

### Оркестрация
Роль **Session Manager** выполняет основная сессия Claude Code при запуске команд `/sdx:*` (`commands/` плагина): она управляет `session_state.json`, переходами между этапами и вызывает субагентов через Task. Субагенты НЕ могут вызывать друг друга — вся координация идет из основной сессии. Формат состояния и логирования описан в `sdx/protocol.md`.

Каждый субагент объявляет `model` во frontmatter **алиасом тира**, а не конкретным model ID: фронтир-тир `opus` — на оценочное суждение (`reviewer`, эскалация `architect` на Technical Design), дешёвый тир `haiku` — на механику (`tech-writer`), рабочий тир `sonnet` — остальным. Алиас резолвится в актуальное поколение тира — при смене поколения моделей правки фреймворка не требуются. Источник истины раскладки — frontmatter агентов; конкретные ID в документах фреймворка не дублируются (обоснование — ADR-3, ADR-008 в `docs/DECISIONS.md`). Поле `effort` во frontmatter не используется (не поддерживается в схеме агента).

## 3. SDX Unified Lifecycle (SUL)
Процесс разделен на изолированные сессии (`.claude/sessions/<id>/`) в отдельных git-ветках `sdx/<id>`.

### Адаптивные треки (Tracks)
Жизненный цикл НЕ является жёсткой фиксированной цепочкой. Каждая сессия проходит по одному из **профилей флоу (треков)**, который определяет, какие этапы активны. Это масштабирует церемонию под размер задачи и экономит токены на мелких правках:
- **patch** — точечный багфикс без изменения логики/контрактов: Execution → Verification (лёгкая, обязательная — ADR-014) → Closeout, дельты в одном `change_note.md`.
- **standard** — малая фича/рефактор: лёгкий Discovery → Change (Spec+Design слитно) → Execution → Verification → Closeout.
- **full** — крупная задача: все этапы ниже.
- **doc** — процессная/рефлексивная работа без кода (типы `grooming`/`retro`/`postmortem`/`intake`, жёстко привязаны к треку на `/sdx:start` — ADR-017): Discovery (лёгкий, инлайн) → Update → Verification (лёгкая, обязательная) → Closeout, единый гейт-артефакт `change_note.md`; параллельная ветка, не часть линейной шкалы `patch → standard → full`.

Трек выбирается при `/sdx:start` как гипотеза (на основе `type`, но не привязан к нему жёстко) и **адаптивно меняется** по ходу работы через `/sdx:retrack`: эскалация при выходе за рамки (задеты контракты/архитектура) или деэскалация при избыточной церемонии. Полное описание треков и переходов — в `sdx/protocol.md`. `/sdx:next` всегда ведёт к следующему _активному_ этапу текущего трека.

Ортогонально треку пользователь может включить **авторежим гейтов** (`gate_mode: auto`, opt-in: `/sdx:start --auto` или согласие на Discovery/Change-гейте — ADR-014): прозаические подтверждения принимают дефолты с записью в `auto_decisions.md` и обязательным дисклоузом перед Closeout (единственная точка подтверждения); остановка — только на стоп-рубрике протокола (коллизии триады, FAIL, внешние контракты, деструктив, эскалация трека, ручные шаги). Детерминированные хуки авторежим не смягчает.

### Этапы (Stages):
1. **Discovery**: Исследование контекста и кодовой базы.
2. **Business Spec**: Создание/обновление `SPEC.md`. Используется Delta-формат ([ADDED], [MODIFIED], [REMOVED]).
3. **Technical Design**: Создание/обновление `DESIGN.md`. Отражение технических решений.
4. **Task Planning**: Создание `PLAN.md` с атомарными задачами.
5. **Execution**: Цикл TDD (Test -> Code -> Verify) для каждой задачи.
6. **Documentation**: Обновление производных артефактов (Swagger, мануалы).
7. **Verification**: Финальное тестирование и Gate для пользователя.
8. **Deployment**: Применение инфраструктурных изменений.
9. **Closeout** (Archiving): Завершение сессии по **обязательному чек-листу** (`/sdx:archive`): коммиты в ветку → перенос дельт из `change_note.md`/`SPEC.md`/`DESIGN.md` в постоянные `docs/` и документацию → архив `PLAN.md` → обновление глобального лога и бэклога (`docs/backlog/`: статусы закрытых записей, новые `DEBT-`/`IDEA-`записи из отложенного — ADR-015) → слияние ветки в `main` → инварианты 1/5/6 enforced скриптом `sdx/hooks/archive-verify.sh` плагина (вызывается `/sdx:archive` после мёржа) → финальный лог. Сессия не считается закрытой, пока все пункты не отмечены. Контракт закрытия описан в `sdx/protocol.md`.

### Перенос между проектами (Import / Export)
Для переноса функциональности между двумя SDX-проектами:
- **`/sdx:export <feature>`** — read-only команда: пакует фичу (срезы `SPEC`/`DESIGN` + код + тесты + замыкание зависимостей + допущения) в переносимый бандл `.sdx/bundles/<feature>/`.
- **`/sdx:import <bundle>`** — сессия типа `import` (обычно трек `full`): реконсилирует бандл с целевым проектом покомпонентно (`transplant` / `reimplement` / `gap`), мёржит срезы в триаду цели и проходит полный цикл. Детали — в `sdx/protocol.md`.

## 4. Источники правды и Коллизии
- **Триада**: [SPEC.md] <-> [DESIGN.md] <-> [Codebase].
- При обнаружении логических противоречий между источниками агент ОБЯЗАН остановиться и запросить решение у пользователя.
- Этот контроль операционализирован на этапе Verification: структурированная проверка по трём осям (полнота/корректность/когерентность) через `/sdx:verify`, где ось когерентности — детектор коллизий триады, запускаемый независимым fresh-eyes ревьюером (`reviewer`), а не автором изменений. Находки `FAIL` блокируют гейт, `WARN` требуют квитирования (в авторежиме — отложенного, через дисклоуз на Closeout; ADR-014). Коллизии триады останавливают работу в ЛЮБОМ `gate_mode`.
- В случае багфикса, не требующего изменения логики (трек `patch`), допускается правка только кода без изменения `SPEC.md`/`DESIGN.md`; суть правки фиксируется в `change_note.md` сессии. Если в ходе работы выясняется, что затронуты логика, контракты или архитектура — выполни эскалацию трека через `/sdx:retrack`, и тогда дельты переносятся в `SPEC.md`/`DESIGN.md` штатно.

## 5. Изоляция и Контекст
- Каждая сессия хранится в `.claude/sessions/<id>/`.
- Для борьбы с деградацией контекста используются периодические "сбросы": `/sdx:checkpoint` фиксирует срез состояния в `checkpoint.md`, после чего пользователь выполняет `/clear`, а состояние восстанавливается чтением `.md` файлов сессии (`/sdx:status`).

## 6. Структура и Инициализация (плагинная модель, ADR-013)
Фреймворк поставляется плагином; структура этого репозитория (= структура плагина):
- `.claude-plugin/plugin.json` / `marketplace.json`: манифест плагина и marketplace.
- `commands/`: Слэш-команды SDX (`/sdx:start`, `/sdx:next` и т.д. — неймспейс даёт имя плагина).
- `agents/`: Инструкции специализированных агентов (YAML frontmatter: `name`, `description`, `tools`, `model`).
- `hooks/hooks.json`: Проводка enforcement-хуков (пути через `${CLAUDE_PLUGIN_ROOT}`).
- `sdx/`: Протокол сессий (`protocol.md`), hook-скрипты с тестами (`hooks/`), шаблоны per-project конфигов (`templates/`).

В каждом целевом проекте (включая этот мета-репозиторий) живёт только **per-project слой**:
- `.claude/sessions/<id>/`: Данные активной сессии. Сессия = ветка `sdx/<id>`, checkout'нутая в основном рабочем дереве (один CLI на весь жизненный цикл, без worktree — ADR-012); артефакты версионируются (tracked) инкрементальными коммитами на этой ветке; только эфемерные `.stopgate.*`-файлы игнорируются точечным паттерном.
- `.claude/sdx/`: Per-project конфиги enforcement-слоя (`prod-guard.conf`, `stage-gate.allow`, `verify-cmd.sh`).
- `docs/specs/`: Постоянное хранилище бизнес-спецификаций.
- `docs/designs/`: Постоянное хранилище технических дизайнов.
- `docs/history/plans/`: Архив выполненных планов реализации.
- `docs/backlog/`: Трекаемый бэклог проекта (ADR-015): файл-на-запись с машиночитаемым YAML frontmatter (`id`, `type`, `status`, `priority`, `wave`, `source`, `session`, `links`), префиксы ID по типам (`FEAT-`/`BUG-`/`DEBT-`/`IDEA-`/`PROC-`), индекс `README.md`. Просмотр и пополнение — команда `/sdx:backlog`; актуализация статусов — обязательный пункт Closeout-чек-листа.

### Инициализация (/sdx:init)
При подключении существующего проекта:
1. Установить плагин один раз на машину: `/plugin marketplace add <репо sdx.cld>` → `/plugin install sdx@sdx --scope user`.
2. В проекте выполнить `/sdx:init` (или `/sdx:init --existing` для реверс-инжиниринга текущей кодовой базы в базовые `SPEC.md` и `DESIGN.md`) — команда создаст структуру `docs/`, `.claude/sessions/`, per-project конфиги из шаблонов и targeted-паттерны `.gitignore`. `/sdx:init` автоматически вызывает `/sdx:reconcile` — сверку легаси-материала проекта и устаревших структур хранения с актуальным форматом фреймворка; та же сверка обязательна после каждого обновления версии плагина (`/sdx:start` предлагает её при расхождении версии). Политика и границы — раздел «Сверка форматов после релиза и миграции» в `sdx/protocol.md`.
