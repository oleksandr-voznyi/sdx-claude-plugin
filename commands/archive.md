---
description: Закрытие SDX-сессии по обязательному чек-листу (Closeout).
---

Ты выполняешь роль Session Manager. Твоя задача — корректно и ПОЛНОСТЬЮ закрыть активную сессию. Этот этап — самый частый источник пропусков (особенно перенос дельт в постоянные документы и удаление файлов сессии), поэтому он оформлен как обязательный чек-лист.

**Модель (ADR-012, вариант A удаления).** Весь Closeout выполняется последовательно в ОДНОМ CLI: сначала шаги 1–5 на ветке сессии `sdx/<id>`, затем checkout основной ветки, мёрж и enforcement-проверка. Инженерный инвариант: HEAD-дерево основной ветки никогда не содержит файлы сессии — удаление оформляется коммитом `git rm -r` НА ВЕТКЕ ДО мёржа.

**Правило:** распечатай чек-лист пользователю с отметками `[ ]/[x]`, выполняй пункты по порядку и обновляй отметки. НЕ объявляй сессию закрытой, пока ВСЕ пункты не отмечены `[x]`. Если пункт неприменим к треку сессии — отметь `[x]` с пометкой `(n/a)` и поясни почему.

## Шаги 1–5 — на ветке сессии `sdx/<id>`

1. Прочитай `session_state.json`: `session_id`, `type`, `track`, `gate_mode`, `git_branch`, список артефактов.
2. **Гейт входа в Closeout (ADR-014):**
   - При `gate_mode: "auto"`: выполни **дисклоуз** — выведи `auto_decisions.md` одним блоком (автоквитированные WARN — первыми, затем прочие дефолты) и получи ОДНО явное подтверждение пользователя (AskUserQuestion). Несогласие → предложи `/sdx:backtrack`/`/sdx:retrack` и прерви Closeout. Без подтверждения дисклоуза чек-лист не начинается.
   - Затем войди в `Closeout` — единственный писатель `stage`, тот же механизм, что и `/sdx:next`. **Важно:** `next` продвигает РОВНО на один активный этап и гейтит только этап, с которого уходит; `Closeout` не обязан быть следующим (например, если `/sdx:archive` вызван раньше срока — ещё на `Execution`, до `Verification`). Поэтому вызывай `next` **в цикле**, шаг за шагом, до фактического достижения `Closeout`, и критерием остановки считай не код возврата, а РЕЗУЛЬТИРУЮЩЕЕ значение `stage` в файле:
     ```bash
     while :; do
       out="$(bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" next "<session_id>")"; rc=$?
       [ "$rc" -eq 0 ] || break
       git add .claude/sessions/<session_id>/session_state.json .claude/sessions/<session_id>/session.log
       git commit -m "sdx(<session_id>): $out"
       cur_stage="$(jq -r '.stage' .claude/sessions/<session_id>/session_state.json)"
       [ "$cur_stage" = "Closeout" ] && break
     done
     ```
     - **Каждый успешный шаг (код `0`)**: скрипт уже проверил объективный гейт УХОДЯЩЕГО этапа этой итерации (существование/непустота артефакта и, где по матрице применимо, отсутствие `^### [FAIL]`) и атомарно записал `[STAGE_CHANGE]` в лог — закоммить `session_state.json`+`session.log` (`OK <old> -> <new>` из stdout как текст коммита), затем проверить `cur_stage`:
       - `cur_stage = "Closeout"` → цикл окончен успешно, продолжай к шагу 3.
       - `cur_stage` — промежуточный этап (значит вызвали раньше срока) → НЕ считай это ошибкой и НЕ угадывай, что этап «наверняка последний»: повтори итерацию цикла — она провалидирует гейт СЛЕДУЮЩЕГО уходящего этапа тем же механизмом. Пример на треке `patch`, если вызвано из `Execution`: 1-я итерация проверяет `change_note.md` и переводит в `Verification`; 2-я итерация проверяет `verification_report.md` без `FAIL` (это и есть гейт ADR-014/ADR-017 «на треках `patch` и `doc` — `verification_report.md` без находок FAIL») и переводит в `Closeout`.
     - **Код `1` на любой итерации** (гейт уходящего этапа ЭТОЙ итерации не пройден — какой именно артефакт и этап, называет stderr; на `patch`/`standard`/`doc` перед `Closeout` это почти всегда `verification_report.md`, на `full` — гейт того этапа, где цикл остановился, см. таблицу подкоманд `next` в `sdx/protocol.md`): сообщи пользователю текст stderr дословно, направь на команду, устраняющую именно этот гейт (обычно `/sdx:verify`, если не хватает `verification_report.md`; иначе — на этап, названный в сообщении), запиши в лог `[ERROR] Ошибка перехода: <текст stderr>`; чек-лист Closeout **не начинается**. Уже закоммиченные предыдущие итерации цикла (если были) — легитимные промежуточные переходы, их откатывать не нужно; сессия просто остаётся на промежуточном этапе до устранения гейта.
     - **Код `2`** (нет `jq` / нет `session_state.json` / состояние повреждено): сообщи текст stderr дословно и остановись.
3. Пройди чек-лист закрытия, отмечая каждый пункт:

   - `[ ]` **1. Коммиты.** Рабочее дерево чистое (`git status`), все изменения закоммичены на ветку `sdx/<id>`.
   - `[ ]` **2. Перенос дельт в постоянные документы.** Это ключевой пункт:
     - Бизнес-дельты (`change_note.md` или `SPEC.md` сессии) → `docs/specs/`.
     - Технические дельты (`DESIGN.md` сессии) → `docs/designs/`.
     - Пользовательская и техническая документация (README, мануалы, OpenAPI) актуализированы под внесённые изменения. При необходимости вызови субагента `tech-writer` (Task).
     - Убедись, что в постоянных доках не осталось расхождений с фактическим кодом (триада SPEC ↔ DESIGN ↔ Codebase).
     - Закоммить перенос на ветку `sdx/<id>` (это отдельный коммит от коммита удаления в п.5).
   - `[ ]` **3. Архив плана.** `PLAN.md` (если создавался) перенесён в `docs/history/plans/<session_id>.md`. Закоммить.
   - `[ ]` **4. Глобальный лог знаний и бэклог.** Краткая запись об итогах сессии добавлена в `docs/history/` (что сделано, какие документы затронуты, ссылка на коммит/ветку). Бэклог `docs/backlog/` актуализирован (ADR-015): записи, закрытые этой сессией, получили `status: closed` и `session: <id>`; отложенные решения и неквитированные WARN-находки верификации оформлены новыми `DEBT-`/`IDEA-`записями (`/sdx:backlog add` или напрямую по схеме); индекс `README.md` обновлён. На треке `doc` (ADR-017) этот пункт — основная точка выхода: сессия обязана произвести минимум одно изменение `docs/backlog/`, помеченное `session: <session_id>` (для типа `intake` минимум одна из этих записей обязана быть НОВОЙ, не правкой существующей). Для сессий типа `retro`/`postmortem`/`intake` этот же пункт включает перенос постоянного документа разбора из каталога сессии в `docs/history/retro/<session_id>.md`, `docs/history/postmortem/<session_id>.md` или `docs/history/intake/<session_id>.md` соответственно (по аналогии с переносом `PLAN.md` в п.3); для `grooming` отдельного документа нет — записи бэклога являются единственным и достаточным выходом. Закоммить.
   - `[ ]` **5. Удаление файлов сессии (коммит на ветке, ДО мёржа — вариант A).**
     ```bash
     git rm -r .claude/sessions/<session_id>
     git commit -m "sdx(<session_id>): drop session artifacts pre-merge"
     ```
     Это отдельный, явно поименованный коммит — НЕ смешивать с коммитом переноса дельт (п.2). После него HEAD-дерево ветки `sdx/<id>` больше не содержит каталог сессии (REQ-SESS-3).

## Шаги 6–8 — мёрж в основную ветку (тот же CLI)

   - `[ ]` **6. Резолв основной ветки и слияние.**
     ```bash
     def=$(bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/lib/default-branch.sh")
     git checkout "$def"
     git merge --no-ff sdx/<session_id> -m "Merge sdx/<session_id>"
     ```
     Docs-дельты вливаются в основную ветку; каталога сессии в HEAD-дереве нет (он был удалён коммитом п.5). `--no-ff` гарантирует, что вся история ветки `sdx/<id>` достижима из merge-коммита (REQ-SESS-4).
   - `[ ]` **7. Enforcement-проверка закрытия.** Вызови:
     ```bash
     bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/archive-verify.sh" <session_id>
     ```
     Скрипт резолвит основную ветку через `lib/default-branch.sh` (без хардкода `main`) и проверяет инвариант 1 (чистое дерево), 5 (ветка слита в основную) и 6 (каталог сессии не tracked в дереве основной ветки — детектор пропуска варианта A), после чего сам удаляет слитую ветку (`git branch -d`; для legacy-сессий, созданных до ADR-012 в отдельном worktree, дополнительно выполняет `git worktree remove --force`). Ненулевой код выхода → Closeout НЕ завершён: устрани FAIL (обычно — пропущенный п.5) и повтори.
   - `[ ]` **8. Финальная запись.** Факт закрытия фиксируется в глобальном логе (п.4, уже выполнен) — `session.log` отдельной записи не требует, так как он был удалён вместе с каталогом сессии в п.5.

После того как все пункты отмечены `[x]`, выведи итоговую сводку закрытия: что перенесено, куда слито, что удалено.

Протокол сессий, профили флоу и полный контракт закрытия: прочитай (Read) `${CLAUDE_PLUGIN_ROOT}/sdx/protocol.md`
