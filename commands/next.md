---
description: Переход на следующий этап жизненного цикла SDX.
---

Ты выполняешь роль Session Manager. Твоя задача — проверить готовность текущего этапа и перевести сессию на следующий **активный** этап трека.

Алгоритм:
1. Прочитай `session_state.json`: определи `track` и текущий `stage`. Набор активных этапов берётся из профиля трека (см. протокол).
2. **Scope-check (адаптивность).** Прежде чем двигаться дальше, оцени, не вышла ли задача за рамки текущего трека:
   - Затронуты бизнес-логика, публичные контракты, схемы данных или архитектура, которых трек не предполагал → предложи эскалацию через `/sdx:retrack`.
   - Церемония оказалась явно избыточной → предложи деэскалацию.
   Не меняй трек молча — сообщи пользователю и дождись решения, если масштаб изменился существенно.
2а. **Авторежим (по протоколу, ADR-014).** Прочитай `gate_mode` из state:
   - Если `"auto"`: гейт-подтверждения, обычно запрашиваемые у пользователя, принимай дефолтом без остановки; каждый дефолт — одна строка в `auto_decisions.md` (коммить вместе со сменой состояния). Остановка ОБЯЗАТЕЛЬНА на стоп-рубрике протокола: коллизии триады, FAIL/красный тест-пол, внешние контракты/схемы данных, деструктив/prod-guard, эскалация трека (сбрасывает `gate_mode → interactive`), ручные шаги.
   - Если `"interactive"` и это первый содержательный гейт сессии (Discovery/Change/Update): при тривиальном масштабе задачи предложи пользователю включить авторежим (AskUserQuestion); решение зафиксируй в `gate_mode`.
   - На треке `vibe` предложение включить авторежим не делается.
3. **Гейт текущего этапа.** Объективно проверяемая часть гейта (существование/непустота требуемого артефакта, отсутствие `FAIL` в `verification_report.md`) проверяется автоматически вызовом `sdx-stage.sh next` (шаг 4) — не описывай её прозой заново. Смысловые условия («решение согласовано», `SPEC.md` подтверждён по смыслу — `status: approved`, декомпозиция `PLAN.md` адекватна) остаются твоим суждением **до** вызова — скрипт их не проверяет (DEBT-007, ограничение унаследовано).
4. Вызови единственный писатель `stage`:
   ```bash
   bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" next "<session_id>"
   ```
   - **Код `0`** (включая no-op `OK no-op <stage>`, когда текущий этап уже терминальный для трека (`Closeout` на линейных треках и `doc`; `Prototype` на `vibe` — без собственного `Closeout`) — файл не тронут): `stage` уже переведён и `[STAGE_CHANGE]` уже записан в лог скриптом атомарно; используй `OK <old> -> <new>` из stdout как текст коммита. Закоммить смену состояния на ветку `sdx/<id>` (REQ-SESS-1):
     ```bash
     git add .claude/sessions/<id>/session_state.json .claude/sessions/<id>/session.log && git commit -m "sdx(<id>): stage <old> -> <new>"
     ```
     (для `OK no-op` коммитить нечего — файл не менялся). Затем запусти работу нового этапа: вызови соответствующего субагента (architect, lead-dev, developer, qa, tech-writer, devops через Task) либо выполни этап инлайн, если трек это предполагает. Если новый этап — `Closeout`, направь пользователя на `/sdx:archive`. Если трек — `vibe` и получен `OK no-op Prototype` — это легитимный no-op, не ошибка: направь пользователя на `/sdx:proto` или `/sdx:retrack standard|full`; коммитить нечего.
   - **Код `1`** (гейт отклонён — недостающий/пустой артефакт или находка `FAIL`): сообщи пользователю текст stderr дословно (он уже называет недостающий артефакт и команду для исправления — не формулируй заново), запиши в лог `[ERROR] Ошибка перехода: <текст stderr>`. `stage` не менялся, коммитить нечего.
   - **Код `2`** (ошибка окружения/аргументов — нет `jq`, нет `session_state.json`, состояние повреждено): сообщи пользователю текст stderr дословно, запиши в лог `[ERROR] Ошибка перехода: <текст stderr>` и остановись.

Протокол сессий, профили флоу и контракт закрытия: прочитай (Read) `${CLAUDE_PLUGIN_ROOT}/sdx/protocol.md`
