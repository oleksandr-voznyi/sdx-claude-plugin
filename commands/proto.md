---
description: Гейт принятия/отклонения прототипа (трек vibe): показать результат, принять с легализацией или откатить точечно.
---

Ты выполняешь роль Session Manager. Твоя задача — провести пользователя через гейт принятия/отклонения экстремального прототипа (трек `vibe`, тип сессии `proto`, этап `Prototype`): честно предъявить фактический результат текущей попытки и, по явному решению пользователя, либо направить прототип в легализацию, либо откатить его точечно до состояния baseline. Эта команда — единственный путь из `Prototype`, кроме `/sdx:retrack`; вызывай её самостоятельно, не полагаясь на контекст, накопленный ранее в этой же сессии.

Алгоритм:

1. **Проверка track/stage.** Определи текущую сессию по активной git-ветке (`git branch --show-current`, ветки вида `sdx/<id>`) и прочитай `.claude/sessions/<id>/session_state.json`.
   - Если `track != "vibe"` → ОСТАНОВИСЬ: сообщи, что команда `/sdx:proto` применима только к сессиям трека `vibe`.
   - Если `stage != "Prototype"` → ОСТАНОВИСЬ и укажи пользователю текущий этап: сессия уже легализована (либо ещё не запущен прототип), дальнейшая работа — через `/sdx:next`.

2. **Чтение `prototype_baseline.txt` — fail-closed.** Прочитай `.claude/sessions/<id>/prototype_baseline.txt`. Формат (машиночитаемый, `.txt`, не `.md`):
   ```
   # SDX vibe prototype baseline
   attempt 1
   started 2026-07-20 14:05:11
   rev 9f3c1a2b4d5e6f708192a3b4c5d6e7f8091a2b3c
   untracked notes/scratch.txt
   untracked tmp/local.env
   ```
   - `attempt N` — номер текущей попытки прототипа (растёт при каждом отклонении).
   - `rev` — коммит, относительно которого считается tracked-дельта прототипа.
   - `untracked <path>` (0..N строк) — untracked-файлы, существовавшие ДО начала этой попытки; project-relative пути, ровно как их печатает `git ls-files --others`. Эти строки исключаются из списка отката, чтобы не задеть посторонние файлы пользователя.

   Запомни `N` (значение `attempt` — номер ЗАВЕРШАЮЩЕЙСЯ попытки) — понадобится на шаге 6; там же используется `N+1` — номер НОВОЙ попытки, стартующей после отката.

   **Если файл отсутствует, пуст, не содержит строку `rev`, либо `rev` не резолвится в существующий коммит** (`git rev-parse --verify --quiet "<rev>^{commit}"` не возвращает ничего) — **ОСТАНОВИСЬ. Baseline НИКОГДА не додумывается из `HEAD` и откат в этом случае НЕ предлагается.** Точечность отката без валидного baseline недоказуема — это ровно тот класс ошибки, который уничтожил бы чужие файлы. Сообщи пользователю дословно, что произошло, и предложи разобрать состояние дерева вручную (`git status`, `git diff`).

3. **Вычисление списка файлов прототипа.** Выполни дословно:
   ```bash
   sdir=".claude/sessions/<id>"
   base_rev="$(awk '$1=="rev"{print $2; exit}' "$sdir/prototype_baseline.txt")"
   git rev-parse --verify --quiet "${base_rev}^{commit}" >/dev/null || {
     echo "SDX proto: baseline-ревизия ${base_rev} не найдена — точечный откат недоказуем." >&2; exit 1; }

   tmp_tracked="$(mktemp)"; tmp_new="$(mktemp)"

   # (1) tracked-дельта: изменённые, удалённые и добавленные-в-индекс файлы против baseline.
   #     `git diff --name-only <rev>` сравнивает РАБОЧЕЕ ДЕРЕВО с ревизией, поэтому покрывает
   #     и staged, и unstaged. Каталог артефактов сессии исключён — он живёт своей жизнью.
   #     `-c core.quotePath=false` — иначе не-ASCII пути печатаются в C-кавычках
   #     (`"\320\272..."`) и не совпадают с реальными именами файлов ниже по пайплайну.
   git -c core.quotePath=false diff --name-only "$base_rev" -- . ':!.claude/sessions/**' > "$tmp_tracked"

   # (2) новые untracked = untracked сейчас МИНУС untracked, записанные в baseline.
   comm -23 \
     <(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' \
         | grep -v '^\.claude/sessions/' | LC_ALL=C sort) \
     <(sed -n 's/^untracked //p' "$sdir/prototype_baseline.txt" | LC_ALL=C sort) \
     > "$tmp_new"
   ```
   Если оба файла (`$tmp_tracked` и `$tmp_new`) пусты — сообщи пользователю «прототип не произвёл изменений» и заверши команду БЕЗ гейта (нечего принимать, нечего откатывать); удали временные файлы (`rm -f "$tmp_tracked" "$tmp_new"`) и остановись здесь.

4. **Предъявление результата.** Покажи пользователю:
   - сводку: `attempt N`, число изменённых tracked-файлов (строк в `$tmp_tracked`), число новых файлов (строк в `$tmp_new`), `git diff --stat "$base_rev" -- . ':!.claude/sessions/**'`;
   - полный список новых файлов из `$tmp_new` (путь + число строк каждого файла);
   - дифф tracked-файлов: `git diff "$base_rev" -- . ':!.claude/sessions/**'`. Если он длиннее примерно 400 строк — вместо полного текста покажи `--stat` и первые 200 строк, предложи развернуть дифф полностью по отдельному запросу.

5. **`AskUserQuestion` — безусловно, независимо от `gate_mode` (REQ-VIBE-10).** Этот гейт интерактивен ВСЕГДА, в том числе при `gate_mode: "auto"` — не пропускай его и не подставляй дефолт вместо ответа пользователя.
   - Если `gate_mode == "auto"` — перед вопросом выведи одну строку: «стоп-рубрика протокола: деструктивная операция + продуктовое суждение — авторежим здесь не действует». В `auto_decisions.md` НИЧЕГО не записывай: дефолта, который можно было бы принять автоматически, для этого решения не существует.
   - Три варианта ответа:
     - **Принять** — прототип достиг цели, переходим к легализации.
     - **Отклонить** — откатить показанный список файлов до состояния baseline.
     - **Продолжить работу** — выйти без действий: пользователь не обязан принимать решение прямо сейчас (неразрушающий третий выход).

6. **Ветка «Отклонить» — точечный откат.** Выполни ТОЛЬКО после явного ответа «Отклонить». Список и дифф уже были предъявлены на шаге 4 как часть самого решения (REQ-VIBE-5) — повторного подтверждения НЕ запрашивай, второе «вы уверены?» обесценило бы первое.

   **Пересчёт списка обязателен — не полагайся на переменные шага 3.** Между шагом 3 и этой точкой пройдена `AskUserQuestion` (шаг 5): shell-сессия, в которой на шаге 3 были вычислены `$base_rev`/`$tmp_tracked`/`$tmp_new`, к этому моменту гарантированно завершена — ни переменные, ни временные файлы `mktemp` больше не существуют. Рабочее дерево между шагом 3 и этой точкой не менялось (единственное действие пользователя — ответ на вопрос), поэтому пересчёт по тому же `prototype_baseline.txt` даёт ровно тот список, что был показан на шаге 4. Весь блок ниже — **один непрерывный bash-блок**, исполняемый одним ходом: сначала пересчёт (дословно тот же фрагмент, что и на шаге 3, построчно совпадающий с блоком «Вычисли дифф прототипа» в `commands/retrack.md`), затем сам откат. **Никогда не используй `git clean -fd`, `git checkout .` или `git reset --hard`** — только поимённые операции по спискам `$tmp_tracked`/`$tmp_new`:
   ```bash
   sdir=".claude/sessions/<id>"
   base_rev="$(awk '$1=="rev"{print $2; exit}' "$sdir/prototype_baseline.txt")"
   git rev-parse --verify --quiet "${base_rev}^{commit}" >/dev/null || {
     echo "SDX proto: baseline-ревизия ${base_rev} не найдена — точечный откат недоказуем." >&2; exit 1; }

   tmp_tracked="$(mktemp)"; tmp_new="$(mktemp)"

   # (1) tracked-дельта: изменённые, удалённые и добавленные-в-индекс файлы против baseline.
   #     `git diff --name-only <rev>` сравнивает РАБОЧЕЕ ДЕРЕВО с ревизией, поэтому покрывает
   #     и staged, и unstaged. Каталог артефактов сессии исключён — он живёт своей жизнью.
   #     `-c core.quotePath=false` — иначе не-ASCII пути печатаются в C-кавычках
   #     (`"\320\272..."`) и не совпадают с реальными именами файлов ниже по пайплайну.
   git -c core.quotePath=false diff --name-only "$base_rev" -- . ':!.claude/sessions/**' > "$tmp_tracked"

   # (2) новые untracked = untracked сейчас МИНУС untracked, записанные в baseline.
   comm -23 \
     <(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' \
         | grep -v '^\.claude/sessions/' | LC_ALL=C sort) \
     <(sed -n 's/^untracked //p' "$sdir/prototype_baseline.txt" | LC_ALL=C sort) \
     > "$tmp_new"

   # (а) tracked: восстановить содержимое baseline; файлы, созданные прототипом и попавшие
   #     в индекс, в baseline отсутствуют — их надо удалить, а не «восстанавливать».
   while IFS= read -r f; do
     [ -z "$f" ] && continue
     if git cat-file -e "$base_rev:$f" 2>/dev/null; then
       git checkout "$base_rev" -- "$f"
     else
       git rm -f --quiet -- "$f"
     fi
   done < "$tmp_tracked"

   # (б) новые untracked: удалить поимённо. НИКАКОГО `git clean -fd`.
   while IFS= read -r f; do
     [ -z "$f" ] && continue
     rm -f -- "$f"
   done < "$tmp_new"

   # (в) прибрать каталоги, опустевшие после (б); непустые каталоги не трогаются.
   while IFS= read -r f; do
     d="$(dirname -- "$f")"
     [ "$d" = "." ] || rmdir -p --ignore-fail-on-non-empty -- "$d" 2>/dev/null || true
   done < "$tmp_new"

   # (г) верификация отката: вне каталога сессии дерево обязано стать чистым.
   leftover="$(git status --porcelain -- . ':!.claude/sessions/**')"
   rm -f "$tmp_tracked" "$tmp_new"
   ```
   - Если `leftover` непуст — откат НЕ объявляй успешным: выведи остаток пользователю дословно. Не приписывай остаток файлам под `.gitignore` — `git status --porcelain` без `--ignored` игнорируемые файлы структурно не показывает, эта причина недостижима. Вероятные причины (называй их прозаически, не додумывая другие): (а) путь был изменён или создан УЖЕ ПОСЛЕ вычисления списка на шаге выше — окно между вычислением `$tmp_tracked`/`$tmp_new` и проверкой (в) не защищено блокировкой; (б) сбой отдельной операции отката — `git checkout`, `git rm` или `rm` для конкретного файла в циклах (а)/(б) не выполнился (правами доступа, занятым файлом и т.п.), а код возврата цикла не проверяется построчно; (в) изменения в путях, исключённых из вычисления списка, — в первую очередь untracked-файлы, поимённо записанные в `prototype_baseline.txt` и потому намеренно исключённые из `$tmp_new` (`comm -23`): они не считаются частью прототипа и остаются как есть, поэтому появляются в `leftover` как `?? path`, даже когда откат полностью успешен. Автоматическая доуборка не выполняется в любом случае — удалять файл без явного решения пользователя опаснее, чем оставить его. Сессия при этом всё равно остаётся на `Prototype`; ручная доуборка (или подтверждение, что остаток — ожидаемый файл из причины (в)) — решение пользователя.
   - Независимо от наличия остатка, зафиксируй новую точку восстановления. Захват baseline структурно тот же самый bash-блок, что и на `/sdx:start` для типа `proto` (сверить построчно с `commands/start.md` за вычетом номера попытки: там при первом запуске записывается `1`, здесь — `<N+1>`, где `<N>` — запомненный на шаге 2 `attempt`, номер только что завершённой/отклонённой попытки):
     ```bash
     sdir=".claude/sessions/<id>"
     # Гард безопасности: путь с переводом строки сделал бы построчный формат неоднозначным.
     n_nul=$(git ls-files --others --exclude-standard -z | tr -dc '\0' | wc -c)
     n_ln=$(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' | wc -l)
     if [ "$n_nul" != "$n_ln" ]; then
       echo "SDX proto: среди untracked-файлов есть путь с переводом строки — baseline не может быть записан безопасно." >&2
       exit 1
     fi
     {
       printf '# SDX vibe prototype baseline\n'
       printf 'attempt %s\n' "<N+1>"
       printf 'started %s\n' "$(date '+%Y-%m-%d %H:%M:%S')"
       printf 'rev %s\n' "$(git rev-parse HEAD)"
       git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' | sed 's|^|untracked |'
     } > "$sdir/prototype_baseline.txt"
     git add "$sdir/prototype_baseline.txt" && git commit -m "sdx(<id>): vibe baseline (attempt <N+1>)"
     ```
     Если гард сработал (путь с переводом строки среди untracked) — новый baseline НЕ записывается: сообщи пользователю об ошибке и остановись; попытка `<N+1>` не стартует, пока проблемный путь не будет убран/переименован. Это честный отказ, а не тихая деградация к `git clean`.
   - Затем запиши строку в лог и закоммить её вместе с уже обновлённым baseline (`K` — суммарное число строк, которое было в `$tmp_tracked`+`$tmp_new` до их удаления в шаге (г) выше):
     ```bash
     echo "[$(date '+%Y-%m-%d %H:%M:%S')] [CHECKPOINT] Прототип отклонён (попытка <N>), откат по списку из <K> файлов; новая точка — попытка <N+1>" >> "$sdir/session.log"
     git add "$sdir/prototype_baseline.txt" "$sdir/session.log" && git commit -m "sdx(<id>): vibe prototype rejected (attempt <N>), baseline reset"
     ```
     (Категория лога — существующая `CHECKPOINT`; отдельная пятая категория не вводится, `STAGE_CHANGE` здесь неверен — `stage` не меняется и `sdx-stage.sh` не вызывается.)
   - Сообщи итог пользователю: сколько файлов восстановлено, сколько удалено, остаток (если есть). Напомни, что сессия осталась на `Prototype` и попытка `<N+1>` может начинаться немедленно.

7. **Ветка «Принять» — передача в легализацию.** Вторым `AskUserQuestion` определи целевой трек легализации:
   - варианты ответа — `standard` и `full`;
   - сформулируй **рекомендованный** вариант прозаически, тем же суждением, каким выбирается трек на `/sdx:start`: `full`, если прототип задел публичные контракты, схемы данных или архитектуру, либо получился крупным по объёму диффа; `standard` — в остальных случаях.
   Автоматически, без подтверждения, трек НЕ меняй — целевой трек легализации это второе, отдельное решение пользователя, а не следствие ответа «Принять».
   После получения ответа выполни алгоритм легализации, описанный в `commands/retrack.md`, ветка «Легализация из `vibe`» (spec-after): она пересчитывает тот же дифф прототипа тем же кодом, что и шаг 3 этой команды (заново, в собственных bash-блоках — переменные `$tmp_tracked`/`$tmp_new` шага 3 к этому моменту не переживают ни `AskUserQuestion`, ни последующие `Task`-вызовы субагентов), выполняет реверс-инжиниринг требований ДО коммитов, затем коммитит код прототипа по заново вычисленному списку (никогда `git add -A`) с обязательной пост-проверкой, что коммит состоялся, затем артефакты реверс-инжиниринга, и лишь после этого переводит `track`/`stage` через `sdx-stage.sh retrack`.

Граничные случаи (fail-closed по построению — ничего из перечисленного не додумывай самостоятельно):
- **Прототип успел закоммитить код** (нарушение REQ-VIBE-3, например ручным `git commit` пользователя) — `git diff --name-only "$base_rev"` всё равно перечислит такие файлы (сравнение с ревизией, а не с индексом), и откат восстановит их содержимое, но сами коммиты останутся в истории ветки. Сообщи об этом честно: «откат восстановил содержимое файлов, но коммиты `<sha...>` остались в истории ветки — при необходимости используй `git reset` вручную». Автоматический `git reset` НЕ выполняй — переписывание истории вне объёма этой команды.
- **Прототип удалил tracked-файл** — попадает в `$tmp_tracked`, восстанавливается веткой `git checkout "$base_rev" -- "$f"` в шаге (а). Отдельной обработки не требует.
- **Прототип создал файл и сделал `git add`** — не виден `git ls-files --others`, но виден `git diff --name-only "$base_rev"`; поскольку в baseline-ревизии его нет, срабатывает ветка `git rm -f` в шаге (а).
- **Повторные попытки** — `attempt` инкрементируется, новый baseline перезаписывает старый; история старых baseline доступна через `git log -p -- prototype_baseline.txt`, отдельный архив попыток не ведётся сознательно.
