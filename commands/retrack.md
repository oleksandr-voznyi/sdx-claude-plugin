---
argument-hint: <patch|standard|full|doc>
description: Адаптивная смена трека текущей SDX-сессии (эскалация/деэскалация).
---

Ты выполняешь роль Session Manager. Твоя задача — безопасно изменить трек активной сессии, сохранив уже проделанную работу.

Аргументы: $ARGUMENTS
(Ожидается: целевой трек — `patch`, `standard`, `full` или `doc`)

Трек `vibe` в `argument-hint` намеренно отсутствует: вход в него извне запрещён (REQ-VIBE-11, ADR-018) — единственный путь в `vibe` это `/sdx:start proto "<цель>"` с чистым рабочим деревом. Выход ИЗ `vibe` эта же команда обрабатывает, но не как обычный аргумент — см. правило направления в шаге 2 и ветку легализации в шаге 3.

Алгоритм:
1. Прочитай `session_state.json`: текущий `track` и `stage`.
2. Определи направление смены:
   - **Эскалация** (`patch → standard → full`, а также `doc → {patch, standard, full}`): задача оказалась глубже — затронуты бизнес-логика, контракты, схемы данных или архитектура.
   - **Деэскалация** (`full → standard → patch`): церемония избыточна для реального масштаба. Требует явного подтверждения пользователя.
   - **Правило направления для `doc` (ADR-017).** Эскалация `doc → {patch, standard, full}` разрешена (см. ниже). Обратное направление — `{patch, standard, full} → doc` — ЗАПРЕЩЕНО прозаически (не enforced скриптом, аналог класса DEBT-007): при целевом треке `doc` и текущем треке ≠ `doc` ОСТАНОВИСЬ и НЕ выполняй retrack — объясни пользователю причину (лёгкая Verification `doc` спроектирована в расчёте на нулевой кодовый дифф и не вызывает `qa`; допуск сюда сессии, уже накопившей код, создал бы дыру в покрытии) и предложи альтернативу: довести текущую сессию до `Closeout` её треком, затем открыть отдельную `grooming`/`retro`/`postmortem`/`intake`-сессию через `/sdx:start`. Исключение — целевой трек `doc` при уже текущем `doc` (репозиционирование внутри сессии, не смена трека) остаётся разрешённым.
   - **Правило направления для `vibe` (ADR-018).** (а) `vibe → {standard, full}` — не «эскалация по желанию», а **единственный легитимный выход** из трека и обязательный путь после принятия прототипа на `/sdx:proto`. (б) `vibe → patch` — ЗАПРЕЩЕНО: трек `patch` не имеет этапа `Change`/спека вовсе, а легализация обязана произвести реверс-инжиниринг требований. (в) `{patch, standard, full, doc} → vibe` — ЗАПРЕЩЕНО прозаически: если целевой трек `vibe`, а текущий ≠ `vibe`, ОСТАНОВИСЬ и объясни пользователю, что `vibe` доступен только как стартовая точка сессии типа `proto` (`/sdx:start proto "<цель>"`), потому что режим строится на чистом baseline и некоммиченном рабочем дереве, чего у сессии с уже накопленной историей больше нет. Для перехода `vibe → {standard, full}` не действует ни правило переноса дельт из шага 3 (обычные ветки «эскалация»/«деэскалация»), ни общий алгоритм шага 4.1 определения целевого `stage` — используй отдельную ветку «Легализация из `vibe`» ниже.
3. Перенеси накопленные дельты под новый трек:
   - При эскалации из patch/standard/doc в full: промоутни содержимое `change_note.md` (накопленного на этапе `Change` или, для `doc`, на этапе `Update` — то же имя файла, тот же перенос) в полноценные `SPEC.md` (бизнес-дельты) и `DESIGN.md` (технические дельты). Для этого вызови субагентов `ba` и `architect` (Task), передав им `change_note.md` как вход. Ранее пропущенные этапы (Business Spec, Technical Design, Task Planning) становятся активными. При эскалации из `doc` учти: `change_note.md` груминг/retro/postmortem/intake-сессии заведомо тоньше типового `change_note.md` кодовой сессии — содержательная компенсация остаётся прозаическим суждением при вызове `ba`/`architect`.
   - При деэскалации в patch/standard: уже созданные `SPEC.md`/`DESIGN.md`/`PLAN.md` сохраняются как справочные; дальнейшие дельты копятся в `change_note.md`.
   - **Ветка «Легализация из `vibe` (spec-after)».** Предусловия: текущий `track == "vibe"`, `stage == "Prototype"`, прототип уже принят пользователем на `/sdx:proto` (эта ветка — прямое продолжение того гейта), целевой трек — `standard` или `full`. Источник дельт — не `change_note.md`, а **фактический дифф прототипа**, вычисляемый ровно тем же способом, что и в `/sdx:proto` (по `prototype_baseline.txt`), а не «по памяти» и не по `git status`. Порядок операций строгий — не переставляй шаги:

     1. **Вычисли дифф прототипа** тем же кодом, что и `/sdx:proto` (см. `commands/proto.md` шаг 3):
        ```bash
        sdir=".claude/sessions/<id>"
        base_rev="$(awk '$1=="rev"{print $2; exit}' "$sdir/prototype_baseline.txt")"
        git rev-parse --verify --quiet "${base_rev}^{commit}" >/dev/null || {
          echo "SDX proto: baseline-ревизия ${base_rev} не найдена — точечный откат недоказуем." >&2; exit 1; }

        tmp_tracked="$(mktemp)"; tmp_new="$(mktemp)"

        # (1) tracked-дельта: изменённые, удалённые и добавленные-в-индекс файлы против baseline.
        #     `git diff --name-only <rev>` сравнивает РАБОЧЕЕ ДЕРЕВО с ревизией, поэтому покрывает
        #     и staged, и unstaged. Каталог артефактов сессии исключён — он живёт своей жизнью.
        git diff --name-only "$base_rev" -- . ':!.claude/sessions/**' > "$tmp_tracked"

        # (2) новые untracked = untracked сейчас МИНУС untracked, записанные в baseline.
        comm -23 \
          <(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' \
              | grep -v '^\.claude/sessions/' | LC_ALL=C sort) \
          <(sed -n 's/^untracked //p' "$sdir/prototype_baseline.txt" | LC_ALL=C sort) \
          > "$tmp_new"
        ```
        Это единственный допустимый источник «что было сделано» — не память диалога и не `git status`.

     2. **Spec-after реверс-инжиниринг — ДО коммитов.** `retrack` проверяет артефакты на диске, а не в git, поэтому порядок «сначала артефакты, потом коммиты» полностью легален:
        - цель **`standard`** — `change_note.md` создаётся **инлайн основной сессией** (симметрия с треком `standard`, где `Discovery`/`Change` идут инлайн, без `architect`). Содержимое: что и зачем сделано (переформулировано как требование, а не как пересказ диффа), перечень затронутых файлов, сознательно отложенное. Если объём прототипа таков, что инлайн реверс-инжиниринг честно не помещается, — это сигнал, что целевой трек `full`, а не повод писать тонкий `change_note.md`;
        - цель **`full`** — последовательно, через Task: `architect` → `context_report.md` (ретроспективный: что прототип задел в кодовой базе, какие интеграционные точки и допущения принял де-факто); `ba` → `SPEC.md` (Delta-формат, `[ADDED]`); `architect` → `DESIGN.md`; `lead-dev` → `PLAN.md` (задачи вида «покрыть тестами / довести до контракта / убрать срезанные углы», не «реализовать с нуля»). Всем четверым на вход даются ПУТИ + команда получения диффа, а не тело диффа (экономия токенов, ADR-005), кроме случая, когда дифф мал и передать его дешевле.

     3. **Первый коммит кода (REQ-VIBE-7) и пост-проверка, что он состоялся.** Между п.1 и этой точкой пройдена серия `Task`-вызовов субагентов (п.2): shell-сессия, в которой п.1 вычислил `$base_rev`/`$tmp_tracked`/`$tmp_new`, к этому моменту гарантированно завершена — ни переменные, ни временные файлы `mktemp` больше не существуют. Весь блок ниже — **один непрерывный bash-блок**, исполняемый одним ходом: сначала пересчёт списка (дословно тот же фрагмент, что и в п.1, построчно совпадающий с шагом 3 `commands/proto.md`), затем коммит только по вычисленному списку (никогда `git add -A`), затем обязательная пост-проверка ДО смены трека (п.5) того, что коммит действительно состоялся. Учти: `git add --` без аргументов завершается кодом `0` и ничего не стажирует, а последующий `git commit` в этом случае падает «nothing to commit» (ненулевой код) — оба симптома («add» ничего не стажировал и рабочее дерево вне каталога сессии осталось грязным) трактуются одинаково: коммит не состоялся, REQ-VIBE-7 не выполнен, легализация прерывается прежде, чем `track` будет изменён.
        ```bash
        sdir=".claude/sessions/<id>"
        base_rev="$(awk '$1=="rev"{print $2; exit}' "$sdir/prototype_baseline.txt")"
        git rev-parse --verify --quiet "${base_rev}^{commit}" >/dev/null || {
          echo "SDX proto: baseline-ревизия ${base_rev} не найдена — точечный откат недоказуем." >&2; exit 1; }

        tmp_tracked="$(mktemp)"; tmp_new="$(mktemp)"

        # (1) tracked-дельта: изменённые, удалённые и добавленные-в-индекс файлы против baseline.
        #     `git diff --name-only <rev>` сравнивает РАБОЧЕЕ ДЕРЕВО с ревизией, поэтому покрывает
        #     и staged, и unstaged. Каталог артефактов сессии исключён — он живёт своей жизнью.
        git diff --name-only "$base_rev" -- . ':!.claude/sessions/**' > "$tmp_tracked"

        # (2) новые untracked = untracked сейчас МИНУС untracked, записанные в baseline.
        comm -23 \
          <(git ls-files --others --exclude-standard -z | tr '\0' '\n' | sed '/^$/d' \
              | grep -v '^\.claude/sessions/' | LC_ALL=C sort) \
          <(sed -n 's/^untracked //p' "$sdir/prototype_baseline.txt" | LC_ALL=C sort) \
          > "$tmp_new"

        git add -- $(cat "$tmp_tracked") $(cat "$tmp_new")
        git commit -m "sdx(<id>): vibe prototype code (legalized)"
        commit_rc=$?
        rm -f "$tmp_tracked" "$tmp_new"

        # Пост-проверка REQ-VIBE-7: коммит кода обязан состояться ДО смены трека. Проверяем ДВА
        # независимых симптома отказа: код возврата коммита и фактическую чистоту дерева —
        # `git add --` без аргументов молча возвращает 0, поэтому одного кода возврата мало.
        leftover_code="$(git status --porcelain -- . ':!.claude/sessions/**')"
        if [ "$commit_rc" -ne 0 ] || [ -n "$leftover_code" ]; then
          echo "SDX retrack: коммит кода прототипа (REQ-VIBE-7) не состоялся — трек НЕ меняется." >&2
          [ -n "$leftover_code" ] && printf '%s\n' "$leftover_code" >&2
          exit 1
        fi
        ```
        (при путях с пробелами — построчный `xargs -d'\n' git add --`). Запрет `git add -A`/`git add .` симметричен запрету слепого авто-коммита в `switch.md` и служит той же цели: посторонние untracked-файлы пользователя не должны попасть в ветку сессии. **Если пост-проверка остановила выполнение (код `1`) — ОСТАНОВИСЬ.** Не переходи ни к п.4 (артефакты), ни тем более к п.5 (смена трека, `Edit` поля `track`): `track` в `session_state.json` в этой точке ещё не тронут, рассинхронизации нет. Сообщи пользователю остаток `git status` дословно и предложи разобраться вручную (`git status`, `git diff`) перед повторным вызовом легализации.

     4. **Второй коммит — артефакты реверс-инжиниринга:**
        ```bash
        git add .claude/sessions/<id> && git commit -m "sdx(<id>): legalize prototype — spec-after artifacts"
        ```
        Порядок «код → артефакты» выбран для читаемости `git log`: первый коммит — точное замыкание прототипа (`git show` даёт ровно его), второй — выведенный из него контракт. Для `reviewer` порядок безразличен: `verify.md` шаг 4 берёт дифф всей ветки против основной.

     5. **Смена трека и этапа** — дальше выполняется существующий механизм этой команды, шаг 4, БЕЗ изменений: `Edit` поля `track` (`vibe` → `standard`/`full`), сброс `gate_mode` в `interactive` (см. уточнение к шагу 4.2 ниже), затем вызов `sdx-stage.sh retrack "<id>" "Execution"` (целевой `stage` для легализации — всегда `Execution`, см. уточнение к шагу 4.1 ниже) с fallback на `Discovery` нового трека при отказе guard-а.

     6. Дальше — обычный жизненный цикл целевого трека: `Execution` (доводка, `developer`; с этого момента `stop-gate` снова активен, а норма инкрементальных коммитов действует без исключений — REQ-VIBE-9) → `Verification` (`qa` + fresh-eyes `reviewer`) → `Closeout` (`/sdx:archive`), уже с собственным `Closeout` целевого трека.
4. Пересчитай трек и переведи `stage` через единственный писатель:
   1. Определи целевой `stage` тем же прозаическим правилом, что и раньше — **ближайший активный этап нового трека, не дальше, чем позволяют фактически существующие гейт-артефакты** (это смысловое суждение остаётся за тобой — скрипт его не вычисляет, только валидирует). Учти: «тот же этап под тем же именем» проходит НЕ всегда — при эскалации `patch → full` этап `Execution` формально активен в `full`, но будет отклонён, пока нет `context_report.md`/`SPEC.md`/`DESIGN.md`/`PLAN.md`. Безопасный fallback при любом отказе — первый активный этап нового трека, он достижим всегда. **Исключение — легализация из `vibe`:** целевой `stage` всегда `Execution` для обоих целевых треков (`standard`/`full`) — артефакты реверс-инжиниринга уже созданы шагом 3-«Легализация из `vibe`» пунктом 2, а не «ближайшим активным этапом по общему правилу»; безопасный fallback при отказе guard-а тот же самый — первый активный этап нового трека (`Discovery`).
   2. Обнови `track` в `session_state.json` через `Edit` — это легитимный прямой путь (deny-хук `stage-write-guard.sh` специфичен полю `stage`, поле `track` не блокирует). При **эскалации** (включая переходы ИЗ `doc` в `{patch, standard, full}` и легализацию ИЗ `vibe` в `{standard, full}`) дополнительно сбрось `gate_mode` в `"interactive"`, если был `"auto"`, и сообщи об этом пользователю: масштаб изменился — исходное решение «задача тривиальна» недействительно (ADR-014). Деэскалация `gate_mode` не меняет.
   3. В ОДНОМ Bash-ходе запиши лог и вызови перевод `stage` (уже с обновлённым на шаге 4.2 `track`):
      ```bash
      echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] Смена трека <old_track> → <new_track>" >> .claude/sessions/<id>/session.log \
        && bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<id>" "<target-stage>"
      ```
      Скрипт проверяет ДВА условия: (а) «`<target-stage>` активен в уже обновлённом `track`»; (б) evidence-based forward-skip guard (REQ-RETRACK-2) — у каждого этапа НОВОГО трека, предшествующего `<target-stage>` в порядке матрицы, гейт-артефакт должен фактически существовать и быть непустым на диске (то же самое, что проверил бы `next` при реальном проходе через этот этап; для `Change` — `change_note.md` ИЛИ пара `SPEC.md`+`DESIGN.md`). Ни то, ни другое условие не требует forward-гейта САМОГО `<target-stage>` (это по-прежнему не продвижение вперёд, REQ-RETRACK-1).

      Практическое следствие для шага 4.1: прямая эскалация в `full` дальше собственного `Discovery` (то есть сразу в `Business Spec`/`Technical Design`/`Task Planning`) теперь требует, чтобы `context_report.md` (и, соответственно, `SPEC.md`/`DESIGN.md` для более дальних целей) реально существовал на диске — предполагать этот выбор целью «по умолчанию» без факта на диске больше нельзя, вызов будет отклонён кодом `1`.
      - **Код `0`** (включая no-op `OK no-op <stage>`, если цель совпала с текущим этапом): закоммить `track`/`gate_mode` (шаг 4.2), лог и `stage` одним коммитом на ветку `sdx/<id>`: `git add .claude/sessions/<id>/session_state.json .claude/sessions/<id>/session.log && git commit -m "sdx(<id>): retrack -> <new_track>"`.
      - **Код `1`** (цель из шага 4.1 не активна в уже обновлённом треке, ЛИБО не пройден forward-skip guard — какой-то предшествующий ей этап нового трека не подтверждён гейт-артефактом на диске): сообщи пользователю текст stderr дословно (он называет конкретный непройденный этап и его артефакт). В этот момент `track` в файле уже изменён этим ходом, а `stage` — ещё нет: файл находится в промежуточном состоянии (значение `stage` может быть неактивно в новом `track`). Порядок восстановления, НЕ оставляя это состояние висеть между ходами:
        1. Не коммить ничего на этом шаге — `track` в рабочем дереве изменён, но ещё не закоммичен (шаг 4.2 предшествует шагу 4.3 в рамках одного незакоммиченного изменения).
        2. Пересмотри целевой `stage` по правилу шага 4.1, но уже глядя на список активных этапов НОВОГО `track` (таблица «Профили флоу» в `sdx/protocol.md`, либо явно перечисли этапы для конкретного трека) — выбери из них ближайший, чей гейт-артефакт (и артефакты всех этапов ПЕРЕД ним) реально есть на диске.
        3. **Собственный первый активный этап нового трека — всегда безопасный fallback.** У него по построению пустая цепочка предшественников (нечего проверять), поэтому он никогда не отклоняется guard-ом из шага выше (б) — если переоценка неочевидна или на диске недостаточно артефактов для более продвинутой цели, откатись именно на него, а не оставляй ход незавершённым.
        4. Немедленно, в том же ходе, повтори вызов шага 4.3 с исправленной целью. Не переходи к шагу 5 и не завершай ход, пока повторный вызов не вернёт код `0`.
        5. Если даже собственный первый активный этап нового трека не подходит по смыслу задачи (редкий случай — обычно это сигнал, что смена трека вообще была ошибкой) — верни `track` к прежнему значению через `Edit` (легитимный прямой путь, поле `track` deny-хуком не блокируется), чтобы файл не оставался в рассинхронизированном состоянии, и только затем задай пользователю уточняющий вопрос.
        В любом случае ход не считается завершённым, пока `track` и `stage` не согласованы между собой (т.е. `stage` — активный этап `track` по матрице, и его гейт-цепочка подтверждена) и оба закоммичены одним коммитом.
