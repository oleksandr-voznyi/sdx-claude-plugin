argument-hint: <patch|standard|full|doc>
description: Адаптивная смена трека текущей SDX-сессии (эскалация/деэскалация).
---

Ты выполняешь роль Session Manager. Твоя задача — безопасно изменить трек активной сессии, сохранив уже проделанную работу.

Аргументы: $ARGUMENTS
(Ожидается: целевой трек — `patch`, `standard`, `full` или `doc`)

Алгоритм:
1. Прочитай `session_state.json`: текущий `track` и `stage`.
2. Определи направление смены:
   - **Эскалация** (`patch → standard → full`, а также `doc → {patch, standard, full}`): задача оказалась глубже — затронуты бизнес-логика, контракты, схемы данных или архитектура.
   - **Деэскалация** (`full → standard → patch`): церемония избыточна для реального масштаба. Требует явного подтверждения пользователя.
   - **Правило направления для `doc` (ADR-017).** Эскалация `doc → {patch, standard, full}` разрешена (см. ниже). Обратное направление — `{patch, standard, full} → doc` — ЗАПРЕЩЕНО прозаически (не enforced скриптом, аналог класса DEBT-007): при целевом треке `doc` и текущем треке ≠ `doc` ОСТАНОВИСЬ и НЕ выполняй retrack — объясни пользователю причину (лёгкая Verification `doc` спроектирована в расчёте на нулевой кодовый дифф и не вызывает `qa`; допуск сюда сессии, уже накопившей код, создал бы дыру в покрытии) и предложи альтернативу: довести текущую сессию до `Closeout` её треком, затем открыть отдельную `grooming`/`retro`/`postmortem`/`intake`-сессию через `/sdx:start`. Исключение — целевой трек `doc` при уже текущем `doc` (репозиционирование внутри сессии, не смена трека) остаётся разрешённым.
3. Перенеси накопленные дельты под новый трек:
   - При эскалации из patch/standard/doc в full: промоутни содержимое `change_note.md` (накопленного на этапе `Change` или, для `doc`, на этапе `Update` — то же имя файла, тот же перенос) в полноценные `SPEC.md` (бизнес-дельты) и `DESIGN.md` (технические дельты). Для этого вызови субагентов `ba` и `architect` (Task), передав им `change_note.md` как вход. Ранее пропущенные этапы (Business Spec, Technical Design, Task Planning) становятся активными. При эскалации из `doc` учти: `change_note.md` груминг/retro/postmortem/intake-сессии заведомо тоньше типового `change_note.md` кодовой сессии — содержательная компенсация остаётся прозаическим суждением при вызове `ba`/`architect`.
   - При деэскалации в patch/standard: уже созданные `SPEC.md`/`DESIGN.md`/`PLAN.md` сохраняются как справочные; дальнейшие дельты копятся в `change_note.md`.
4. Пересчитай трек и переведи `stage` через единственный писатель:
   1. Определи целевой `stage` тем же прозаическим правилом, что и раньше — **ближайший активный этап нового трека, не дальше, чем позволяют фактически существующие гейт-артефакты** (это смысловое суждение остаётся за тобой — скрипт его не вычисляет, только валидирует). Учти: «тот же этап под тем же именем» проходит НЕ всегда — при эскалации `patch → full` этап `Execution` формально активен в `full`, но будет отклонён, пока нет `context_report.md`/`SPEC.md`/`DESIGN.md`/`PLAN.md`. Безопасный fallback при любом отказе — первый активный этап нового трека, он достижим всегда.
   2. Обнови `track` в `session_state.json` через `Edit` — это легитимный прямой путь (deny-хук `stage-write-guard.sh` специфичен полю `stage`, поле `track` не блокирует). При **эскалации** (включая переходы ИЗ `doc` в `{patch, standard, full}`) дополнительно сбрось `gate_mode` в `"interactive"`, если был `"auto"`, и сообщи об этом пользователю: масштаб изменился — исходное решение «задача тривиальна» недействительно (ADR-014). Деэскалация `gate_mode` не меняет.
   3. В ОДНОМ Bash-ходе запиши лог и вызови перевод `stage` (уже с обновлённым на шаге 4.2 `track`):
      ```bash
      echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] Смена трека <old_track> → <new_track>" >> .claude/sessions/<id>/session.log \
        && bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<id>" "<target-stage>"
      ```
      Скрипт проверяет ДВА условия: (а) «`<target-stage>` активен в уже обновлённом `track`»; (б) evidence-based forward-skip guard (REQ-RETRACK-2) — у каждого этапа НОВОГО трека, предшествующего `<target-stage>` в порядке матрицы, гейт-артефакт должен фактически существовать и быть непустым на диске (то же самое, что проверил бы `next` при реальном проходе через этот этап; для `Change` — `change_note.md` ИЛИ пара `SPEC.md`+`DESIGN.md`). Ни то, ни другое условие не требует forward-гейта САМОГО `<target-stage>` (это по-прежнему не продвижение вперёд, REQ-RETRACK-1).

      Практическое следствие для шага 4.1: прямая эскалация в `full` дальше собственного `Discovery` (то есть сразу в `Business Spec`/`Technical Design`/`Task Planning`) теперь требует, чтобы `context_report.md` (и, соответственно, `SPEC.md`/`DESIGN.md` для более дальних целей) реально существовал на диске — предполагать этот выбор целью «по умолчанию» без факта на диске больше нельзя, вызов будет отклонён кодом `1`.
      - **Код `0`** (включая no-op `OK no-op <stage>`, если цель совпала с текущим этапом): закоммить `track`/`gate_mode` (шаг 4.2), лог и `stage` одним коммитом на ветку `sdx/<id>`: `git add .claude/sessions/<id>/session_state.json .claude/sessions/<id>/session.log && git commit -m "sdx(<id>): retrack -> <new_track>"`.
      - **Код `1`** (цель из шага 4.1 не активна в уже обновлённом треке, ЛИБО не пройден forward-skip guard — какой-то предшествующий ей этап нового трека не подтверждён гейт-артефактом на диске): сообщи пользователю текст stderr дословно (он называет конкретный непройденный этап и его артефакт). В этот момент `track` в файле уже изменён этим ходом, а `stage` — ещё нет: файл находится в промежуточном состоянии (значение `stage` может быть неактивно в новом `track`). Порядок восстановления, НЕ оставляя это состояние висеть между ходами:
        1. Не коммить ничего на этом шаге — `track` в рабочем дереве изменён, но ещё не закоммичен (шаг 4.2 предшествует шагу 4.3 в рамках одного незакоммиченного изменения).
        2. Пересмотри целевой `stage` по правилу шага 4.1, но уже глядя на список активных этапов НОВОГО `track` (таблица «Профили флоу» в `sdx/protocol.md`, либо явно перечисли этапы для конкретного трека) — выбери из них ближайший, чей гейт-артефакт (и артефакты всех этапов ПЕРЕД ним) реально есть на диске.
        3. **Собственный первый активный этап нового трека — всегда безопасный fallback.** У него по построению пустая цепочка предшественников (нечего проверять), поэтому он никогда не отклоняется guard-ом из шага выше (б) — если переоценка неочевидна или на диске недостаточно артефактов для более продвинутой цели, откатись именно на него, а не оставляй ход незавершённым.
        4. Немедленно, в том же ходе, повтори вызов шага 4.3 с исправленной целью. Не переходи к шагу 5 и не завершай ход, пока повторный вызов не вернёт код `0`.
        5. Если даже собственный первый активный этап нового трека не подходит по смыслу задачи (редкий случай — обычно это сигнал, что смена трека вообще была ошибкой) — верни `track` к прежнему значению через `Edit` (легитимный прямой путь, поле `track` deny-хуком не блокируется), чтобы файл не оставался в рассинхронизированном состоянии, и только затем задай пользователю уточняющий вопрос.
        В любом случае ход не считается завершённым, пока `track` и `stage` не согласованы между собой (т.е. `stage` — активный этап `track` по матрице, и его гейт-цепочка подтверждена) и оба закоммичены одним коммитом.
      - **Код `2`** (нет `jq` / нет `session_state.json` / состояние повреждено): сообщи текст stderr дословно и остановись.
5. Сообщи пользователю, какие этапы теперь активны и какие артефакты требуется создать или актуализировать.

Протокол сессий, профили флоу и контракт закрытия: прочитай (Read) `${CLAUDE_PLUGIN_ROOT}/sdx/protocol.md`
