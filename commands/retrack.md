argument-hint: <patch|standard|full>
description: Адаптивная смена трека текущей SDX-сессии (эскалация/деэскалация).
---

Ты выполняешь роль Session Manager. Твоя задача — безопасно изменить трек активной сессии, сохранив уже проделанную работу.

Аргументы: $ARGUMENTS
(Ожидается: целевой трек — `patch`, `standard` или `full`)

Алгоритм:
1. Прочитай `session_state.json`: текущий `track` и `stage`.
2. Определи направление смены:
   - **Эскалация** (`patch → standard → full`): задача оказалась глубже — затронуты бизнес-логика, контракты, схемы данных или архитектура.
   - **Деэскалация** (`full → standard → patch`): церемония избыточна для реального масштаба. Требует явного подтверждения пользователя.
3. Перенеси накопленные дельты под новый трек:
   - При эскалации из patch/standard в full: промоутни содержимое `change_note.md` в полноценные `SPEC.md` (бизнес-дельты) и `DESIGN.md` (технические дельты). Для этого вызови субагентов `ba` и `architect` (Task), передав им `change_note.md` как вход. Ранее пропущенные этапы (Business Spec, Technical Design, Task Planning) становятся активными.
   - При деэскалации в patch/standard: уже созданные `SPEC.md`/`DESIGN.md`/`PLAN.md` сохраняются как справочные; дальнейшие дельты копятся в `change_note.md`.
4. Пересчитай трек и переведи `stage` через единственный писатель:
   1. Определи целевой `stage` тем же прозаическим правилом, что и раньше — **как правило текущий этап, если он активен в новом треке; иначе ближайший активный этап нового трека** (это смысловое суждение остаётся за тобой — скрипт его не вычисляет, только валидирует).
   2. Обнови `track` в `session_state.json` через `Edit` — это легитимный прямой путь (deny-хук `stage-write-guard.sh` специфичен полю `stage`, поле `track` не блокирует). При **эскалации** дополнительно сбрось `gate_mode` в `"interactive"`, если был `"auto"`, и сообщи об этом пользователю: масштаб изменился — исходное решение «задача тривиальна» недействительно (ADR-014). Деэскалация `gate_mode` не меняет.
   3. В ОДНОМ Bash-ходе запиши лог и вызови перевод `stage` (уже с обновлённым на шаге 4.2 `track`):
      ```bash
      echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] Смена трека <old_track> → <new_track>" >> .claude/sessions/<id>/session.log \
        && bash "${CLAUDE_PLUGIN_ROOT}/sdx/hooks/sdx-stage.sh" retrack "<id>" "<target-stage>"
      ```
      Скрипт проверяет только «`<target-stage>` активен в уже обновлённом `track`» — без forward-гейта (это не продвижение вперёд).
      - **Код `0`** (включая no-op `OK no-op <stage>`, если цель совпала с текущим этапом): закоммить `track`/`gate_mode` (шаг 4.2), лог и `stage` одним коммитом на ветку `sdx/<id>`: `git add .claude/sessions/<id>/session_state.json .claude/sessions/<id>/session.log && git commit -m "sdx(<id>): retrack -> <new_track>"`.
      - **Код `1`** (цель из шага 4.1 не активна в уже обновлённом треке — ошибка в определении цели): сообщи пользователю текст stderr дословно. `track` в файле уже изменён этим ходом, а `stage` — ещё нет; пересмотри целевой `stage` (вернись к шагу 4.1) и повтори вызов шага 4.3, не оставляя `track`/`stage` рассинхронизированными.
      - **Код `2`** (нет `jq` / нет `session_state.json` / состояние повреждено): сообщи текст stderr дословно и остановись.
5. Сообщи пользователю, какие этапы теперь активны и какие артефакты требуется создать или актуализировать.

Протокол сессий, профили флоу и контракт закрытия: прочитай (Read) `${CLAUDE_PLUGIN_ROOT}/sdx/protocol.md`
