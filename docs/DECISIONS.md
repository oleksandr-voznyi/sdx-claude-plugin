# SDX — Журнал дизайн-решений (ADR)

Краткие записи о принятых архитектурных решениях фреймворка SDX, их обоснование и **инварианты**, которые нельзя нарушать при дальнейшем развитии. Формат: Контекст → Решение → Обоснование → Инварианты.

---

## ADR-001. Адаптивные треки вместо жёсткого phase-lock
- **Контекст.** Исходный конвейер гнал любую задачу через все 9 этапов. Для мелких правок (багфикс в строку) это расточительно по токенам и времени.
- **Решение.** Введены профили флоу (треки) `patch` / `standard` / `full` — наборы _активных_ этапов. `track` — отдельное поле состояния, `type` — лишь стартовая гипотеза.
- **Обоснование.** Церемония должна быть пропорциональна размеру задачи (идея из OpenSpec), но при этом сохраняются детерминированные гейты и триада источников правды.
- **Инварианты.**
  - Трек НЕ привязан жёстко к типу: багфикс может потребовать архитектуры, фича — оказаться минорной. Смена трека (`/sdx:retrack`) — первоклассная операция.
  - `/sdx:next` ведёт к следующему _активному_ этапу трека, пропуская неактивные.
  - Оркестратор на каждом гейте выполняет scope-check и предлагает эскалацию/деэскалацию.

## ADR-002. Контракт закрытия сессии (Closeout) как обязательный чек-лист
- **Контекст.** Этап архивации систематически выполнялся неполно — особенно перенос дельт в постоянные доки и удаление файлов сессии.
- **Решение.** `/sdx:archive` с обязательным чек-листом из 7 пунктов; оркестратор печатает его с `[ ]/[x]` и не закрывает сессию, пока всё не отмечено.
- **Обоснование.** Полнота завершения — повторяющаяся боль; её нужно контролировать механически, а не «по памяти».
- **Инварианты.**
  - Сессия не считается закрытой без: коммитов в ветку, переноса дельт в `docs/`+документацию, архива `PLAN.md`, обновления глобального лога, слияния в `main` и **удаления каталога сессии**.
  - Удаление `.claude/sessions/<id>/` — пункт, который пропускается чаще всего; выполняется явно.

## ADR-003. `change_note.md` — единый аккумулятор дельт лёгких треков
- **Контекст.** Размазывание мелких дельт по `SPEC.md`/`DESIGN.md` дорого и избыточно для мелких правок.
- **Решение.** В треках `patch`/`standard` дельты копятся в одном `change_note.md`; `SPEC.md`/`DESIGN.md` не трогаются по ходу.
- **Обоснование.** Один файл дешевле в сопровождении; перенос в постоянные доки — отдельный контролируемый шаг.
- **Инварианты.**
  - При эскалации до `full` содержимое `change_note.md` промоутится в полноценные `SPEC.md`/`DESIGN.md`.
  - На Closeout дельты из `change_note.md` ОБЯЗАТЕЛЬНО переносятся в постоянные документы (см. ADR-002).

## ADR-004. Структурированная верификация: 3 оси + fresh-eyes ревьюер
- **Контекст.** Верификация сводилась к «написать тесты и запустить»; автор изменений рационализирует собственные несоответствия.
- **Решение.** Этап Verification = три оси (полнота / корректность / когерентность) + независимый агент `reviewer`, который сверяет контракт (артефакты) с поставкой (diff). Команда `/sdx:verify`, артефакт `verification_report.md`.
- **Обоснование.** Три оси — рубрика (что проверять), fresh-eyes — позиция наблюдателя (откуда). Ревьюер без памяти о решениях ловит то, что автор оправдал бы.
- **Инварианты.**
  - Fresh-eyes ревью выполняется ТОЛЬКО через делегирование Task, НИКОГДА инлайн основной сессией (даже на лёгких треках) — иначе рушится стена контекста.
  - Ревьюеру передаются только артефакты-контракт + diff; НЕ передаются `session.log`, история рассуждений, пояснения «почему так».
  - `FAIL` блокирует гейт; `WARN` требует явного квитирования пользователем. Покрытие из diff — «видимое», не доказанное.
  - Ось когерентности = детектор коллизий триады из CLAUDE.md §4.

## ADR-005. Экономный режим логирования (ходы дороже строк)
- **Контекст.** Обязательный `echo`-в-лог на каждое действие тратит ходы агента (каждый ход — полный round-trip контекста).
- **Решение.** Логируются только `START` / `STAGE_CHANGE` / `CHECKPOINT` / `ERROR`. ARTIFACT выводится из git, INFO/статус не логируется.
- **Обоснование.** Доминирующая стоимость — число tool calls, а не размер записи. Информация не теряется: git — авторитетный реестр артефактов.
- **Инварианты.**
  - Артефакты коммитятся в ветку сессии инкрементально (поддерживает вывод истории из git и Closeout).
  - `/sdx:status` — чисто читающая операция, не мутирует состояние.
  - В Task субагентам передаются **пути**, а не содержимое файлов (двойная токенизация). Исключение — diff для `reviewer` по контракту изоляции.
  - Время + лог + правка state — одним ходом (батч).

## ADR-006. Интерактивное ручное тестирование
- **Контекст.** `MANUAL_TEST.md` отдавался пользователю как статичный файл «иди разбирайся сам».
- **Решение.** Команда `/sdx:manual` ведёт пользователя по шагам: детальная инструкция + ожидаемый результат → сбор фактического результата (`Passed/Failed/Blocked` + наблюдение) через `AskUserQuestion`. `qa` готовит атомарные шаги; `/sdx:verify` вызывает прогон при наличии `MANUAL_TEST.md`.
- **Обоснование.** Интерактивное ведение снижает порог и даёт структурированный сбор результатов; провалы попадают в общий гейт.
- **Инварианты.**
  - Проваленные ручные шаги = находки `FAIL` верификации наравне с автотестами и fresh-eyes ревью.
  - Результаты пишутся в `MANUAL_TEST.md` одним блоком в конце; тривиальные шаги группируются (экономия ходов, см. ADR-005).

## ADR-007. Перенос функциональности между SDX-проектами (Import / Export)
- **Контекст.** Нужен перенос фичи из одного SDX-проекта в другой. Наивное копирование файлов не работает: разные архитектура, стек, нейминг, зависимости.
- **Решение.** Асимметричная пара: `export` — лёгкая read-only команда, пакующая фичу в переносимый бандл; `import` — тяжёлый тип сессии (полный жизненный цикл), который реконсилирует бандл с целью. Бандл несёт **полное замыкание** (срезы `SPEC`/`DESIGN` + код + тесты + зависимости + допущения + точки интеграции). Интеграция — **адаптивно покомпонентно**: `transplant` / `reimplement` / `gap`.
- **Обоснование.** Оба проекта на SDX → перенос triad-aware (нативные дельты, а не только код). Асимметрия отражает реальность: экспорт = извлечение (read), импорт = интеграция (порт). Покомпонентная адаптация согласуется с адаптивной философией фреймворка (ADR-001).
- **Инварианты.**
  - `export` не модифицирует источник; `import` создаётся через `/sdx:import <bundle>`, не через `/sdx:start`.
  - Срезы мёржатся в `SPEC`/`DESIGN` цели через Delta-формат; коллизии триады → стоп и решение пользователя (ADR-004).
  - `gap` (нет опоры в цели) и несовместимость версий SDX не замалчиваются — выводятся пользователю.
  - Решения реконсиляции фиксируются в `import_map.md`; перенос проходит обычный Verification + Closeout.
  - Извлечение и реконсиляция — режимы агента `architect` (без отдельной роли).

## ADR-008. Алиасы моделей вместо пина поколения
- **Контекст.** Фаза 1 (REQ-ROUTE-1) закрепила во frontmatter агентов полные model ID (`claude-sonnet-4-6` и т.д.) — DESIGN Фазы 1 сознательно выбрал ID, «точнее пиннящие поколение», с обязательством обновить раскладку при смене поколения. Поколение сменилось (семейство Claude 5) — раскладка протухла, шесть агентов остались на устаревшем Sonnet 4.6 (аудит 2026-07-01, находка B1). Кроме того, раскладка с конкретными ID была продублирована в `CLAUDE.md` §2 и `architect.md` — три точки дрейфа (находка B2).
- **Решение.** Frontmatter агентов объявляет `model` **алиасом тира**: `reviewer`→`opus`, `tech-writer`→`haiku`, остальные шесть→`sonnet`. Конкретные model ID из документов фреймворка убраны; `CLAUDE.md` описывает только принцип раскладки. Это отменяет решение DESIGN Фазы 1 «полные ID пиннят поколение».
- **Обоснование.** Для ролей фреймворка важен **тир** (фронтир — оценочное суждение; рабочий — код и требования; дешёвый — механика), а не конкретное поколение. Алиас всегда резолвится в актуальное поколение тира — класс проблемы «пин протух» устраняется, а не откладывается до следующей смены поколения. Алиасы — документированная основная форма поля (zod-схема: «Model alias this agent uses»). Трейд-офф «меньше контроля: модель тира может смениться под ногами» принят осознанно.
- **Инварианты.**
  - Привязка ролей к тирам фиксирована: `opus` — оценочное суждение (`reviewer`, эскалация `architect`); `haiku` — механика (`tech-writer`); `sonnet` — остальное. Пересмотр привязки — новый ADR.
  - Конкретные model ID не дублируются в документах фреймворка; источник истины раскладки — frontmatter агентов.
  - Проект-потребитель вправе запиннить конкретный ID per-project override'ом — это не нарушение инварианта.
  - Исторические документы (`docs/specs|designs/phase1-*`, `docs/history/*`) не переписываются.

## ADR-009. git worktree + версионируемые артефакты сессии
> Статус: worktree-часть пересмотрена **ADR-012** (сессия = ветка в основном дереве, один CLI);
> часть «версионируемые артефакты + вариант A» действует без изменений.
- **Контекст.** ADR-005 декларировал «артефакты коммитятся в ветку сессии инкрементально», но
  `.claude/sessions/` целиком в `.gitignore` — коммит физически невозможен (аудит 2026-07-01,
  C7). Параллельно `switch.md` делал слепой `git add -A && git commit` в общем дереве (C2).
  Обе находки решаются одной связкой.
- **Решение.** Одна сессия = один **git worktree** на ветке `sdx/<id>`, расположенный в
  gitignored `.sdx/worktrees/<id>/` внутри репозитория. Содержательные `.md`/`.json`-артефакты
  сессии (`.claude/sessions/<id>/…`) версионируются на этой ветке; эфемерные `.stopgate.*`
  игнорируются точечным паттерном корневого `.gitignore`. Удаление каталога сессии оформляется
  коммитом `git rm -r` НА ВЕТКЕ до мёржа (**вариант A**); мёрж `--no-ff` вносит историю в DAG
  основной ветки, но не в её HEAD-дерево. `archive-verify.sh` переходит с активного `rm -rf` на
  проверку отсутствия + `git worktree remove`. `switch.md` больше не трогает общее дерево:
  переключение = запуск CLI в каталоге нужного worktree (следствие консервативного допущения о
  `CLAUDE_PROJECT_DIR`, §8). Чистый **cutover** без миграции существующих не-worktree сессий.
- **Обоснование.** Делает ADR-005 буквально истинным (устраняет C7); полностью снимает C2 (нет
  общего дерева для порчи); физическая изоляция каталогами закрывает REQ-WT-3 без stash/локов.
  Вариант A предпочтён варианту B (пост-мёрж `rm`+commit прямо в основную ветку), т.к. сохраняет
  инвариант «основная ветка получает только merge-коммиты» и роль хука как gate, а не источника
  прямых коммитов в основную ветку. Расположение внутри репо (vs снаружи) выбрано за
  предсказуемость путей, унификацию с `.sdx/bundles/` и простоту хендофф-инструкций;
  самовложенность закрыта gitignore, watcher-шум — рекомендацией/escape-hatch `SDX_WORKTREE_ROOT`.
- **Инварианты.**
  - Содержательные артефакты сессии tracked на `sdx/<id>`; `.stopgate.*` — никогда (REQ-SESS-1/2).
  - После Closeout HEAD-дерево основной ветки не содержит `.claude/sessions/<id>/`; история
    достижима через merge-DAG до `git branch -d` и после него — через merge-коммит (REQ-SESS-3/4).
  - `switch.md` НЕ выполняет `git add -A`/`commit`/`checkout` в общем дереве (REQ-WT-2).
  - Деструктив (`worktree remove`, `branch -d`) — только после доказанных инвариантов 1/5/6.
  - Пересмотр расположения worktree или варианта удаления — новый ADR.
- **Связь.** Уточняет/операционализирует ADR-005 (не переписывает его); прецедент нового ADR при
  развороте — ADR-008.

## ADR-010. Автоопределение основной ветки вместо хардкода `main`
- **Контекст.** Хуки и команды хардкодят `main` (`archive-verify.sh` инвариант 5, `verify.md`
  diff, prose протокола). На репозиториях с `master`/иным именем инвариант «ветка слита» ложно
  падает/пропускается, diff считается относительно несуществующей ветки (запрос 2026-07-03).
- **Решение.** Единый шелл-хелпер `.claude/sdx/hooks/lib/default-branch.sh` резолвит имя
  основной ветки: `git symbolic-ref refs/remotes/origin/HEAD` → `git config init.defaultBranch`
  (если ветка существует) → эвристика существующих локальных `main`/`master` → last-resort `main`.
  Хелпер переиспользуется хуками (`archive-verify.sh`) и документированно — агентом в prose-
  командах (`verify.md`, `archive.md`, protocol.md). Хелпер безопасен при отсутствии remote и
  всегда печатает непустое имя.
- **Обоснование.** Один источник истины устраняет дрейф между точками хардкода; `origin/HEAD` —
  наиболее авторитетный сигнал «что команда считает основной веткой», с деградацией к локальным
  сигналам при отсутствии remote. Не требует per-project настройки (REQ-BRANCH-4).
- **Инварианты.**
  - Ни хук, ни команда не содержат литерала `main` как имени основной ветки (REQ-BRANCH-1);
    иллюстративные упоминания в доках допускаются с явной пометкой.
  - Резолв — только через `lib/default-branch.sh` (или его документированный one-liner); новые
    места, нуждающиеся в имени основной ветки, обязаны использовать хелпер.
  - Изменение порядка фолбэков — новый ADR.

## ADR-011. Green-run cache в stop-gate (кэш зелёного прогона по отпечатку дерева)
- **Контекст.** На стадиях Execution/Verification stop-gate запускал verify-команду при **каждом**
  `Stop`. Длинная Execution из десятков ходов = десятки прогонов, в том числе когда код между
  ходами не менялся (находка аудита A4, `docs/audit-2026-07-01-recommendations.md`).
- **Решение.** Перед прогоном stop-gate вычисляет отпечаток рабочего дерева
  (`git rev-parse HEAD` + md5 от `git status --porcelain`) и сверяет с отпечатком последнего
  зелёного прогона в `.claude/sessions/<id>/.stopgate.ok`. Совпадение → `exit 0` без перезапуска
  verify. Отпечаток пишется только при зелёном прогоне; красный кэш не создаёт. Проверка
  размещена ПОСЛЕ резолва verify-команды и ДО инкремента loop-guard-счётчика (кэш-хит не крутит
  `.stopgate.count`). `SDX_STOP_GATE=1` обходит чтение кэша (форс всегда честный), но зелёный
  форс всё равно обновляет `.stopgate.ok`.
- **Обоснование.** Отпечаток по `HEAD` + porcelain учитывает и коммиты, и незакоммиченные
  правки → инвалидация автоматическая и точная, без ручного сброса. Опирается на существующий
  инвариант: `.stopgate.*` игнорируются точечным паттерном `.gitignore` (ADR-009), поэтому сам
  кэш-файл не попадает в porcelain и не меняет отпечаток. Семантика гейта сохранена — пол под
  красным прогоном остаётся, кэшируется только повторный зелёный на неизменном дереве.
- **Инварианты.**
  - Кэш пишется исключительно в ветке успешного (зелёного) прогона; красный/hung прогон
    `.stopgate.ok` не создаёт и не обновляет.
  - Проверка кэша — до инкремента loop-guard; кэш-хит не изменяет `.stopgate.count`.
  - `SDX_STOP_GATE=1` обходит чтение кэша (но не запись при зелёном форсе).
  - Изменение состава отпечатка (что входит в fingerprint) — новый ADR.
- **Границы.** Кэш per-session (worktree-safe): `.stopgate.ok` в каталоге сессии, кросс-сессия
  не пересекается. Покрыт тестами `test-stop-gate.sh` [8]–[12] (13/13 зелёные).

## ADR-012. Сессия = ветка в основном рабочем дереве (отказ от worktree, один CLI)
- **Контекст.** Worktree-модель (ADR-009) оказалась операционно тяжёлой: `/sdx:start`
  заканчивался хендоффом «откройте новый CLI в `.sdx/worktrees/<id>/`», Closeout требовал двух
  CLI-инстансов (Фаза 1 в worktree, Фаза 2 в корне). Непрерывность работы в одном CLI ломалась
  на каждом старте и закрытии сессии (запрос пользователя, 2026-07-05). При этом enforcement-хуки
  (`stage-gate.sh`, `stop-gate.sh`) резолвят сессию по имени текущей ветки, а не по worktree —
  физическая изоляция не была для них необходимой.
- **Решение.** Одна сессия = одна ветка `sdx/<id>`, checkout'нутая в ОСНОВНОМ рабочем дереве;
  весь жизненный цикл — в одном CLI, запущенном в корне репозитория. `/sdx:start` = `git
  checkout -b sdx/<id>` при чистом дереве (без worktree, без хендоффа); `/sdx:switch` = `git
  checkout sdx/<id>` с гардом чистого дерева (при грязном — стоп, без авто-коммита); `/sdx:archive`
  = однофазный чек-лист (шаги на ветке → checkout основной → merge --no-ff → archive-verify) в
  том же CLI. Из ADR-009 сохранены без изменений: версионирование артефактов на ветке
  (REQ-SESS-1/2, ADR-005), вариант A (`git rm -r` коммитом на ветке ДО мёржа, REQ-SESS-3/4),
  инварианты archive-verify 1/5/6, запрет слепого авто-коммита (ядро REQ-WT-2), ADR-010.
- **Обоснование.** Цена worktree-модели (перезапуск CLI на каждый старт, два CLI на закрытие)
  оказалась выше её пользы (изоляция параллельных грязных деревьев, REQ-WT-3). Хуки инвариантны
  к модели — переход стоит только правок prose-команд и протокола, ноль правок enforcement-логики.
  Риск потери незакоммиченных правок при переключении купируется гардом чистого дерева и нормой
  инкрементальных коммитов (REQ-SESS-1).
- **Инварианты.**
  - Полный жизненный цикл сессии проходит в одном CLI; команды `/sdx:*` не инструктируют
    открыть другой CLI-инстанс в штатном флоу.
  - `checkout` ветки сессии (start/switch) выполняется только при чистом рабочем дереве;
    авто-коммит чужих изменений запрещён.
  - REQ-SESS-1..4, вариант A и инварианты 1/5/6 archive-verify действуют дословно как при ADR-009.
  - `archive-verify.sh` сохраняет условное `git worktree remove` как legacy-совместимость с
    сессиями, созданными до ADR-012; паттерн `.sdx/worktrees/` в `.gitignore` существующих
    проектов безвреден и может оставаться.
  - REQ-WT-1/3/4/5 сняты (потеряли предмет); возврат к физической изоляции сессий — новый ADR.
- **Связь.** Пересматривает worktree-часть ADR-009 (прецедент разворота — ADR-008); часть
  ADR-009 про версионируемые артефакты и вариант A остаётся в силе и унаследована дословно.

## ADR-013. Дистрибуция фреймворка как плагина Claude Code (root-as-plugin)
- **Контекст.** SDD нужен в нескольких проектах на одном сервере; модель «скопировать
  `.claude/` в каждый проект» (CLAUDE.md §6 до 2026-07-19) тиражирует фреймворк, порождая
  дрейф версий и ручную синхронизацию правок (запрос пользователя, 2026-07-19).
- **Решение.** Фреймворк упакован в плагин Claude Code: корень репозитория — одновременно
  плагин (`.claude-plugin/plugin.json`, `name: sdx`) и marketplace
  (`.claude-plugin/marketplace.json`, `source: "./"`). `commands/`, `agents/`,
  `hooks/hooks.json`, `sdx/{protocol.md, hooks/, templates/}` переносятся из `.claude/` в
  корень; пути фреймворка в контенте — через `${CLAUDE_PLUGIN_ROOT}` (подставляется рантаймом
  в markdown команд/агентов и hooks.json — проверено эмпирически). Установка один раз на
  машину: `/plugin marketplace add <репо>` → `/plugin install sdx@sdx --scope user`. В проекте
  остаётся только per-project слой (`docs/`, `.claude/sessions/`, конфиги `.claude/sdx/`),
  разворачиваемый `/sdx:init` из шаблонов плагина.
- **Обоснование.** Имя плагина `sdx` сохраняет неймспейс `/sdx:*` — UX не меняется. Хуки
  инвариантны к местоположению скриптов (project-данные читаются из `$CLAUDE_PROJECT_DIR`,
  соседи — через `BASH_SOURCE`): релокация без правок логики, 54/54 юнит-тестов зелёные.
  Глобальная (user-scope) активация хуков безопасна благодаря их safe-by-default свойству
  (no-op вне веток `sdx/<id>` и без opt-in конфигов).
- **Инварианты.**
  - Источник истины фреймворка — этот репозиторий; в целевых проектах файлы фреймворка не
    копируются и не форкаются (`/sdx:init` детектирует legacy-копии и предлагает удалить).
  - Контент плагина не пишет в свой корень в рантайме: всё изменяемое — в
    `$CLAUDE_PROJECT_DIR` (сессии, конфиги, docs) на стороне проекта.
  - CLAUDE.md плагином не поставляется; переносимый минимум правил — сниппет
    `sdx/templates/claude-md-snippet.md`, добавляется в проект только с согласия пользователя.
  - Мета-репозиторий разрабатывается через установленный плагин (догфудинг); собственная
    hook-проводка в `.claude/settings.json` удалена.
- **Связь.** Наследует ADR-012 (модель сессий) и ADR-010 (динамическая основная ветка) без
  изменений; операционализирует REQ-PLUGIN-1..6 (`docs/specs/plugin-distribution.md`).

## ADR-014. Авторежим гейтов (gate_mode) + обязательная лёгкая верификация patch-трека
- **Контекст.** Фидбек после реального использования (2026-07-19, бэклог E4): для тривиальных
  задач подтверждение каждого прозаического гейта дороже, чем одна проверка перед закрытием.
  Параллельно находка C3: у patch-трека не было ни одного обязательного слоя независимой
  проверки — Verification «по запросу», fresh-eyes «опционально»; правка уходила в основную
  ветку только с самооценкой автора. Наложение авторежима на такой patch дало бы «в main
  вообще без чьего-либо взгляда».
- **Решение.** (1) Поле `gate_mode: "interactive" | "auto"` в `session_state.json`
  (default `interactive`; opt-in per-session: `/sdx:start --auto` или согласие на
  Discovery/Change-гейте). При `auto` прозаические подтверждения принимают дефолт без
  остановки; каждый дефолт — строка в трек-независимом файле сессии `auto_decisions.md`.
  Единственная обязательная остановка — дисклоуз на входе в Closeout: `auto_decisions.md`
  одним блоком (автоквитированные WARN первыми) + одно явное подтверждение пользователя.
  Стоп-рубрика (авто не пропускает): коллизии триады, FAIL/красный тест-пол, внешние
  контракты/схемы данных, деструктив/prod-guard, эскалация трека (сбрасывает
  `gate_mode → interactive`), шаги `/sdx:manual`. (2) На треке `patch` Verification становится
  обязательной в лёгком объёме независимо от `gate_mode`: регрессионный тест + fresh-eyes
  `reviewer` суженно против `change_note.md`; `/sdx:archive` не начинает чек-лист без
  `verification_report.md` без FAIL. (3) Инвариант ADR-004 «WARN требует явного квитирования
  пользователем» уточнён: в `gate_mode: "auto"` квитирование выполняется отложенно — через
  дисклоуз (прецедент пересмотра инварианта — ADR-008).
- **Обоснование.** Авторежим касается только прозаического слоя — детерминированные хуки
  (stage-gate, stop-gate, prod-guard, archive-verify) не смягчаются, машинный пол сохранён.
  Обязательный лёгкий fresh-eyes на patch превращает авторежим из «доверяю на слово» в
  «доверяю, но проверяю один раз»: diff мал, стоимость копеечная, а независимая проверка
  появляется у самого рискованного сочетания (лёгкий трек + автопрохождение гейтов).
  Отложенное квитирование WARN сохраняет суть инварианта ADR-004 (пользователь видит и явно
  принимает каждый WARN) — меняется только момент предъявления.
- **Инварианты.**
  - Детерминированные хуки не читают и не учитывают `gate_mode` — авторежим никогда не
    ослабляет enforcement-слой. Смягчение хука по `gate_mode` — новый ADR.
  - Дисклоуз перед Closeout неотключаем: сессия с `gate_mode: "auto"` не может быть закрыта
    без предъявления `auto_decisions.md` и явного подтверждения пользователя.
  - Стоп-рубрика — закрытый список протокола; её сокращение — новый ADR (расширение допустимо).
  - Verification на `patch` обязательна и не может быть пропущена ни в одном `gate_mode`;
    контракт изоляции fresh-eyes (ADR-004) действует дословно и в лёгком объёме.
  - Эскалация трека всегда возвращает `gate_mode: "interactive"`.
- **Связь.** Уточняет ADR-004 (момент квитирования WARN); операционализирует находки E4 и C3
  бэклога (`docs/audit-2026-07-01-recommendations.md`); прецедент пересмотра инварианта — ADR-008.

## ADR-015. Трекаемый бэклог как артефакт фреймворка (docs/backlog/)
- **Контекст.** Бэклог жил неструктурированно (находка E2, фидбек 2026-07-19):
  `docs/audit-2026-07-01-recommendations.md` + roadmap в gitignored-бандле; без ID-префиксов,
  трассировки в документации, команд просмотра и планирования волн. Техдолг фиксировался
  нерегулярно; отложенные решения и WARN-находки верификации терялись после закрытия сессий.
- **Решение.** Постоянный трекаемый каталог `docs/backlog/` в per-project слое:
  одна запись = один файл `<ID>-<slug>.md` с машиночитаемым YAML frontmatter
  (`id`, `type`, `status`, `priority`, `wave`, `source`, `session`, `links`) и телом
  `## Суть` / `## Рекомендация` (+`## Резолюция` у закрытых). Префиксы ID по типам:
  `FEAT-`/`BUG-`/`DEBT-`/`IDEA-`/`PROC-`, сквозная нумерация внутри типа; статусы
  `open | in-progress | closed | deferred`; поле `wave` — волна планирования. Индекс —
  `docs/backlog/README.md` (таблицы «Открытые»/«Закрытые»). Команда `/sdx:backlog`:
  список/фильтры/деталь/`add`. Closeout-интеграция: пункт 4 чек-листа расширен —
  вместе с глобальным логом актуализируется бэклог (закрытые записи → `status: closed`
  + `session`; отложенное и неквитированные WARN → новые `DEBT-`/`IDEA-`записи).
  `/sdx:init` создаёт `docs/backlog/` с индексом из шаблона. Находки A*–E* аудита
  2026-07-01 мигрированы в новый формат; файл аудита оставлен историческим снапшотом
  с баннером-указателем.
- **Обоснование.** Файл-на-запись вместо единого файла — атомарные диффы, адресуемость
  по ID, отсутствие merge-конфликтов при параллельных сессиях. Машиночитаемый frontmatter —
  интеграционная точка будущего плагина портфельного управления (парсится без LLM).
  Расширение существующего пункта 4 чек-листа вместо нового пункта — сознательное решение:
  нумерация инвариантов 1/5/6 зашита в `archive-verify.sh` и ADR-009/012, ренумерация
  сломала бы ссылки. Enforcement-слой не затронут: бэклог — прозаический слой,
  `docs/**` всегда открыт stage-gate.
- **Инварианты.**
  - Схема frontmatter расширяется только аддитивно (новые поля — можно, переименование/
    удаление существующих — новый ADR): на неё завязана будущая машинная интеграция.
  - ID записи неизменяем после создания (адресуемость из истории/логов); ре-классификация
    типа — новая запись со ссылкой на старую, а не переименование.
  - Нумерация пунктов Closeout-чек-листа и инвариантов archive-verify не меняется
    расширениями бэклога.
- **Связь.** Операционализирует находку E2; закрывает риск C1 частично (roadmap-идеи получают
  трекаемый дом — `IDEA-`записи); продолжает линию ADR-014 (auto_decisions → DEBT-записи
  на Closeout — механизм «отложенное не теряется»).

## ADR-016. Единственный писатель `stage` + deny-хук (закрытие DEBT-001, DEBT-011)
- **Контекст.** `stage` в `session_state.json` писали шесть прозаических путей без
  единообразной проверки гейта; `backtrack` не проверял вообще ничего и содержал
  нерешённую развилку («бэкап или обнуление» PLAN.md) — DEBT-001 + DEBT-011. При
  верификации (проход 2) отдельно вскрылась находка WARN-4: `retrack` не имел
  собственного forward-skip guard и мог посадить `stage` сразу на `Closeout`, минуя все
  промежуточные гейты того же трека. Первая версия guard'а (проход 2) закрывала это
  общей межтрековой шкалой рангов (`SDX_CANON_ORDER`): `target` допустим, если его ранг
  ≤ ранга текущего `stage`. Проход 3 показал, что этот ранг сам читается из
  самодекларированного `stage` — а `stage` мог быть поднят предыдущим легитимным вызовом
  `retrack` без единого пройденного гейта, то есть два вызова `retrack` подряд образуют
  храповик (F-1): позиция растёт, не будучи подтверждена ни одним артефактом на диске.
- **Решение.** `sdx/hooks/sdx-stage.sh` (4 подкоманды: `init|next|backtrack|retrack`) —
  единственный писатель, со встроенной машиночитаемой матрицей «трек → активные этапы →
  гейт-артефакт» (REQ-STAGE-3). `sdx/hooks/stage-write-guard.sh` — новый PreToolUse-хук
  (`Write|Edit|MultiEdit`), блокирующий прямую правку поля `stage`, узко (REQ-DENY-2:
  только `stage`, не весь файл; создание нового файла не блокируется) — обнаружение
  apply-and-compare (правка симулируется над реальным содержимым файла, денится только
  при фактическом изменении распарсенного `.stage`, а не по текстовому паттерну ключа).
  Backward-переход (`backtrack`) помечает артефакты строго после целевой точки, до конца
  активных этапов трека, без верхней границы по текущему этапу, HTML-баннером в самом
  файле (без бэкапа/зануления, REQ-BACKTRACK-2). `retrack`'ов forward-skip guard
  (REQ-RETRACK-2) переписан на **evidence-based** правило (F-1): целевой этап нового
  трека достижим, только если у КАЖДОГО этапа этого трека, предшествующего цели в
  порядке матрицы, гейт-артефакт фактически существует и непуст на диске — те же
  критерии, что `next` применяет к уходящему этапу, с единственным исключением: этап
  `Change` (patch/standard) засчитывается по `change_note.md` **или** паре
  `SPEC.md`+`DESIGN.md`, тогда как `next` требует именно `change_note.md` — иначе
  честная деэскалация из `full` была бы заблокирована. `stage`/старый трек в проверке не
  участвуют вовсе; ранговая таблица `SDX_CANON_ORDER`/`canon_rank` полностью удалена из
  `sdx-stage.sh`. `sdx-stage.sh` fail-closed без `jq`; `stage-write-guard.sh` fail-open
  без `jq` (REQ-DENY-4, явное решение пользователя, пересмотр исходной формулировки).
- **Обоснование.** Единая точка входа — предпосылка для доказуемого гейта;
  деном-граница технически закрывает случайный (не намеренный) обход тем же инструментом,
  каким писали раньше. Матрица внутри скрипта устраняет дублирование знания о порядке
  этапов, остававшееся ранее только в прозе `protocol.md`+`next.md`. Отказ от рангового
  guard'а в пользу evidence-based — не косметика: позиция (`stage`/ранг) — это
  **утверждение о прогрессе**, artefact на диске — **доказательство** этого прогресса;
  выведение потолка `retrack` из позиции возвращает ровно ту самую самодекларируемость,
  против которой всё это решение изначально направлено (единственный писатель + deny-хук
  бессмысленны, если после них остаётся путь нарастить недоказанный прогресс легитимными
  вызовами). Ключевое свойство, ради которого evidence-based правило безопасно и без
  верхней границы у собственного первого активного этапа трека: у него по построению
  пустая цепочка предшественников, поэтому войти в трек с его стартовой точки всегда
  можно — деэскалация в `patch`/`standard` не блокируется отсутствием артефактов треков
  с более широким скоупом.
- **Инварианты.**
  - Ни одна команда `/sdx:*` не пишет `.stage` в `session_state.json` напрямую
    Write/Edit/MultiEdit-инструментами — только через `sdx-stage.sh`.
  - Deny-хук специфичен полю `stage`; правки `track`/`gate_mode`/`artifacts`/`history`
    остаются легитимными без прохождения через скрипт.
  - Матрица «трек → этапы → гейт-артефакт» существует в ЕДИНСТВЕННОМ машиночитаемом
    месте — внутри `sdx-stage.sh`; `protocol.md` — человекочитаемая проекция.
  - Backward-переход не деструктивен: маркировка — баннер в файле, не удаление/бэкап.
  - **`retrack` не продвигает `stage` вперёд мимо непройденных гейтов; допустимость
    цели определяется фактическим наличием гейт-артефактов предшествующих ей этапов на
    диске, а не позицией `stage`/рангом трека.** Ранговая/позиционная схема — запрещённый
    класс реализации этого guard'а (её возвращение — регресс к F-1, застрахован
    отдельным sanity-тестом на отсутствие `SDX_CANON_ORDER`/`canon_rank` в коде).
  - Схема `session_state.json` не меняется (REQ-COMPAT-1).
- **Границы (не решается этим ADR, названы явно).**
  - **DEBT-003 — Bash-обход.** `session_state.json` остаётся редактируемым напрямую
    через `Bash` (`sed -i`/`jq` in-place) — deny-хук матчится только на
    `Write|Edit|MultiEdit`, эта поверхность им принципиально не покрывается.
  - **Сегмент `..` в пути.** Нормализация пути в deny-хуке ограничена схлопыванием
    `/./` и повторных `/`; сегмент `..` не разрешается. Практически недостижимо (контракт
    инструментов Claude Code требует уже абсолютный путь без относительных сегментов),
    но технически возможный обход остаётся неустранённым этим решением.
  - **Карве-аут восстановления.** Если содержимое `session_state.json` ДО операции уже
    не является валидным JSON (пусто/пробельно, либо реальная ошибка парсинга — валидные
    скалярные документы `null`/`false` НЕ считаются повреждением и защищены обычным
    путём), deny-хук трактует операцию как починку уже сломанного состояния и не денит
    её независимо от результата. Это единственный штатно предусмотренный путь, которым
    правка через `Write`/`Edit`/`MultiEdit` может миновать проверку `stage` — но
    воспользоваться им можно, только если файл уже повреждён способом, который сам этот
    хук не покрывает (в первую очередь — тем же Bash-обходом DEBT-003 выше).
  - **Доказательность гейта — факт наличия, а не качество.** Гейт-артефакт считается
    пройденным по существованию непустого файла (и отсутствию маркера `[FAIL]` там, где
    он применим). Файл-заглушка в одну строку гейт проходит: механизм отличает «нет
    артефакта» от «есть», но не «осмыслен». Содержательная оценка остаётся за
    прозаическим суждением и fresh-eyes ревью. Следствие: баннер `<!-- SDX-OUTDATED -->`,
    проставляемый `backtrack`, **не блокирует ни один гейт** — это сигнал человеку и
    агенту, а не механическое ограничение.
- **Связь.** Закрывает DEBT-001, DEBT-011. Пересекается, но не решает DEBT-003
  (Bash-обход) и DEBT-007 (`status`/`artifacts`/`history` мёртвые поля) — оба явно
  зафиксированы как унаследованные границы. Расширяет паттерн `deny()` из ADR-001
  (`docs/designs/phase1-enforcement-routing.md`) на новый класс операции (диф значения
  поля, а не только путь файла).

## ADR-017. Трек `doc` и типы сессий grooming/retro/postmortem/intake
- **Контекст.** Работа над самим бэклогом (груминг), периодическая ретроспектива
  завершённых сессий, разбор инцидентов (постмортем) и разбор нового значительного
  блока внешних требований (intake) не имели своего типа сессии и трека в SDX —
  делались вне процесса, без структурированного входа/выхода и следа в бэклоге
  (находка E1 аудита, `docs/backlog/PROC-002`). `intake` добавлен позже трёх остальных
  типов — пользователь обнаружил недостающий тип на возврате (`/sdx:backtrack`) с
  Technical Design на Business Spec этой же сессии; вошёл в решение вместе с ними, не
  отдельным follow-up ADR.
- **Решение.** Новый профиль флоу `doc`: `Discovery (лёгкий, инлайн) → Update →
  Verification (лёгкая, обязательная) → Closeout`, единственный гейт-артефакт —
  `change_note.md` (этап `Update`). Четыре новых значения `type` —
  `grooming`/`retro`/`postmortem`/`intake` — жёстко привязаны к `track: "doc"` на
  `/sdx:start` (без диалога о выборе трека, в отличие от триажа `patch`/`standard`/
  `full`). Discovery собирает вход, специфичный типу (срез бэклога / выборка из
  `sessions-log.md` за период-список / интервью об инциденте / приём внешнего
  продуктового материала), без вызова `architect`. Обязательный выход — минимум одно
  изменение `docs/backlog/`; для `retro`/`postmortem`/`intake` дополнительно —
  постоянный документ разбора `docs/history/retro|postmortem|intake/<session_id>.md`
  (перенос из шаблонов `sdx/templates/retro.md`/`postmortem.md`/`intake.md`, первый
  прецедент шаблона сессионного, а не per-project, артефакта). `SDX_STAGE_MATRIX`
  расширяется четырьмя строками; логика переходов `sdx-stage.sh` не меняется — `doc`
  полностью укладывается в существующий формат `track|stage|artifact|fail_marker`.
  **Число типов, привязанных к треку, — величина, к которой матрица безразлична**: три
  типа или четыре, строк в `SDX_STAGE_MATRIX` ровно четыре в обоих случаях, потому что
  тип не параметр матрицы (см. «Инварианты»).
  _Уточнение по факту реализации:_ ровно одно место процедурного кода всё же потребовало
  правки — и не из-за нового трека как такового, а потому что новый трек вскрыл ранее
  скрытый дефект. Текст отказа FAIL-гейта содержал захардкоженную remediation-подсказку
  `/sdx:backtrack --to Execution`; на треке `doc` этапа `Execution` нет, и буквальное
  следование подсказке дало бы пользователю вторую ошибку. Подсказка сделана
  трек-зависимой (при отсутствии `Execution` в треке называется предыдущий активный
  этап — для `doc` это `Update`). Находка обнаружена `qa` на Verification этой же сессии
  и исправлена штатным возвратом `/sdx:backtrack --to Execution`; покрыта сценарием 41
  `test-sdx-stage.sh`, который теперь проверяет не только факт отказа, но и что
  названный в подсказке этап активен в треке.
- **`intake` и `grooming` — разные роли в одном потоке, не синонимы и не альтернативные
  имена одного типа.** Это явное разграничение фиксируется здесь намеренно, чтобы
  будущий читатель ADR не схлопнул их в один тип по сходству контракта файлов (оба
  пишут `docs/backlog/` через один и тот же `change_note.md`, оба на треке `doc`):
  - `intake` **порождает** записи бэклога из НОВОГО внешнего материала (эпик, пачка
    багрепортов, продуктовый материал) — create-путь; стоит по потоку ВЫШЕ `grooming`.
  - `grooming` **перераспределяет** атрибуты (`status`/`priority`/`wave`) уже
    существующих записей — update-путь; не создаёт записи из внешнего материала.
  - Типичный порядок использования: сначала `intake` (материал → записи), затем,
    возможно спустя время и накопление нескольких `intake`/`feature`/`bug`-сессий,
    `grooming` (пересмотр приоритетов среди уже накопленного). Порядок не enforced —
    оба типа доступны в любой момент через `/sdx:start`, это прозаическая рекомендация
    потока, а не жёсткая последовательность.
  - Различие живёт целиком в прозе (Discovery-инструкция `start.md`, состав шаблона
    `intake.md`, этот параграф ADR) — ни один enforcement-хук его не проверяет, оба
    типа в равной мере проходят через один и тот же `doc|Update|change_note.md|no`.
- **Два явных отступления от прецедентов, зафиксированных сознательно:**
  1. **Расширение роли `change_note.md` за пределы ADR-003.** Там файл определён как
     «единый аккумулятор дельт лёгких треков [кода]». Здесь он становится ЕДИНЫМ
     гейт-артефактом ещё и для процессной/рефлексивной работы четырёх разных по смыслу
     типов (груминг-резюме vs резюме ретро/постмортема/intake со ссылкой на параллельный
     содержательный файл). Альтернатива — четыре отдельных трека с разными
     гейт-артефактами (`change_note.md`/`retro.md`/`postmortem.md`/`intake.md`) — была
     отклонена: она противоречит буквальной формулировке PROC-002 («общий... трек») и
     умножает матрицу/тесты/документацию без выигрыша в безопасности (гейт и так —
     только факт-наличия, не содержание). Компенсация размытия семантики — обязательный
     fresh-eyes прогон Verification (см. ниже), а не молчаливое расширение роли файла.
  2. **Обязательная лёгкая Verification без регрессионного тест-прогона.** Это
     сознательно ДРУГОЕ решение, чем «doc-track вообще без Verification», которое
     рассматривалось на Discovery этой же сессии (см. `context_report.md`, развилка 3)
     и было отклонено пользователем на Discovery-гейте в пользу присутствия
     Verification — но БЕЗ вызова `qa` (нет кода/тестов, вызывать нечего). Это
     ДРУГОЙ класс отступления, чем ADR-014: там прецедент был «риск смержить код без
     единого взгляда» (`patch`, автономная работа `developer`); здесь контент по
     конструкции создаётся ИНТЕРАКТИВНО с пользователем в реальном времени — роль
     независимого взгляда частично закрыта самим присутствием пользователя. Тем не
     менее лёгкая Verification сохранена (а не опущена вовсе) — единственная
     объективная проверка когерентности (не противоречит ли новая запись бэклога уже
     существующим, включая проверку, что `intake`-сессия не подменила собой `grooming`)
     остаётся за независимой стороной (`reviewer`), а не за автором правок, симметрично
     оси когерентности CLAUDE.md §4.
- **Обоснование.** `doc` переиспользует существующий контракт матрицы буквально без
  нового кода: `Discovery|-` — прецедент `standard`; `Verification|...|yes` — прецедент
  всех трёх существующих треков; имя файла `change_note.md` у `Update` совпадает с
  `Change`/`Execution`, поэтому эскалация `doc → {patch,standard,full}` работает
  бесплатно через существующий forward-skip guard (REQ-RETRACK-2), без специального
  кода для стадии `Update`. Обратное направление (деэскалация `{patch,standard,full} →
  doc`) сознательно НЕ поддерживается — прозаическим правилом в `retrack.md`, не
  скриптом: лёгкая Verification `doc` рассчитана на нулевой кодовый дифф (`qa` не
  вызывается), пропуск кодовой сессии через `doc` создал бы дыру в покрытии кода,
  а существующий путь `full → standard → patch` уже штатно обслуживает случай «работа
  оказалась мельче ожидаемого» без этого риска.
- **Инварианты.**
  - `SDX_STAGE_MATRIX` — единственный машиночитаемый источник истины и для `doc`;
    `protocol.md` — проекция; sanity-тест 34 расширен на `doc` (закрывает риск дрейфа
    для нового трека тем же механизмом, что уже страхует три существующих).
  - `type` не валидируется enforcement-скриптами и на `doc` — тот же принцип, что и для
    остальных значений (ADR не меняет схему `session_state.json`, REQ-COMPAT-1). Это
    прямое следствие того, что `SDX_STAGE_MATRIX` параметризована `track`, а не `type` —
    добавление `intake` к трём уже принятым типам не меняет ни строки матрицы: тип как
    измерение не пересекается со скриптовым контрактом ни в одной точке.
  - Жёсткая привязка `type ∈ {grooming,retro,postmortem,intake} → track=doc` действует
    ТОЛЬКО в момент `/sdx:start`; после старта `type`/`track` могут разойтись легитимно
    при эскалации (типовое для протокола: `type` — исходная гипотеза, не пожизненный
    инвариант).
  - `intake` и `grooming` семантически различны (create vs update над `docs/backlog/`),
    но технически неразличимы для `sdx-stage.sh` — граница между ними держится
    исключительно прозой (Discovery-инструкция, шаблон `intake.md`, этот ADR).
  - Обратный вход (`{patch,standard,full} → doc` через `retrack`) отклоняется
    прозаически в `retrack.md`, НЕ enforcement-хуком — известная, явно принятая
    граница (аналог класса DEBT-007), не тестируется в `test-sdx-stage.sh`.
  - Верификация `doc` не вызывает `qa`; отсутствие тест-прогона компенсируется
    интерактивностью Discovery/Update (присутствие пользователя в реальном времени) и
    обязательным fresh-eyes прогоном `reviewer` (ось когерентности, обязательно; ось
    полноты — в границах `change_note.md`). **Это не делает хук `stop-gate` прозрачным
    на `doc` «по построению»**: `stop-gate` включается по значению `stage`
    (`Execution|Verification`) независимо от трека и, если в проекте настроен
    `.claude/sdx/verify-cmd.sh`, реально прогоняет его и на `doc|Verification` —
    прозрачность (no-op) возникает только в проектах без известной тест-команды, тот же
    случай, что и на любом другом треке в такой конфигурации.
  - Постоянные документы `docs/history/retro/`, `docs/history/postmortem/`,
    `docs/history/intake/` — новый паттерн хранения по аналогии с
    `docs/history/plans/`; собственного индекса (в отличие от `docs/backlog/README.md`)
    не заводится этим решением.
- **Границы (не решается этим ADR, названы явно).**
  - Качество входа при эскалации `doc → {patch,standard,full}` (тонкий
    груминг-`change_note.md` как вход промоушена в `SPEC.md`/`DESIGN.md`) — не
    объективно проверяемо, остаётся прозаическим суждением при вызове `ba`/`architect`.
  - Отсутствие индекса для `docs/history/retro|postmortem|intake/` при накоплении
    файлов — естественный будущий `IDEA-`-кандидат бэклога, не решается сейчас.
  - Запрет деэскалации в `doc` — процессная, не enforcement-граница, ПРИНЯТЫЙ (не
    просто отложенный) trade-off: явное решение, а не упущение; обходится
    игнорированием инструкции `retrack.md` тем же классом риска, что и прочие
    прозаические условия фреймворка.
  - **Отсутствие структурированного места для входящих артефактов `intake`
    (зависимость от `docs/backlog/PROC-009`, deferred).** У SDX нет каталога/конвенции
    для входящего материала (BRD, макеты, багрепорты со скриншотами) — вопрос сознательно
    отложен `PROC-009` и вне скоупа этого ADR; до его закрытия вход `intake` подаётся
    пользователем явно, а состав фиксируется прозаически (см. «Обработка ошибок»,
    подраздел про `intake`).
- **Связь.** Закрывает `PROC-002` (находка E1 аудита); зависимость `PROC-002 → E2`
  закрыта заранее `PROC-003`/ADR-015 (сессия `fw-backlog-20260719`). Расширяет паттерн
  «лёгкая обязательная Verification без `qa`» из ADR-014 на второй трек. Расширяет роль
  `change_note.md`, определённую ADR-003. Не пересматривает и не ослабляет ADR-016
  (единственный писатель `stage`, evidence-based `retrack`) — `doc` полностью укладывается
  в существующий контракт без исключений в коде. Порождает `PROC-009` (зависимость,
  зафиксированная как отдельная запись бэклога, не закрывается этим ADR).

## Процессные соглашения
- **Git-флоу правок фреймворка.** Изменения коммитятся в отдельную ветку (`feature/*`, `chore/*`), затем сливаются в **основную ветку (автоопределяется, ADR-010 — не хардкод `main`)** и ветка удаляется. Коммит — отдельный явный шаг по запросу пользователя. _Уточнение (ADR-009):_ для SDX-сессий (`sdx/<id>`) слияние выполняется `git merge --no-ff` (не fast-forward), чтобы вся история ветки, включая коммит удаления артефактов до мёржа, была достижима из merge-коммита основной ветки (REQ-SESS-4); линейность истории — соглашение для обычных `feature/*`/`chore/*` правок, но не для сессий.
- **Триггер коммита в сообщениях.** Завершать commit-сообщения трейлером `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- **Языковая политика.** Общение, спеки, дизайн, пользовательская документация — русский; код и комментарии, техническая OpenAPI-документация — английский (см. CLAUDE.md §1).
- **Источники правды.** Состояние живёт в файлах (`.claude/sessions/`, `docs/`), а не в контексте диалога — это позволяет восстанавливать работу после `/clear` и суммаризации. Этот журнал — durable-носитель «почему», которого нет в коде.
