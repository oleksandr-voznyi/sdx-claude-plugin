# PROC-010. Параллельные субагенты Execution пишут в общие файлы сессии без protocol'а разрешения гонок

## Суть
На Execution сессии `fw-session-types-20260720` два субагента `developer` были запущены
параллельно на непересекающихся наборах файлов поставки — но оба по инструкции дописывали
строки в общие файлы сессии `auto_decisions.md` и `PLAN.md`. Возникла реальная гонка:
второй агент обнаружил, что его номера строк заняты, и перенумеровал свои записи вручную.
Потерь не произошло только потому, что агент заметил конфликт сам.

Фреймворк поощряет параллельный Execution (см. IDEA-006 про escalate-тир), но не
определяет ни протокола записи в общие файлы сессии, ни того, кто разрешает конфликт.
Оркестратор узнаёт о гонке только из отчёта субагента — то есть постфактум.

## Рекомендация
Определить правило записи в общие файлы сессии при параллельном исполнении. Варианты:
- писать только оркестратору — субагенты возвращают свои записи в ответе, оркестратор
  дописывает их одним ходом после сбора (симметрично тому, как он уже владеет коммитами);
- файл-на-агента (`auto_decisions.<agent>.md`) со склейкой на Closeout;
- явный запрет параллельного запуска агентов, пишущих в один и тот же файл сессии, —
  проверяется оркестратором при планировании волны.
Первый вариант выглядит предпочтительным: он не плодит файлы и совпадает с уже принятым
разделением ответственности (субагенты не коммитят — коммитит оркестратор).
