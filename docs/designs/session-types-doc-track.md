# Технический дизайн: Типы сессий grooming / retro / postmortem / intake и трек doc

Сессия: `fw-session-types-20260720` (feature, full). Решение и инварианты: ADR-017 (`docs/DECISIONS.md`).

## Архитектурный обзор

Вводится четвёртый профиль флоу `doc` — процессный/рефлексивный трек с этапами
`Discovery → Update → Verification (лёгкая, обязательная) → Closeout`, механически
устроенный **точно по тому же контракту**, что уже несёт `SDX_STAGE_MATRIX` для
`patch`/`standard`/`full`: строка `track|stage|artifact|fail_marker`. Ключевое
архитектурное решение всего дизайна — **описать новый трек данными, не процедурной
логикой переходов**: 4 новые строки матрицы добавлены в `cmd_init`/`cmd_next`/
`cmd_backtrack`/`cmd_retrack`/`stage_artifact_ok`/`matrix_*` без изменений самой логики.
Это возможно благодаря переиспользованию уже существующих паттернов:

_Уточнение по факту реализации (см. также ADR-017 ниже):_ ровно одно место процедурного
кода внутри `cmd_next` всё же изменено — и не логика перехода, а текст диагностики:
remediation-подсказка FAIL-гейта была захардкожена на `Execution`, которого в треке `doc`
нет. Подсказка сделана трек-зависимой (~17 строк с fallback и комментариями). Новый трек
не потребовал этого кода — он вскрыл ранее скрытый дефект существующего.

1. **`Discovery` без гейт-артефакта** — не новый механизм. Прецедент уже есть:
   `standard|Discovery|-|no`. `doc|Discovery|-|no` — буквально та же семантика (гейт не
   проверяется объективно, решение прозаическое), заимствованная один в один.

2. **`Update` — тот же физический файл, что `Change`.** Гейт-артефакт `doc|Update` —
   `change_note.md`, ИМЕНЕМ файла совпадающий с гейт-артефактом `standard|Change` и
   `patch|Execution`. `stage_artifact_ok()`/`matrix_row()` не знают о трек-специфичных
   именах стадий — они читают путь артефакта из матрицы и проверяют файл на диске. Из
   этого следует нетривиальное, но корректное следствие: **эскалация `doc → standard/full`
   переносит накопленный `change_note.md` бесплатно**, без единой строки нового кода в
   `sdx-stage.sh` — файл на диске тот же, читает его тот же генерик-путь
   `stage_artifact_ok`. Подробности — раздел «Обработка ошибок и Граничные случаи».

Второе архитектурное решение — **fresh-eyes `reviewer` на этапе Verification без `qa`**
воспроизводит уже провалидированный паттерн `patch` (ADR-014: обязательная лёгкая
верификация, без вызова `qa`, суженный `change_note.md`-скоуп). Отличие `doc` от `patch` —
не в механике верификации (та же), а в том, что регрессионного тест-прогона в принципе нет:
`qa` не вызывается, потому что нет кода. Уточнение: это НЕ означает, что хук `stop-gate`
прозрачен на `doc` «по построению» — он включается по значению `stage` (`Execution|Verification`)
независимо от трека и в проекте с настроенным `.claude/sdx/verify-cmd.sh` реально прогонит эту
команду и на `doc|Verification`. Прозрачность возникает лишь в проектах без известной тест-команды.

Третье архитектурное решение — `sdx/templates/retro.md`, `postmortem.md` и `intake.md`
становятся **первым прецедентом шаблона артефакта СЕССИИ** (в отличие от существующих
`sdx/templates/*` — все они per-project конфиги, копируемые `/sdx:init`). Копирование
выполняется не при инициализации проекта, а при `/sdx:start` (шаг 7), и только для
`type ∈ {retro, postmortem, intake}`.

**Четвёртое архитектурное решение — тип сессии НЕ является параметром `SDX_STAGE_MATRIX` и
никогда им не был.** Матрица параметризована исключительно `track`, а `track` для всех четырёх
типов (`grooming`/`retro`/`postmortem`/`intake`) — один и тот же `doc`. Отсюда прямое
архитектурное следствие: добавление четвёртого типа — это **изменение нулевой мощности** для
`sdx-stage.sh`, `SDX_STAGE_MATRIX` и таблицы «Профили флоу». Вся специфика `intake` — что
подаётся на Discovery, какой шаблон копируется на старте, куда переносится постоянный документ
на Closeout — живёт исключительно в прозаической логике команд (`start.md`, `archive.md`) и в
текстовых шаблонах, а не в enforcement-скрипте. Тип — чисто описательное измерение (REQ-DOC-13),
трек — единственное измерение, которое видит контракт скрипта.

**Смысловое разграничение `intake` и `grooming` — не архитектурный, а прозаический инвариант,
но требует явной фиксации (REQ-DOC-14).** Оба типа читают и пишут один и тот же `docs/backlog/`,
оба используют один и тот же гейт-артефакт `change_note.md`. Разница — не в контракте файлов
(тот же для обоих), а в НАПРАВЛЕНИИ операции над бэклогом: `intake` порождает записи из внешнего
материала (create-путь), `grooming` перераспределяет атрибуты уже существующих записей
(update-путь). Ничто в enforcement-слое эту разницу не проверяет и не может проверить (оба типа
в равной мере проходят через один и тот же `doc|Update|change_note.md|no`) — граница держится
исключительно текстом инструкций (`start.md` Discovery-lite ветка, шаблон `intake.md`) и ADR-017.

## Схема данных / API

### `session_state.json`
```
"type":  "feature | bug | refactor | init | import | grooming | retro | postmortem | intake"
"track": "patch | standard | full | doc"
```
Поле `type` остаётся описательным (ни один enforcement-скрипт его не читает), но приобретает
прозаическую значимость: жёстко определяет `track = "doc"` на `/sdx:start` и выбор шаблона на
этапе Update.

### `SDX_STAGE_MATRIX` — новые строки (дословно)
```
doc|Discovery|-|no
doc|Update|change_note.md|no
doc|Verification|verification_report.md|yes
doc|Closeout|-|no
```
Формат идентичен существующим строкам: `track|stage|artifact|fail_marker`. Порядок строк =
порядок активных этапов. **Не вводится ни одного нового значения `fail_marker`, ни новой ветки
в `stage_artifact_ok()`** — трек `doc` полностью укладывается в существующий словарь.

### `change_note.md` на `doc`-треке — контракт содержимого
Единственный гейт-артефакт этапа `Update` для ВСЕХ ЧЕТЫРЁХ типов. Минимальный контракт:
- краткое резюме сессии (что разобрано/поправлено и почему);
- список ID затронутых записей `docs/backlog/` (создано/изменено);
- для `retro`/`postmortem`/`intake` — ссылка на параллельно ведущийся
  `retro.md`/`postmortem.md`/`intake.md`;
- для `intake` дополнительно — состав внешнего материала, фактически взятого в разбор.

### Шаблоны `sdx/templates/{retro,postmortem,intake}.md`
Обычные `.md` с плейсхолдерами, без frontmatter (парсятся только человеком).

**`retro.md`** содержит разделы: Период/охват, Наблюдения (что сработало/трение/паттерны),
Выводы → Бэклог (таблица), Открытые вопросы.

**`postmortem.md`** содержит: Инцидент, Хронология (таблица), Воздействие, Корневая причина,
Что сработало/не сработало в реагировании, План действий (таблица), Извлечённые уроки.

**`intake.md`** содержит: Вход (источник материала, перечень/ссылки, что осталось за пределами),
Разложение по записям (таблица), Порождённые записи бэклога (таблица), Сознательно не взято в
скоуп, Открытые вопросы к дальнейшему груминг-циклу. Последний раздел — явная точка передачи
эстафеты `grooming`.

### Постоянное хранение (после Closeout)
`docs/history/retro/<session_id>.md`, `docs/history/postmortem/<session_id>.md`,
`docs/history/intake/<session_id>.md` (по аналогии с `docs/history/plans/<session_id>.md`).
Индекс/README для этих каталогов **не вводится** этим дизайном.

## Обработка ошибок и Граничные случаи

### Discovery без гейт-артефакта — переиспользование, не новый прецедент
`doc|Discovery|-|no` работает идентично `standard|Discovery|-|no`: `stage_artifact_ok`
и `matrix_row` уже трактуют `"-"` как «гейт не проверяется объективно». Вход в `doc` с его
первого активного этапа всегда безусловно достижим.

### Эскалация `doc → {patch, standard, full}` — разрешена
`change_note.md`, накопленный на `Update`, **читается напрямую** как гейт-артефакт `Change`
(standard) или `Execution` (patch) — совпадение имени файла делает это бесплатным на уровне
`sdx-stage.sh`. Для эскалации в `full` действует существующее ограничение: `context_report.md`
физически не создавался на `doc`-Discovery (её нет — это лёгкий инлайн-Discovery), поэтому
прямая посадка на `full|Business Spec` **блокируется**.

**Сброс `gate_mode`.** Любой переход ИЗ `doc` в `{patch, standard, full}` трактуется
как эскалация для целей сброса `gate_mode → interactive`.

### Деэскалация `{patch, standard, full} → doc` — ЗАПРЕЩЕНА (прозаически, не скриптом)
Явное архитектурное решение. Обоснование: лёгкая Verification `doc`-трека рассчитана на
нулевой кодовый дифф (`qa` не вызывается), пропуск кодовой сессии через `doc` создал бы
дыру в покрытии кода. **Это НЕ enforced скриптом** — граница исключительно прозаическая
инструкция `retrack.md`.

### Verification на `doc` — исключение `qa`, не новая ветка логики
`verify.md` шаг 1 получает третий пункт наравне с `patch`, шаг 2 — третье условие исключения
вызова `qa`. Полнота (completeness) на `doc` явно ограничена рамками `change_note.md`.

### Intake — вход зависит от нерешённой PROC-009
Discovery типа `intake` принимает внешний материал, для которого на момент этой поставки нет
структурированного места в репозитории. Единственный структурированный след входа — прозаический
(текст в `change_note.md` + раздел «Вход» шаблона `intake.md`).

### Closeout — `(n/a)` по умолчанию для пп.2/3, с исключением
- П.2 — как правило `(n/a)`: doc-сессия не производит `SPEC.md`/`DESIGN.md`. **Исключение**:
  если сама сессия приходит к выводу об изменении протокола/`CLAUDE.md`/нового ADR.
- П.3 — всегда `(n/a)`: doc-трек не включает Task Planning.
- П.4 — основная точка выхода + перенос постоянных документов разбора.

### Backtrack на `doc` — без изменений механики
`backtrack --to Discovery` из любого этапа помечает артефакты `<!-- SDX-OUTDATED -->` баннером —
тот же generic путь, что и для остальных треков.

## Безопасность

- **Enforcement-слой не расширяется.** `stage-gate.sh` уже открывает `docs/**`,
  `.claude/**`, `*.md` независимо от этапа.
- **`archive-verify.sh`** проверяет инварианты 1/5/6 трек-агностично.
- **`stage-write-guard.sh`** денит правку поля `stage` вне `sdx-stage.sh` трек-агностично.
- **Новая прозаическая граница «деэскалация в `doc` запрещена»** — это ПРОЦЕССНАЯ, не
  enforcement-мера. Явно зафиксировано в ADR-017 как известный незакрытый класс.
- **Доказательность гейта `Update`/`doc` — факт наличия, не качество** (унаследовано из ADR-016).

## Тестовая стратегия (`sdx/hooks/test-sdx-stage.sh`)

**Расширение сценария 34:** `for track in full standard patch` → `for track in full standard patch doc`.
Без единой другой правки сценария — цикл уже параметричен.

**Новые сценарии** (36+, по образцу существующих): `init` track=doc; `next` и `backtrack` для
всех этапов; `retrack` эскалации `doc → {standard, full}` и безопасный fallback на `full|Discovery`.
Все 13 сценариев проверены мутацией — тавтологичности нет.

## Риски и границы

1. **Дрейф дублирующихся копий матрицы** (унаследованный риск). `stage-transition-guard.md` не
   защищён sanity-тестом — ручная синхронизация при реализации.

2. **Прозаическая, не enforced скриптом граница «деэскалация в `doc` запрещена»** — явно
   принятый trade-off, не тестируется в `test-sdx-stage.sh`.

3. **Качество входа при эскалации `doc → {patch,standard,full}`** — тонкий груминг/intake-резюме
   как вход `ba`/`architect`. Не блокирует, но снижает содержательность.

4. **Отсутствие индекса `docs/history/retro|postmortem|intake/`** — при накоплении файлов
   естественный будущий `IDEA-`-кандидат.

5. **Зависимость `intake` от нерешённой `docs/backlog/PROC-009`** — вход `intake`-сессий не
   имеет структурированного места хранения. Воспроизводимость разбора опирается на дисциплину
   заполнения `change_note.md` и раздела «Вход» шаблона `intake.md`.

6. **Смешение ролей `intake`/`grooming` — прозаический, не enforcement риск.** Ничто не мешает
   `intake`-сессии по факту начать перераспределять приоритеты. Граница держится текстом
   инструкций и ADR-017, не кодом.

## Архитектурное решение и обоснование

Полный текст архитектурного решения и анализ trade-off приведён в **ADR-017** (`docs/DECISIONS.md`).
Ключевые инварианты:
- `SDX_STAGE_MATRIX` — единственный машиночитаемый источник истины; `protocol.md` — проекция.
- `type` не валидируется enforcement-скриптами; трек — единственное измерение, видимое скрипту.
- Жёсткая привязка `type ∈ {grooming,retro,postmortem,intake} → track=doc` действует ТОЛЬКО на
  `/sdx:start`; после старта `type`/`track` могут разойтись при эскалации.
- Верификация `doc` не вызывает `qa`; отсутствие тест-прогона компенсируется интерактивностью
  Discovery/Update и обязательным fresh-eyes прогоном `reviewer`.

## Версионирование плагина

Новая функциональность — рекомендуемый bump: **MINOR** (`1.3.0 → 1.4.0`). Новый трек + 4 типа
сессий + 3 шаблона + новый ADR: сопоставимый по значимости прирост функциональности.
Добавление четвёртого типа `intake` не меняет категорию бампа (SDX_STAGE_MATRIX и enforcement-контракт
неизменны; прирост — описательный и документационный).
