# Business Specification: SDX Enforcement & Routing — Фаза 1

## Контекст и Цели

SDX-фреймворк уже реализует экономную оркестрацию (одно-агентная модель, передача путей вместо контента) и сильный слой верификации (fresh-eyes `reviewer` с контрактом изоляции). Однако ряд ключевых инвариантов закреплён только в прозе: модель обещает соблюдать их, но никакой детерминированной защиты нет.

**Цель Фазы 1** — перевести четыре инварианта из «обещаний в тексте» в автоматически применяемые правила, а также установить явное объявление модели для каждого субагента:

1. **Код не пишется до одобрения плана** (сейчас — доверие к дисциплине стадий).
2. **Ход не завершается на красных тестах** (сейчас — обязательство QA-агента).
3. **Завершение сессии не пропускает удаление артефактов** (пункт 6 Closeout «чаще всего пропускается»).
4. **Прод-деплой — только явное действие человека** (сейчас нет никакого enforcement).
5. **Каждый агент явно привязан к модели** (сейчас все работают на Opus по глобальному умолчанию пользователя, что избыточно дорого).

Фаза 1 является фундаментом; без неё последующие фазы (headless/batch-контур, кэширование, self-improving loop) небезопасны.

---

## Требования

### [ADDED] REQ-GATE-1 — Stage-gate (блокировка записи в код до гейта Execution)

**Что**: Запись в файлы кодовой базы должна быть физически невозможна, пока сессия не достигла стадии Execution. Записи в `docs/`, каталог `.claude/` и любые `*.md`-файлы (артефакты планирования) разрешены на всех стадиях без ограничений.

**Зачем**: Инвариант «человек одобряет план до кода» — сейчас это обещание модели, которое нарушается при деградации контекста или спешке. Enforcement переводит его в детерминированную защиту. Без этого гейта headless-контур (Фаза 4) небезопасен.

**Скоуп**: Активируется только в рамках SDX-сессии (определяется по ветке `sdx/<id>`). Вне SDX-сессии ограничений нет. Треки `patch` и `standard`, стартующие сразу с Execution, ограничений не испытывают — gate прозрачен. Дополнительный per-project allowlist позволяет расширить список разрешённых путей.

**Важно (из Discovery)**: Конкретный механизм блокировки (выходной код хука vs JSON-поле `permissionDecision`) — техническое решение, закрепляется в Technical Design. Здесь фиксируется только поведение: запись заблокирована.

---

### [ADDED] REQ-GATE-2 — Stop-gate (детерминированный тест-пол на Verification)

**Что**: Ход агента не должен завершаться, пока тест-прогон красный, в рамках стадий Execution и Verification (а также при принудительном флаге для headless-режима). После трёх последовательных неудачных прогонов gate прекращает блокировать и возвращает управление человеку.

**Зачем**: «Корректность исполнением» — текущий лозунг фреймворка. Сейчас это декларация. Stop-gate делает её доказанной: зелёный прогон становится необходимым условием завершения хода, а не пожеланием.

**Деградация**: При отсутствии известной тест-команды (нет `verify-cmd.sh`, нет распознанного менеджера пакетов) gate безопасно переходит в режим no-op — не блокирует. Это допущение критично для мета-проекта SDX, у которого нет тест-сьюта. Обнаружение команды — per-project ответственность.

**Скоуп**: Gate стоит под слоем суждений `qa` (покрытие/тавтологичность) и `reviewer` (когерентность) — он самый нижний, детерминированный, без суждения.

---

### [MODIFIED] REQ-CLOSEOUT-1 — Archive-verify (инварианты Closeout 1/5/6 в скрипте)

**Что**: Три инварианта завершения сессии, ранее описанных только в прозе чек-листа, переводятся в скрипт, вызываемый командой `/sdx:archive` после мёржа ветки:
- Инвариант 1: рабочее дерево чистое (нет незакоммиченных изменений).
- Инвариант 5: ветка `sdx/<id>` слита в `main`.
- Инвариант 6: каталог сессии `.claude/sessions/<id>/` удалён и удаление проверено.

При нарушении любого инварианта скрипт завершается с ошибкой, Closeout считается незавершённым.

**Зачем**: Пункт 6 («удаление файлов сессии») систематически пропускается по данным ретроспектив. Prose-чек-лист зависит от памяти модели в конце длинной сессии — это самое ненадёжное место. Скрипт не требует памяти.

**Изменение**: До этого требования инварианты существовали как описание в `protocol.md` и `CLAUDE.md`. После — они enforced скриптом; проза заменяется ссылкой на него.

---

### [ADDED] REQ-PROD-1 — Prod-guard (блокировка прод-деплоя агентом)

**Что**: Bash-команды, совпадающие с паттернами из per-project конфигурационного файла (`prod-guard.conf`), блокируются автоматически. Деплой в продакшн — только явное действие человека, не агента.

**Зачем**: Стадия Deployment в SDX формально существует, но без enforcement агент может случайно или по ошибке выполнить прод-команду на любой стадии. Для headless-контура (Фаза 4) это неприемлемо.

**Opt-in**: Защита активируется только при наличии непустого `prod-guard.conf`. Пустой или отсутствующий файл — нет защиты. Это намеренно: паттерны прод-хостов и команд специфичны для каждого проекта, фреймворк не может знать их заранее. Файл поставляется как задокументированный шаблон.

**Скоуп мета-проекта**: SDX мета-проект не имеет продакшн-окружения — `prod-guard.conf` остаётся пустым шаблоном с комментариями.

---

### [ADDED] REQ-ROUTE-1 — Model routing (явное объявление модели в frontmatter агентов)

**Что**: Каждый из восьми субагентов объявляет поле `model` в YAML frontmatter своего файла. Распределение: фронтир-модель (Opus) — только на ролях с оценочным суждением (`reviewer`); дешёвые модели — на механических ролях (`tech-writer` — Haiku); остальные — Sonnet.

**Зачем**: Сейчас все агенты наследуют глобальный дефолт пользователя (`~/.claude/settings.json: model: opus`), что означает Opus на каждом вызове. Явная раскладка снижает стоимость на механических ролях без ущерба качеству, где оценочное суждение не требуется.

**Поле `effort`**: Бандл рекомендует `effort: low` для `reviewer`. По данным Discovery, поддержка поля `effort` во frontmatter субагентов не подтверждена (обнаружено только как CLI-флаг). Включение `effort` — условно: подлежит верификации в Technical Design. Если не поддерживается — исключается без влияния на остальные требования.

**Целевая раскладка**:

| Агент | Роль | model |
|---|---|---|
| `reviewer` | adversarial-суждение (Opus, редко) | claude-opus-4-8 |
| `architect` | Discovery (Sonnet); DESIGN(full) — суждение (эскалация) | claude-sonnet-4-6 |
| `ba` | формулировка требований | claude-sonnet-4-6 |
| `lead-dev` | декомпозиция плана | claude-sonnet-4-6 |
| `developer` | код + unit-тесты | claude-sonnet-4-6 |
| `qa` | тесты, прогон | claude-sonnet-4-6 |
| `devops` | конфиги / деплой | claude-sonnet-4-6 |
| `tech-writer` | механическая генерация документации | claude-haiku-4-5 |

## Критерии приёмки

### REQ-GATE-1 (Stage-gate)
- [ ] На стадии ≠ Execution/Deployment попытка записи в файл кодовой базы блокируется с объяснением причины; ход не прерывается, агент получает сообщение об ошибке.
- [ ] На тех же стадиях запись в `docs/**`, `.claude/**` и любой `*.md`-файл проходит без ограничений.
- [ ] Вне SDX-ветки (`sdx/<id>`) блокировки нет (gate прозрачен).
- [ ] При наличии per-project `stage-gate.allow` файлы из allowlist проходят.
- [ ] На стадии Execution/Deployment любая запись разрешена.

### REQ-GATE-2 (Stop-gate)
- [ ] На стадии Execution/Verification при красном тест-прогоне ход не завершается; агент получает хвост вывода теста.
- [ ] При зелёном тест-прогоне ход завершается штатно.
- [ ] После трёх последовательных красных прогонов gate перестаёт блокировать и возвращает управление человеку с поясняющим сообщением.
- [ ] При отсутствии известной тест-команды (`verify-cmd.sh` не существует, автодетект не находит команду) gate безопасно пропускает без ошибки (no-op). Это обязательное поведение для мета-проекта SDX, не имеющего тест-сьюта.
- [ ] Вне стадий Execution/Verification (без принудительного флага) gate прозрачен.
- [ ] **Green-run cache (ADR-011):** при неизменном рабочем дереве после зелёного прогона повторный `Stop` пропускается без перезапуска verify (кэш-хит); изменение дерева инвалидирует кэш и запускает прогон заново; красный прогон кэш не пишет; `SDX_STOP_GATE=1` кэш обходит; кэш-хит не крутит loop-guard-счётчик.

### REQ-CLOSEOUT-1 (Archive-verify)
- [ ] `/sdx:archive` вызывает скрипт ПОСЛЕ мёржа ветки.
- [ ] При нечистом рабочем дереве скрипт завершается с ненулевым кодом, Closeout прерывается.
- [ ] При неслитой ветке `sdx/<id>` скрипт завершается с ненулевым кодом, Closeout прерывается.
- [ ] При выполненных условиях 1 и 5: каталог сессии `.claude/sessions/<id>/` удаляется; скрипт верифицирует отсутствие каталога.
- [ ] Ветка `sdx/<id>` удаляется после верификации.
- [ ] При успехе скрипт выводит явное подтверждение всех трёх инвариантов.

### REQ-PROD-1 (Prod-guard)
- [ ] Bash-команда, совпадающая с паттерном из `prod-guard.conf`, блокируется; агент получает объяснение.
- [ ] При пустом или отсутствующем `prod-guard.conf` блокировок нет (opt-in enforcement).
- [ ] Файл `prod-guard.conf` поставляется как шаблон с документирующими комментариями и примерами паттернов.
- [ ] Для мета-проекта SDX `prod-guard.conf` существует как пустой шаблон (нет продакшн-окружения).

### REQ-ROUTE-1 (Model routing)
- [ ] Все 8 файлов `.claude/agents/*.md` содержат поле `model` в YAML frontmatter.
- [ ] `reviewer` объявляет `model: claude-opus-4-8`.
- [ ] `tech-writer` объявляет `model: claude-haiku-4-5`.
- [ ] Оставшиеся 6 агентов объявляют `model: claude-sonnet-4-6`.
- [ ] Поле `effort:` добавляется только если подтверждена его поддержка в frontmatter субагентов (решение Technical Design); при неподтверждении — исключается.

## Намеренно НЕ делаем (анти-требования)

### [NOOP] REQ-NOOP-PLANMODE — Plan Mode не адоптируется

Plan mode Claude Code как глобальный режим не вводится. Причина: он блокирует все записи на диск, включая `.md`-артефакты стадий Business Spec, Technical Design и Task Planning. SDX-фреймворк реализует ту же дисциплину «исследование до кода» через систему стадий и гейтов. Инвариант «код read-only до одобрения плана» обеспечивается REQ-GATE-1, который, в отличие от plan mode, разрешает запись артефактов планирования.

### [NOOP] REQ-NOOP-TEAMS — Agent Teams не вводятся

Agent Teams не активируются по умолчанию. SDX уже одно-агентный (обходит ~7× множитель стоимости). Параллельное исполнение — только опциональный escalate-тир для доказуемо независимых `[CODE]`-задач, не дефолт и не предмет Фазы 1.

## Отложено за пределы Фазы 1

Следующие требования из бандла осознанно исключены из Фазы 1 и не реализуются в текущей сессии:

- **REQ-LANE-1** (fanout-контур headless/batch) — Фаза 4
- **REQ-LOOP-1** (self-improving loop, стоимостный сигнал в Closeout) — Фаза 4
- **REQ-CACHE-1** (prompt caching, детерминированный порядок сборки контекста) — Фаза 2
- **REQ-LEAN-1** (lean-аудит/абляция, конвертация инвариантов из прозы в хуки) — Фаза 3
- **REQ-CHECKPOINT-1** (расщепление назначения `/sdx:checkpoint`) — Фаза 3
- **Parallel-execution escalate-тир** (git worktrees для независимых задач) — не в скоупе

## Риски и Ограничения

1. **Механизм блокировки PreToolUse (РИСК-1)**: В версии Claude Code 2.1.195 JSON-поле `decision: "block"` объявлено deprecated для PreToolUse; актуальный подход — `hookSpecificOutput.permissionDecision: "deny"`. Поведение `exit 2` в PreToolUse не верифицировано практически. SPEC не привязывается к технике — конкретный механизм фиксируется в Technical Design после тестирования.

2. **SubagentStop vs Stop (РИСК-2)**: Stop-хук генерируется при завершении хода главной сессии, SubagentStop — при завершении субагента (Task). Если stop-gate привязан только к Stop, тест-прогон выполняется при завершении хода оркестратора, а не developer-субагента. Покрытие и задержка — вопрос Technical Design.

3. **Поле `effort` в frontmatter (РИСК-3)**: Не подтверждено как поддерживаемое поле frontmatter субагентов (только CLI-флаг). REQ-ROUTE-1 обязателен без `effort`; включение `effort` — условно по результатам Design/тестирования.

4. **Отсутствие тест-сьюта в мета-проекте SDX**: Stop-gate деградирует в no-op. Это приемлемо и зафиксировано как допущение. Ответственность за `verify-cmd.sh` — на проектах-потребителях.

5. **Самоприменение stage-gate**: После установки хука сессия в стадии до Execution сама подпадает под блокировку записи в код. Это корректное поведение — хуки устанавливаются только на стадии Execution (когда gate разрешает запись). Порядок действий критичен для PLAN.md.

6. **Prod-guard без конфига = нет защиты**: Пустой `prod-guard.conf` означает полное отсутствие enforcement для REQ-PROD-1. Это осознанный opt-in дизайн, но требует явного документирования в шаблоне.
