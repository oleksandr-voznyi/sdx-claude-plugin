# Roadmap: отложенные требования Фаз 2–4 (deferred)

**Источник:** бандл `sdx-efficiency-automation-2026` (2026-06-27, локальный
`.sdx/bundles/upgrade_2026-06-27/` — gitignored; этот документ — его трекаемый промоут,
сессия `fw-roadmap-20260720`, закрывает DEBT-008). Фаза 1 (stage-gate, stop-gate,
archive-verify, prod-guard, model routing) реализована — см. `phase1-enforcement-routing.md`.
Ниже — полные формулировки требований, осознанно отложенных за пределы Фазы 1, с их
дизайн-срезами. Трекинг — записи `IDEA-*` в `docs/backlog/`.

## Фаза 2

### REQ-CACHE-1 (prompt caching) — бэклог: IDEA-001
`[ADDED]` В протоколе фиксируется детерминированный порядок сборки контекста
(framework → конституция проекта → паттерны → сессия), стабильный префикс не
редактируется в середине сессии; для fanout-контура — явные cache-брейкпоинты.

**Дизайн-срез (бандл §2.9).** Стабильный префикс = framework `CLAUDE.md` + конституция
проекта + релевантные `docs/patterns/*`. Правила: детерминированный порядок сборки;
не редактировать префикс в середине сессии; для fanout — явные cache-брейкпоинты,
помнить TTL (5 мин дефолт / 1 час). Изоляция `reviewer` даёт другой префикс (только
артефакты+diff) — это корректно, кэш не делится.

## Фаза 3

### REQ-LEAN-1 (lean-аудит) — бэклог: IDEA-005
`[ADDED]` В протокол добавляется процедура lean-аудита (абляция) и правило
«инвариант-в-прозе → хук».

**Дизайн-срез (бандл §2.11).** Документированная абляция + правило конвертации.
Кандидаты на конвертацию в хук: Closeout 1/5/6 (реализовано Фазой 1 — archive-verify),
языковая политика для кода (опционально lint-хук). Под вопросом до проверки: наследуют
ли субагенты `CLAUDE.md`; если НЕТ — повтор языковой политики в `developer`/`tech-writer`
несущий, не дублирующий. Проверять абляцией, не угадывать (процедура — бандл §4.D).

### REQ-CHECKPOINT-1 (checkpoint dual-purpose) — бэклог: IDEA-004
`[MODIFIED]` Назначение `/sdx:checkpoint` расщепляется: durable-state для возобновления
(оставить) vs ручной сброс контекстного давления (устарел — это делает авто-компакция).

**Дизайн-срез (бандл §2.10).** `/sdx:checkpoint` оставить как durable-state для
возобновления многосессионного флоу; убрать его роль ручного anti-overflow-клапана.
Правка текста §5 `CLAUDE.md` и раздела checkpoint в `protocol.md`.

## Фаза 4

### REQ-LANE-1 (fanout contour) — бэклог: IDEA-002
`[ADDED]` Вводится второй контур `sdx fanout`: stateless-задачи по портфелю репозиториев
(регенерация доков, аудиты, `/sdx:init --existing` массово), пригодные для headless и
Batch API. Интерактивный посессионный пайплайн не трогается.

**Дизайн-срез (бандл §2.6).** Второй контур рядом с интерактивным `sdx run`:
`sdx run` (есть) — stateful, один проект, человек на гейтах; `sdx fanout` (новое) —
stateless по N репозиториям, headless/Batch. Элемент = самодостаточный промпт; роль
SDX — дать шаблон элемента (формат спеки/отчёта), не сессионную машину. Кандидаты:
`tech-writer` по всем репо, аудиты, массовый `/sdx:init --existing`. Биллинг:
headless/Agent SDK с 15.06.2026 идёт из отдельного пула кредитов по API-ставкам —
закладывать бюджет явно.

### REQ-LOOP-1 (self-improving loop) — бэклог: IDEA-003
`[ADDED]` Closeout-рефлексия расширяется стоимостным сигналом; запись идёт в существующий
`docs/history/`; из неё рождаются проектные оверрайды агентов (`model` per-project).

**Дизайн-срез (бандл §2.8).** Петля замыкается на существующем `docs/history/`
(Closeout п.4 уже его обновляет): (1) jsonl-анализатор (`~/.claude/projects/*.jsonl`,
офлайн) → атрибуция расхода по агенту/сессии; (2) Closeout-рефлексия пишет стоимостный
вывод («оправдал ли `reviewer`/фронтир-тир расход на типичных задачах проекта»);
(3) из лога рождаются проектные оверрайды агентов. Ядро SDX не трогается.

## Вне фаз (опциональный тир)

### Parallel-execution escalate-тир — бэклог: IDEA-006
Часть анти-требования REQ-NOOP-TEAMS (бандл §2.7): не дефолт. Включается только когда
full-`PLAN.md` содержит доказуемо независимые `[CODE]`-задачи: запуск в git worktrees
с файловыми локами, ревью хендоффов человеком. По умолчанию остаётся одно-агентная
последовательная оркестрация.

## Анти-требования (действуют бессрочно, зафиксированы Фазой 1)
- **REQ-NOOP-PLANMODE.** Plan mode как глобальный режим не адоптируется: он блокирует
  ВСЕ записи, включая `.md`-артефакты стадий Spec/Design/Plan. Его инвариант (код
  read-only до аппрува) реплицирован REQ-GATE-1 (stage-gate).
- **REQ-NOOP-TEAMS.** Agent Teams не вводятся по умолчанию: SDX одно-агентный (обходит
  ~7× множитель стоимости). Параллелизм — только опциональный escalate-тир (IDEA-006).

## Критерии приёмки отложенных требований (из бандла)
- [ ] Существует `sdx fanout`-входная точка, не использующая сессионную машину (REQ-LANE-1).
- [ ] В `protocol.md` зафиксирован порядок сборки контекста и кэш-правила (REQ-CACHE-1).
