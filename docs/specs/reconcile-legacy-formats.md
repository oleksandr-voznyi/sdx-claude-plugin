# Спецификация: сверка легаси-структур с актуальным форматом (`/sdx:reconcile`)

Постоянная спецификация, перенесённая на Closeout сессии `fw-reconcile-20260720`
(трек `standard`, версия плагина 1.5.0). Реализация — `commands/reconcile.md`; политика —
раздел «Сверка форматов после релиза и миграции» в `sdx/protocol.md`.

## Задача

Две связанные боли, обе воспроизводившиеся регулярно:

1. **Незакрытые задачи теряются при подключении SDX.** `/sdx:init` разворачивал структуру
   `docs/`, но не смотрел внутрь уже накопленного в проекте: файлов памяти, старых
   каталогов сессий, свободных заметок, TODO-списков, roadmap-блоков README, отчётов
   аудитов. Незакрытые задачи и техдолг оттуда в бэклог не попадали — пользователю
   приходилось каждый раз просить об этом отдельно. Прецеденты в самом репозитории SDX:
   запись `DEBT-008` («roadmap живёт в gitignored-файле — риск потери») и миграция находок
   `docs/audit-2026-07-01-recommendations.md` в `docs/backlog/` выполнялись вручную.
2. **Старые структуры хранения не приводились к обновлённому формату.** Формат данных
   фреймворка эволюционирует (ADR-015 ввёл `docs/backlog/`, ADR-017 — `docs/history/retro|
   postmortem|intake/`, ADR-012 сменил модель сессий). После обновления плагина проект
   продолжал жить в старом формате, пока кто-нибудь не замечал расхождение. Правило
   «привести старое к новому» существовало только в голове пользователя.

## Требования

- **REQ-RECON-1 (команда).** Существует команда `/sdx:reconcile`, не требующая активной
  SDX-сессии и работающая как на свежеинициализированном проекте, так и в середине
  жизненного цикла.
- **REQ-RECON-2 (фазы).** Прогон состоит из фаз: режим → скан → извлечение → пропуск уже
  мигрированного → дедупликация → детект форматных находок → сводка и подтверждение →
  перенос (неотделимо с пометкой источника) → приведение форматов → отчёт → маркер версии →
  коммит.
- **REQ-RECON-3 (границы скана).** Сканируются: файлы памяти внутри рабочего дерева,
  свободные заметки и аудиты в `docs/` вне `docs/backlog/` и `docs/history/`, TODO/FIXME и
  roadmap-блоки документации, остаточные каталоги `.claude/sessions/*` без соответствующей
  ветки, легаси-записи прежних форматов. **Исключены:** `docs/backlog/` (цель переноса),
  `docs/history/` (архив, read-only по смыслу), исходный код (TODO в коде — предмет
  код-ревью), `.git/`.
- **REQ-RECON-4 (кандидат).** Кандидат — незакрытая задача, техдолг, отложенное решение
  или открытый вопрос, для которого формулируется конкретное действие. Констатации без
  последствий кандидатами не являются.
- **REQ-RECON-5 (дедупликация).** Кандидат сверяется с существующим бэклогом по смыслу, а
  не по строке; при совпадении дубль не заводится, при необходимости дополняется
  существующая запись.
- **REQ-RECON-6 (перенос).** Записи создаются в формате ADR-015; `source` — файл-источник,
  `session` — `null`, `priority` по умолчанию `normal` (нейтральный дефолт, не создающий
  ложной срочности; уточняется грумингом).
- **REQ-RECON-7 (пометка источников, идемпотентность).** Исходный фрагмент НИКОГДА не
  удаляется и не редактируется — рядом ставится маркер `<!-- SDX-MIGRATED: → <ID> (<дата>) -->`.
  Контекст возникновения задачи ценнее чистоты файла-источника. Помеченное пропускается
  последующими сканами — это и делает повторный прогон идемпотентным. Пометка — неотъемлемая
  часть переноса: комбинация «перенесено, но не помечено» недостижима.
- **REQ-RECON-8 (сверка форматов).** Отдельная фаза приводит легаси-структуры к актуальному
  формату: дозаполнение обязательных полей frontmatter записей бэклога, создание
  отсутствующих каталогов текущей раскладки `docs/`. Остаточные артефакты отменённых
  моделей (например worktree-каталоги до ADR-012) НИКОГДА не применяются автоматически —
  только сообщаются с предложением ручного решения: это потенциально ценные данные.
- **REQ-RECON-9 (режимы).** Три режима: интерактивный (подтверждение по группам),
  `--scan-only` (read-only), `--auto` (без подтверждений). `--auto` — НЕ эквивалент
  `gate_mode: auto` протокола (ADR-014): стоп-рубрики, `auto_decisions.md` и дисклоуза у
  него нет. Предохранители `--auto`: чистое рабочее дерево на входе, верхняя граница
  операций за прогон, рекомендация предварительного `--scan-only`.
- **REQ-RECON-10 (отчёт).** `docs/sdx-reconcile-report.md` пишется всегда, включая
  `--scan-only` — единственное признанное исключение из read-only-семантики: отчёт есть
  вывод команды, а не запись бэклога/маркер/структура проекта.
- **REQ-RECON-11 (маркер версии).** `.claude/sdx/sdx-version` пишется ИСКЛЮЧИТЕЛЬНО
  `/sdx:reconcile` и только по факту завершённой сверки (не в `--scan-only`, не при
  отклонении всех групп). `/sdx:init` маркер не пишет. Инвариант: состояние «сверка
  предложена и отклонена» обязано оставаться отличимым от «сверка пройдена».
- **REQ-RECON-12 (автовызов).** `/sdx:init` вызывает `/sdx:reconcile` автоматически после
  разворачивания структуры и реверс-инжиниринга `--existing`. `/sdx:start` сверяет маркер
  версии с версией плагина и при расхождении **предлагает** сверку; у предложения ровно два
  терминальных исхода — отказ (старт продолжается) и согласие (`/sdx:start` останавливается
  без создания сессии, сверка выполняется отдельной командой в своей ветке).
- **REQ-RECON-13 (политика).** Приведение накопленных данных проекта к актуальному формату —
  обязательный шаг после подключения SDX к существующему проекту и после каждого обновления
  версии плагина. Зафиксировано в `sdx/protocol.md`.

## Границы решения

- Скан прозаический (агент читает и судит), а не детерминированный парсер: полнота не
  гарантируется — отсюда обязательный отчёт со списком просканированного.
- Скан памяти покрывает только пути внутри рабочего дерева; автономная память Claude Code
  (`~/.claude/projects/<slug>/memory/`) вне дерева и требует ручного просмотра.
- Маркер `SDX-MIGRATED` — конвенция в тексте файла, не enforced хуком: ручное удаление
  маркера приведёт к повторному заведению записи.
- Проверка версии в `/sdx:start` не покрывает работу вне команд `/sdx:*`.
- `--auto` не даёт гарантий авторежима протокола (ADR-014) — только отсутствие вопросов.
- TODO/FIXME в исходном коде сознательно вне скоупа: иначе бэклог утонет в шуме реализации.
- Выбор `/sdx:start` как точки автопроверки, а не SessionStart-хука, сознателен: хук
  потребовал бы отдельного скрипта и тест-сьюта ради напоминания, тогда как ни одна
  SDX-сессия не начинается мимо `/sdx:start`.

## История верификации

Поставка прошла четыре круга fresh-eyes ревью (10 WARN → 1 FAIL/5 WARN → 4 WARN → 4 WARN,
итог PASS). Дважды исправление находки вводило новый дефект: сначала автономный коммит в
пути согласия (нарушение «коммит — явный шаг по запросу пользователя»), затем коммит на
основной ветке (нарушение branch-first, ADR-009). Итоговое решение — `/sdx:start` вообще не
выполняет реконсиляцию инлайн — устранило оба нарушения разом и сделало обе ветви шага
терминальными. Это прецедент, который стоит помнить: для прозаических инструкций
исправление находки требует такого же ревью, как исходная поставка.
