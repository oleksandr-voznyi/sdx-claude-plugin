# Спецификация: Типы сессий grooming / retro / postmortem / intake и трек doc

Сессия: `fw-session-types-20260720` (feature, full). Решение и инварианты: ADR-017 (`docs/DECISIONS.md`).

## Контекст и Цели

Работа над самим бэклогом (пересмотр приоритетов, актуализация статусов), периодическая
ретроспектива завершённых сессий, разбор инцидентов (постмортем) и разбор нового значительного
блока внешних требований (intake) сейчас не имеют своего типа сессии и трека в SDX —
делаются вне процесса, «по наитию», без структурированного входа, выхода и следа в бэклоге
(`docs/backlog/PROC-002`).

Цель — ввести четыре новых типа сессии (`grooming`, `retro`, `postmortem`, `intake`), которые
проходят по общему лёгкому треку `doc` (Discovery → Update → Verification (лёгкая) → Closeout,
без стадий Execution/кода) и производят прослеживаемый результат: изменения `docs/backlog/`, а
для `retro`/`postmortem`/`intake` — дополнительно постоянный документ разбора. Это распространяет
дисциплину SDX (гейты, трассируемость, Closeout-чек-лист) на процессную/рефлексивную работу,
которая раньше её не проходила.

## Требования

### Назначение и вход четырёх типов сессий

- [ADDED] **REQ-DOC-1 (grooming — назначение и вход).** Тип `grooming` — сессия пересмотра
  ТЕКУЩЕГО состояния бэклога: актуализация статусов/приоритетов/волн (`wave`) существующих
  записей `docs/backlog/`, закрытие неактуальных, устранение дублей, перегруппировка. Вход —
  срез текущего открытого/deferred бэклога (`docs/backlog/README.md` + сами записи); сессия НЕ
  привязана к конкретному периоду времени или конкретной завершённой сессии.

- [ADDED] **REQ-DOC-2 (retro — назначение и вход).** Тип `retro` — сессия разбора набора уже
  ЗАВЕРШЁННЫХ SDX-сессий за выбранный период (или по явному списку `session_id`, заданному
  пользователем на старте): цель — выявить системные наблюдения (повторяющиеся паттерны,
  что сработало хорошо/плохо в процессе) и превратить их в новые записи бэклога. Вход —
  `docs/history/sessions-log.md` (срез за период/по списку) плюс записи `docs/backlog/`,
  которые эти сессии породили или закрыли.

- [ADDED] **REQ-DOC-3 (postmortem — назначение и вход).** Тип `postmortem` — сессия разбора
  ОДНОГО конкретного инцидента (не обязательно привязанного к SDX-сессии — может быть
  прод-инцидент, процессный сбой, найденный дефект с серьёзными последствиями): цель —
  зафиксировать хронологию, корневую причину и план действий. Вход — описание инцидента,
  собираемое интерактивно от пользователя на этапе Discovery, плюс релевантные записи
  `docs/backlog/`/`docs/history/sessions-log.md`, если применимо.

- [ADDED] **REQ-DOC-14 (intake — назначение, вход и место в потоке относительно grooming).**
  Тип `intake` — сессия разбора НОВОГО значительного блока внешних требований (эпик, пачка
  багрепортов, набор продуктового материала) с целью **породить** записи `docs/backlog/`.
  Ключевое смысловое отличие от `grooming` (REQ-DOC-1): `intake` стоит по потоку **выше**
  `grooming` — `intake` создаёт записи бэклога из внешнего материала, `grooming` затем
  перераспределяет приоритеты и волны среди уже существующих записей; `intake` не правит
  статусы/приоритеты/волны существующих записей (это зона `grooming`), а `grooming` не создаёт
  записи из внешнего продуктового материала (это зона `intake`). Вход — внешний материал: BRD/
  бриф, юзер-стори, макеты дизайна, референсы, багрепорты со скриншотами/скринкастами. У
  такого материала на момент этой поставки нет определённого места в репозитории — это
  известное ограничение (см. «Риски и Ограничения», зависимость от `docs/backlog/PROC-009`):
  до закрытия PROC-009 материал подаётся пользователем явно (ссылками, путями к файлам или
  вложением в диалог), а состав фактически взятого в разбор фиксируется в `change_note.md`
  этапа Update (REQ-DOC-9).

### Выход

- [MODIFIED] **REQ-DOC-4 (обязательный минимум — записи бэклога).** Каждая doc-сессия
  (независимо от типа) обязана произвести минимум одно наблюдаемое изменение
  `docs/backlog/`. Характер изменения зависит от типа: для `grooming` — это, как правило,
  правка статуса/приоритета/волны СУЩЕСТВУЮЩЕЙ записи; для `intake` — это, как правило, НОВАЯ
  запись (в первую очередь `FEAT-`/`BUG-`), порождённая из внешнего материала (REQ-DOC-14); для
  `retro`/`postmortem` — новая или изменённая запись, вытекающая из разбора. Отсутствие такого
  изменения на момент Closeout означает, что сессия не достигла цели.

- [MODIFIED] **REQ-DOC-5 (дополнительный постоянный документ для retro/postmortem/intake).**
  Для типов `retro`, `postmortem` и `intake`, помимо записей бэклога, обязателен постоянный
  документ разбора, переживающий закрытие сессии: он ведётся в каталоге сессии как
  `retro.md`/`postmortem.md`/`intake.md` по единому шаблону и на Closeout переносится в
  постоянное хранилище — `docs/history/retro/`, `docs/history/postmortem/` и
  `docs/history/intake/` соответственно (по аналогии с уже существующим `docs/history/plans/`
  для `PLAN.md`), с именем файла по `session_id`. Для типа `grooming` отдельного постоянного
  документа не создаётся — записи бэклога являются единственным и достаточным выходом.

- [ADDED] **REQ-DOC-15 (intake — состав постоянного документа разбора).** Документ
  `docs/history/intake/<session_id>.md` (REQ-DOC-5) обязан содержать: что было на входе
  (перечень/ссылки на разобранный внешний материал — REQ-DOC-14), как этот материал разложен
  по записям бэклога, какие записи `docs/backlog/` порождены (с ID), и что сознательно не взято
  в скоуп разбора и почему. Это симметрично составу `retro.md`/`postmortem.md` по духу (разбор
  → выводы → бэклог), но с фокусом на прослеживаемость входного материала, а не на хронологию
  инцидента или период.

### Поведение трека `doc`

- [MODIFIED] **REQ-DOC-6 (активные этапы, единые для всех четырёх типов).** Все четыре типа
  сессии (`grooming`, `retro`, `postmortem`, `intake`) проходят по единому треку `doc`:
  Discovery → Update → Verification (лёгкая, обязательная) → Closeout. Состав и порядок
  активных этапов определяется треком, а не типом сессии — сам тип влияет только на содержание
  (что собирается на Discovery и что производится на Update), не на маршрут.

- [MODIFIED] **REQ-DOC-7 (жёсткая привязка типа к треку).** При `/sdx:start` со значением типа
  `grooming`, `retro`, `postmortem` или `intake` трек `doc` выбирается безусловно, без диалога
  об альтернативном треке — в отличие от триажа `patch`/`standard`/`full`, здесь связь тип↔трек
  жёсткая 1:1 по построению задачи.

- [MODIFIED] **REQ-DOC-8 (Discovery на doc-треке).** Discovery выполняется инлайн, без вызова
  субагента `architect`, и состоит в сборе входного среза, специфичного для типа сессии
  (REQ-DOC-1/2/3/14): для `grooming` — обзор текущего бэклога; для `retro` — выборка из
  `sessions-log.md` за период/по списку; для `postmortem` — интервью пользователя об
  инциденте; для `intake` — приём внешнего материала, явно поданного пользователем (ссылками,
  путями или вложением — ограничение по PROC-009, REQ-DOC-14), и его первичный обзор. Своего
  гейтуемого файла-артефакта у Discovery на этом треке нет ни для одного из четырёх типов.

- [MODIFIED] **REQ-DOC-9 (Update на doc-треке — единый гейт-артефакт).** `change_note.md` —
  единственный объективно проверяемый гейт-артефакт этапа Update для всех четырёх типов. Правки
  `docs/backlog/` выполняются НА ЭТОМ этапе (не откладываются до Closeout); для `retro`/
  `postmortem`/`intake` на этом же этапе параллельно заполняется содержательный, но не
  гейтуемый скриптом документ (`retro.md`/`postmortem.md`/`intake.md`). `change_note.md`
  фиксирует краткое резюме сессии и список ID затронутых записей бэклога (и ссылку на
  `retro.md`/`postmortem.md`/`intake.md`, если применимо); для `intake` дополнительно
  фиксирует состав внешнего материала, фактически взятого в разбор (REQ-DOC-14) — это
  единственный след входа, пока не закрыт `docs/backlog/PROC-009`.

- [ADDED] **REQ-DOC-10 (обязательная лёгкая Verification).** doc-трек включает обязательный
  лёгкий этап Verification: fresh-eyes прогон `reviewer` (через делегирование Task) против
  `change_note.md` — оси корректности и когерентности (не противоречит ли новая/изменённая
  запись бэклога уже существующим записям и решениям, зафиксированным в постоянных
  документах). Ось полноты — в границах `change_note.md`. Субагент `qa` не вызывается,
  автоматизированного тестового прогона нет — на этом треке нет кода.

### Закрытие сессии (Closeout)

- [ADDED] **REQ-DOC-11 (гейт входа в Closeout).** Как и на треке `patch`, вход в Closeout для
  `doc`-сессии требует существования `verification_report.md` без находок `FAIL`; при
  отсутствии — стоп и направление на `/sdx:verify`.

- [MODIFIED] **REQ-DOC-12 (адаптация чек-листа Closeout).** Стандартный чек-лист Closeout
  применяется к `doc`-сессии целиком, со следующими типовыми отметками:
  - п.2 (перенос дельт в `docs/specs/`/`docs/designs/`) — как правило `(n/a)`: doc-сессия не
    производит `SPEC.md`/`DESIGN.md`. Исключение: если сессия по итогам разбора приходит к
    выводу об изменении самого процесса/протокола SDX (правки `sdx/protocol.md`, `CLAUDE.md`,
    новый ADR) — такие правки переносятся по п.2 штатно, и пункт не `(n/a)`.
  - п.3 (архив `PLAN.md`) — всегда `(n/a)`: doc-трек не включает Task Planning.
  - п.4 (глобальный лог + бэклог) — основная точка выхода doc-сессии: актуализация статусов
    закрытых/изменённых записей бэклога и обновление индекса (REQ-DOC-4), а для `retro`/
    `postmortem`/`intake` — перенос постоянного документа разбора в `docs/history/retro/`,
    `docs/history/postmortem/` или `docs/history/intake/` соответственно (REQ-DOC-5) как часть
    обновления `docs/history/`.
  - п.1, п.5–п.8 (коммиты, удаление каталога сессии до мёржа, мёрж, enforcement-проверка) —
    применяются без изменений, трек-агностичны.

- [MODIFIED] **REQ-DOC-13 (расширение допустимых значений типа сессии).** Поле `type` в
  `session_state.json` расширяется значениями `grooming`, `retro`, `postmortem`, `intake`.
  Поле остаётся описательным (не валидируется enforcement-скриптами), но становится значимым
  для прозаической логики `/sdx:start` (жёсткая привязка к треку `doc`, REQ-DOC-7) и для
  выбора шаблона на этапе Update (REQ-DOC-9).

## Критерии приемки

- Все четыре типа сессии (`grooming`, `retro`, `postmortem`, `intake`) создаются через
  `/sdx:start <type> <goal>` с автоматической привязкой `track: "doc"`.
- На старте сессий типа `retro`/`postmortem`/`intake` в каталог сессии копируются
  соответствующие шаблоны.
- `intake` и `grooming` текстом спецификации и протокола явно различены как разные роли.
- Порядок этапов для всех четырёх типов: Discovery → Update → Verification → Closeout.
- Переход Update → Verification требует непустого `change_note.md`.
- `/sdx:verify` на doc-сессии не вызывает `qa`.
- На Closeout `docs/backlog/` содержит минимум одно изменение, отмеченное `session: <session_id>`.
- Для `retro`/`postmortem`/`intake` файлы разбора переносятся в `docs/history/*/` каталоги.

## Риски и Ограничения

- **Единый `change_note.md` для четырёх разных по смыслу типов.** Один и тот же файл служит
  гейт-артефактом для разных типов — без дисциплины может стать формальной опиской. Компенсация
  — обязательный fresh-eyes прогон Verification.
- **Отсутствие Execution/кода на этом треке** означает, что единственная защита от «пустой»
  doc-сессии — критерий REQ-DOC-4 и Verification.
- **Перенос постоянного документа разбора** — новый паттерн хранения в `docs/history/`.
- **Отсутствие структурированного места для входящих артефактов `intake`** (зависимость от
  `docs/backlog/PROC-009`): до закрытия PROC-009 материал подаётся пользователем явно.
