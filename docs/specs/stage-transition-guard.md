# Спецификация: детерминированный владелец поля `stage` (DEBT-001 + DEBT-011)

Сессия: `fw-stage-guard-20260720` (feature, full). Решение и инварианты: ADR-016 (`docs/DECISIONS.md`).

## Контекст и Цели

Поле `stage` в `session_state.json` определяет, какой enforcement-слой (гейты, разрешения на код/тесты) считает активным. Сегодня это поле пишется по прозе — несколько независимых путей (`/sdx:start`, `/sdx:next`, `/sdx:backtrack`, `/sdx:retrack`, `/sdx:archive`, `/sdx:import`) реализуют каждый свою логику гейта (или не реализуют вообще). Между решением оркестратора перейти на новый этап и фактической записью `stage` в файл нет детерминированного барьера: при деградации контекста модель может записать произвольное значение в обход проверок, обнуляя всю защиту enforcement-слоя без технического препятствия.

**Цель**: ввести единственного писателя поля `stage` — скрипт `sdx-stage.sh`, который проверяет объективно проверяемые условия перехода (наличие гейт-артефактов, отсутствие маркера `FAIL` и т.п.) и блокирует любую попытку изменить `stage` в обход него. Решение закрывает **случайный** обход (прозаическая ошибка, деградация контекста), не претендуя на защиту от **намеренного** обхода через Bash.

## Требования

### REQ-STAGE-1 — Единственный писатель поля `stage`

Вводится единый детерминированный механизм перехода, через который проходит **любое** изменение значения поля `stage` — вне зависимости от инициирующей команды (`/sdx:start`, `/sdx:import`, `/sdx:next`, `/sdx:backtrack`, `/sdx:retrack`, `/sdx:archive`).

### REQ-STAGE-2 — Проверка объективных гейт-условий forward-перехода

При forward-переходе (текущий этап → следующий активный этап трека) механизм перехода проверяет объективно проверяемые по артефактам условия готовности уходящего этапа:
- существование и непустоту требуемого артефакта этапа (`context_report.md`, `SPEC.md`, `DESIGN.md`, `PLAN.md`, `change_note.md`, `verification_report.md`);
- отсутствие маркера `FAIL` там, где применимо (выход из Verification).

При невыполнении условия переход отклоняется, `stage` не меняется. Смысловые условия остаются прозаическим суждением оркестратора; механизм перехода проверяет только то, что проверяемо по факту наличия/содержимого артефакта.

### REQ-STAGE-3 — Единый источник истины матрицы «трек → этапы → гейт-артефакты»

Соответствие «трек → упорядоченный список активных этапов → требуемый гейт-артефакт(ы)» существует в **одном** машиночитаемом виде, используемом механизмом перехода. `sdx/protocol.md` ссылается на эту матрицу как на описание, сохраняя текущую таблицу треков в протоколе как человекочитаемую проекцию.

### REQ-STAGE-4 — Идемпотентность и обработка невалидного имени этапа

- Вызов перехода с целевым этапом, совпадающим с текущим `stage`, завершается успешно без изменения файла (безопасный no-op).
- Вызов с именем этапа, не входящим в протокол, отклоняется без изменения файла, с указанием, что имя не распознано.

### REQ-STAGE-5 — Наблюдаемость отказа перехода

При отказе в переходе сообщение отказа обязано явно называть:
- (а) какого именно условия/артефакта не хватает;
- (б) какой командой это исправить.

Отказ не должен ограничиваться немым ненулевым кодом возврата.

### REQ-BACKTRACK-1 — Допустимые цели backward-переходов

`/sdx:backtrack` переводит `stage` только на этап, который (а) активен в текущем треке сессии и (б) не позже текущего этапа в порядке трека. Переход на неактивный этап отклоняется с указанием на `/sdx:retrack`. Backward-переход **не проверяет** гейт-артефакты уходящего этапа.

### REQ-BACKTRACK-2 — Единое правило для артефактов отменяемых этапов

При backward-переходе артефакты, относящиеся к этапам **после** целевой точки в порядке трека (без верхней границы, через конец активных этапов), помечаются как устаревшие. Артефакт самого целевого этапа не метится.

### REQ-RETRACK-1 — `/sdx:retrack` как отдельный режим перехода

Смена трека через `/sdx:retrack` переводит `stage` на ближайший активный этап **нового** трека, вычисленный из текущего этапа и накопленного прогресса, **без** повторной проверки forward гейт-артефактов уходящего этапа. Проверка «этап активен в новом треке» обязательна.

### REQ-RETRACK-2 — Защита `retrack` от форвард-скачка мимо гейта

`/sdx:retrack` переводит `stage` на этап, который достижим только если гейт-артефакты **всех** предшествующих ему этапов нового трека фактически существуют на диске по тем же объективным критериям, что применяет `next`. Проверка не опирается на текущий `stage` и не сравнивает его ни с чем — только на то, что реально есть в `.claude/sessions/<id>/`. Собственный первый активный этап нового трека всегда достижим (пустая цепочка предшественников). Особый случай — объединённый этап `Change`: считается подтверждённым, если существует **либо** `change_note.md`, **либо** одновременно `SPEC.md` **и** `DESIGN.md`.

### REQ-CLOSEOUT-ENTRY-1 — Вход в Closeout через единственный писатель

`/sdx:archive` переводит `stage` в `Closeout` через тот же механизм в forward-режиме. `Closeout` трактуется как обычный последний активный этап трека.

### REQ-DENY-1 — Блокировка прямой правки поля `stage`

Попытка изменить значение поля `stage` в `session_state.json` инструментами `Write`/`Edit`/`MultiEdit` в обход механизма перехода блокируется детерминированно (deny), с объяснением причины и указанием легитимного пути.

### REQ-DENY-2 — Deny-граница ограничена полем `stage`

Блокировка срабатывает **только** если операция изменяет значение поля `stage` относительно текущего содержимого файла. Разрешены без блокировки:
- правки полей `artifacts`, `history`, `track`, `gate_mode`;
- первичное создание файла `session_state.json` (создание, а не изменение).

### REQ-DENY-3 — Деградация вне SDX-сессии

Вне ветки `sdx/<id>` (нет активной сессии) либо при отсутствии `session_state.json` — deny-хук прозрачен (no-op), как и остальные хуки enforcement-слоя.

### REQ-DENY-4 — Поведение при отсутствии `jq`

Если зависимость `jq` недоступна, deny-хук деградирует в **fail-open**: пропускает правку `session_state.json`, но громко предупреждает.

### REQ-COMPAT-1 — Обратная совместимость

Схема `session_state.json` не меняется. Существующие сессии продолжают работать без миграции — единственное изменение: с момента внедрения дальнейшие правки `stage` идут через механизм перехода.

## Явная граница ответственности

Решение закрывает **случайный** обход поля `stage` (прозаическая ошибка, деградация контекста, использование неверного инструмента). Оно **не защищает** от:
- намеренного вызова механизма перехода с произвольным этапом, минуя проверки (скрипт проверяет только артефакты, не авторизацию вызвавшей стороны);
- редактирования `session_state.json` через Bash (`sed -i`, `jq` in-place) — deny-хук покрывает только `Write`/`Edit`/`MultiEdit`. Это пересечение с DEBT-003 (обход enforcement-слоя через Bash).

Мёртвые поля `artifacts`/`history` и недоспецифицированное поле `status` остаются как есть. Условие гейта Business Spec «`status: approved`» остаётся объективно непроверяемым механизмом перехода — это унаследованный пробел DEBT-007, а не новый дефект этой спецификации.

## Критерии приёмки

### Механизм перехода
- Forward-переход с выполненным гейт-условием → успешный переход, `stage` меняется на следующий активный этап трека.
- Forward-переход без требуемого гейт-артефакта → отклоняется; `stage` не меняется; сообщение называет недостающий артефакт и команду.
- Переход в `Closeout` → выполняется через механизм без отдельного кода-пути.
- Backward-переход на активный, не позже текущего этап → выполняется без проверки гейт-артефактов уходящего этапа; артефакты отменяемых этапов помечаются как устаревшие.
- Backward-переход на неактивный этап → отклоняется с указанием на `/sdx:retrack`.
- `/sdx:retrack` пересчитывает `stage` на ближайший активный этап нового трека без повторной проверки forward-гейта уходящего этапа.
- `/sdx:retrack` на этап с непроверённым гейтом предшествующего этапа → отклоняется с указанием на `/sdx:next`.
- Повторный вызов перехода с целевым этапом, равным текущему, → успешный no-op.
- Вызов с нераспознанным именем этапа → отклоняется без изменения файла.

### Deny-хук
- Прямая правка `stage` через `Edit` / `Write` → блокируется с объяснением.
- Правка `artifacts`, `history`, `track`, `gate_mode` (без `stage`) → не блокируется.
- Первичное создание `session_state.json` → не блокируется.
- Вне ветки `sdx/<id>` → deny-хук прозрачен.
- При отсутствии `jq` → deny-хук пропускает правку и предупреждает (fail-open).

### Обратная совместимость
- Существующие сессии продолжают функционировать без миграции.
