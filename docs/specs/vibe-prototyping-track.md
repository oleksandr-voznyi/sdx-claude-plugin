# Спецификация: Режим экстремального прототипирования — трек `vibe`, тип сессии `proto`

Сессия: `feat-vibe-track-20260720` (feature, full). Источник требования: `docs/backlog/PROC-004-vibe-prototyping-mode.md`.
Решение и инварианты: ADR-018 (`docs/DECISIONS.md`). Технический дизайн: `docs/designs/vibe-prototyping-track.md`.

## Контекст и Цели

Сегодня любая SDX-сессия, производящая код, обязана нести церемонию, соразмерную своему треку
(минимум — Execution → Verification → Closeout на `patch`), и следовать норме инкрементальных
коммитов (ADR-005/REQ-SESS-1). Это оправдано, когда решение уже известно — но плохо подходит
для исследовательской работы, где ценность промпта в том, чтобы **быстро увидеть результат
целиком**, не платя заранее за спеки и дизайн для идеи, которая может быть отброшена целиком
или частично.

Цель — ввести новый профиль флоу `vibe`: разработчик формулирует цель одним промптом, получает
реализацию «как есть», без промежуточной церемонии и без инкрементальных коммитов, лично
оценивает результат на интерактивном гейте и либо отбрасывает его (точечный откат до
исходного состояния, с правом повторной попытки), либо переводит в легализацию — обязательный
проход через реверс-инжиниринг бизнес- и технических требований, тестовое покрытие и
верификацию, после которого прототип получает те же гарантии качества, что и код, прошедший
обычный трек. Прототип, оставленный без легализации, не должен иметь пути в основную ветку —
это структурный инвариант, а не договорённость.

## Требования

### Назначение и вход

- [ADDED] **REQ-VIBE-1 (назначение режима).** Новый профиль флоу `vibe` — режим быстрого
  экспериментального прототипирования: разработчик получает работающую реализацию идеи «в один
  промпт» — без предварительных бизнес- и технических спецификаций, без декомпозиции задач и
  без обязанности коммитить изменения по ходу работы. Режим предназначен для ситуаций, когда
  ценность идеи ещё не доказана и целесообразность полной церемонии SDX неочевидна до того, как
  результат увиден целиком.

- [ADDED] **REQ-VIBE-2 (вход в режим).** Режим запускается новым типом сессии `proto`
  (`/sdx:start proto "<цель>"`), который жёстко и безусловно привязан к треку `vibe` — по
  прецеденту жёсткой привязки типа к треку (ADR-017, типы `grooming`/`retro`/`postmortem`/
  `intake` → трек `doc`): диалог о выборе альтернативного трека для типа `proto` не
  предлагается. Единственный активный этап трека до момента гейта принятия/отклонения —
  стадия `Prototype`, на которой реализация выполняется как одно непрерывное действие, без
  промежуточных подтверждений и без промежуточных артефактов Discovery/Business Spec/Technical
  Design/Task Planning.

### Жизненный цикл

- [ADDED] **REQ-VIBE-3 (стадия Prototype).** На стадии `Prototype` разработка ведётся без
  церемонии, применимой к другим трекам: не создаются `SPEC.md`/`DESIGN.md`/`PLAN.md`/
  `change_note.md`, промежуточные коммиты кода не производятся. Рабочее дерево на этой стадии
  может содержать незакоммиченные изменения — это осознанное и временное исключение из нормы
  инкрементальных коммитов (см. REQ-VIBE-9), действующее исключительно в пределах стадии
  `Prototype` этого трека.

- [ADDED] **REQ-VIBE-4 (гейт принятия/отклонения).** По завершении реализации на стадии
  `Prototype` разработчику предъявляется результат для интерактивного решения: **принять**
  (перейти к легализации, REQ-VIBE-6) или **отклонить** (откатить прототип, REQ-VIBE-5).
  Гейт судится ЛИЧНО разработчиком — независимое fresh-eyes ревью (`reviewer`) на этом гейте
  не вызывается; `reviewer` включается в работу штатно позже, уже на этапе Verification
  легализованной сессии.

- [ADDED] **REQ-VIBE-5 (отклонение и точечный откат).** При отклонении прототипа выполняется
  откат рабочего дерева, ограниченный **точечным списком файлов**, фактически изменённых или
  созданных за время стадии `Prototype` этой попытки: перед откатом пользователю предъявляется
  этот список (и дифф) как часть решения. Тотальный откат всего рабочего дерева
  (эквивалент `git clean -fd` по всему дереву без ограничения списком) не применяется — файлы,
  не относящиеся к прототипу, откатом не затрагиваются. После отклонения сессия **остаётся**
  на стадии `Prototype` — допускается повторная попытка в рамках той же сессии; лимита числа
  попыток нет.

- [ADDED] **REQ-VIBE-6 (легализация).** При принятии прототипа сессия обязана пройти
  легализацию — единственный легитимный способ покинуть трек `vibe`, выполняемый через
  `/sdx:retrack` на трек `standard` или `full`. Легализация обязана произвести реверс-
  инжиниринг фактически реализованных изменений в требования (`change_note.md` либо
  `SPEC.md`+`DESIGN.md` — по образцу `/sdx:init --existing`), покрытие тестами, полный проход
  верификации (включая fresh-eyes `reviewer`, действующий на легализованной сессии по
  правилам своего целевого трека) и штатное закрытие через `/sdx:archive`. Целевой трек
  легализации (`standard` или `full`) не фиксирован жёстко заранее — решение принимается по
  фактическому масштабу получившегося прототипа тем же порядком, каким трек выбирается на
  `/sdx:start` для обычной задачи.

- [ADDED] **REQ-VIBE-7 (первый коммит кода — при легализации).** Поскольку на стадии
  `Prototype` код не коммитится (REQ-VIBE-3), легализация обязана произвести первый коммит
  кода прототипа как часть перехода — именно этим восстанавливается норма инкрементальных
  коммитов (REQ-VIBE-9) для дальнейшего хода легализованной сессии.

### Инварианты и границы

- [ADDED] **REQ-VIBE-8 (инвариант: нет мёржа без легализации).** Сессия трека `vibe`, не
  прошедшая легализацию (REQ-VIBE-6), не может быть закрыта и не может попасть в основную
  ветку. Это структурный инвариант жизненного цикла трека, а не прозаическая договорённость:
  единственный выход из трека `vibe` — переход на другой трек через `/sdx:retrack`; трек
  `vibe` в принципе не оканчивается собственным Closeout.

- [ADDED] **REQ-VIBE-9 (объём приостановления REQ-SESS-1).** Приостановление обязательности
  инкрементальных коммитов (ADR-005/REQ-SESS-1) действует ТОЛЬКО в пределах стадии `Prototype`
  трека `vibe` и является явно поименованным исключением, а не отменой нормы: на всех прочих
  треках и стадиях, а также на самой `vibe`-сессии после легализации (REQ-VIBE-7), норма
  действует без изменений.

- [ADDED] **REQ-VIBE-10 (безусловная интерактивность гейта принятия/отклонения).** Гейт
  принятия/отклонения (REQ-VIBE-4) ОБЯЗАН запрашивать явное решение пользователя независимо
  от `gate_mode` сессии, включая `gate_mode: auto`. Это отражение стоп-рубрики протокола:
  решение содержит деструктивную операцию (откат при отклонении) и продуктовое суждение
  («достигла ли идея цели»), которые не подлежат автоматическому дефолту ни в каком режиме.

- [ADDED] **REQ-VIBE-11 (запрет входа в vibe извне).** Переход в трек `vibe` из любого другого
  трека через `/sdx:retrack` запрещён — прозаически, по прецеденту симметричного запрета
  деэскалации в трек `doc` (ADR-017). Трек `vibe` доступен только как стартовая точка сессии
  типа `proto` (REQ-VIBE-2).

- [ADDED] **REQ-VIBE-12 (граница: без параллельного прототипирования).** Одновременное
  прототипирование нескольких вариантов решения одной задачи в рамках режима `vibe`
  сознательно не входит в объём этой поставки. Пересекается с открытым `docs/backlog/PROC-010`
  (параллельные субагенты за общие файлы сессии) и рассматривается отдельно, при закрытии
  этой зависимости.

- [ADDED] **REQ-VIBE-13 (граница: без структурированного входа материала).** Режим `vibe` не вводит отдельного структурированного места для входящих
  материалов (BRD, макеты) — это зависимость `docs/backlog/PROC-009`, вне объёма этой
  поставки; цель прототипа формулируется пользователем текстом промпта на `/sdx:start`.

## Критерии приемки

- [ ] `/sdx:start proto "<цель>"` создаёт сессию с типом `proto` и треком `vibe`, стартующую
      сразу на стадии `Prototype`, без диалога о выборе альтернативного трека.
- [ ] На стадии `Prototype` реализация выполняется без создания `SPEC.md`/`DESIGN.md`/
      `PLAN.md`/`change_note.md` и без промежуточных коммитов кода.
- [ ] По завершении реализации пользователю предъявляется интерактивный гейт
      принятия/отклонения — независимо от значения `gate_mode` (включая `--auto`).
- [ ] При отклонении: пользователю предъявлен список фактически тронутых/созданных файлов
      прототипа (и дифф) до выполнения отката; откат ограничен этим списком; файлы вне
      списка не затронуты; сессия остаётся на стадии `Prototype`; повторная попытка возможна
      без ограничения по числу попыток.
- [ ] При принятии: единственный доступный путь — `/sdx:retrack` на `standard` или `full`;
      попытка закрыть vibe-сессию (`/sdx:archive`) без предварительной легализации
      останавливается с указанием на необходимость `/sdx:retrack`.
- [ ] Легализованная сессия производит `change_note.md` либо пару `SPEC.md`+`DESIGN.md`,
      тестовое покрытие, проходит верификацию (включая fresh-eyes `reviewer`) и закрывается
      штатным `/sdx:archive` — код прототипа при этом закоммичен впервые в рамках легализации.
- [ ] Сессия трека `vibe`, не прошедшая легализацию, не может быть слита в основную ветку.
- [ ] `/sdx:retrack` из любого другого трека в `vibe` отклоняется.
- [ ] Документация фреймворка (`sdx/protocol.md`, `CLAUDE.md`) явно называет `vibe` исключением
      из нормы инкрементальных коммитов и параллельной веткой, не входящей в линейную шкалу
      `patch → standard → full`.

## Риски и Ограничения

- **Широта отката при отклонении.** Точечный откат (REQ-VIBE-5) требует, чтобы список
  затронутых файлов был вычислен точно относительно момента входа в стадию `Prototype`;
  ошибка в его вычислении либо теряет изменения прототипа не полностью, либо задевает файлы,
  не относящиеся к нему. Способ вычисления списка — предмет Technical Design, не этой спеки.
- **Отсутствие fresh-eyes ревью до легализации.** Решение принять/отклонить прототип целиком
  доверено пользователю — при недостаточной внимательности слабый прототип может быть
  легализован без независимой проверки на этом конкретном шаге (компенсация — обязательное
  ревью на Verification уже после легализации, REQ-VIBE-6).
- **Неограниченное число попыток отклонения.** Отсутствие лимита попыток (REQ-VIBE-5) — по
  решению пользователя на Discovery-гейте; при необходимости ограничение может быть введено
  отдельной доработкой.
- **Открытая зависимость PROC-009/PROC-010.** Параллельное прототипирование (REQ-VIBE-12) и
  структурированный вход материалов (REQ-VIBE-13) сознательно оставлены вне объёма — не
  устранены, а отложены до закрытия соответствующих записей бэклога.
