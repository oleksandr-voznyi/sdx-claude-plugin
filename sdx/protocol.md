# Протокол SDX-сессий

Этот документ описывает формат состояния и логирования сессий. Им руководствуется основная сессия Claude Code при выполнении команд `/sdx:*` (роль Session Manager). Оркестрация субагентов (architect, ba, lead-dev, developer, qa, tech-writer, devops) выполняется только из основной сессии через инструмент Task — субагенты не могут вызывать друг друга.

## Состояние сессии (session_state.json)
```json
{
  "session_id": "string",
  "type": "feature | bug | refactor | init | import | grooming | retro | postmortem | intake",
  "track": "patch | standard | full | doc",
  "stage": "Discovery | Business Spec | Technical Design | Task Planning | Execution | Documentation | Verification | Deployment | Closeout",
  "status": "draft | review | approved | executing",
  "gate_mode": "interactive | auto",
  "git_branch": "sdx/...",
  "artifacts": [],
  "history": []
}
```

- **`type`** — исходная категория задачи. Это лишь _стартовая гипотеза_ для выбора трека, а не жёсткая привязка.
- **`track`** — фактический режим прохождения сессии (профиль флоу). Может меняться по ходу работы через `/sdx:retrack`.
- **`stage`** — текущий этап. Допустимые этапы зависят от активного трека (см. ниже).
- **`gate_mode`** — режим прохождения прозаических гейтов: `interactive` (по умолчанию) или `auto` (opt-in, см. «Авторежим гейтов»). Отсутствие поля = `interactive`.

## Модель сессии = ветка в основном рабочем дереве (ADR-012)
Одна сессия = одна **git-ветка** `sdx/<id>`, checkout'нутая в ОСНОВНОМ рабочем дереве
репозитория. Весь жизненный цикл — от `/sdx:start` до `/sdx:archive` включительно — проходит
в **одном CLI-инстансе**, запущенном в корне репозитория; хендоффов на другой CLI нет.
Содержательные артефакты сессии (`.claude/sessions/<id>/session_state.json`, `session.log`,
`SPEC.md`/`DESIGN.md`/`PLAN.md`/`change_note.md` и т.д.) **версионируются** на этой ветке
инкрементальными коммитами; эфемерные `.stopgate.*` игнорируются точечным паттерном корневого
`.gitignore` (REQ-SESS-1/2). Основная ветка (автоопределяется, см. ниже) никогда не видит файлы
сессии в своём HEAD-дереве: удаление каталога сессии оформляется коммитом `git rm -r` НА ВЕТКЕ
`sdx/<id>` ДО мёржа (**вариант A**), после чего мёрж `--no-ff` вносит всю историю ветки в DAG
основной ветки, но не в её HEAD-дерево.

**Операционные следствия.** Enforcement-хуки резолвят активную сессию по имени текущей ветки
(`sdx/<id>`) — при работе в основном дереве `$CLAUDE_PROJECT_DIR` (корень репозитория) корректен
по построению, отдельный CLI не нужен. Переключение сессий (`/sdx:switch`) — обычный
`git checkout sdx/<id>` с гардом чистого дерева: при незакоммиченных изменениях — стоп и
инструкция закоммитить их в текущую сессию (слепой авто-коммит запрещён, ядро REQ-WT-2).
Физической изоляции параллельных «грязных» деревьев (бывший REQ-WT-3) больше нет — осознанный
трейд-офф в пользу непрерывности работы в одном CLI; дисциплина инкрементальных коммитов
(REQ-SESS-1) — штатная компенсация.

Обоснование и границы — ADR-012 в `docs/DECISIONS.md` (worktree-модель ADR-009 пересмотрена).

## Профили флоу (Tracks)
Трек — это упорядоченный набор _активных_ этапов. `/sdx:next` переводит сессию на следующий **активный** этап текущего трека, пропуская неактивные. Это позволяет масштабировать церемонию под размер задачи.

| Трек | Назначение | Активные этапы | Артефакты дельт |
|------|------------|----------------|------------------|
| **patch** | багфикс / точечная правка без изменения логики и контрактов | Execution → Verification (лёгкая, обязательная — ADR-014) → Closeout | `change_note.md` |
| **standard** | малая фича / рефактор | Discovery (лёгкий, инлайн) → Change → Execution → Verification → Closeout | `change_note.md` |
| **full** | крупная фича, затрагивающая контракты/архитектуру | Discovery → Business Spec → Technical Design → Task Planning → Execution → Documentation → Verification → Deployment → Closeout | `SPEC.md`, `DESIGN.md`, `PLAN.md` |
| **doc** | процессная/рефлексивная работа без кода: груминг бэклога / ретроспектива / постмортем / intake (типы `grooming`/`retro`/`postmortem`/`intake`, жёстко привязаны к этому треку на `/sdx:start` — ADR-017) | Discovery (лёгкий, инлайн) → Update → Verification (лёгкая, обязательная — ADR-017) → Closeout | `change_note.md` |

**Примечание:** Машиночитаемый источник истины порядка активных этапов каждого трека и их гейт-артефактов — `SDX_STAGE_MATRIX` внутри `sdx/hooks/sdx-stage.sh`. Таблица выше — человекочитаемая проекция для справки. При расхождении скрипт имеет приоритет. Sanity-тест `test-sdx-stage.sh` («sanity: SDX_STAGE_MATRIX matches sdx/protocol.md's track table for full/standard/patch/doc, in order») сверяет упорядоченные цепочки названий этапов из таблицы и матрицы для всех четырёх треков и выявляет явное пропущенное/добавленное/переставленное различие.

Пояснения:
- **Change** (треки patch/standard) — объединённый этап: бизнес- и технические решения фиксируются _слитно_ внутри `change_note.md`, без отдельных `SPEC.md`/`DESIGN.md`.
- **Discovery** в `standard` выполняется инлайн в основной сессии без вызова субагента `architect`.
- **Closeout** — обязательный завершающий этап для ВСЕХ треков (бывший Archiving). Контракт закрытия описан ниже.
- **Трек `doc`** — параллельная ветка, а не часть линейной шкалы `patch → standard → full`: церемония кода (Execution/Documentation/Deployment) здесь не активна вовсе, а активные этапы обслуживают процессную/рефлексивную работу над самим SDX-проектом. Четыре значения `type` жёстко и безусловно привязаны к нему на `/sdx:start` (без диалога о выборе трека, ADR-017):
  - **grooming** — пересмотр ТЕКУЩЕГО состояния `docs/backlog/`: актуализация статусов/приоритетов/волн уже существующих записей (update-путь над бэклогом).
  - **retro** — разбор набора уже ЗАВЕРШЁННЫХ SDX-сессий за период/список (`docs/history/sessions-log.md`), выводы оформляются новыми записями бэклога и постоянным документом `retro.md`.
  - **postmortem** — разбор ОДНОГО конкретного инцидента (прод, процессный сбой, серьёзный дефект): хронология, корневая причина, план действий; постоянный документ `postmortem.md`.
  - **intake** — разбор НОВОГО значительного блока внешних требований (эпик, пачка багрепортов, продуктовый материал), цель — **породить** записи бэклога из этого материала (create-путь); постоянный документ `intake.md`.
  - **`intake` vs `grooming` — не синонимы и не альтернативные имена одного типа**, несмотря на то, что оба читают/пишут один и тот же `docs/backlog/` через один и тот же гейт-артефакт `change_note.md`. Разница — в направлении операции: `intake` **порождает** записи из внешнего материала, `grooming` **перераспределяет** атрибуты уже существующих записей и сам новых записей из внешнего материала не создаёт. Типичный порядок — сначала `intake` (материал → записи), затем, отдельной сессией, `grooming` (пересмотр приоритетов среди уже накопленного); порядок не enforced, оба типа доступны в любой момент. Ничто в enforcement-слое эту границу не проверяет — она держится текстом инструкций `/sdx:start` и шаблоном `intake.md` (раздел «Открытые вопросы к дальнейшему груминг-циклу» как явная точка передачи эстафеты), см. ADR-017.
  - **Что производит этап `Update` (контракт содержимого `change_note.md`, ADR-017).** `change_note.md` — единственный объективно проверяемый гейт-артефакт этого этапа для всех четырёх типов; минимальный состав: краткое резюме сессии (что разобрано/поправлено и почему) + список ID затронутых/порождённых записей `docs/backlog/` +, для `retro`/`postmortem`/`intake`, ссылка на параллельно ведущийся содержательный документ `retro.md`/`postmortem.md`/`intake.md` в каталоге сессии + для `intake` дополнительно — состав внешнего материала, фактически взятого в разбор (ссылки/пути/описание вложений; единственный структурированный след входа, пока не закрыт `docs/backlog/PROC-009`). **Правки `docs/backlog/` выполняются НА этом этапе, а не откладываются до Closeout** — п.4 чек-листа Closeout (см. ниже) — это финальная сверка/актуализация статусов, не первичная работа над бэклогом.

### Адаптивность трека (re-routing)
Трек выбирается при `/sdx:start` как гипотеза, но **не фиксирован**. На каждом гейте (`/sdx:next`) и при любом существенном открытии оркестратор ОБЯЗАН переоценить адекватность трека:
- **Эскалация** (`patch → standard → full`): задача оказалась глубже — затронуты бизнес-логика, публичные контракты, схемы данных или архитектура. При эскалации ранее накопленный `change_note.md` промоутится в полноценные `SPEC.md`/`DESIGN.md`, а пропущенные этапы становятся активными.
- **Деэскалация** (`full → standard → patch`): задача оказалась мельче ожидаемого — церемония избыточна. Допускается с явного подтверждения пользователя; уже созданные артефакты сохраняются как справочные.

Смена трека выполняется командой `/sdx:retrack` и логируется категорией `STAGE_CHANGE`.

## Логирование (экономный режим)
В журнал `.claude/sessions/<session_id>/session.log` пишутся ТОЛЬКО durable-события смены состояния. Каждая отдельная запись в лог — это лишний ход агента (полный round-trip контекста), поэтому логирование намеренно минимально.

### Формат строки:
`[YYYY-MM-DD HH:mm:ss] [КАТЕГОРИЯ] [ЭТАП] Сообщение`

### Логируемые категории (только эти четыре):
- `START`: Инициализация сессии или переключение на неё.
- `STAGE_CHANGE`: Переход между этапами / смена трека.
- `CHECKPOINT`: Фиксация точки восстановления.
- `ERROR`: Ошибки команд или проваленные гейты.

### Что НЕ логируется (выводится иначе):
- **Артефакты (создание/изменение файлов)** — НЕ логируются. История изменений файлов выводится из git (`git log --stat`, `git diff --name-status`), а текущий состав артефактов — из `git status`/листинга папки сессии по запросу. Для этого артефакты коммитятся в ветку сессии **инкрементально** по ходу работы (это же поддерживает чек-лист Closeout).
- **INFO / просмотр статуса** — НЕ логируется вообще. Чтение состояния не должно мутировать его.

### Правила записи:
- **Время**: не выдумывай — получай `date '+%Y-%m-%d %H:%M:%S'`. Захватывай время и пиши запись в том же ходе, что и правку `session_state.json` (батч), а не отдельным вызовом.
- **ЭТАП**: текущее значение `stage`. **Сообщение**: кратко, на русском.
- Запись — в режиме **append** (`>>`):
```bash
echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] [Execution] -> Verification" >> .claude/sessions/<session_id>/session.log
```

## Передача контекста субагентам (экономия токенов)
При вызове субагента через Task передавай **пути к артефактам и краткую задачу**, а не вставляй содержимое файлов в промпт. Субагент прочитает нужное сам. Вставка контента в промпт ведёт к двойной токенизации (в контексте оркестратора и заново в субагенте). Исключение — fresh-eyes `reviewer`, которому diff передаётся как данные по контракту изоляции.

## Гейты (Gates)
Перед переходом на следующий **активный** этап трека проверяется наличие необходимых артефактов (`SPEC.md`, `DESIGN.md`, `PLAN.md`, `change_note.md` и т.д.). Объективно проверяемая часть гейта (существование/непустота артефактов, отсутствие `FAIL` в `verification_report.md`) **enforced автоматически** скриптом `sdx-stage.sh` (см. ниже, раздел «Механизм перехода этапов»). Смысловые условия («решение согласовано», контракты подтверждены) остаются **прозаическим суждением оркестратора** (не проверяются скриптом). Гейт также включает scope-check: оркестратор оценивает, не вышла ли задача за рамки текущего трека, и при необходимости предлагает `/sdx:retrack`.

## Механизм перехода этапов (`sdx-stage.sh`)

Единственный детерминированный писатель поля `.stage` в `session_state.json` — CLI-скрипт `sdx/hooks/sdx-stage.sh` (не Hook в терминах `hooks.json`, а самостоятельный скрипт, вызываемый командами через инструмент `Bash`, как уже происходит с `archive-verify.sh`). Скрипт инкапсулирует машиночитаемую матрицу «трек → упорядоченные активные этапы → гейт-артефакт» и выполняет все проверки гейта автоматически.

### Четыре подкоманды

| Подкоманда | Назначение | Вызывается из | Коды возврата |
|---|---|---|---|
| `init <sid> <type> <track> <stage> <gate_mode> <git_branch>` | Первичное создание `session_state.json` (REQ-STAGE-1) | `/sdx:start`, `/sdx:import` | `0` успех; `2` ошибка (файл уже существует, неверный стартовый этап) |
| `next <sid>` | Forward-переход на следующий активный этап; проверяет гейт уходящего этапа (REQ-STAGE-2/4/5) | `/sdx:next`, `/sdx:archive` (вход в Closeout) | `0` успех/no-op; `1` гейт не пройден; `2` ошибка (нет `session_state.json`, повреждённое состояние) |
| `backtrack <sid> <target>` | Backward-переход на указанный этап без проверки гейта уходящего этапа; помечает артефакты как устаревшие (REQ-BACKTRACK-1/2) | `/sdx:backtrack --to <stage>` | `0` успех/no-op; `1` отказ (этап не найден, не активен в треке, позже текущего); `2` ошибка |
| `retrack <sid> <target>` | Переход на другой этап после смены трека; вызывается **после** того, как `track` уже обновлён через `Edit` (REQ-RETRACK-1) | `/sdx:retrack` | `0` успех/no-op; `1` целевой этап не распознан/не активен в новом треке, либо какой-то из предшествующих ему этапов не подтверждён гейт-артефактом на диске (REQ-RETRACK-2, evidence-based); `2` ошибка |

### stdout и stderr (REQ-STAGE-5)

- **Успех (код 0):** stdout содержит одну из строк: `OK <old-stage> -> <new-stage>` или `OK no-op <stage>`. Для `backtrack` дополнительно выводятся строки `OUTDATED: <path>` для каждого помеченного артефакта.
- **Отказ (код 1/2):** stderr содержит одну developer-facing строку на русском языке, называющую (а) чего не хватает и (б) какой командой чинить.

### Forward-skip guard в `retrack` (REQ-RETRACK-2)

`backtrack` не приземляет `stage` позже текущего этапа в порядке трека (REQ-BACKTRACK-1). `retrack` симметрично не приземляет `stage` дальше, чем позволяет уже накопленный прогресс — но поскольку целевой и текущий этап могут принадлежать РАЗНЫМ трекам (в этом весь смысл `retrack`), «накопленный прогресс» не может определяться позицией `stage` внутри старого трека: два легитимных вызова `retrack` подряд способны раздуть самодекларируемую позицию без единого пройденного гейта (это и было содержанием отменённой ранговой механики `SDX_CANON_ORDER`/`canon_rank`, полностью удалённой из `sdx-stage.sh` как источник этого класса дефекта).

Правило — **evidence-based**, не позиционное: целевой этап нового трека допустим, только если у КАЖДОГО этапа этого трека, предшествующего целевому в порядке матрицы (`SDX_STAGE_MATRIX`), собственный гейт-артефакт фактически существует и непуст на диске (те же критерии, что `next` применяет к уходящему этапу — существование+непустота, плюс отсутствие маркера `[FAIL]` там, где он применим). `stage`/старый трек в этой проверке вообще не участвуют — ранг никогда не выводится из самодекларированной позиции. У собственного первого активного этапа трека цепочка предшественников пуста по построению, поэтому войти в трек с его стартовой точки всегда можно; приземлиться дальше требует тех же артефактов, которые потребовал бы реальный проход через `/sdx:next` — с единственным исключением: этап `Change` (см. ниже) засчитывается и по паре `SPEC.md`+`DESIGN.md`, тогда как `next` на треках patch/standard требует именно `change_note.md`.

Для объединённого этапа `Change` (patch/standard) — у которого нет собственного файла-артефакта — гейт считается подтверждённым, если существует непустой `change_note.md` **ИЛИ** одновременно непустые `SPEC.md` и `DESIGN.md` (эквивалент, который `retrack.md` шаг 3 безусловно создаёт при эскалации из `Change`, и который у сессии `full`, дошедшей своим ходом до `Technical Design`, уже есть по факту).

Guard по-прежнему не проверяет гейт-артефакт САМОГО целевого этапа и уходящего этапа (это вне скоупа `retrack`, REQ-RETRACK-1 не меняется) — только цепочку предшественников цели. Форвард-движение мимо гейта остаётся исключительно прерогативой `/sdx:next`.

**Границы доказательности (общие для `next` и `retrack`).** Доказательством пройденного гейта считается факт существования непустого файла-артефакта (плюс отсутствие маркера `[FAIL]` там, где он применим) — не его содержательное качество. Файл-заглушка в одну строку гейт проходит: механизм отличает «артефакта нет» от «артефакт есть», но не «артефакт осмыслен» — последнее остаётся предметом прозаического суждения оркестратора и fresh-eyes ревью на Verification. Из того же следует, что баннер `<!-- SDX-OUTDATED -->`, который `backtrack` проставляет отменённым артефактам, **не влияет ни на один гейт**: помеченный устаревшим `DESIGN.md` продолжает считаться доказательством для цепочки предшественников. Баннер — сигнал человеку и агенту, что содержимое требует актуализации, а не механическая блокировка. Наконец, `retrack` не требует, чтобы `track` фактически изменился: вызов с прежним треком допустим и работает как перепозиционирование в пределах тех же артефактных требований.

### Источник истины: матрица внутри скрипта

**Машиночитаемый источник истины** порядка активных этапов для каждого трека и соответствующих гейт-артефактов находится **исключительно** внутри `sdx/hooks/sdx-stage.sh` как heredoc-матрица. Таблица «Профили флоу» выше в этом документе (раздел «Профили флоу (Tracks)») — **человекочитаемая проекция той же данных** для справки; она **не используется** скриптом. При расхождении между таблицей выше и матрицей в скрипте — прав скрипт. Sanity-тест сравнивает упорядоченные цепочки названий этапов из обеих мест для каждого трека и сигнализирует о явном пропуске/добавлении/перестановке этапа в одном из мест (см. `sdx/hooks/test-sdx-stage.sh`, сценарий «sanity: SDX_STAGE_MATRIX matches sdx/protocol.md's track table»). Отдельный sanity-сценарий («sanity: SDX_CANON_ORDER / canon_rank dead code fully removed») страхует от повторного появления в скрипте отменённой ранговой межтрековой шкалы (см. «Forward-skip guard в `retrack`» ниже) — она была вынесена оттуда как мёртвый/вредный код.

## Авторежим гейтов (gate_mode, opt-in — ADR-014)
По умолчанию сессия интерактивна: прозаические гейты (аппрувы артефактов, переходы `/sdx:next`, квитирование WARN) требуют подтверждения пользователя на каждом шаге. Для тривиальных задач пользователь может включить **авторежим** — `gate_mode: "auto"` в `session_state.json`: флагом `/sdx:start --auto` либо согласием на предложение оркестратора на Discovery/Change-гейте (там, где согласуются развилки задачи). Решение per-session; возврат в `interactive` — в любой момент по слову пользователя.

При `gate_mode: "auto"`:
- Прозаические подтверждения и несущественные уточнения по ходу принимают **дефолт без остановки**. Каждый принятый дефолт фиксируется одной строкой в файле сессии `auto_decisions.md` (формат: `развилка → выбранный дефолт → почему несущественно`). Файл трек-независим (работает и на `full`, где нет `change_note.md`) и версионируется как обычный артефакт сессии.
- Детерминированные хуки НЕ смягчаются: stage-gate, stop-gate, prod-guard и archive-verify работают без изменений. Авторежим касается только прозаического слоя.
- Единственная обязательная точка подтверждения — **дисклоуз** на входе в Closeout (см. `/sdx:archive`): `auto_decisions.md` выводится пользователю одним блоком (автоквитированные WARN — первыми), после чего запрашивается одно явное подтверждение. Несогласие с любым дефолтом → штатный `/sdx:backtrack`/`/sdx:retrack`.

**Стоп-рубрика.** Авторежим ОБЯЗАН остановиться и вернуть решение пользователю на:
- коллизиях триады SPEC ↔ DESIGN ↔ Codebase (CLAUDE.md §4 — без исключений);
- находках FAIL верификации и красном тест-поле stop-gate;
- изменениях внешних контрактов: публичные API, схемы данных, форматы файлов, миграции;
- деструктивных/необратимых операциях и территории prod-guard;
- эскалации трека: `/sdx:retrack` вверх сбрасывает `gate_mode → interactive` (масштаб изменился — исходное решение «задача тривиальна» недействительно); деэскалация режим не меняет;
- шагах ручного тестирования (`/sdx:manual` интерактивен по определению, ADR-006).

Критерий по умолчанию для не перечисленного: если цена отката ошибочного дефолта выше цены остановки — остановись.

## Enforcement-слой (хуки)
Инварианты, обязанные выполняться всегда, вынесены из прозы в детерминированные хуки.
Скрипты поставляются плагином (`${CLAUDE_PLUGIN_ROOT}/sdx/hooks/`), проводка — `hooks/hooks.json`
плагина. Per-project конфиги слоя (`prod-guard.conf`, `stage-gate.allow`, `verify-cmd.sh`)
живут в `.claude/sdx/` целевого проекта и разворачиваются `/sdx:init` из шаблонов
`${CLAUDE_PLUGIN_ROOT}/sdx/templates/`. Все хуки активны только в
SDX-ветке `sdx/<id>` и деградируют в no-op при отсутствии сессии/конфига (safe-by-default).
Механизм блокировки PreToolUse: JSON `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"<причина>"}}` на stdout, exit 0 (НЕ exit 2 — подтверждено на бинарнике Claude Code 2.1.195).

- **stage-gate** (PreToolUse `Write|Edit|MultiEdit`): запись в код заморожена до гейта
  Execution; `docs/**`, `.claude/**`, `*.md` открыты всегда (артефакты планирования).
  На стадии **Verification** дополнительно открыты тестовые каталоги (`tests/**`, `test/**`,
  `spec/**`, вкл. вложенные) — там `qa` пишет интеграционные тесты (`/sdx:verify` шаг 2);
  не-тестовый код остаётся заморожен, правки FAIL-находок — через `/sdx:backtrack --to Execution`.
  Со-локованные тесты (напр. `foo_test.go` рядом с исходником) покрываются per-project
  `stage-gate.allow`. Блокировка — JSON `permissionDecision:"deny"`. Plan mode НЕ используется
  намеренно — он заблокировал бы и `.md`-артефакты Spec/Design/Plan.
- **stop-gate** (Stop): тест-прогон как пол под Verification; ход не завершается на красном
  (скоуп Execution/Verification или `SDX_STOP_GATE=1`). После 3 красных подряд — возврат
  человеку. Привязан к `Stop` (не `SubagentStop`), чтобы не воевать с TDD red-green developer'а.
  Деградирует в no-op без известной тест-команды (`verify-cmd.sh` — per-project, ADR-4).
  **Green-run cache (ADR-011):** отпечаток дерева (`HEAD` + хэш `git status --porcelain`)
  последнего зелёного прогона хранится в `.stopgate.ok`; при неизменном дереве повторный `Stop`
  пропускается без перезапуска verify (кэш-хит до инкремента loop-guard). `SDX_STOP_GATE=1`
  кэш обходит; красный прогон кэш не пишет.
- **prod-guard** (PreToolUse `Bash`): прод-команды по `prod-guard.conf` блокируются; пустой
  или отсутствующий conf (нет ни одной активной строки-паттерна) = нет защиты (opt-in
  per-project) — проверка conf выполняется **раньше** проверки `jq`. Прод-деплой — только явное
  действие человека, не агента. **Граница:** покрывает только инструмент `Bash` — деплой
  через иной механизм (MCP-tool, шелл-обёртка) его минует. При отсутствии `jq` prod-guard
  деградирует в **fail-closed** (блокирует команду из осторожности) **только если защита
  сконфигурирована**; без активных паттернов отсутствие `jq` не блокирует команды.
- **stage-write-guard** (PreToolUse `Write|Edit|MultiEdit`): узкая деном-граница,
  блокирующая прямую правку поля `stage` в `session_state.json` вне механизма `sdx-stage.sh`.
  Резолвит активную SDX-сессию по имени текущей ветки; вне ветки `sdx/*` — no-op.
  Специфичен исключительно к файлу `session_state.json` текущей сессии (сравнение по точному
  пути, не basename; сегменты `/./` и повторные `/` нормализуются на обеих сторонах сравнения
  перед сопоставлением — путь с `/./` НЕ обходит хук). **Первичное создание не блокируется**
  (если файл ещё не существует на диске) — карве-аут defence-in-depth поверх того, что
  инициализация уже идёт через `sdx-stage.sh init` (вызов `Bash`, вне этого хука в принципе).
  **Обнаружение — apply-and-compare для всех трёх инструментов:** для `Write` новое содержимое
  дано целиком; для `Edit`/`MultiEdit` правка СИМУЛИРУЕТСЯ (литеральная, не regex/glob
  подстановка через `awk`) над реально прочитанным содержимым файла — в обоих случаях денится,
  только если распарсенное значение `.stage` фактически изменилось, паттерн-поиск по ключу
  `"stage"` не используется. **Сбой инструмента ≠ тихий обход:** если симуляция правки не
  может выполниться (`awk`-процесс не запускается, например `E2BIG` на аномально большом
  файле) — это отличимо по коду возврата от штатного «`old_string` не найден» и трактуется
  как «не могу доказать, что `stage` не меняется» → **`deny`** с сообщением о сбое инструмента,
  а не пропуск. **Карве-аут восстановления:** если содержимое файла ДО операции уже не
  является валидным JSON (пусто/пробельно, либо реальная ошибка парсинга — не то же самое,
  что валидные скалярные документы `null`/`false`, которые повреждением не считаются и
  защищены обычным путём), операция трактуется как починка уже сломанного состояния и не
  денится — независимо от того, что получится после. Это единственный путь, которым правка
  через `Write`/`Edit`/`MultiEdit` может миновать проверку `stage`; чтобы им воспользоваться,
  файл сперва должен быть повреждён способом, ЭТИМ же хуком не покрываемым (см. границы ниже).
  **Реальные, не устранённые границы:** (а) сегмент `..` в пути НЕ нормализуется (нормализация
  ограничена `/./` и повторными `/`) — практически недостижимо, т.к. контракт инструментов
  Claude Code требует абсолютный путь без относительных сегментов, но технически возможно;
  (б) `session_state.json` остаётся редактируемым напрямую через `Bash` (`sed -i`/`jq`
  in-place) — этот хук матчится только на `Write|Edit|MultiEdit`, Bash-обход — осознанно
  принятая граница (DEBT-003), через неё же единственно достижим карве-аут восстановления
  выше. **Деградация без `jq`:** fail-open (exit 0) с громким предупреждением в stderr
  (REQ-DENY-4, асимметрия с `sdx-stage.sh` — у deny-хука выше цена отказа писать, чем
  случайной правки в уже деградированной среде).
- **preflight** (SessionStart): однократная проверка зависимостей хуков (`jq`). При
  отсутствии `jq` громко предупреждает (prod-guard → fail-closed, stage-gate → fail-open);
  опционально может жёстко отказывать в сессии (закомментированный `exit 2`).
- **archive-verify** (вызов из `/sdx:archive` после мёржа, вариант A): полагается только на
  переданный `$proj` и git. Резолвит основную ветку через `lib/default-branch.sh` (без хардкода
  `main`, ADR-010) и проверяет Closeout-инварианты: 1 (рабочее дерево чистое), 5 (ветка
  `sdx/<id>` доказуемо слита в автоопределённую основную ветку — имя ветки сверяется точно
  `grep -qx`, при отсутствии ветки слияние считается недоказуемым и деструктив отменяется без
  явного подтверждения `SDX_ARCHIVE_NO_BRANCH_OK=1`), 6 (каталог сессии отсутствует/не tracked
  в дереве основной ветки — детектор пропуска коммита `git rm` до мёржа). Деструктив
  (`git branch -d`; условное `git worktree remove --force` — legacy-совместимость с
  worktree-сессиями до ADR-012, при их отсутствии no-op) выполняется ТОЛЬКО после того, как все
  три инварианта доказаны — `rm -rf` каталога сессии остаётся лишь страховкой на остаточный
  физический путь, а не основным механизмом удаления.

## Верификация (этап Verification)
Этап верификации структурирован по **трём осям** и сочетает корректность-исполнением с независимым fresh-eyes ревью. Команда — `/sdx:verify`, артефакт — `verification_report.md` в папке сессии.

### Три оси
- **Полнота (completeness)** — у каждого требования/критерия есть реализация и тест; нет кода без прослеживаемого требования (scope creep). Источник предупреждений вида «сценарий X не имеет тестового покрытия».
- **Корректность (correctness)** — поведение соответствует намерению, граничные случаи из `DESIGN.md` обработаны, тесты не тавтологичны.
- **Когерентность (coherence)** — непротиворечивость триады `SPEC ↔ DESIGN ↔ Codebase`. Это операционализация детектора коллизий из CLAUDE.md §4: проверку ведёт независимая сторона, а не автор изменений.

### Статусы и гейт
- **FAIL** блокирует гейт (возврат к `developer`/`backtrack`) — в любом `gate_mode`.
- **WARN** не блокирует, но требует явного квитирования пользователем. При `gate_mode: "auto"` квитирование **отложенное**: WARN автоквитируются с записью в `auto_decisions.md` и предъявляются первым блоком дисклоуза на входе в Closeout (ADR-014).
- Покрытие, выведенное из чтения diff, — «видимое» (apparent), а не доказанное.

### Ручное тестирование (интерактивно)
Сценарии, которые нельзя покрыть автотестом, исполняются **интерактивно** командой `/sdx:manual`, а не отдаются пользователю как статичный файл. Оркестратор ведёт пользователя по шагам: на каждый шаг даёт детальную инструкцию и ожидаемый результат, затем собирает фактический результат (`Passed/Failed/Blocked` + наблюдение) через `AskUserQuestion`. План шагов готовит `qa` в `MANUAL_TEST.md` (атомарные шаги: действие / предусловия / ожидаемый результат); результаты прогона дописываются туда же одним блоком. Проваленные шаги становятся находками `FAIL` для гейта верификации наравне с автотестами и fresh-eyes ревью.

### Fresh-eyes: контракт изоляции
Статическое ревью поставки выполняет агент `reviewer` и **только через делегирование Task — никогда инлайн основной сессией** (даже на лёгких треках). Цель — стена контекста: ревьюеру передаются ТОЛЬКО артефакты-контракт и diff поставки (`git diff $(default-branch.sh)...sdx/<id> -- . ':!.claude/sessions/**'`, основная ветка — автоопределяется, REQ-BRANCH-3; каталог артефактов сессии исключён из diff, т.к. он tracked на ветке `sdx/<id>` и иначе замусорил бы fresh-eyes-обзор), но НЕ `session.log`, не история рассуждений и не пояснения «почему так». Отсутствие памяти о решениях — это и есть механизм: автор изменений рационализировал бы несоответствия, ревьюер без памяти — ловит. Решение, отсутствующее в артефактах, но присутствующее в коде, классифицируется как находка (недокументированное решение).

Объём ревью масштабируется по треку: `full` — обязательно против `SPEC.md`/`DESIGN.md`; `standard` — суженно против `change_note.md`; `patch` — **обязательный лёгкий прогон** против `change_note.md` (ADR-014): оси корректности и когерентности, полнота — в границах change_note. Корректность-исполнением (`qa`) на patch не требуется — регрессионный тест и stop-gate покрывают; независимо от `gate_mode`. `doc` — **обязательный лёгкий прогон**, симметричный `patch` (ADR-017): fresh-eyes `reviewer` против `change_note.md`, оси корректности и когерентности (не противоречит ли новая/изменённая запись бэклога уже существующим), полнота — в границах change_note. Субагент `qa` не вызывается вовсе — на этом треке нет кода, поэтому регрессионный тест-прогон не имеет предмета проверки. **Уточнение по факту (не по замыслу):** хук `stop-gate` (`sdx/hooks/stop-gate.sh`) остаётся включённым по значению `stage` (`Execution|Verification`) **независимо от трека** — он не знает о `doc`. В проекте, где настроен `.claude/sdx/verify-cmd.sh`, `stop-gate` фактически прогонит эту команду и на `doc|Verification` — трек здесь не даёт иммунитета. Прозрачность (no-op) возникает только в проектах без известной тест-команды — тот же случай, что и на любом другом треке в такой конфигурации, а не doc-специфичное свойство «прозрачен по построению».

## Контракт закрытия сессии (Closeout)
Этап Closeout завершает ЛЮБУЮ сессию (любой трек). Это самый частый источник пропусков, поэтому он оформлен как **обязательный чек-лист**: оркестратор печатает его пользователю с отметками `[ ]/[x]` и НЕ объявляет сессию закрытой, пока все пункты не отмечены `[x]`. Если пункт неприменим к треку — он помечается `[x]` с пометкой `(n/a)`.

**Порядок под вариант A: удаление ДО мёржа.** Весь Closeout выполняется последовательно в
ОДНОМ CLI (ADR-012): шаги 1–5 — на ветке сессии `sdx/<id>`, затем checkout основной ветки,
мёрж и enforcement-проверка. Инженерный инвариант сохранён: HEAD-дерево основной ветки не
должно содержать файлы сессии — удаление оформляется коммитом `git rm -r` НА ВЕТКЕ до мёржа.

**Гейт входа в Closeout** (до начала чек-листа, см. `/sdx:archive`):
- на треках `patch` и `doc` — существует `verification_report.md` без находок FAIL (лёгкая верификация обязательна, ADR-014/ADR-017); отсутствует → стоп и направление на `/sdx:verify`;
- при `gate_mode: "auto"` — выполнен **дисклоуз**: `auto_decisions.md` выведен пользователю одним блоком (автоквитированные WARN первыми) и получено одно явное подтверждение. Это единственная обязательная остановка авторежима; несогласие → `/sdx:backtrack`/`/sdx:retrack`, Closeout прерывается.

Чек-лист закрытия (см. `/sdx:archive`):

**На ветке сессии `sdx/<id>`:**
1. `[ ]` Рабочее дерево чистое, все изменения закоммичены в ветку сессии `sdx/<id>`.
2. `[ ]` Дельты перенесены в постоянные документы: `change_note.md`/`SPEC.md` → `docs/specs/`, `DESIGN.md` → `docs/designs/`, пользовательская и техническая документация актуализированы. Закоммичено на `sdx/<id>`. На треке `doc` — как правило `(n/a)`: doc-сессия не производит `SPEC.md`/`DESIGN.md`. **Исключение** — если сама сессия (типично `postmortem`) приходит к выводу об изменении протокола `sdx/`, `CLAUDE.md` или порождает новый ADR (как эта сессия): тогда пункт выполняется штатно, не `(n/a)`.
3. `[ ]` `PLAN.md` (если создавался) заархивирован в `docs/history/plans/`. Закоммичено. На треке `doc` — всегда `(n/a)`: этот трек не включает Task Planning.
4. `[ ]` Глобальный лог знаний обновлён (`docs/history/`) и бэклог актуализирован (`docs/backlog/`, ADR-015): закрытые этой сессией записи получили `status: closed` + `session`; отложенные решения и неквитированные WARN-находки верификации оформлены новыми `DEBT-`/`IDEA-`записями; индекс `README.md` обновлён. На треке `doc` (ADR-017) правки `docs/backlog/` уже выполнены на этапе `Update` (см. «Профили флоу» выше, «Что производит этап `Update`») — здесь, на Closeout, этот пункт **финальная сверка/актуализация статусов, не первичная работа**: подтвердить, что минимум одно изменение `docs/backlog/` (новая запись либо правка `status`/`priority`/`wave` существующей) действительно присутствует и помечено `session: <session_id>`. Оговорка: «не первичная работа» относится к продукту разбора, а не к находкам верификации — неквитированные WARN и отложенные решения возникают ПОСЛЕ `Update` и физически не могли быть оформлены на нём, поэтому соответствующие `DEBT-`/`IDEA-`записи заводятся здесь, на Closeout, на любом треке включая `doc`; для типа `intake` минимум одна из таких записей обязана быть НОВОЙ (не правкой существующей) — единственное место чек-листа, где Closeout содержательно различает `intake` и `grooming`. Для сессий типа `retro`/`postmortem`/`intake` этот же пункт включает перенос постоянного документа разбора из каталога сессии в `docs/history/retro/<session_id>.md`, `docs/history/postmortem/<session_id>.md` или `docs/history/intake/<session_id>.md` соответственно (по аналогии с переносом `PLAN.md` в п.3) — для `grooming` отдельного документа нет, записи бэклога являются единственным и достаточным выходом. Закоммичено.
5. `[ ]` Файлы сессии удалены коммитом `git rm -r .claude/sessions/<id>` НА ВЕТКЕ `sdx/<id>`, ДО мёржа (вариант A) — отдельный, явно поименованный коммит, не смешанный с коммитом переноса дельт (п.2).

**На основной ветке (автоопределяется), тот же CLI:**
6. `[ ]` Ветка `sdx/<id>` слита в основную ветку (`git checkout $(default-branch.sh) && git merge --no-ff sdx/<id>`).
7. `[ ]` Enforcement-проверка закрытия (`archive-verify.sh <id>`) пройдена: инварианты 1 (дерево чисто), 5 (ветка слита), 6 (каталог сессии не tracked в дереве основной ветки — подтверждает, что п.5 выполнен). Скрипт сам выполняет `git branch -d` после подтверждения инвариантов (и `git worktree remove --force` для legacy-worktree-сессий, если такая обнаружится).
8. `[ ]` Факт закрытия зафиксирован в глобальном логе (п.4, уже выполнено) — `session.log` отдельной записи не требует, так как удаляется вместе с каталогом сессии в п.5.

Пункт 5 (удаление файлов сессии, коммит ДО мёржа) исторически пропускается чаще всего —
оркестратор обязан выполнить его явно; его пропуск ловится инвариантом 6 хука `archive-verify.sh` в п.7.

## Import / Export (перенос функциональности между SDX-проектами)
Перенос фичи из проекта-источника в целевой проект, где **оба используют SDX**. Перенос triad-aware: бандл несёт нативные дельты `SPEC`/`DESIGN`, а не только код.

### Асимметрия
- **`/sdx:export` — команда** (без сессии, read-only на источнике): извлекает фичу и пакует в переносимый бандл. Источник не модифицируется.
- **`import` — тип сессии** (полный жизненный цикл, обычно трек `full`): принимает бандл, реконсилирует с целью и интегрирует. Создаётся через `/sdx:import <путь-к-бандлу>`.

### Формат бандла
Самоописательный каталог `.sdx/bundles/<feature>/` с **полным замыканием**:
```
manifest.yaml        # источник, версия SDX, имя фичи, дата, исходная сессия
spec.delta.md        # срез бизнес-требований (Delta-формат)
design.slice.md      # компоненты, схемы данных, API, обработка ошибок
code/                # файлы кода фичи
tests/               # связанные тесты
deps.yaml            # замыкание зависимостей: внешние либы+версии, внутренние модули
assumptions.md       # допущения о среде (рантайм, фреймворк, общие сервисы)
integration_points.md# точки подключения (роуты, DI-регистрации, entrypoints)
```

### Поток import-сессии
1. Валидация `manifest` и **совместимости версии SDX** (предупреждение при расхождении).
2. Discovery по цели + **реконсиляция** (`architect`): каждый компонент бандла классифицируется —
   - `transplant` — чисто ложится в цель (адаптировать пути/нейминг/импорты);
   - `reimplement` — не ложится; реализуется заново по `spec.delta`/`design.slice`, код бандла = референс;
   - `gap` — в цели нет опоры (зависимость/паттерн отсутствует) → стоп и решение пользователя.
   Результат — артефакт сессии `import_map.md` (компонент → решение → место в цели → заметки).
3. Мёрж `spec.delta.md`/`design.slice.md` в `SPEC`/`DESIGN` цели через Delta-формат, адаптированно. Коллизии триады → стоп и решение пользователя (ось когерентности).
4. Далее обычный full-цикл: Task Planning (задачи transplant + reimplement) → Execution → Verification → Closeout. Тесты из бандла переносятся и служат проверкой сохранности поведения; `reviewer` сверяет diff с мёрженым контрактом; недостающие зависимости добавляются (при необходимости — `devops`).

## Непрерывное улучшение (рекомендации по ходу сессии)
Оркестратор ОБЯЗАН активно озвучивать пользователю замеченные по ходу реальной работы
возможности улучшения фреймворка и процесса, а не молча их обходить. Сигналы: любой workaround
(обход хука через shell, ручная правка состояния), ложное срабатывание enforcement-слоя,
избыточная церемония трека, повторяющаяся ручная операция, отсутствующая автоматизация.
Правило: столкнулся с трением → сообщи одной-двумя строками и предложи оформить запись бэклога
(`/sdx:backlog add`) либо, для мета-репозитория SDX, отдельную сессию доработки. Молчаливый
обход запрещён — так теряются дефекты enforcement-слоя (прецедент: BUG-006 — stage-gate на
Windows долго обходили через shell вместо репорта). В авторежиме это не гейт: рекомендация
озвучивается без остановки работы, решение об оформлении — за пользователем (или на дисклоузе
Closeout).

## Checkpoint и сброс контекста
Команда `/sdx:checkpoint` записывает суммаризацию состояния в `.claude/sessions/<id>/checkpoint.md`. После этого пользователь может выполнить `/clear`; восстановление контекста происходит чтением файлов сессии (`/sdx:status`, `checkpoint.md`). Менять системный промпт или очищать контекст программно невозможно.
