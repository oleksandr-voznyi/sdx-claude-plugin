# Протокол SDX-сессий

Этот документ описывает формат состояния и логирования сессий. Им руководствуется основная сессия Claude Code при выполнении команд `/sdx:*` (роль Session Manager). Оркестрация субагентов (architect, ba, lead-dev, developer, qa, tech-writer, devops) выполняется только из основной сессии через инструмент Task — субагенты не могут вызывать друг друга.

## Состояние сессии (session_state.json)
```json
{
  "session_id": "string",
  "type": "feature | bug | refactor | init | import",
  "track": "patch | standard | full",
  "stage": "Discovery | Business Spec | Technical Design | Task Planning | Execution | Documentation | Verification | Deployment | Closeout",
  "status": "draft | review | approved | executing",
  "git_branch": "sdx/...",
  "artifacts": [],
  "history": []
}
```

- **`type`** — исходная категория задачи. Это лишь _стартовая гипотеза_ для выбора трека, а не жёсткая привязка.
- **`track`** — фактический режим прохождения сессии (профиль флоу). Может меняться по ходу работы через `/sdx:retrack`.
- **`stage`** — текущий этап. Допустимые этапы зависят от активного трека (см. ниже).

## Модель сессии = ветка в основном рабочем дереве (ADR-012)
Одна сессия = одна **git-ветка** `sdx/<id>`, checkout'нутая в ОСНОВНОМ рабочем дереве
репозитория. Весь жизненный цикл — от `/sdx:start` до `/sdx:archive` включительно — проходит
в **одном CLI-инстансе**, запущенном в корне репозитория; хендоффов на другой CLI нет.
Содержательные артефакты сессии (`.claude/sessions/<id>/session_state.json`, `session.log`,
`SPEC.md`/`DESIGN.md`/`PLAN.md`/`change_note.md` и т.д.) **версионируются** на этой ветке
инкрементальными коммитами; эфемерные `.stopgate.*` игнорируются точечным паттерном корневого
`.gitignore` (REQ-SESS-1/2). Основная ветка (автоопределяется, см. ниже) никогда не видит файлы
сессии в своём HEAD-дереве: удаление каталога сессии оформляется коммитом `git rm -r` НА ВЕТКЕ
`sdx/<id>` ДО мёржа (**вариант A**), после чего мёрж `--no-ff` вносит всю историю ветки в DAG
основной ветки, но не в её HEAD-дерево.

**Операционные следствия.** Enforcement-хуки резолвят активную сессию по имени текущей ветки
(`sdx/<id>`) — при работе в основном дереве `$CLAUDE_PROJECT_DIR` (корень репозитория) корректен
по построению, отдельный CLI не нужен. Переключение сессий (`/sdx:switch`) — обычный
`git checkout sdx/<id>` с гардом чистого дерева: при незакоммиченных изменениях — стоп и
инструкция закоммитить их в текущую сессию (слепой авто-коммит запрещён, ядро REQ-WT-2).
Физической изоляции параллельных «грязных» деревьев (бывший REQ-WT-3) больше нет — осознанный
трейд-офф в пользу непрерывности работы в одном CLI; дисциплина инкрементальных коммитов
(REQ-SESS-1) — штатная компенсация.

Обоснование и границы — ADR-012 в `docs/DECISIONS.md` (worktree-модель ADR-009 пересмотрена).

## Профили флоу (Tracks)
Трек — это упорядоченный набор _активных_ этапов. `/sdx:next` переводит сессию на следующий **активный** этап текущего трека, пропуская неактивные. Это позволяет масштабировать церемонию под размер задачи.

| Трек | Назначение | Активные этапы | Артефакты дельт |
|------|------------|----------------|------------------|
| **patch** | багфикс / точечная правка без изменения логики и контрактов | Execution → (Verification, опц.) → Closeout | `change_note.md` |
| **standard** | малая фича / рефактор | Discovery (лёгкий, инлайн) → Change → Execution → Verification → Closeout | `change_note.md` |
| **full** | крупная фича, затрагивающая контракты/архитектуру | Discovery → Business Spec → Technical Design → Task Planning → Execution → Documentation → Verification → Deployment → Closeout | `SPEC.md`, `DESIGN.md`, `PLAN.md` |

Пояснения:
- **Change** (треки patch/standard) — объединённый этап: бизнес- и технические решения фиксируются _слитно_ внутри `change_note.md`, без отдельных `SPEC.md`/`DESIGN.md`.
- **Discovery** в `standard` выполняется инлайн в основной сессии без вызова субагента `architect`.
- **Closeout** — обязательный завершающий этап для ВСЕХ треков (бывший Archiving). Контракт закрытия описан ниже.

### Адаптивность трека (re-routing)
Трек выбирается при `/sdx:start` как гипотеза, но **не фиксирован**. На каждом гейте (`/sdx:next`) и при любом существенном открытии оркестратор ОБЯЗАН переоценить адекватность трека:
- **Эскалация** (`patch → standard → full`): задача оказалась глубже — затронуты бизнес-логика, публичные контракты, схемы данных или архитектура. При эскалации ранее накопленный `change_note.md` промоутится в полноценные `SPEC.md`/`DESIGN.md`, а пропущенные этапы становятся активными.
- **Деэскалация** (`full → standard → patch`): задача оказалась мельче ожидаемого — церемония избыточна. Допускается с явного подтверждения пользователя; уже созданные артефакты сохраняются как справочные.

Смена трека выполняется командой `/sdx:retrack` и логируется категорией `STAGE_CHANGE`.

## Логирование (экономный режим)
В журнал `.claude/sessions/<session_id>/session.log` пишутся ТОЛЬКО durable-события смены состояния. Каждая отдельная запись в лог — это лишний ход агента (полный round-trip контекста), поэтому логирование намеренно минимально.

### Формат строки:
`[YYYY-MM-DD HH:mm:ss] [КАТЕГОРИЯ] [ЭТАП] Сообщение`

### Логируемые категории (только эти четыре):
- `START`: Инициализация сессии или переключение на неё.
- `STAGE_CHANGE`: Переход между этапами / смена трека.
- `CHECKPOINT`: Фиксация точки восстановления.
- `ERROR`: Ошибки команд или проваленные гейты.

### Что НЕ логируется (выводится иначе):
- **Артефакты (создание/изменение файлов)** — НЕ логируются. История изменений файлов выводится из git (`git log --stat`, `git diff --name-status`), а текущий состав артефактов — из `git status`/листинга папки сессии по запросу. Для этого артефакты коммитятся в ветку сессии **инкрементально** по ходу работы (это же поддерживает чек-лист Closeout).
- **INFO / просмотр статуса** — НЕ логируется вообще. Чтение состояния не должно мутировать его.

### Правила записи:
- **Время**: не выдумывай — получай `date '+%Y-%m-%d %H:%M:%S'`. Захватывай время и пиши запись в том же ходе, что и правку `session_state.json` (батч), а не отдельным вызовом.
- **ЭТАП**: текущее значение `stage`. **Сообщение**: кратко, на русском.
- Запись — в режиме **append** (`>>`):
```bash
echo "[$(date '+%Y-%m-%d %H:%M:%S')] [STAGE_CHANGE] [Execution] -> Verification" >> .claude/sessions/<session_id>/session.log
```

## Передача контекста субагентам (экономия токенов)
При вызове субагента через Task передавай **пути к артефактам и краткую задачу**, а не вставляй содержимое файлов в промпт. Субагент прочитает нужное сам. Вставка контента в промпт ведёт к двойной токенизации (в контексте оркестратора и заново в субагенте). Исключение — fresh-eyes `reviewer`, которому diff передаётся как данные по контракту изоляции.

## Гейты (Gates)
Перед переходом на следующий **активный** этап трека проверяется наличие необходимых артефактов (`SPEC.md`, `DESIGN.md`, `PLAN.md`, `change_note.md` и т.д.) — см. `/sdx:next`. Гейт также включает scope-check: оркестратор оценивает, не вышла ли задача за рамки текущего трека, и при необходимости предлагает `/sdx:retrack`.

## Enforcement-слой (хуки)
Инварианты, обязанные выполняться всегда, вынесены из прозы в детерминированные хуки.
Скрипты поставляются плагином (`${CLAUDE_PLUGIN_ROOT}/sdx/hooks/`), проводка — `hooks/hooks.json`
плагина. Per-project конфиги слоя (`prod-guard.conf`, `stage-gate.allow`, `verify-cmd.sh`)
живут в `.claude/sdx/` целевого проекта и разворачиваются `/sdx:init` из шаблонов
`${CLAUDE_PLUGIN_ROOT}/sdx/templates/`. Все хуки активны только в
SDX-ветке `sdx/<id>` и деградируют в no-op при отсутствии сессии/конфига (safe-by-default).
Механизм блокировки PreToolUse: JSON `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"<причина>"}}` на stdout, exit 0 (НЕ exit 2 — подтверждено на бинарнике Claude Code 2.1.195).

- **stage-gate** (PreToolUse `Write|Edit|MultiEdit`): запись в код заморожена до гейта
  Execution; `docs/**`, `.claude/**`, `*.md` открыты всегда (артефакты планирования).
  На стадии **Verification** дополнительно открыты тестовые каталоги (`tests/**`, `test/**`,
  `spec/**`, вкл. вложенные) — там `qa` пишет интеграционные тесты (`/sdx:verify` шаг 2);
  не-тестовый код остаётся заморожен, правки FAIL-находок — через `/sdx:backtrack --to Execution`.
  Со-локованные тесты (напр. `foo_test.go` рядом с исходником) покрываются per-project
  `stage-gate.allow`. Блокировка — JSON `permissionDecision:"deny"`. Plan mode НЕ используется
  намеренно — он заблокировал бы и `.md`-артефакты Spec/Design/Plan.
- **stop-gate** (Stop): тест-прогон как пол под Verification; ход не завершается на красном
  (скоуп Execution/Verification или `SDX_STOP_GATE=1`). После 3 красных подряд — возврат
  человеку. Привязан к `Stop` (не `SubagentStop`), чтобы не воевать с TDD red-green developer'а.
  Деградирует в no-op без известной тест-команды (`verify-cmd.sh` — per-project, ADR-4).
  **Green-run cache (ADR-011):** отпечаток дерева (`HEAD` + хэш `git status --porcelain`)
  последнего зелёного прогона хранится в `.stopgate.ok`; при неизменном дереве повторный `Stop`
  пропускается без перезапуска verify (кэш-хит до инкремента loop-guard). `SDX_STOP_GATE=1`
  кэш обходит; красный прогон кэш не пишет.
- **prod-guard** (PreToolUse `Bash`): прод-команды по `prod-guard.conf` блокируются; пустой
  или отсутствующий conf (нет ни одной активной строки-паттерна) = нет защиты (opt-in
  per-project) — проверка conf выполняется **раньше** проверки `jq`. Прод-деплой — только явное
  действие человека, не агента. **Граница:** покрывает только инструмент `Bash` — деплой
  через иной механизм (MCP-tool, шелл-обёртка) его минует. При отсутствии `jq` prod-guard
  деградирует в **fail-closed** (блокирует команду из осторожности) **только если защита
  сконфигурирована**; без активных паттернов отсутствие `jq` не блокирует команды.
- **preflight** (SessionStart): однократная проверка зависимостей хуков (`jq`). При
  отсутствии `jq` громко предупреждает (prod-guard → fail-closed, stage-gate → fail-open);
  опционально может жёстко отказывать в сессии (закомментированный `exit 2`).
- **archive-verify** (вызов из `/sdx:archive` после мёржа, вариант A): полагается только на
  переданный `$proj` и git. Резолвит основную ветку через `lib/default-branch.sh` (без хардкода
  `main`, ADR-010) и проверяет Closeout-инварианты: 1 (рабочее дерево чистое), 5 (ветка
  `sdx/<id>` доказуемо слита в автоопределённую основную ветку — имя ветки сверяется точно
  `grep -qx`, при отсутствии ветки слияние считается недоказуемым и деструктив отменяется без
  явного подтверждения `SDX_ARCHIVE_NO_BRANCH_OK=1`), 6 (каталог сессии отсутствует/не tracked
  в дереве основной ветки — детектор пропуска коммита `git rm` до мёржа). Деструктив
  (`git branch -d`; условное `git worktree remove --force` — legacy-совместимость с
  worktree-сессиями до ADR-012, при их отсутствии no-op) выполняется ТОЛЬКО после того, как все
  три инварианта доказаны — `rm -rf` каталога сессии остаётся лишь страховкой на остаточный
  физический путь, а не основным механизмом удаления.

## Верификация (этап Verification)
Этап верификации структурирован по **трём осям** и сочетает корректность-исполнением с независимым fresh-eyes ревью. Команда — `/sdx:verify`, артефакт — `verification_report.md` в папке сессии.

### Три оси
- **Полнота (completeness)** — у каждого требования/критерия есть реализация и тест; нет кода без прослеживаемого требования (scope creep). Источник предупреждений вида «сценарий X не имеет тестового покрытия».
- **Корректность (correctness)** — поведение соответствует намерению, граничные случаи из `DESIGN.md` обработаны, тесты не тавтологичны.
- **Когерентность (coherence)** — непротиворечивость триады `SPEC ↔ DESIGN ↔ Codebase`. Это операционализация детектора коллизий из CLAUDE.md §4: проверку ведёт независимая сторона, а не автор изменений.

### Статусы и гейт
- **FAIL** блокирует гейт (возврат к `developer`/`backtrack`).
- **WARN** не блокирует, но требует явного квитирования пользователем.
- Покрытие, выведенное из чтения diff, — «видимое» (apparent), а не доказанное.

### Ручное тестирование (интерактивно)
Сценарии, которые нельзя покрыть автотестом, исполняются **интерактивно** командой `/sdx:manual`, а не отдаются пользователю как статичный файл. Оркестратор ведёт пользователя по шагам: на каждый шаг даёт детальную инструкцию и ожидаемый результат, затем собирает фактический результат (`Passed/Failed/Blocked` + наблюдение) через `AskUserQuestion`. План шагов готовит `qa` в `MANUAL_TEST.md` (атомарные шаги: действие / предусловия / ожидаемый результат); результаты прогона дописываются туда же одним блоком. Проваленные шаги становятся находками `FAIL` для гейта верификации наравне с автотестами и fresh-eyes ревью.

### Fresh-eyes: контракт изоляции
Статическое ревью поставки выполняет агент `reviewer` и **только через делегирование Task — никогда инлайн основной сессией** (даже на лёгких треках). Цель — стена контекста: ревьюеру передаются ТОЛЬКО артефакты-контракт и diff поставки (`git diff $(default-branch.sh)...sdx/<id> -- . ':!.claude/sessions/**'`, основная ветка — автоопределяется, REQ-BRANCH-3; каталог артефактов сессии исключён из diff, т.к. он tracked на ветке `sdx/<id>` и иначе замусорил бы fresh-eyes-обзор), но НЕ `session.log`, не история рассуждений и не пояснения «почему так». Отсутствие памяти о решениях — это и есть механизм: автор изменений рационализировал бы несоответствия, ревьюер без памяти — ловит. Решение, отсутствующее в артефактах, но присутствующее в коде, классифицируется как находка (недокументированное решение).

Объём ревью масштабируется по треку: `full` — обязательно против `SPEC.md`/`DESIGN.md`; `standard` — суженно против `change_note.md`; `patch` — по запросу.

## Контракт закрытия сессии (Closeout)
Этап Closeout завершает ЛЮБУЮ сессию (любой трек). Это самый частый источник пропусков, поэтому он оформлен как **обязательный чек-лист**: оркестратор печатает его пользователю с отметками `[ ]/[x]` и НЕ объявляет сессию закрытой, пока все пункты не отмечены `[x]`. Если пункт неприменим к треку — он помечается `[x]` с пометкой `(n/a)`.

**Порядок под вариант A: удаление ДО мёржа.** Весь Closeout выполняется последовательно в
ОДНОМ CLI (ADR-012): шаги 1–5 — на ветке сессии `sdx/<id>`, затем checkout основной ветки,
мёрж и enforcement-проверка. Инженерный инвариант сохранён: HEAD-дерево основной ветки не
должно содержать файлы сессии — удаление оформляется коммитом `git rm -r` НА ВЕТКЕ до мёржа.

Чек-лист закрытия (см. `/sdx:archive`):

**На ветке сессии `sdx/<id>`:**
1. `[ ]` Рабочее дерево чистое, все изменения закоммичены в ветку сессии `sdx/<id>`.
2. `[ ]` Дельты перенесены в постоянные документы: `change_note.md`/`SPEC.md` → `docs/specs/`, `DESIGN.md` → `docs/designs/`, пользовательская и техническая документация актуализированы. Закоммичено на `sdx/<id>`.
3. `[ ]` `PLAN.md` (если создавался) заархивирован в `docs/history/plans/`. Закоммичено.
4. `[ ]` Глобальный лог знаний обновлён (`docs/history/`). Закоммичено.
5. `[ ]` Файлы сессии удалены коммитом `git rm -r .claude/sessions/<id>` НА ВЕТКЕ `sdx/<id>`, ДО мёржа (вариант A) — отдельный, явно поименованный коммит, не смешанный с коммитом переноса дельт (п.2).

**На основной ветке (автоопределяется), тот же CLI:**
6. `[ ]` Ветка `sdx/<id>` слита в основную ветку (`git checkout $(default-branch.sh) && git merge --no-ff sdx/<id>`).
7. `[ ]` Enforcement-проверка закрытия (`archive-verify.sh <id>`) пройдена: инварианты 1 (дерево чисто), 5 (ветка слита), 6 (каталог сессии не tracked в дереве основной ветки — подтверждает, что п.5 выполнен). Скрипт сам выполняет `git branch -d` после подтверждения инвариантов (и `git worktree remove --force` для legacy-worktree-сессий, если такая обнаружится).
8. `[ ]` Факт закрытия зафиксирован в глобальном логе (п.4, уже выполнено) — `session.log` отдельной записи не требует, так как удаляется вместе с каталогом сессии в п.5.

Пункт 5 (удаление файлов сессии, коммит ДО мёржа) исторически пропускается чаще всего —
оркестратор обязан выполнить его явно; его пропуск ловится инвариантом 6 хука `archive-verify.sh` в п.7.

## Import / Export (перенос функциональности между SDX-проектами)
Перенос фичи из проекта-источника в целевой проект, где **оба используют SDX**. Перенос triad-aware: бандл несёт нативные дельты `SPEC`/`DESIGN`, а не только код.

### Асимметрия
- **`/sdx:export` — команда** (без сессии, read-only на источнике): извлекает фичу и пакует в переносимый бандл. Источник не модифицируется.
- **`import` — тип сессии** (полный жизненный цикл, обычно трек `full`): принимает бандл, реконсилирует с целью и интегрирует. Создаётся через `/sdx:import <путь-к-бандлу>`.

### Формат бандла
Самоописательный каталог `.sdx/bundles/<feature>/` с **полным замыканием**:
```
manifest.yaml        # источник, версия SDX, имя фичи, дата, исходная сессия
spec.delta.md        # срез бизнес-требований (Delta-формат)
design.slice.md      # компоненты, схемы данных, API, обработка ошибок
code/                # файлы кода фичи
tests/               # связанные тесты
deps.yaml            # замыкание зависимостей: внешние либы+версии, внутренние модули
assumptions.md       # допущения о среде (рантайм, фреймворк, общие сервисы)
integration_points.md# точки подключения (роуты, DI-регистрации, entrypoints)
```

### Поток import-сессии
1. Валидация `manifest` и **совместимости версии SDX** (предупреждение при расхождении).
2. Discovery по цели + **реконсиляция** (`architect`): каждый компонент бандла классифицируется —
   - `transplant` — чисто ложится в цель (адаптировать пути/нейминг/импорты);
   - `reimplement` — не ложится; реализуется заново по `spec.delta`/`design.slice`, код бандла = референс;
   - `gap` — в цели нет опоры (зависимость/паттерн отсутствует) → стоп и решение пользователя.
   Результат — артефакт сессии `import_map.md` (компонент → решение → место в цели → заметки).
3. Мёрж `spec.delta.md`/`design.slice.md` в `SPEC`/`DESIGN` цели через Delta-формат, адаптированно. Коллизии триады → стоп и решение пользователя (ось когерентности).
4. Далее обычный full-цикл: Task Planning (задачи transplant + reimplement) → Execution → Verification → Closeout. Тесты из бандла переносятся и служат проверкой сохранности поведения; `reviewer` сверяет diff с мёрженым контрактом; недостающие зависимости добавляются (при необходимости — `devops`).

## Checkpoint и сброс контекста
Команда `/sdx:checkpoint` записывает суммаризацию состояния в `.claude/sessions/<id>/checkpoint.md`. После этого пользователь может выполнить `/clear`; восстановление контекста происходит чтением файлов сессии (`/sdx:status`, `checkpoint.md`). Менять системный промпт или очищать контекст программно невозможно.
